# Transactions, ACID & Isolation

Όταν πολλοί χρήστες γράφουν ταυτόχρονα, χωρίς transactions η βάση καταλήγει σε **ασυνεπή κατάσταση** (χαμένα λεφτά, διπλές κρατήσεις). Στα interviews εξετάζουν τι σημαίνει ACID, τα isolation levels, και πώς λύνεις race conditions.

## Τι πρέπει να μπορείς στο τέλος

- να εξηγείς ACID με παραδείγματα,
- να γράφεις transactions (`BEGIN`/`COMMIT`/`ROLLBACK`),
- να ξεχωρίζεις τα isolation levels και τι ανωμαλίες λύνουν,
- να αναγνωρίζεις race conditions σε concurrent writes,
- να χειρίζεσαι deadlocks και lost updates.

## Τι είναι transaction

Μια ομάδα operations που εκτελούνται **ως μία αδιαίρετη μονάδα**: ή όλα ή τίποτα.

```sql
BEGIN;
  UPDATE accounts SET balance = balance - 100 WHERE id = 1;  -- χρέωση
  UPDATE accounts SET balance = balance + 100 WHERE id = 2;  -- πίστωση
COMMIT;  -- αν κάτι αποτύχει ενδιάμεσα → ROLLBACK, καμία αλλαγή
```

Χωρίς transaction, αν ο server κρασάρει μετά την πρώτη γραμμή, τα 100 «εξαφανίζονται».

## ACID

```txt
Atomicity   → όλα ή τίποτα. Αποτυχία ⇒ rollback σε όλο το transaction.
Consistency → η βάση πάει από έγκυρη κατάσταση σε έγκυρη (constraints τηρούνται).
Isolation   → concurrent transactions δεν «βλέπουν» η μία τα μισοτελειωμένα της άλλης.
Durability  → μετά το COMMIT, τα δεδομένα επιβιώνουν ακόμα και σε crash.
```

## Isolation levels & ανωμαλίες

Όσο πιο αυστηρό το isolation, τόσο λιγότερες ανωμαλίες — αλλά λιγότερο concurrency:

```txt
READ UNCOMMITTED → επιτρέπει dirty reads (διαβάζεις uncommitted αλλαγές)
READ COMMITTED   → όχι dirty reads· αλλά non-repeatable reads (default σε Postgres)
REPEATABLE READ  → ίδιο read δίνει ίδιο αποτέλεσμα· αλλά (κλασικά) phantom rows
SERIALIZABLE     → σαν να έτρεξαν σειριακά· καμία ανωμαλία, μέγιστο κόστος
```

```txt
Ανωμαλίες:
- Dirty read       → διαβάζεις αλλαγή που μπορεί να γίνει rollback
- Non-repeatable   → ίδιο SELECT, διαφορετικό αποτέλεσμα (κάποιος έκανε commit ενδιάμεσα)
- Phantom          → νέα rows εμφανίζονται σε επανάληψη range query
- Lost update      → δύο reads-modify-writes, η μία πατάει την άλλη
```

## Λύσεις σε race conditions

```sql
-- (α) Atomic conditional update — η βάση κάνει check+write μαζί
UPDATE products SET stock = stock - 1
WHERE id = 1 AND stock > 0;   -- 0 affected rows = sold out

-- (β) Pessimistic lock — κλείδωσε το row μέχρι το COMMIT
BEGIN;
  SELECT stock FROM products WHERE id = 1 FOR UPDATE;  -- άλλοι περιμένουν
  UPDATE products SET stock = stock - 1 WHERE id = 1;
COMMIT;

-- (γ) Optimistic locking — version column
UPDATE products SET stock = stock - 1, version = version + 1
WHERE id = 1 AND version = 7;  -- αν κάποιος άλλαξε, version != 7 → retry
```

## Deadlocks

Δύο transactions περιμένουν locks το ένα του άλλου → κανένα δεν προχωρά. Η βάση συνήθως ανιχνεύει deadlock και κάνει abort το ένα. Πρόληψη: **απόκτησε locks με σταθερή σειρά** (π.χ. πάντα account με μικρότερο id πρώτα), κράτα τα transactions μικρά.

## Πώς απαντάς στη συνέντευξη

«How do transactions and isolation prevent concurrency bugs?»

> A transaction groups operations into an all-or-nothing unit, and ACID describes the guarantees: atomicity rolls everything back on failure, consistency keeps constraints valid, isolation stops concurrent transactions from seeing each other's half-done work, and durability means a committed change survives a crash. Isolation levels trade safety for concurrency — read committed avoids dirty reads, repeatable read stabilizes rows you've read, serializable behaves as if transactions ran one at a time. The classic bug is a lost update: two requests read the same stock of 1, both pass the check, and both decrement, so I oversell. I fix it by making the check-and-write atomic — a conditional `UPDATE ... WHERE stock > 0` and checking affected rows, a `SELECT ... FOR UPDATE` row lock, or optimistic locking with a version column. For deadlocks I acquire locks in a consistent order and keep transactions short.
