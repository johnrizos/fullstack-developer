# Command Line Basics

Κάθε deploy, κάθε log, κάθε server τρέχει πίσω από ένα terminal. Όσο μένεις μόνο στο GUI, εξαρτάσαι από κουμπιά που σε production δεν υπάρχουν. Το command line είναι το κοινό λεξιλόγιο κάθε backend/DevOps δουλειάς.

## Τι θα μάθεις

- να διακρίνεις **terminal** από **shell** και να διαβάζεις τη δομή μιας εντολής,
- να κινείσαι στο filesystem και να χειρίζεσαι αρχεία με σιγουριά,
- να συνδυάζεις εντολές με **pipes** και **redirection**,
- να διαβάζεις exit codes και να αλυσιδώνεις εντολές υπό συνθήκη.

## Terminal vs shell

Δύο διαφορετικά πράγματα που συχνά μπερδεύονται:

- **Terminal**: το πρόγραμμα-παράθυρο που δείχνει κείμενο (Terminal.app, Windows Terminal, iTerm).
- **Shell**: ο διερμηνέας που εκτελεί τις εντολές σου (`bash`, `zsh`, `sh`, `fish`). Σε Linux servers σχεδόν πάντα `bash`.

Όταν λέμε «τρέξε στο terminal», στην ουσία δίνεις εντολή στο shell.

## Η δομή μιας εντολής

Κάθε εντολή διαβάζεται το ίδιο:

```bash
ls -l -a /var/log
# │  │  │  └── argument (σε ποιο πράγμα)
# │  └──┴───── options/flags (πώς)
# └─────────── command (τι)
```

- Τα short flags ενώνονται: `ls -la` == `ls -l -a`.
- Τα long flags είναι αυτο-επεξηγηματικά: `ls --all`.
- Όταν κολλήσεις: `man ls` ή `ls --help`.

## Πού είμαι; — πλοήγηση

```bash
pwd                 # print working directory — πού βρίσκομαι τώρα
ls                  # τι υπάρχει εδώ
ls -la              # τα πάντα (και κρυφά .files) με λεπτομέρειες
cd /var/www         # πήγαινε σε absolute path
cd projects         # πήγαινε σε relative path (μέσα από εδώ)
cd ..               # ένα επίπεδο πάνω
cd ~                # home directory σου
cd -                # πίσω στον προηγούμενο φάκελο
```

Κλειδί που μπερδεύει αρχάριους — τα ειδικά paths:

| Σύμβολο | Σημαίνει |
| --- | --- |
| `/` | root του filesystem (η κορυφή) |
| `~` | το home σου (π.χ. `/home/john`) |
| `.` | ο τρέχων φάκελος |
| `..` | ο γονικός φάκελος |

**Absolute path** ξεκινά από `/` και δουλεύει από οπουδήποτε. **Relative path** εξαρτάται από το `pwd` σου.

## Δημιουργία & χειρισμός αρχείων

```bash
mkdir logs                  # νέος φάκελος
mkdir -p app/src/utils      # -p: φτιάξε και τους ενδιάμεσους
touch app.js                # δημιούργησε άδειο αρχείο
cp config.json config.bak   # copy
cp -r src/ backup/          # -r: αναδρομικά για φακέλους
mv old.txt new.txt          # rename ή move
rm temp.log                 # διαγραφή αρχείου
rm -r node_modules/         # -r: διαγραφή φακέλου με περιεχόμενο
```

> ⚠️ Δεν υπάρχει «κάδος ανακύκλωσης». Το `rm -rf /` ή το `rm -rf $VAR/` με κενό `$VAR` έχουν διαλύσει production servers. Κοίτα δύο φορές πριν πατήσεις Enter.

## Διάβασμα αρχείων

```bash
cat app.log            # όλο το αρχείο στην οθόνη
less app.log           # σελιδοποιημένα (q για έξοδο, / για search)
head -n 20 app.log     # τις πρώτες 20 γραμμές
tail -n 50 app.log     # τις τελευταίες 50 γραμμές
tail -f app.log        # follow: live νέες γραμμές (το #1 εργαλείο για logs)
```

Το `tail -f` είναι αυτό που τρέχεις όταν θες να δεις τι κάνει ένα app **την ώρα** που το χτυπάς με requests.

## Αναζήτηση — grep & find

```bash
grep "ERROR" app.log              # γραμμές που περιέχουν ERROR
grep -i "error" app.log           # case-insensitive
grep -r "TODO" src/               # αναδρομικά σε φάκελο
grep -n "timeout" app.log         # με αριθμούς γραμμής

find . -name "*.log"              # όλα τα .log από εδώ και κάτω
find /var -type f -mtime -1       # αρχεία που άλλαξαν τις τελευταίες 24h
which node                        # πού βρίσκεται το εκτελέσιμο
```

## Pipes & redirection — η υπερδύναμη του shell

Εδώ το command line σταματά να είναι «μενού εντολών» και γίνεται γλώσσα. Το **pipe** `|` στέλνει το output της μιας εντολής ως input στην επόμενη:

```bash
cat app.log | grep "ERROR" | wc -l       # πόσα ERROR συνολικά;
ps aux | grep node                        # ποιες node διεργασίες τρέχουν;
history | grep ssh                         # ποιες ssh εντολές έτρεξα;
```

**Redirection** στέλνει output σε αρχείο αντί για την οθόνη:

```bash
echo "hello" > out.txt        # > overwrite (σβήνει ό,τι είχε)
echo "world" >> out.txt       # >> append (προσθέτει στο τέλος)
node app.js > app.log 2>&1    # stdout ΚΑΙ stderr στο ίδιο αρχείο
node app.js 2> errors.log     # μόνο τα errors (stderr) χωριστά
```

Τα τρία streams που πρέπει να ξέρεις: **stdin** (0), **stdout** (1), **stderr** (2). Το `2>&1` σημαίνει «στείλε το stderr εκεί που πάει το stdout».

## Αλυσίδωση εντολών

```bash
npm run build && npm start     # τρέξε το 2ο ΜΟΝΟ αν το 1ο πέτυχε
npm test || echo "tests failed" # τρέξε το 2ο ΜΟΝΟ αν το 1ο απέτυχε
git pull ; npm install          # τρέξε και τα δύο, ανεξάρτητα
```

Πίσω από αυτά κρύβεται το **exit code**: κάθε εντολή επιστρέφει `0` για επιτυχία, ≠`0` για αποτυχία.

```bash
ls /tmp
echo $?        # 0  → πέτυχε
ls /nope
echo $?        # 2  → απέτυχε
```

Το `$?` και τα `&&`/`||` είναι ο πυρήνας κάθε deploy script και CI pipeline.

## Wildcards & μεταβλητές

```bash
rm *.tmp                  # όλα τα αρχεία που τελειώνουν σε .tmp
ls log-202?.txt           # ? = ένας χαρακτήρας
echo $HOME                # μεταβλητή περιβάλλοντος
echo $PATH                # πού ψάχνει το shell για εκτελέσιμα
export NODE_ENV=production # όρισε env var για τη session
```
