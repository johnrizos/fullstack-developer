# Git & GitHub

Fullstack developer χωρίς Git δεν στέκεται σε ομάδα. Σε συνέντευξη πρέπει να μπορείς να εξηγήσεις πώς δουλεύεις με commits, branches και pull requests.

## Mental model

Το Git είναι local version control. Το GitHub είναι remote platform για collaboration.

Βασική ροή:

```bash
git status
git add .
git commit -m "Add login form"
git push
```

Πριν κάνεις commit, πάντα κοίτα:

```bash
git status
git diff
```

Το `status` δείχνει ποια αρχεία άλλαξαν. Το `diff` δείχνει τι ακριβώς άλλαξε.

## Commit

Ένα commit είναι snapshot αλλαγών με μήνυμα. Καλό commit:

- είναι μικρό,
- έχει ξεκάθαρο σκοπό,
- δεν ανακατεύει άσχετες αλλαγές.

Παράδειγμα καλού μηνύματος:

```txt
Add validated contact form
```

Απόφυγε μηνύματα όπως:

```txt
fix
changes
update stuff
```

Ένα καλό commit message λέει τι άλλαξε, όχι απλώς ότι άλλαξε κάτι.

## Branch

Branch είναι απομονωμένη γραμμή δουλειάς.

```bash
git switch -c feature/contact-form
```

Δουλεύεις σε branch ώστε το main να μένει σταθερό.

Συχνά branch names:

- `feature/auth-flow`,
- `fix/navbar-overflow`,
- `chore/update-eslint`.

## Pull και sync

Πριν ανοίξεις PR ή πριν συνεχίσεις δουλειά μετά από ώρα, φέρε τις τελευταίες αλλαγές:

```bash
git pull
```

Αν δουλεύεις σε ομάδα, αυτό μειώνει εκπλήξεις και conflicts.

## Pull request

Pull request είναι πρόταση αλλαγής. Εκεί γίνονται:

- code review,
- automated checks,
- discussion,
- merge.

Ένα καλό PR έχει:

- μικρό scope,
- καθαρή περιγραφή,
- screenshots για UI αλλαγές,
- τι tests έτρεξες,
- links σε issue/task αν υπάρχει.

## Merge conflicts

Conflict συμβαίνει όταν δύο αλλαγές ακουμπάνε το ίδιο σημείο. Η σωστή απάντηση σε interview δεν είναι “φοβάμαι τα conflicts”, αλλά:

1. Διαβάζω και τις δύο αλλαγές.
2. Κρατάω τη σωστή τελική εκδοχή.
3. Τρέχω tests/build.
4. Κάνω commit το resolved conflict.

## Merge vs rebase — η κλασική ερώτηση

Δύο τρόποι να φέρεις τις αλλαγές του main στο branch σου (ή το αντίστροφο):

```bash
git merge main    # δημιουργεί merge commit — το ιστορικό δείχνει τη συγχώνευση
git rebase main   # "ξαναπαίζει" τα commits σου ΠΑΝΩ στο νέο main — ευθεία γραμμή
```

- **Merge**: ασφαλές, διατηρεί την πραγματική ιστορία, αλλά γεμίζει το log με merge commits.
- **Rebase**: καθαρό γραμμικό ιστορικό, αλλά **ξαναγράφει commits** — γι' αυτό ο χρυσός κανόνας: *ποτέ rebase σε branch που έχουν τραβήξει άλλοι* (shared/main).

Συνηθισμένη ομαδική πρακτική: rebase το δικό σου feature branch πάνω στο main όσο δουλεύεις, και squash merge στο PR — μικρά βρώμικα commits γίνονται ένα καθαρό commit στο main.

## Undo: τα τρία επίπεδα διορθώσεων

Αυτό ξεχωρίζει όσους «ξέρουν Git» από όσους κάνουν panic:

```bash
# 1. Αλλαγές ΠΟΥ ΔΕΝ έγιναν commit:
git restore file.ts            # πέτα τις αλλαγές του αρχείου
git restore --staged file.ts   # βγάλε το από το staging (κρατά τις αλλαγές)

# 2. Λάθος commit ΠΟΥ ΔΕΝ έχει γίνει push:
git commit --amend             # διόρθωσε το τελευταίο commit (μήνυμα ή περιεχόμενο)
git reset --soft HEAD~1        # αναίρεσε το commit, κράτα τις αλλαγές staged
git reset --hard HEAD~1        # ⚠️ αναίρεσε commit ΚΑΙ αλλαγές — χάνονται!

# 3. Λάθος commit ΠΟΥ ΕΧΕΙ γίνει push:
git revert <hash>              # ΝΕΟ commit που αναιρεί το παλιό — ασφαλές σε shared history
```

Ο κανόνας για συνέντευξη: **reset για τοπικά, revert για ό,τι έχει δημοσιευτεί** — το reset ξαναγράφει ιστορία και σε pushed branches δημιουργεί χάος στην ομάδα. Και το `git stash` για το «θέλω να αλλάξω branch αλλά έχω μισοτελειωμένα»: κρύβει τις αλλαγές, `git stash pop` τις επαναφέρει.

## .gitignore

Δεν πρέπει όλα τα αρχεία να μπαίνουν στο Git. Συνήθως αγνοείς:

- `node_modules`,
- `.env`,
- build outputs,
- local logs,
- editor/system files.

Το `.env.example` μπορεί να μπει στο Git για να δείχνει ποια env vars χρειάζονται, χωρίς values.

## GitHub Actions

Συχνά στις αγγελίες ζητάνε CI/CD. Με GitHub Actions μπορείς να τρέχεις lint/build/tests σε κάθε PR.

```yaml
name: CI
on: [pull_request]
jobs:
  checks:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
      - run: npm ci
      - run: npm run lint
      - run: npm run build
```

## Πώς απαντάς στη συνέντευξη

«How do you work with Git in a team?»

> I create focused branches, make small meaningful commits, open pull requests for review, rely on CI checks, and keep main stable. I rebase my own feature branches to stay current but never rebase shared history — for anything already pushed, I use revert instead of reset. If conflicts happen, I read both changes, keep the correct final version, and rerun tests before committing the resolution.
