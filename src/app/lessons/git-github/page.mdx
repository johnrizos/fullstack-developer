# Git & GitHub

Fullstack developer χωρίς Git δεν στέκεται σε ομάδα. Σε συνέντευξη πρέπει να μπορείς να εξηγήσεις πώς δουλεύεις με commits, branches και pull requests.

## Mental model

Το Git είναι local version control. Το GitHub είναι remote platform για collaboration.

Βασική ροή:

```bash
git status
git add .
git commit -m "Add login form"
git push
```

Πριν κάνεις commit, πάντα κοίτα:

```bash
git status
git diff
```

Το `status` δείχνει ποια αρχεία άλλαξαν. Το `diff` δείχνει τι ακριβώς άλλαξε.

## Commit

Ένα commit είναι snapshot αλλαγών με μήνυμα. Καλό commit:

- είναι μικρό,
- έχει ξεκάθαρο σκοπό,
- δεν ανακατεύει άσχετες αλλαγές.

Παράδειγμα καλού μηνύματος:

```txt
Add validated contact form
```

Απόφυγε μηνύματα όπως:

```txt
fix
changes
update stuff
```

Ένα καλό commit message λέει τι άλλαξε, όχι απλώς ότι άλλαξε κάτι.

## Branch

Branch είναι απομονωμένη γραμμή δουλειάς.

```bash
git switch -c feature/contact-form
```

Δουλεύεις σε branch ώστε το main να μένει σταθερό.

Συχνά branch names:

- `feature/auth-flow`,
- `fix/navbar-overflow`,
- `chore/update-eslint`.

## Pull και sync

Πριν ανοίξεις PR ή πριν συνεχίσεις δουλειά μετά από ώρα, φέρε τις τελευταίες αλλαγές:

```bash
git pull
```

Αν δουλεύεις σε ομάδα, αυτό μειώνει εκπλήξεις και conflicts.

## Pull request

Pull request είναι πρόταση αλλαγής. Εκεί γίνονται:

- code review,
- automated checks,
- discussion,
- merge.

Ένα καλό PR έχει:

- μικρό scope,
- καθαρή περιγραφή,
- screenshots για UI αλλαγές,
- τι tests έτρεξες,
- links σε issue/task αν υπάρχει.

## Merge conflicts

Conflict συμβαίνει όταν δύο αλλαγές ακουμπάνε το ίδιο σημείο. Η σωστή απάντηση σε interview δεν είναι “φοβάμαι τα conflicts”, αλλά:

1. Διαβάζω και τις δύο αλλαγές.
2. Κρατάω τη σωστή τελική εκδοχή.
3. Τρέχω tests/build.
4. Κάνω commit το resolved conflict.

## .gitignore

Δεν πρέπει όλα τα αρχεία να μπαίνουν στο Git. Συνήθως αγνοείς:

- `node_modules`,
- `.env`,
- build outputs,
- local logs,
- editor/system files.

Το `.env.example` μπορεί να μπει στο Git για να δείχνει ποια env vars χρειάζονται, χωρίς values.

## GitHub Actions

Συχνά στις αγγελίες ζητάνε CI/CD. Με GitHub Actions μπορείς να τρέχεις lint/build/tests σε κάθε PR.

```yaml
name: CI
on: [pull_request]
jobs:
  checks:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
      - run: npm ci
      - run: npm run lint
      - run: npm run build
```

> I create focused branches, make small meaningful commits, open pull requests for review, rely on CI checks, and keep main stable. If conflicts happen, I resolve them by understanding both changes and rerunning tests.
