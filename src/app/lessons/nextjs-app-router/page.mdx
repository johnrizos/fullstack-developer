# Next.js App Router + TypeScript

Το Next.js κυριαρχεί στις fullstack αγγελίες γιατί ενώνει frontend, routing, server rendering, data loading και API endpoints σε ένα framework. Στις συνεντεύξεις, η ερώτηση-κλειδί είναι σχεδόν πάντα η ίδια: **«Server Components vs Client Components — πότε και γιατί;»**. Σε αυτό το μάθημα θα τη λύσεις οριστικά.

## File-based routing: η δομή είναι το API

Κάθε φάκελος μέσα στο `app/` είναι ένα route segment, και ειδικά αρχεία ορίζουν τη συμπεριφορά του:

```txt
app/
  layout.tsx          ← root layout (τυλίγει ΤΑ ΠΑΝΤΑ — html/body)
  page.tsx            ← το "/"
  loading.tsx         ← loading UI (αυτόματο Suspense boundary)
  not-found.tsx       ← 404 UI
  error.tsx           ← error boundary (πρέπει να είναι client component)
  projects/
    layout.tsx        ← κοινό UI για ό,τι ξεκινά με /projects
    page.tsx          ← το "/projects"
    [slug]/
      page.tsx        ← το "/projects/roadmap", "/projects/tasks", ...
```

Δύο κανόνες που ρωτιούνται:

- **Layouts δεν κάνουν re-render στο navigation** — το state τους διατηρείται όταν πλοηγείσαι ανάμεσα σε σελίδες που μοιράζονται το ίδιο layout. Γι' αυτό το sidebar δεν «τρεμοπαίζει».
- **Τα layouts είναι φωλιασμένα**: root layout → projects layout → page. Κάθε επίπεδο τυλίγει το επόμενο.

## Dynamic routes & params

Ο φάκελος `[slug]` πιάνει δυναμικά τμήματα του URL. Στις πρόσφατες εκδόσεις του Next.js, τα `params` είναι **Promise** — τα κάνεις `await`:

```tsx
// app/projects/[slug]/page.tsx
export default async function ProjectPage({
  params,
}: {
  params: Promise<{ slug: string }>;
}) {
  const { slug } = await params;
  const project = await getProject(slug);

  if (!project) notFound(); // δείχνει το πλησιέστερο not-found.tsx

  return <h1>{project.title}</h1>;
}

// Προ-δημιουργία των routes στο build (static generation):
export function generateStaticParams() {
  return [{ slug: "roadmap" }, { slug: "tasks" }];
}
```

Διάκριση που μπερδεύει συχνά: **route params** (`/projects/[slug]` → μέρος του path) vs **search params** (`?filter=active` → query string, διαθέσιμα μέσω του `searchParams` prop στη σελίδα ή `useSearchParams()` σε client component).

## Server Components — το default και γιατί

Στο App Router **όλα τα components είναι Server Components εκτός αν δηλώσεις αλλιώς**. Τρέχουν μόνο στον server:

```tsx
// Κανένα "use client" — αυτό τρέχει ΜΟΝΟ server-side
export default async function ProjectsPage() {
  const projects = await db.project.findMany(); // απευθείας DB πρόσβαση!
  return (
    <ul>
      {projects.map((p) => <li key={p.id}>{p.title}</li>)}
    </ul>
  );
}
```

Τι κερδίζεις:

- **Async data loading χωρίς useEffect** — το component απλά κάνει `await`.
- **Μηδέν JavaScript στο bundle** για αυτά τα components — ο browser παίρνει έτοιμο HTML.
- **Secrets παραμένουν server-side** — DB credentials, API keys δεν φτάνουν ποτέ στον client.

Τι ΔΕΝ μπορούν: state (`useState`), effects, event handlers (`onClick`), browser APIs. Για αυτά υπάρχουν τα Client Components.

## Client Components — το boundary της διαδραστικότητας

```tsx
"use client";

import { useState } from "react";

export function LikeButton({ projectId }: { projectId: string }) {
  const [likes, setLikes] = useState(0);
  return <button onClick={() => setLikes(likes + 1)}>♥ {likes}</button>;
}
```

Τα τρία πράγματα που πρέπει να πεις σωστά:

1. **Το `"use client"` είναι boundary, όχι «ετικέτα»** — ό,τι κάνει import ένα client component γίνεται κι αυτό μέρος του client bundle. Γι' αυτό το βάζεις όσο πιο **χαμηλά** στο δέντρο γίνεται (στο κουμπί, όχι στη σελίδα).
2. **Server → Client props πρέπει να είναι serializable** — objects, strings, numbers ναι· functions, class instances, Dates θέλουν προσοχή. Δεν μπορείς να περάσεις ένα DB client ως prop.
3. **Client components ΜΠΟΡΟΥΝ να δεχτούν Server Components ως `children`** — το pattern «interactive wrapper γύρω από server content»:

```tsx
// ✅ Ο interactive Tabs (client) τυλίγει server-rendered περιεχόμενο
<Tabs>
  <ServerRenderedReport />
</Tabs>
```

## Data fetching & caching

Το data loading στο App Router γίνεται στον server, στο σημείο που χρειάζεται:

```tsx
async function getProject(slug: string): Promise<Project | null> {
  const res = await fetch(`https://api.example.com/projects/${slug}`, {
    next: { revalidate: 3600 }, // ISR: φρεσκάρισμα κάθε ώρα
  });
  if (!res.ok) return null;
  return res.json();
}
```

Οι τρεις στρατηγικές που πρέπει να ονομάζεις:

- **Static** (build time): σελίδες που δεν αλλάζουν συχνά — το ταχύτερο δυνατό.
- **ISR (Incremental Static Regeneration)**: static + περιοδικό φρεσκάρισμα (`revalidate`) — ο συνδυασμός ταχύτητας και φρεσκάδας.
- **Dynamic** (request time): per-user περιεχόμενο (dashboards) — όταν διαβάζεις cookies/headers, η σελίδα γίνεται dynamic.

Επειδή τα Server Components τρέχουν παράλληλα και κάνουν cache τα fetch requests, το ίδιο `fetch` σε πολλά components στο ίδιο render **δεν** ξαναχτυπά το API.

## Πώς απαντάς στη συνέντευξη

«Server Components vs Client Components?»

> In the App Router everything is a Server Component by default — they render on the server, can fetch data directly with async/await, keep secrets server-side, and ship zero JavaScript to the browser. I add "use client" only where I need state, effects, or event handlers, and I push that boundary as low in the tree as possible, because everything a client component imports joins the client bundle. Props crossing the boundary must be serializable, and a client component can still receive server-rendered content through children.
