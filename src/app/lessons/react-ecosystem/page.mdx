# React + TypeScript

Στα React interviews δεν αρκεί να «ξέρεις components». Εξετάζουν αν καταλαβαίνεις **πώς δουλεύει το rendering**, πώς ρέουν τα δεδομένα, και γιατί υπάρχουν οι κανόνες των hooks. Εδώ θα χτίσουμε αυτό το mental model με TypeScript από την αρχή.

## Το mental model: UI = f(state)

Το React έχει μία κεντρική ιδέα: **το UI είναι συνάρτηση του state**. Δεν «πειράζεις» το DOM — αλλάζεις state, και το React ξανατρέχει το component σου και ενημερώνει το DOM όπου χρειάζεται.

```tsx
function Counter() {
  const [count, setCount] = useState(0);

  // Κάθε φορά που αλλάζει το count, αυτή η function ΞΑΝΑΤΡΕΧΕΙ
  // ολόκληρη — αυτό είναι το "re-render".
  return <button onClick={() => setCount(count + 1)}>Count: {count}</button>;
}
```

Δύο συνέπειες που εξηγούν τα περισσότερα «παράξενα» bugs:

1. **Κάθε render είναι ένα στιγμιότυπο.** Οι μεταβλητές και οι functions ξαναδημιουργούνται. Το `count` μέσα σε ένα παλιό closure είναι η παλιά τιμή.
2. **Το setState είναι ασύγχρονο και κάνει batching.** Δεν βλέπεις τη νέα τιμή αμέσως μετά το `setCount`.

## Typed props — το contract του component

Τα props είναι το δημόσιο API ενός component. Με TypeScript το contract γίνεται ρητό:

```tsx
type TodoItemProps = {
  todo: Todo;
  onToggle: (id: string) => void;
  variant?: "compact" | "detailed"; // optional με union — όχι σκέτο string
};

function TodoItem({ todo, onToggle, variant = "detailed" }: TodoItemProps) {
  return (
    <li>
      <input type="checkbox" checked={todo.completed} onChange={() => onToggle(todo.id)} />
      {todo.title}
    </li>
  );
}
```

Σημείωση ορολογίας: τα δεδομένα ρέουν **προς τα κάτω** (props), τα events **προς τα πάνω** (callbacks). Αυτό λέγεται one-way data flow και είναι θεμελιώδης interview απάντηση.

## Typed state & immutable updates

```tsx
const [todos, setTodos] = useState<Todo[]>([]);
```

Στο React **δεν κάνεις ποτέ mutate το state** — δημιουργείς νέες references:

```tsx
// ❌ Mutation — το React ΔΕΝ θα κάνει re-render
todos.push(newTodo);
todo.completed = true;

// ✅ Νέο array / νέο object
setTodos([...todos, newTodo]);
setTodos(todos.map((t) => (t.id === id ? { ...t, completed: !t.completed } : t)));
setTodos(todos.filter((t) => t.id !== id));
```

Γιατί; Το React συγκρίνει references (`Object.is`) για να αποφασίσει αν άλλαξε κάτι. Αν κάνεις mutate το ίδιο array, η reference είναι ίδια → το React νομίζει ότι τίποτα δεν άλλαξε. Αυτή η εξήγηση — «reference equality, όχι deep comparison» — είναι αυτό που ψάχνει ο interviewer.

## Derived state: μην αποθηκεύεις ό,τι υπολογίζεται

Συχνότατο λάθος σε code reviews: δεύτερο state που «καθρεφτίζει» το πρώτο.

```tsx
// ❌ Δύο sources of truth — θα ξεσυγχρονιστούν κάποια στιγμή
const [todos, setTodos] = useState<Todo[]>([]);
const [activeTodos, setActiveTodos] = useState<Todo[]>([]);

// ✅ Ένα state, τα υπόλοιπα υπολογίζονται στο render
const [todos, setTodos] = useState<Todo[]>([]);
const [filter, setFilter] = useState<Filter>("all");

const visibleTodos = useMemo(() => {
  if (filter === "active") return todos.filter((t) => !t.completed);
  if (filter === "completed") return todos.filter((t) => t.completed);
  return todos;
}, [todos, filter]);
```

Το `useMemo` εδώ αποφεύγει τον επανυπολογισμό όταν δεν άλλαξαν τα inputs — αλλά το ουσιαστικό είναι το pattern: **ένα source of truth, όλα τα άλλα derived**.

## useEffect — λιγότερο απ' όσο νομίζεις

Το `useEffect` είναι για **συγχρονισμό με εξωτερικά συστήματα**: fetch, subscriptions, timers, DOM APIs. ΔΕΝ είναι για να «αντιδράς σε state» — αυτό γίνεται στο render.

```tsx
useEffect(() => {
  const controller = new AbortController();

  async function load() {
    try {
      const res = await fetch(`/api/todos?filter=${filter}`, {
        signal: controller.signal,
      });
      setTodos(await res.json());
    } catch (err) {
      if (!controller.signal.aborted) setError(true);
    }
  }

  load();
  return () => controller.abort(); // cleanup: ακύρωσε το προηγούμενο
}, [filter]); // ξανατρέχει ΜΟΝΟ όταν αλλάξει το filter
```

Τα τρία σημεία που εξετάζονται:

- **Dependency array**: `[]` = μία φορά στο mount, `[filter]` = όταν αλλάζει το filter, καθόλου array = κάθε render (σχεδόν πάντα bug).
- **Cleanup function**: τρέχει πριν το επόμενο effect και στο unmount — εκεί ακυρώνεις timers/subscriptions/requests. Χωρίς αυτήν: memory leaks και race conditions.
- **Race condition**: αν ο χρήστης αλλάξει filter γρήγορα, ένα αργό παλιό response μπορεί να «πατήσει» το νέο. Το `AbortController` το λύνει.

## Lists & keys

```tsx
{visibleTodos.map((todo) => (
  <TodoItem key={todo.id} todo={todo} onToggle={toggleTodo} />
))}
```

Το `key` βοηθά το React να ταιριάξει τα στοιχεία μεταξύ renders. Γιατί **όχι** το index ως key; Αν η λίστα αναδιαταχθεί ή διαγραφεί στοιχείο, τα indexes αλλάζουν «ιδιοκτήτη» — το React θα μπερδέψει το state των components (π.χ. checked checkboxes εμφανίζονται σε λάθος row). Με σταθερό `id`, κάθε στοιχείο κρατά την ταυτότητά του.

## Πού ζει το state; (lifting state up)

Κανόνας: το state ζει στον **χαμηλότερο κοινό πρόγονο** όλων των components που το χρειάζονται.

```text
TodoApp        ← state: todos, filter (το χρειάζονται και τα δύο παιδιά)
├── TodoForm   ← callback: onAdd
└── TodoList   ← props: visibleTodos, onToggle
```

Αν το state το χρειάζεται μόνο ένα component, κράτα το εκεί (τοπικά). Αν το χρειάζεται «όλη η εφαρμογή» (theme, current user), τότε Context — αλλά μην ξεκινάς από εκεί.

## Πώς απαντάς στη συνέντευξη

«How does React decide to re-render?»

> A component re-renders when its state changes or its parent re-renders. React compares the new virtual DOM with the previous one and only updates the real DOM where something changed. That's why immutable updates matter — React relies on reference equality to detect changes, so mutating an existing object or array makes the update invisible.

«When do you use useEffect?»

> Only for synchronizing with external systems — data fetching, subscriptions, timers. Derived values belong in the render body or useMemo, not in effects. Every effect that subscribes to something returns a cleanup function, and for fetches I use an AbortController to prevent race conditions.
