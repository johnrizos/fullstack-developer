# Pipes, Guards & Interceptors

Το request pipeline του Nest είναι ο λόγος που το framework νιώθει «enterprise»: validation, authorization, logging και error shaping γίνονται με **declarative** components αντί για if/else μέσα στους handlers. Στη συνέντευξη: «ποια η σειρά εκτέλεσης;», «pipe vs guard vs interceptor;», «πού πιάνεις τα exceptions;».

## Τι πρέπει να μπορείς στο τέλος

- να ξέρεις τη **σειρά** του pipeline,
- να κάνεις validation με ValidationPipe + DTOs,
- να γράφεις guards για authorization,
- να χρησιμοποιείς interceptors για cross-cutting concerns,
- να shape-άρεις errors με exception filters.

## Η σειρά του pipeline

```txt
Request
  → Middleware
  → Guards            (επιτρέπεται; canActivate → true/false)
  → Interceptors (pre)
  → Pipes             (transform/validate input)
  → ROUTE HANDLER
  → Interceptors (post) (map/wrap response)
  → Exception Filters  (αν σκάσει οτιδήποτε)
Response
```

Κλειδί: **guards πριν pipes**. Δεν validate-άρεις input για κάποιον που ούτως ή άλλως θα φάει `403`. Και τα exception filters πιάνουν errors από **όλα** τα προηγούμενα στάδια.

## Pipes — validation & transformation

Το ValidationPipe + class-validator DTO είναι ο standard τρόπος:

```ts
// create-task.dto.ts
export class CreateTaskDto {
  @IsString() @MinLength(2)
  title: string;

  @IsOptional() @IsBoolean()
  completed?: boolean;
}

// controller
@Post()
create(@Body() dto: CreateTaskDto) {   // ValidationPipe ελέγχει το dto
  return this.taskService.create(dto);
}

// main.ts — global
app.useGlobalPipes(new ValidationPipe({ whitelist: true, transform: true }));
```

- `whitelist: true` → **πετά** properties που δεν είναι στο DTO (απόρριψη extra fields → mass-assignment protection).
- `transform: true` → μετατρέπει το plain object σε instance του DTO class και κάνει type coercion (π.χ. `"42"` → `42`).

Invalid input → αυτόματο `400` με λίστα errors. Δεν γράφεις validation logic στον handler.

## Guards — authorization

Ο guard απαντά **ναι/όχι** στο «επιτρέπεται αυτό το request;» μέσω `canActivate`:

```ts
@Injectable()
export class JwtAuthGuard implements CanActivate {
  canActivate(ctx: ExecutionContext): boolean {
    const req = ctx.switchToHttp().getRequest();
    const token = req.headers.authorization?.replace("Bearer ", "");
    if (!token) throw new UnauthorizedException();        // → 401
    req.user = this.jwt.verify(token);                    // attach user
    return true;
  }
}

@UseGuards(JwtAuthGuard)
@Get("me")
me(@Req() req) { return req.user; }
```

Για role-based access συνδυάζεις guard + custom decorator μέσω `Reflector`:

```ts
@Roles("admin")           // custom metadata decorator
@UseGuards(JwtAuthGuard, RolesGuard)
@Delete(":id")
remove(@Param("id") id: string) { /* ... */ }
```

Ο `RolesGuard` διαβάζει το metadata με `reflector.get(Roles, ...)` και συγκρίνει με το `req.user.role`.

## Interceptors — cross-cutting concerns

Ο interceptor τυλίγει τον handler — τρέχει **πριν και μετά**. Ιδανικός για logging, response shaping, caching, timeouts:

```ts
@Injectable()
export class TransformInterceptor implements NestInterceptor {
  intercept(ctx: ExecutionContext, next: CallHandler): Observable<any> {
    const now = Date.now();
    return next.handle().pipe(                 // next.handle() = ο handler
      map((data) => ({ data, tookMs: Date.now() - now })),  // wrap response
    );
  }
}
```

Επειδή δουλεύει με RxJS `Observable`, μπορείς να κάνεις `map`, `tap` (side effects/logging), `timeout`, `catchError`. Π.χ. ένα logging interceptor κάνει `tap` πριν/μετά χωρίς να αγγίζει τον handler.

## Exception filters — error shaping

Όλα τα exceptions καταλήγουν σε filter που τα μετατρέπει σε HTTP response:

```ts
@Catch(HttpException)
export class HttpErrorFilter implements ExceptionFilter {
  catch(ex: HttpException, host: ArgumentsHost) {
    const res = host.switchToHttp().getResponse();
    res.status(ex.getStatus()).json({
      error: ex.message,
      timestamp: new Date().toISOString(),
    });
  }
}
```

Το Nest έχει built-in filter που ήδη κάνει sensible μετατροπή των `HttpException` (π.χ. `NotFoundException` → 404). Γράφεις custom μόνο όταν θες ενιαίο error shape ή logging.

## Πώς απαντάς στη συνέντευξη

«Walk me through the NestJS request pipeline.»

> A request goes middleware, then guards, then interceptors, then pipes, then the handler, and on the way out interceptors again, with exception filters catching anything that throws. Guards run before pipes on purpose — there's no point validating input for a request that'll be rejected with 403. Guards implement `canActivate` and return true/false, so I put JWT auth and role checks there, reading role metadata with a Reflector. Pipes handle validation and transformation; I use `ValidationPipe` with `whitelist: true` so unknown fields are stripped — that's mass-assignment protection — and `transform: true` to get a real DTO instance with coerced types. Interceptors wrap the handler with RxJS, so logging, timing, caching and response shaping live there without touching handlers. Finally exception filters turn thrown exceptions into a consistent HTTP error shape; Nest already maps its built-in `HttpException`s sensibly so I only add a custom filter for a unified envelope or logging.
