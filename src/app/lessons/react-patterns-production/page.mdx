# Patterns, Testing & Production

Το τελευταίο βήμα του React track: τα patterns που κρατούν τον κώδικα καθαρό σε μεγάλη κλίμακα, η προσβασιμότητα, και το testing που δίνει εμπιστοσύνη. Στα interviews ξεχωρίζεις όταν δείχνεις ότι γράφεις React που **αντέχει σε ομάδα και σε production**.

## Τι πρέπει να μπορείς στο τέλος

- να εφαρμόζεις composition patterns (compound components, render-as-children),
- να γράφεις προσβάσιμα (a11y) components,
- να τεστάρεις συμπεριφορά, όχι implementation, με React Testing Library,
- να χειρίζεσαι errors με boundaries σε production,
- να αναγνωρίζεις code smells σε review.

## Composition patterns

**Compound components** — components που δουλεύουν μαζί και μοιράζονται implicit state:

```tsx
<Tabs defaultValue="profile">
  <Tabs.List>
    <Tabs.Trigger value="profile">Profile</Tabs.Trigger>
    <Tabs.Trigger value="settings">Settings</Tabs.Trigger>
  </Tabs.List>
  <Tabs.Panel value="profile">…</Tabs.Panel>
  <Tabs.Panel value="settings">…</Tabs.Panel>
</Tabs>
```

Το `Tabs` κρατά το active state (μέσω Context) και τα παιδιά το διαβάζουν. Ο caller συνθέτει ελεύθερα — καθαρό API χωρίς prop explosion.

**Custom hooks για logic reuse** (από προηγούμενο μάθημα) είναι σήμερα το προτιμώμενο pattern έναντι των παλιών HOCs/render props για το μοίρασμα λογικής.

## Accessibility — δεν είναι προαιρετικό

```tsx
// ❌ div ως κουμπί — μη προσβάσιμο
<div onClick={handleClick}>Delete</div>

// ✅ semantic element + a11y attributes
<button onClick={handleClick} aria-label="Delete item">Delete</button>
```

Βασικά: semantic HTML (`button`, `nav`, `main`), labels σε inputs, keyboard navigation, `aria-*` όπου χρειάζεται, focus management σε modals. Το a11y είναι συχνά interview/review σημείο — και νομική απαίτηση σε πολλές εταιρείες.

## Testing: behavior, όχι implementation

Η φιλοσοφία της React Testing Library: τεστάρεις ό,τι **βλέπει και κάνει ο χρήστης**, όχι internal state ή ονόματα functions.

```tsx
import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";

test("προσθέτει todo", async () => {
  render(<TodoApp />);
  await userEvent.type(screen.getByRole("textbox"), "Buy milk");
  await userEvent.click(screen.getByRole("button", { name: /add/i }));
  expect(screen.getByText("Buy milk")).toBeInTheDocument();
});
```

Query με βάση ρόλους/κείμενο (`getByRole`, `getByText`) — όπως θα έβρισκε το στοιχείο ένας χρήστης. Αυτά τα tests δεν σπάνε σε refactor εσωτερικών — σπάνε μόνο όταν αλλάζει η συμπεριφορά.

## Error boundaries σε production

```tsx
<ErrorBoundary fallback={<ErrorPage />} onError={(e) => reportToSentry(e)}>
  <Dashboard />
</ErrorBoundary>
```

Σε production τυλίγεις λογικές περιοχές (route, widget) σε boundaries ώστε ένα crash σε ένα κομμάτι να μη ρίχνει όλη την εφαρμογή — και στέλνεις το error σε monitoring (Sentry). Συνδυάζεται με loading states (Suspense) για ολοκληρωμένο UX.

## Code smells σε review

```txt
- Mutation του state (push, άμεση ανάθεση πεδίου)
- index ως key σε δυναμική λίστα
- Derived τιμή σε useEffect + δεύτερο state
- useEffect για κάτι που ανήκει σε event handler
- Boolean prop explosion αντί για composition
- Μόνο happy path (χωρίς loading/error/empty)
- div με onClick αντί για button (a11y)
```

## Πώς απαντάς στη συνέντευξη

«How do you keep a large React codebase maintainable?»

> I lean on composition over configuration — compound components that share state through context give a clean API without a pile of boolean props, and I reuse logic through custom hooks rather than HOCs. Accessibility is built in, not bolted on: semantic elements, labels, keyboard support, focus management in modals. I test behavior, not implementation, with React Testing Library — querying by role and text the way a user would — so my tests survive refactors like swapping `useState` for `useReducer` and only break on real behavior changes. In production I wrap routes and widgets in error boundaries that show a fallback and report to monitoring, so one crashing piece doesn't take down the app. And in review I watch for the usual smells: state mutation, index keys, derived state in effects, and missing loading/error/empty states.
