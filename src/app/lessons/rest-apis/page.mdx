# REST APIs

REST API είναι ένας κοινός τρόπος να οργανώνεις επικοινωνία frontend-backend.

## Resources

Σκέψου resources:

- `/tasks`
- `/users`
- `/orders`

Τα URLs πρέπει να μοιάζουν με ουσιαστικά, όχι με functions. Προτίμησε:

- `GET /tasks`
- `POST /tasks`
- `PATCH /tasks/123`

Απέφυγε:

- `GET /getTasks`
- `POST /createTaskNow`

## HTTP methods

- `GET /tasks`: διάβασε tasks.
- `POST /tasks`: δημιούργησε task.
- `PATCH /tasks/:id`: άλλαξε task.
- `DELETE /tasks/:id`: διέγραψε task.

Το method δείχνει intent. Αυτό βοηθά caching, tooling, logs και άλλους developers να καταλάβουν το API χωρίς έξτρα εξήγηση.

## Status codes

- `200 OK`: επιτυχής ανάγνωση/ενημέρωση.
- `201 Created`: δημιουργήθηκε resource.
- `400 Bad Request`: λάθος input.
- `401 Unauthorized`: δεν έχεις login/token.
- `403 Forbidden`: έχεις login αλλά όχι δικαίωμα.
- `404 Not Found`: δεν βρέθηκε.
- `500 Internal Server Error`: server bug/failure.

Μη γυρνάς πάντα `200` με `{ success: false }`. Αν το input είναι λάθος, χρησιμοποίησε `400`. Αν λείπει login, `401`. Αν ο χρήστης δεν έχει δικαίωμα, `403`.

## API contract

Καλό API έχει προβλέψιμο response shape.

```json
{
  "data": [],
  "error": null
}
```

Για errors:

```json
{
  "data": null,
  "error": {
    "message": "Title is required"
  }
}
```

Για validation errors μπορείς να δώσεις και field-level errors:

```json
{
  "data": null,
  "error": {
    "message": "Validation failed",
    "fields": {
      "title": "Title is required"
    }
  }
}
```

## Pagination και filtering

Μην επιστρέφεις “όλα τα rows” αν τα δεδομένα μεγαλώνουν.

```http
GET /tasks?completed=false&page=2&pageSize=20
```

Ένα predictable response μπορεί να έχει metadata:

```json
{
  "data": [],
  "meta": {
    "page": 2,
    "pageSize": 20,
    "total": 135
  }
}
```

## Idempotency

Idempotent σημαίνει ότι αν κάνεις το ίδιο request πολλές φορές, το τελικό αποτέλεσμα παραμένει ίδιο.

- `GET` πρέπει να είναι safe/read-only.
- `PUT` συνήθως είναι idempotent.
- `DELETE` συχνά μπορεί να είναι idempotent.
- `POST` για create δεν είναι πάντα idempotent.

Για payments ή orders μπορεί να χρειαστεί idempotency key, ώστε retry να μη δημιουργήσει διπλή χρέωση.

## Versioning

Αν αλλάξεις API contract, σκέψου compatibility.

Συχνές επιλογές:

- `/api/v1/tasks`,
- header-based versioning,
- additive changes χωρίς breaking response changes.

Μην αφαιρείς fields που χρησιμοποιεί frontend χωρίς migration plan.
