# Utility, Mapped & Conditional Types

Το TypeScript σου επιτρέπει να **παράγεις** types από άλλους types αντί να τους ξαναγράφεις. Αυτό το μάθημα καλύπτει τα built-in utility types (το λεξιλόγιο που εξετάζεται), πώς χτίζονται με **mapped** & **conditional** types (+ `infer`), και το modern `satisfies` operator. Το κεντρικό μήνυμα: **ένα source of truth** — ορίζεις ένα type μία φορά και παράγεις τα συγγενικά.

## Built-in utility types — το λεξιλόγιο

```ts
interface Task {
  id: string;
  title: string;
  status: "todo" | "doing" | "done";
  createdAt: Date;
}

type TaskDraft = Omit<Task, "id" | "createdAt">; // χωρίς server-generated fields
type TaskPatch = Partial<TaskDraft>;             // όλα optional — για PATCH body
type TaskPreview = Pick<Task, "id" | "title">;   // μόνο όσα δείχνει η λίστα
type TasksById = Record<string, Task>;           // map από id σε Task
type FrozenTask = Readonly<Task>;                // όλα readonly
type RequiredDraft = Required<TaskDraft>;        // όλα required

function createTask(draft: TaskDraft): Task { /* ... */ }
type Created = ReturnType<typeof createTask>;     // Task — παράγεται, δεν αντιγράφεται
type Args = Parameters<typeof createTask>;        // [TaskDraft]
```

Τα πιο συχνά στις συνεντεύξεις: `Partial`, `Pick`, `Omit`, `Record`, `Readonly`, `Required`, `ReturnType`, `Parameters`, `Awaited`. Το πραγματικό μάθημα: ορίζεις το `Task` **μία φορά** και παράγεις τα υπόλοιπα — αν αλλάξει το `Task`, ενημερώνονται αυτόματα.

## Mapped types — πώς χτίζονται τα utility types

Ένα mapped type «περπατά» πάνω στα κλειδιά ενός τύπου και φτιάχνει νέο. Έτσι ορίζεται, π.χ., το `Partial`:

```ts
type MyPartial<T> = { [K in keyof T]?: T[K] };   // κάθε property γίνεται optional
type MyReadonly<T> = { readonly [K in keyof T]: T[K] };
```

Το `[K in keyof T]` είναι το «for κάθε κλειδί K του T». Μπορείς να προσθέσεις/αφαιρέσεις modifiers (`?`, `readonly`) και να μετασχηματίσεις τον τύπο `T[K]`. Αυτό ακριβώς (keyof + indexed access από το προηγούμενο μάθημα) στηρίζει όλα τα built-in.

## Conditional types & `infer`

Ένα conditional type διαλέγει τύπο με βάση μια συνθήκη: `T extends U ? X : Y`.

```ts
type IsString<T> = T extends string ? "yes" : "no";
type A = IsString<string>; // "yes"
type B = IsString<number>; // "no"
```

Το `infer` «πιάνει» έναν τύπο μέσα από μια δομή. Έτσι ορίζεται, π.χ., το `Awaited`/`ReturnType`:

```ts
type MyAwaited<T> = T extends Promise<infer U> ? U : T; // βγάλε το U μέσα από το Promise
type R = MyAwaited<Promise<string>>; // string

type MyReturnType<F> = F extends (...args: any[]) => infer R ? R : never;
```

> Mapped & conditional types είναι «προχωρημένα» — δεν θα τα γράφεις κάθε μέρα. Αλλά το να **καταλαβαίνεις** πώς χτίζονται τα utility types ξεχωρίζει στις συνεντεύξεις και σε ξεμπλοκάρει όταν διαβάζεις types βιβλιοθηκών.

## `satisfies` — έλεγξε χωρίς να χάσεις τον στενό τύπο

Πρόβλημα: ένα type annotation **πλαταίνει** τον τύπο, ενώ ένα `as` **παρακάμπτει** τον έλεγχο. Το `satisfies` (modern TS) δίνει το καλύτερο και των δύο: ελέγχει ότι η τιμή ταιριάζει σε ένα type, **κρατώντας** τον στενό inferred τύπο.

```ts
type Config = Record<string, string | number>;

// με annotation: χάνεις τους στενούς τύπους
const a: Config = { port: 3000, host: "localhost" };
a.port; // string | number — δεν ξέρει ότι είναι number 😕

// με satisfies: ελέγχεται ΚΑΙ κρατά τους στενούς τύπους
const b = { port: 3000, host: "localhost" } satisfies Config;
b.port;  // number ✅
b.host;  // string ✅
b.xyz;   // ❌ compile error — ελέγχθηκε ότι ταιριάζει στο Config
```

Χρησιμοποίησέ το όταν θες **και** validation έναντι ενός type **και** τον ακριβή inferred τύπο της τιμής (config objects, routes, palettes).

## Πώς απαντάς στη συνέντευξη

> «How do you avoid duplicating types, and what's `satisfies`?»

> One source of truth: I define the core type once and derive the rest with utility types — `Omit` for a create-draft without server fields, `Partial` for a PATCH body, `Pick` for a list preview, `ReturnType`/`Awaited` to pull a type out of a function. If the core type changes, the derived ones update automatically. Under the hood those are mapped types — `{ [K in keyof T]?: T[K] }` is essentially `Partial` — and conditional types with `infer`, which is how `Awaited` and `ReturnType` extract an inner type. And `satisfies` is the modern operator I reach for on config-like objects: it checks the value against a type without widening it, so I keep the precise inferred literal types while still catching typos — something a plain annotation loses and an `as` cast skips entirely.
