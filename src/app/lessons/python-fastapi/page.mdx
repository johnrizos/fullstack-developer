# FastAPI Services

Όταν ένα Node/Next.js product χρειάζεται Python — για AI, data processing, scraping, background jobs — το standard είναι ένα **FastAPI service** δίπλα στο main app. Στις συνεντεύξεις, το ζητούμενο δεν είναι «ξέρεις FastAPI;» αλλά «καταλαβαίνεις πού μπαίνει η Python σε fullstack architecture, και πώς σχεδιάζεις το HTTP contract ανάμεσα στα services;».

## Τι πρέπει να μπορείς στο τέλος

- να εξηγείς γιατί βάζεις Python service σε Node/Next product (και πότε ΔΕΝ το βάζεις),
- να γράφεις FastAPI endpoints με Pydantic request/response models,
- να καταλαβαίνεις πώς το FastAPI κάνει validation από τα type hints,
- να χρησιμοποιείς dependency injection (`Depends`) για auth/db,
- να σχεδιάζεις το contract main-app ↔ Python service.

## Πού ταιριάζει — και πού όχι — η Python

Καλή χρήση Python service: AI/ML inference, data pipelines, PDF/image processing, scraping, analytics, scheduled jobs — δουλειές όπου το Python ecosystem (numpy, pandas, torch, transformers) είναι ασύγκριτο.

Κακή χρήση: user CRUD, basic login, settings page, business rules που ανήκουν στο core product. Μην προσθέτεις service μόνο και μόνο επειδή «είναι trendy» — κάθε νέο service είναι deployment, monitoring και network hop παραπάνω.

> Η ερώτηση-φίλτρο πριν φτιάξεις Python service: *«Το πρόβλημα χρειάζεται το Python ecosystem, ή απλώς ένα ακόμη backend endpoint;»* Αν είναι το δεύτερο, μένει στο Node.

## FastAPI — το πρώτο service

```py
from fastapi import FastAPI
from pydantic import BaseModel, Field

app = FastAPI()

class AnalyzeRequest(BaseModel):
    text: str = Field(min_length=1, max_length=10_000)
    language: str = "en"

class AnalyzeResponse(BaseModel):
    word_count: int
    sentiment: str

@app.post("/analyze", response_model=AnalyzeResponse)
def analyze(payload: AnalyzeRequest) -> AnalyzeResponse:
    words = payload.text.split()
    sentiment = "positive" if "good" in payload.text.lower() else "neutral"
    return AnalyzeResponse(word_count=len(words), sentiment=sentiment)
```

Τι κερδίζεις «δωρεάν» εδώ — και αυτό είναι το πιο σημαντικό σημείο της συνέντευξης: το FastAPI **διαβάζει τα Pydantic models / type hints** και:
- κάνει **validation** του incoming JSON (λάθος ή λείπον πεδίο → αυτόματα `422` με σαφές μήνυμα),
- κάνει **parsing & coercion** στους σωστούς τύπους,
- κάνει **serialization** του response μέσω του `response_model`,
- παράγει **αυτόματο OpenAPI / Swagger** στο `/docs`.

Δηλαδή ο ίδιος ο type hint που τεκμηριώνει τον κώδικα **είναι** το validation contract. Αυτό είναι το «type hints αποκτούν δόντια» που είδες στο OOP/Typing μάθημα.

## Async endpoints & I/O

Το FastAPI υποστηρίζει `async def` για I/O-bound δουλειά (database, εξωτερικά APIs). Η λογική είναι ίδια με το Node event loop:

```py
import httpx

@app.get("/weather/{city}")
async def weather(city: str):
    async with httpx.AsyncClient() as client:
        resp = await client.get(f"https://api.example.com/weather/{city}")
        return resp.json()
```

Κανόνας: `async def` + `await` για I/O (μη μπλοκάρεις το event loop). Για **CPU-bound** δουλειά (ML inference, image processing) **μην** την τρέχεις inline σε async handler — μπλοκάρει τα πάντα· πάει σε background job ή worker (επόμενο μάθημα).

## Dependency Injection με `Depends`

Το FastAPI έχει built-in DI για κοινές ανάγκες (auth, database session, config). Δηλώνεις τι χρειάζεσαι, το framework το παρέχει:

```py
from fastapi import Depends, Header, HTTPException

def require_api_key(x_api_key: str = Header()):
    if x_api_key != "secret-from-env":
        raise HTTPException(status_code=401, detail="Invalid API key")
    return x_api_key

@app.post("/jobs")
def create_job(payload: AnalyzeRequest, _: str = Depends(require_api_key)):
    return {"status": "queued"}
```

Το `Depends(require_api_key)` τρέχει πριν τον handler· αν πετάξει `HTTPException`, ο handler δεν εκτελείται ποτέ. Ίδια ιδέα με τα middleware/guards του Express/NestJS — testable, reusable, ορατές εξαρτήσεις.

## Το contract main-app ↔ Python service

Σχεδίασε τη ροή ρητά. Δύο pattern:

```txt
1) Synchronous (γρήγορη δουλειά < ~1s):
   Next.js API ──POST /analyze──> FastAPI ──result──> Next.js (περιμένει)

2) Async job (αργή δουλειά: ML, μεγάλα αρχεία):
   Next.js ──POST /jobs──> FastAPI: επιστρέφει {job_id, status:"queued"} αμέσως
   worker επεξεργάζεται στο background
   Next.js ──GET /jobs/{id}──> {status:"done", result:{...}}  (polling ή webhook)
```

Ο κανόνας: **μην κρατάς ανοιχτό HTTP request για δουλειά που αργεί**. Αν ξεπερνά λίγα δευτερόλεπτα, γίνεται job με status flow — αλλιώς γεμίζεις timeouts και κρεμασμένα connections.

## Πώς απαντάς στη συνέντευξη

«When would you add a Python service to a Node/Next.js product?»

> When the problem genuinely needs the Python ecosystem — ML inference, data pipelines, scraping, document processing — not for plain CRUD that the Node backend already handles. I'd expose it as a FastAPI service with Pydantic models, so request validation, response serialization, and the OpenAPI docs all come from the type hints. For fast work I call it synchronously; for anything slow like inference I make it a job — the endpoint returns a job id immediately and the client polls or gets a webhook — so I never hold an HTTP request open for a long task.
