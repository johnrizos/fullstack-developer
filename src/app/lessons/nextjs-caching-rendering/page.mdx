# Caching & Rendering Strategies

Εδώ ξεχωρίζει ο senior Next.js developer. Στο Next.js 16 το caching γίνεται ρητό με **Cache Components** και τη directive `use cache`, και το default rendering είναι **Partial Prerendering (PPR)**: στατικό shell + δυναμικά κομμάτια που streamάρουν. Στα interviews εξετάζουν αν καταλαβαίνεις τι γίνεται build-time vs request-time.

## Τι πρέπει να μπορείς στο τέλος

- να εξηγείς static shell vs request-time rendering (PPR),
- να χρησιμοποιείς `use cache` σε data ή UI level,
- να ρυθμίζεις `cacheLife` και `cacheTag`,
- να χειρίζεσαι runtime APIs (`cookies`, `headers`) με `<Suspense>`,
- να ξέρεις πότε κάτι πρέπει να είναι uncached.

## Το μοντέλο: PPR

Με `cacheComponents: true` στο `next.config.ts`, το Next.js στο build renderάρει το δέντρο και κατατάσσει κάθε κομμάτι:

```txt
use cache              → το αποτέλεσμα μπαίνει στο static shell
<Suspense>             → fallback στο shell, περιεχόμενο streamάρει στο request
deterministic work     → μπαίνει αυτόματα στο shell (pure computation, imports)
runtime APIs (cookies) → πρέπει να είναι σε <Suspense>, αλλιώς build error
```

Το αποτέλεσμα: ο χρήστης παίρνει άμεσα ένα static shell, και τα δυναμικά μέρη streamάρουν. Αυτό είναι **Partial Prerendering**.

## use cache — data level

```tsx
// app/lib/data.ts
import { cacheLife } from "next/cache";

export async function getUsers() {
  "use cache";
  cacheLife("hours"); // πόσο ζει το cache entry
  return db.query("SELECT * FROM users");
}
```

Τα **arguments** της function γίνονται μέρος του cache key — διαφορετικά inputs → διαφορετικά entries (parameterized caching).

## use cache — UI level

Μπορείς να cache-άρεις ολόκληρο component/page/layout:

```tsx
import { cacheLife, cacheTag } from "next/cache";

async function BlogPosts() {
  "use cache";
  cacheLife("hours");
  cacheTag("posts"); // tag για στοχευμένο invalidation

  const res = await fetch("https://api.example.com/blog");
  const posts = await res.json();
  return <PostList posts={posts} />;
}
```

Το `cacheTag("posts")` επιτρέπει αργότερα `updateTag("posts")`/`revalidateTag("posts")` μετά από mutation, ώστε να λήξει στοχευμένα αυτό το cache.

## Uncached / runtime data → Suspense

Για δεδομένα που πρέπει να είναι **φρέσκα σε κάθε request**, ΜΗΝ βάζεις `use cache`. Τύλιξε σε `<Suspense>`:

```tsx
import { cookies } from "next/headers";
import { Suspense } from "react";

async function UserGreeting() {
  const theme = (await cookies()).get("theme")?.value ?? "light"; // runtime API, async
  return <p>Theme: {theme}</p>;
}

export default function Page() {
  return (
    <>
      <h1>Dashboard</h1>
      <Suspense fallback={<p>Loading…</p>}>
        <UserGreeting />
      </Suspense>
    </>
  );
}
```

Τα runtime APIs (`cookies()`, `headers()`, `searchParams`, `params`) είναι **async** (await) και απαιτούν το request — γι' αυτό θέλουν `<Suspense>` ή θα πάρεις build error «uncached data accessed outside of Suspense».

## Non-deterministic operations

`Math.random()`, `Date.now()`, `crypto.randomUUID()` δίνουν διαφορετική τιμή κάθε φορά. Δύο επιλογές:

```tsx
// (α) Μοναδικό ανά request: connection() + Suspense
import { connection } from "next/server";
async function Unique() {
  await connection();
  return <p>{crypto.randomUUID()}</p>;
}

// (β) Ίδιο για όλους μέχρι revalidation: cache το
export default async function Page() {
  "use cache";
  return <p>{crypto.randomUUID()}</p>;
}
```

## Περνώντας runtime τιμές σε cached function

```tsx
async function ProfileContent() {
  const sessionId = (await cookies()).get("session")?.value; // runtime, όχι cached
  return <CachedContent sessionId={sessionId} />;
}

async function CachedContent({ sessionId }: { sessionId?: string }) {
  "use cache";
  const data = await fetchUserData(sessionId); // sessionId μπαίνει στο cache key
  return <div>{data}</div>;
}
```

## Πώς απαντάς στη συνέντευξη

«How does caching and rendering work in Next.js 16?»

> The default with Cache Components is Partial Prerendering: at build time Next.js produces a static shell and defers the dynamic parts to request time. Caching is explicit through the `use cache` directive — I can apply it at data level on a function or at UI level on a whole component, and its arguments become part of the cache key, so different inputs get separate entries. I tune lifetime with `cacheLife` and tag entries with `cacheTag` so a mutation can invalidate them precisely via `revalidateTag`/`updateTag`. Anything that must be fresh per request — `cookies()`, `headers()`, `searchParams` — is async and has to be wrapped in `<Suspense>`; otherwise Cache Components gives me a build-time error instead of silently making the whole route dynamic. Non-deterministic values like `randomUUID` I either defer with `connection()` and Suspense, or cache so everyone sees the same value until revalidation.
