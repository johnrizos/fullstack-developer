# Data Fetching & Streaming

Στο App Router φέρνεις δεδομένα **κατευθείαν μέσα σε async Server Components** — χωρίς `getServerSideProps`/`getStaticProps`. Στα interviews εξετάζουν αν ξέρεις ότι το `fetch` **δεν cache-άρεται πια by default**, πώς γίνεται streaming με `loading.js`/`<Suspense>`, και πώς αποφεύγεις request waterfalls.

## Τι πρέπει να μπορείς στο τέλος

- να φέρνεις δεδομένα σε async Server Components (fetch ή ORM),
- να ξέρεις ότι το `fetch` δεν είναι cached by default,
- να κάνεις streaming με `loading.js` και `<Suspense>`,
- να ξεχωρίζεις parallel από sequential fetching,
- να στέλνεις δεδομένα σε Client Component με το `use` API.

## Fetch σε Server Component

```tsx
// app/blog/page.tsx — async Server Component
export default async function Page() {
  const res = await fetch("https://api.example.com/blog");
  const posts = await res.json();
  return (
    <ul>
      {posts.map((post: { id: string; title: string }) => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  );
}
```

Ή κατευθείαν με ORM/database — επειδή τρέχει στον server, τα credentials δεν φεύγουν στον client:

```tsx
import { db, posts } from "@/lib/db";
export default async function Page() {
  const allPosts = await db.select().from(posts);
  return <List items={allPosts} />;
}
```

## fetch: ΔΕΝ είναι cached by default (Next.js 16)

Κρίσιμο breaking change. Παλιότερα το `fetch` cache-αρόταν αυτόματα· τώρα **όχι** — μπλοκάρει το render μέχρι να ολοκληρωθεί. Για caching χρησιμοποιείς τη directive `use cache` (επόμενο μάθημα), ή τυλίγεις σε `<Suspense>` για streaming. Επίσης, ίδια `fetch` requests στο ίδιο render **memoize-άρονται** (deduplication ανά request).

## Streaming με loading.js

Ένα `loading.tsx` δίπλα στο `page.tsx` δείχνει instant loading UI ενώ φορτώνει η σελίδα — αυτόματα τυλίγει το page σε `<Suspense>`:

```tsx
// app/blog/loading.tsx
export default function Loading() {
  return <BlogSkeleton />;
}
```

Ο χρήστης βλέπει αμέσως το layout + skeleton, και το περιεχόμενο μπαίνει όταν ετοιμαστεί.

## Streaming με `<Suspense>` (granular)

```tsx
import { Suspense } from "react";

export default function BlogPage() {
  return (
    <div>
      <header><h1>Welcome</h1></header> {/* στέλνεται αμέσως */}
      <Suspense fallback={<BlogListSkeleton />}>
        <BlogList /> {/* αργό async — streamάρει όταν ετοιμαστεί */}
      </Suspense>
    </div>
  );
}
```

Το `<Suspense>` σου δίνει έλεγχο: ό,τι είναι εκτός boundary φαίνεται αμέσως, ό,τι μέσα streamάρει.

```tsx
// ✅ Parallel — ξεκινούν μαζί
const artistData = getArtist(username);
const albumsData = getAlbums(username);
const [artist, albums] = await Promise.all([artistData, albumsData]); // ~300ms
```

Όταν το ένα request **εξαρτάται** από το άλλο (χρειάζεσαι το `artist.id`), τότε sequential είναι αναπόφευκτο — αλλά τύλιξε το εξαρτημένο σε `<Suspense>` για να μη μπλοκάρει όλη η σελίδα.

## Client streaming με το use API

Φέρνεις στον server, περνάς το **promise** (χωρίς await) σε Client Component, και το διαβάζεις με `use`:

```tsx
// Server
export default function Page() {
  const posts = getPosts(); // ΧΩΡΙΣ await
  return (
    <Suspense fallback={<div>Loading...</div>}>
      <Posts posts={posts} />
    </Suspense>
  );
}

// Client
"use client";
import { use } from "react";
export default function Posts({ posts }: { posts: Promise<Post[]> }) {
  const all = use(posts); // resolve-άρει το promise
  return <ul>{all.map((p) => <li key={p.id}>{p.title}</li>)}</ul>;
}
```

Για data fetching σε Client Components, εναλλακτικά χρησιμοποιείς SWR/React Query.

## Πώς απαντάς στη συνέντευξη

«How do you fetch data in the App Router?»

> I fetch directly inside async Server Components — no `getServerSideProps`. I can `await fetch` or query a database/ORM, and since it runs on the server, credentials stay off the client. A big Next.js 16 change is that `fetch` is no longer cached by default; it blocks rendering until it resolves, and I opt into caching with the `use cache` directive or stream with Suspense. Identical fetches in one render are still deduped. For perceived performance I stream: a `loading.tsx` gives the whole segment an instant skeleton, and `<Suspense>` lets me stream specific slow parts while the rest shows immediately. I watch for waterfalls — two independent `await`s in a row run sequentially, so I kick them off together and `await Promise.all`. To stream data into a Client Component I pass the unawaited promise and read it with React's `use`.
