# Agile, Scrum & Kanban

Οι ορισμοί («Scrum είναι framework με sprints») δεν σε βοηθούν την πρώτη μέρα στη δουλειά. Αυτό που χρειάζεσαι είναι να **δεις ένα πραγματικό feature να ταξιδεύει** από «ιδέα» μέχρι «σε production». Γι' αυτό όλο αυτό το μάθημα παρακολουθεί **μία ομάδα, ένα feature, ένα sprint** — μέρα με τη μέρα. Τους ορισμούς θα τους μάθεις στην πορεία, μέσα από το παράδειγμα.

## Η σκηνή: μια αληθινή ομάδα

```txt
Προϊόν:   "JobBoard" — site αγγελιών εργασίας
Ομάδα:    Maria (Product Owner), Kostas (Scrum Master / senior),
          Εσύ (junior dev), Lena (dev), Nikos (QA)
Ρυθμός:   2-week sprints, board στο Jira/Linear
Εργαλείο: GitHub για PRs, Slack για επικοινωνία
```

Το feature αυτού του sprint: **«ο χρήστης αποθηκεύει μια αναζήτηση και ειδοποιείται για νέες αγγελίες»**. Δες πώς γίνεται πραγματικότητα.

## Βήμα 0 — Από «ιδέα» σε «ticket» (backlog refinement)

Η Maria (PO) έχει στο **backlog** μια γραμμή: _«saved searches με notifications»_. Είναι πολύ μεγάλο για να το πιάσει ένας dev — λέγεται **epic**. Σε ένα μισάωρο meeting (**refinement/grooming**) η ομάδα το σπάει σε μικρότερα **user stories**:

```txt
EPIC: Saved searches
├─ Story 1: User αποθηκεύει μια αναζήτηση  ← το παίρνεις ΕΣΥ
├─ Story 2: User βλέπει & σβήνει saved searches
└─ Story 3: Nightly job στέλνει email για νέα matches
```

Κάθε story γράφεται από την οπτική του **χρήστη**, όχι σαν τεχνική εντολή:

```txt
TICKET JOB-142
Story: As a job seeker, I want to save my current search filters,
       so that I don't have to re-enter them every visit.

Acceptance criteria:
  ✓ "Save search" button εμφανίζεται όταν υπάρχουν ενεργά filters
  ✓ Click → η αναζήτηση αποθηκεύεται για τον logged-in user
  ✓ Μη logged-in user → prompt για login
  ✓ Επιτυχία δείχνει confirmation· διπλό save δεν δημιουργεί duplicate
  ✓ Λάθος/σφάλμα δείχνει error message, όχι σιωπή

Out of scope: το email notification (Story 3), η σελίδα διαχείρισης (Story 2)
Estimate: 3 points
```

Τα **acceptance criteria** είναι το συμβόλαιό σου: όταν ισχύουν όλα, το ticket είναι «done». Το **out of scope** σε προστατεύει από το να φτιάξεις μισό feature που δεν ζητήθηκε.

> **Story points**: σχετική εκτίμηση δυσκολίας (όχι ώρες). 1 = τετριμμένο, 3 = μέτριο, 8 = «πρέπει να το σπάσουμε». Η ομάδα ψηφίζει — αν εσύ λες 2 και η Lena 8, η συζήτηση **γιατί** αποκαλύπτει κρυμμένη πολυπλοκότητα.

## Βήμα 1 — Sprint planning (Δευτέρα, εβδομάδα 1)

Στο **sprint planning** η ομάδα διαλέγει από το backlog τι **χωράει** σε 2 εβδομάδες (το **capacity** της). Δεν παίρνεις ό,τι θες — δεσμεύεσαι σε ένα ρεαλιστικό σύνολο:

```txt
SPRINT 24 — Commitment (σύνολο 18 points για 4 devs)
  JOB-142  Save a search           3pts  → Εσύ
  JOB-143  Manage saved searches   5pts  → Lena
  JOB-150  Fix pagination bug      2pts  → Εσύ
  JOB-138  Nightly email job       8pts  → Kostas
```

Το **sprint goal** σε μία πρόταση: _«Ο χρήστης μπορεί να αποθηκεύει και να βλέπει saved searches»_. Το Story 3 (email) μπορεί να γλιστρήσει στο επόμενο sprint — γι' αυτό σπάσαμε το epic.

## Βήμα 2 — Η ζωή του JOB-142 στο board, μέρα με τη μέρα

Το board έχει στήλες· το ticket κινείται από αριστερά προς τα δεξιά. Παρακολούθησε:

```txt
Backlog | To Do | In Progress | Code Review | QA | Done
```

**Δευτέρα απόγευμα** — Παίρνεις το JOB-142, το σέρνεις σε **In Progress**. Πρώτα διαβάζεις ξανά τα acceptance criteria και εντοπίζεις μια **ανοιχτή ερώτηση**: «τι ακριβώς αποθηκεύουμε — τα filters ή το URL;». Δεν μαντεύεις — ρωτάς τη Maria στο Slack. Απαντά: «τα filters ως JSON». Τώρα ξέρεις τι χτίζεις.

**Τρίτη** — Φτιάχνεις το migration (πίνακας `saved_searches`), το API endpoint και το κουμπί. Στο μέσο της μέρας **κολλάς**: το endpoint χρειάζεται τον current user, αλλά δεν ξέρεις πώς η ομάδα παίρνει το session. Αυτό είναι **blocker**.

**Τετάρτη, 10:00 — Daily standup** (15 λεπτά, όρθιοι). Κάθε άτομο λέει 3 πράγματα. Πραγματικός διάλογος:

```txt
Lena:   "Χθες τελείωσα το UI του JOB-143. Σήμερα ξεκινάω το delete flow.
         Κανένα blocker."
Εσύ:    "Χθες έφτιαξα το migration και το κουμπί του JOB-142. ΜΠΛΟΚΑΡΩ:
         δεν ξέρω πώς παίρνουμε τον current user στο API layer."
Kostas: "Έχουμε ένα getCurrentUser() helper στο lib/auth. Δείξε μου
         μετά το standup, 2 λεπτά."
Nikos:  "Σήμερα ετοιμάζω test cases για το saved-search flow."
```

Πρόσεξε: το standup **δεν** είναι report στον manager. Είναι sync για να **βγουν τα blockers στην επιφάνεια**. Το δικό σου blocker λύθηκε σε 2 λεπτά αντί να χαθεί μισή μέρα. Αυτό είναι όλο το νόημα.

**Τετάρτη μεσημέρι** — Ξεμπλοκάρεις, τελειώνεις τη λογική, γράφεις 2 tests. Ανοίγεις **Pull Request** και σέρνεις το ticket στο **Code Review**:

```txt
PR #341: Add "save search" feature (JOB-142)

What & why:
  Logged-in users can save current filters. Closes JOB-142.

How it maps to acceptance criteria:
  ✓ Button shows only with active filters (SearchBar.tsx)
  ✓ POST /api/saved-searches με ownership (userId from session)
  ✓ Unique constraint (user_id, filters_hash) → no duplicates
  ✓ 401 για guests → frontend δείχνει login prompt
  ✓ Error toast σε failure

Testing: unit για το dedup, manual για το guest flow
Screenshots: [confirmation toast]
```

Πρόσεξε ότι το PR **απαντά ένα-ένα στα acceptance criteria** — έτσι ο reviewer ξέρει ότι κάλυψες το ticket, όχι κάτι άλλο.

**Πέμπτη** — Ο Kostas κάνει review. Σχόλιο: «το filters_hash είναι καλό, αλλά τι γίνεται αν αλλάξει η σειρά των filters;». Σωστή παρατήρηση — κάνεις normalize πριν το hash, push, ο Kostas approve-άρει. Merge. Το ticket πάει **QA**.

**Παρασκευή** — Ο Nikos (QA) δοκιμάζει: αποθηκεύει, ξανα-αποθηκεύει (no duplicate ✓), δοκιμάζει ως guest (login prompt ✓), προκαλεί error (toast ✓). Όλα τα acceptance criteria περνούν → **Done**. 🎉

## Βήμα 3 — Definition of Done (το ίδιο για ΟΛΑ τα tickets)

Το «Done» δεν είναι «δουλεύει στο laptop μου». Είναι **team agreement** που ισχύει για κάθε ticket:

```txt
✓ Acceptance criteria όλα ισχύουν
✓ Code review approved
✓ lint + tests + build περνούν (CI πράσινο)
✓ Merged στο main
✓ QA verified
✓ Deployed (ή ready to deploy)
```

Το JOB-142 πέρασε όλα αυτά. Αν είχε μείνει κάτι (π.χ. κόκκινο CI), θα γυρνούσε πίσω — όχι στο Done.

## Βήμα 4 — Τέλος sprint: review & retrospective

**Sprint review** (Παρασκευή εβδομάδα 2): η ομάδα **δείχνει** τι παραδόθηκε σε PO/stakeholders. Demo του saved-search flow live. Το JOB-138 (email) δεν πρόλαβε — μεταφέρεται στο επόμενο sprint. Αυτό είναι φυσιολογικό, όχι αποτυχία.

**Retrospective** (μετά το review): η ομάδα κοιτά **τη διαδικασία**, όχι το προϊόν. Με συγκεκριμένα action items:

```txt
✅ Πήγε καλά:  τα blockers βγήκαν γρήγορα στα standups
⚠️ Βελτίωση:   2 tickets κόλλησαν στο Code Review 2 μέρες
🎯 Action:     "Όποιος ανοίγει PR, ζητά review μέσα σε 1 ώρα στο Slack"
               (owner: Kostas, από επόμενο sprint)
```

Χωρίς action item, το retro είναι απλώς παράπονα. Το νόημα είναι η ομάδα να **βελτιώνεται κάθε sprint**.

## Scrum ή Kanban; — μέσα από το παράδειγμα

Η ομάδα του JobBoard δούλεψε **Scrum**: σταθερά 2-week sprints, commitment, planning/review/retro. Καλό όταν θες ρυθμό και προβλεψιμότητα σε feature work.

Αν η ίδια ομάδα ήταν **support/ops team** (συνεχής ροή από επείγοντα bugs που δεν περιμένουν το επόμενο sprint), θα προτιμούσε **Kanban**: χωρίς sprints, συνεχής ροή, με **WIP limits**.

```txt
To Do | In Progress (max 2) | Code Review (max 3) | Done
                  ↑ WIP limit: κανείς δεν ανοίγει 3ο ticket
                    πριν τελειώσει ένα — "stop starting, start finishing"
```

Το **WIP limit** είναι η καρδιά του Kanban: αν όλοι ανοίγουν 5 tasks ταυτόχρονα, πολλά ξεκινούν και λίγα τελειώνουν. Όταν το «In Progress» γεμίσει, σταματάς και **τελειώνεις** πριν ξεκινήσεις νέο. Αν το «Code Review» γεμίζει συνέχεια, είναι ορατό σήμα ότι η ομάδα πρέπει να κάνει reviews αντί να ανοίγει νέα δουλειά.

## Πώς δουλεύεις ΕΣΥ ως junior σε αυτό

Η ροή που είδες, σε 6 κανόνες που εφαρμόζεις από μέρα 1:

```txt
1. Πριν ξεκινήσεις: διάβασε acceptance criteria, βρες ανοιχτές ερωτήσεις, ΡΩΤΑ.
2. Σύρε το ticket στο σωστό column — η ομάδα βλέπει την κατάσταση από το board.
3. Κράτα ένα ticket σε In Progress τη φορά (μην ανοίγεις 5).
4. Κολλημένος; Πες το στο standup ΑΜΕΣΩΣ — blocker, όχι σιωπή.
5. PR focused, που εξηγεί τι άλλαξες και πώς καλύπτει τα acceptance criteria.
6. "Done" μόνο όταν περνά το Definition of Done — όχι "δούλεψε σε μένα".
```
