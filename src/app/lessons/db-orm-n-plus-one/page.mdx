# ORMs, N+1 & Data Access

Οι περισσότερες εφαρμογές μιλάνε στη βάση μέσω **ORM — Object-Relational Mapping** (Prisma, SQLAlchemy, TypeORM): μια βιβλιοθήκη που αντιστοιχίζει τους πίνακες της βάσης σε objects/classes του κώδικά σου. Ο ORM είναι βολικός αλλά κρύβει το SQL — και μαζί το πιο διάσημο performance bug: το **N+1 problem**. Στα interviews το N+1 είναι σχεδόν σίγουρη ερώτηση.

## Τι πρέπει να μπορείς στο τέλος

- να εξηγείς τι κάνει ένας ORM και τα trade-offs,
- να αναγνωρίζεις και να λύνεις το N+1 problem,
- να ξεχωρίζεις eager από lazy loading,
- να ξέρεις πότε γράφεις raw SQL,
- να εξηγείς connection pooling.

## Τι κάνει ένας ORM

Αντιστοιχίζει tables σε objects/classes, ώστε να γράφεις κώδικα αντί για SQL:

```ts
// Prisma-style
const user = await prisma.user.findUnique({ where: { id: 1 } });
const posts = await prisma.post.findMany({ where: { published: true } });
```

```txt
Πλεονεκτήματα: type safety, λιγότερο boilerplate, migrations, ταχύτητα ανάπτυξης
Trade-offs:    κρύβει το SQL → εύκολο να γράψεις αργά queries (N+1) χωρίς να το δεις
```

## Το N+1 problem

Το πιο συχνό ORM bug. Φέρνεις μια λίστα, και μετά για **κάθε** στοιχείο κάνεις ξεχωριστό query:

```ts
const posts = await prisma.post.findMany();        // 1 query: φέρε posts
for (const post of posts) {
  post.author = await prisma.user.findUnique({     // N queries: ένα ανά post 🐢
    where: { id: post.authorId },
  });
}
// 100 posts → 1 + 100 = 101 queries!
```

## Λύση: eager loading (include/join)

```ts
// ✅ Ένα query (ή δύο batched) με join — όχι N+1
const orders = await db.order.findMany({
  include: { customer: true },   // φέρε και τον customer μαζί
});

// ✅ Πολλά parents → φέρε όλα τα children με ένα IN query
const posts = await db.post.findMany({ include: { author: true } });
```

Ο ORM μετατρέπει το `include` σε `JOIN` ή σε ένα δεύτερο `WHERE id IN (...)` query — όχι σε N ξεχωριστά.

## Eager vs lazy loading

```txt
Lazy loading:  τα related δεδομένα φορτώνονται ΟΤΑΝ τα ζητήσεις (συχνά → N+1)
Eager loading: τα φέρνεις ΜΑΖΙ εκ των προτέρων (include/join) → αποφεύγεις N+1
```

Default σε πολλά ORMs είναι lazy — γι' αυτό το N+1 κρύβεται. Όταν ξέρεις ότι θα χρειαστείς τα related, ζήτα τα **eager**.

## Πότε raw SQL

Ο ORM δεν είναι πάντα η απάντηση:

```txt
- Πολύπλοκα analytics queries (window functions, σύνθετα joins/CTEs)
- Bulk operations / performance-critical paths
- Database-specific features που δεν εκθέτει ο ORM
```

Οι περισσότεροι ORMs επιτρέπουν raw SQL escape hatch (`$queryRaw`, `text()`). Χρησιμοποίησέ το συνειδητά — με **parameterized** queries (όχι string concatenation, για να αποφύγεις SQL injection).

## Connection pooling

Το να ανοίγεις νέα σύνδεση ανά request είναι ακριβό. Ένα **connection pool** κρατά ένα σύνολο επαναχρησιμοποιούμενων συνδέσεων:

```txt
- Επαναχρησιμοποιεί συνδέσεις αντί για open/close κάθε φορά
- Όριο max connections → προστατεύει τη βάση από εξάντληση
- Serverless: πρόσεξε — πολλές instances × pool μπορεί να ξεπεράσει το όριο της βάσης
  (λύση: external pooler, π.χ. PgBouncer)
```

## Πώς απαντάς στη συνέντευξη

«What is the N+1 problem and how do you fix it?»

> An ORM maps tables to objects so I write code instead of SQL — great for type safety and speed, but it hides the SQL, which is how the N+1 problem sneaks in. I fetch a list with one query, then for each row I lazily access a relation, firing one query per row: a hundred posts become a hundred-and-one queries. In production that hammers the database. The fix is eager loading — `include` or a join — so the ORM fetches the related data in one query or a single batched `IN`, not N separate ones. Many ORMs default to lazy loading, which is why it's easy to miss, so I check query logs. I drop to raw, parameterized SQL for complex analytics, bulk operations, or database-specific features the ORM doesn't expose. And I use a connection pool to reuse connections and cap the max, being careful in serverless where many instances can exhaust the database's connection limit.
