## Το GIL — η μία πρόταση που πρέπει να ξέρεις

Ο **Global Interpreter Lock** σημαίνει ότι σε CPython **μόνο ένα thread εκτελεί Python bytecode κάθε στιγμή**, ακόμη κι αν έχεις 16 πυρήνες. Συνέπειες:

- **CPU-bound δουλειά ΔΕΝ επιταχύνεται με threads** — τα threads μοιράζονται έναν πυρήνα για Python code. Για παράλληλο CPU χρειάζεσαι **processes** (`multiprocessing`), που έχουν δικό τους interpreter, άρα δικό τους GIL.
- **I/O-bound δουλειά ΕΠΩΦΕΛΕΙΤΑΙ από threads/async** — όταν ένα thread περιμένει δίκτυο/δίσκο, αφήνει το GIL και τρέχει άλλο. Εδώ η συγχρονικότητα δουλεύει.

Έτσι ο πίνακας απόφασης είναι απλός:

```txt
I/O-bound (δίκτυο, db, αρχεία):  asyncio  (ή threads)  → ένας πυρήνας φτάνει
CPU-bound (ML, math, encoding):  multiprocessing       → πολλοί πυρήνες, παρακάμπτεις GIL
Πολλά+βαριά:                     queue + worker process (Celery/RQ)
```

## asyncio — concurrency σε ένα thread

Για I/O-bound δουλειά, το `asyncio` δίνει concurrency χωρίς threads — ίδιο μοντέλο με το Node event loop:

```py
import asyncio
import httpx

async def fetch(client, url):
    resp = await client.get(url)
    return resp.status_code

async def main():
    urls = ["https://a.com", "https://b.com", "https://c.com"]
    async with httpx.AsyncClient() as client:
        # Όλα ΤΑΥΤΟΧΡΟΝΑ — όχι σειριακά
        results = await asyncio.gather(*(fetch(client, u) for u in urls))
    print(results)

asyncio.run(main())
```

Σημεία-κλειδιά:
- `await` = «παράτησε το control όσο περιμένω I/O, ώστε να τρέξει άλλη coroutine». Δεν μπλοκάρει τον thread.
- `asyncio.gather(...)` τρέχει πολλά awaitables **ταυτόχρονα** — αν τα κάνεις `await` ένα-ένα σε loop, γίνονται σειριακά (κλασικό λάθος).
- **Κανόνας μη-ανάμειξης**: μέσα σε `async` κώδικα μη βάζεις blocking κλήσεις (`time.sleep`, sync `requests`, βαρύ CPU loop) — παγώνουν όλο το event loop. Χρησιμοποίησε `asyncio.sleep`, async clients, ή πέτα τη CPU δουλειά σε executor/process.

## Background jobs — queue + worker

Όταν η δουλειά είναι αργή ή βαριά, δεν την κάνεις μέσα στο request. Τη βάζεις σε **queue** και ένας ξεχωριστός **worker process** την επεξεργάζεται:

```txt
FastAPI endpoint ──push job──> Queue (Redis/RabbitMQ) ──> Worker process(es)
       │                                                        │
       └── επιστρέφει job_id αμέσως          γράφει result/status στη βάση
```

Με Celery (το πιο διαδεδομένο):

```py
# tasks.py
from celery import Celery

celery = Celery("jobs", broker="redis://localhost:6379/0")

@celery.task(bind=True, max_retries=3)
def process_document(self, doc_id: str):
    try:
        # βαριά δουλειά: OCR, ML, parsing...
        result = heavy_processing(doc_id)
        save_result(doc_id, result)
    except TransientError as exc:
        # exponential backoff retry
        raise self.retry(exc=exc, countdown=2 ** self.request.retries)
```

```py
# στο FastAPI endpoint
@app.post("/documents/{doc_id}/process")
def enqueue(doc_id: str):
    task = process_document.delay(doc_id)   # μπαίνει στην queue, δεν τρέχει εδώ
    return {"job_id": task.id, "status": "queued"}
```

Τα production patterns που πρέπει να αναφέρεις:
- **Idempotency**: ένα job μπορεί να ξανατρέξει (retry/duplicate) — γράψ' το ώστε να μην προκαλεί διπλά side effects (π.χ. έλεγχος «έχει ήδη γίνει;»).
- **Retries με backoff**: παροδικά σφάλματα (δίκτυο, rate limit) ξαναδοκιμάζονται με αυξανόμενη καθυστέρηση· μόνιμα σφάλματα πάνε σε **dead-letter queue**.
- **Status flow**: `queued → processing → done/failed`, αποθηκευμένο ώστε ο client να κάνει polling ή να δεχτεί webhook.
- **Timeouts**: κάθε job έχει όριο χρόνου, αλλιώς ένα κολλημένο job κρατά worker.

## Πώς απαντάς στη συνέντευξη

«Python has the GIL — how do you run heavy work without blocking your API?»

> It depends on the workload. For I/O-bound work — network, database, external APIs — the GIL is released during the wait, so `asyncio` (or threads) gives me real concurrency on a single core; I use `asyncio.gather` to fan out instead of awaiting in a loop. For CPU-bound work like ML inference the GIL blocks parallelism in threads, so I use multiple processes. In practice, anything slow or heavy goes off the request path entirely: I push a job to a queue like Celery on Redis, return a job id immediately, and a separate worker processes it. I make the job idempotent, add retries with backoff and a dead-letter queue, and expose a status endpoint for polling.
