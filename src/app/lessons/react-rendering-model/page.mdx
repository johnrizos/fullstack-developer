# Rendering & Reconciliation

Πριν μάθεις hooks, πρέπει να καταλάβεις **πώς το React αποφασίζει τι να ζωγραφίσει στην οθόνη**. Σχεδόν κάθε «παράξενο» React bug — stale τιμές, περιττά re-renders, χαμένο state σε λίστες — εξηγείται από αυτό το μοντέλο. Στη συνέντευξη: «what triggers a re-render;», «what is reconciliation;», «why not index as key;».

## Τι πρέπει να μπορείς στο τέλος

- να εξηγείς το `UI = f(state)` και τι σημαίνει «render»,
- να ξέρεις τι προκαλεί re-render και τι όχι,
- να περιγράφεις reconciliation και το virtual DOM diffing,
- να εξηγείς γιατί τα keys πρέπει να είναι σταθερά και μοναδικά,
- να αναγνωρίζεις γιατί το reference equality οδηγεί σε immutable updates.

## UI = f(state)

Η κεντρική ιδέα: το UI είναι **καθαρή συνάρτηση** του state. Δεν πειράζεις το DOM με το χέρι· αλλάζεις state και το React ξανατρέχει το component για να παράγει την περιγραφή του UI.

```tsx
function Counter() {
  const [count, setCount] = useState(0);
  // Αυτή η function ΞΑΝΑΤΡΕΧΕΙ ολόκληρη σε κάθε αλλαγή του count.
  return <button onClick={() => setCount(count + 1)}>Count: {count}</button>;
}
```

«Render» = το React καλεί τη function του component και παίρνει πίσω JSX (μια περιγραφή, όχι DOM). «Commit» = το React εφαρμόζει τις διαφορές στο πραγματικό DOM. Render συμβαίνει συχνά· commit μόνο όταν κάτι όντως άλλαξε.

## Τι προκαλεί re-render

Μόνο τρία πράγματα:

```txt
1. Αλλαγή του δικού του state         (setState με ΝΕΑ τιμή)
2. Re-render του parent              (το παιδί ξαναγίνεται render by default)
3. Αλλαγή του context που καταναλώνει
```

Αυτό που **δεν** προκαλεί re-render: αλλαγή props χωρίς re-render του parent (αδύνατο by default), mutation αντικειμένου, ή αλλαγή μεταβλητής εκτός state. Το «το parent ξανα-renderάρει → τα παιδιά ξανα-renderάρουν» είναι ο νούμερο ένα λόγος για περιττά renders.

## Κάθε render είναι ένα στιγμιότυπο

Κάθε φορά που τρέχει το component, οι μεταβλητές και οι functions του **ξαναδημιουργούνται**. Το `count` ενός render είναι μια σταθερή τιμή για εκείνο το render — δεν «ενημερώνεται» αργότερα.

## Reconciliation & virtual DOM

Στο re-render, το React δεν αγγίζει αμέσως το DOM. Παράγει ένα νέο δέντρο (virtual DOM) και το **συγκρίνει** με το προηγούμενο (diffing). Εφαρμόζει στο πραγματικό DOM μόνο τις διαφορές. Αυτό λέγεται **reconciliation**.

Δύο ευρετικοί κανόνες κάνουν το diffing γρήγορο:

- **Διαφορετικός τύπος element → πετάει όλο το subtree.** `<div>` → `<section>` σημαίνει unmount/remount, όχι patch.
- **Λίστες ταιριάζονται με βάση το `key`.** Με σταθερό key, το React ξέρει ποιο στοιχείο μετακινήθηκε αντί να τα ξαναχτίσει.

## Γιατί όχι index ως key

```tsx
// ❌ index ως key
{todos.map((todo, i) => <TodoRow key={i} todo={todo} />)}

// ✅ σταθερό id
{todos.map((todo) => <TodoRow key={todo.id} todo={todo} />)}
```

Αν διαγράψεις το πρώτο στοιχείο, όλα τα indexes μετατοπίζονται: το key=`0` τώρα δείχνει σε διαφορετικό todo. Το React νομίζει ότι «το στοιχείο 0 απλώς άλλαξε περιεχόμενο» και κρατά το παλιό εσωτερικό state (π.χ. το κείμενο ενός input ή ένα checked checkbox) στο **λάθος** row. Με σταθερό `id`, κάθε row κρατά την ταυτότητά του.

## Πώς απαντάς στη συνέντευξη

«What happens when state changes in React?»

> React calls the component function again to produce a new tree of elements — that's a render, and it's just a description, not DOM. Then it reconciles: it diffs the new virtual tree against the previous one and commits only the differences to the real DOM. A component re-renders when its own state changes, when its parent re-renders, or when a context it consumes changes. Because the diff relies on reference equality and on `key` for lists, I always update state immutably and use stable ids as keys — mutating an object leaves the same reference, so React can't see the change, and index keys make React attach the wrong state to the wrong row after a reorder.
