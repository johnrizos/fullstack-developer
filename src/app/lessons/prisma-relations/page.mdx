# Relations & Nested Writes

Το 80% της δουλειάς σε ένα data layer είναι relations: users έχουν posts, posts έχουν tags, orders έχουν items. Στη συνέντευξη: «πώς μοντελοποιείς m-n;», «include vs select;», «τι είναι nested write;». Το Prisma τα κάνει type-safe — αρκεί να καταλάβεις το schema μοντέλο.

## Τι πρέπει να μπορείς στο τέλος

- να μοντελοποιείς 1-1, 1-n, m-n relations στο schema,
- να κάνεις nested create/update σε μία κλήση,
- να διαλέγεις σωστά ανάμεσα σε `include` και `select`,
- να φιλτράρεις με βάση related records.

## Το relation μοντέλο

Κάθε relation έχει δύο πλευρές: το **relation field** (virtual, για navigation) και το **scalar foreign key** (το πραγματικό column).

```prisma
model User {
  id    String @id @default(cuid())
  posts Post[]                          // 1-n: relation field (όχι column)
}

model Post {
  id       String @id @default(cuid())
  title    String
  author   User   @relation(fields: [authorId], references: [id])
  authorId String                        // το πραγματικό foreign key column
}
```

Το `posts Post[]` δεν υπάρχει στη βάση — είναι navigation. Το `authorId` είναι το column. Αυτό μπερδεύει αρχικά: **το relation field είναι για το Prisma, το scalar FK για τη βάση**.

## Many-to-many — implicit vs explicit

```prisma
// IMPLICIT: το Prisma φτιάχνει μόνο του τον join table
model Post { id String @id; tags Tag[] }
model Tag  { id String @id; posts Post[] }

// EXPLICIT: όταν θες extra πεδία στο join (π.χ. πότε προστέθηκε)
model PostTag {
  post     Post @relation(fields: [postId], references: [id])
  postId   String
  tag      Tag  @relation(fields: [tagId], references: [id])
  tagId    String
  addedAt  DateTime @default(now())
  @@id([postId, tagId])
}
```

Implicit όταν το join είναι «καθαρό»· explicit όταν χρειάζεσαι metadata πάνω στη σχέση.

## Nested writes — μία κλήση, πολλά records

Αντί για χωριστά inserts + manual FK wiring, το Prisma φτιάχνει parent + children μαζί:

```ts
const post = await prisma.post.create({
  data: {
    title: "Hello",
    author: { connect: { id: userId } },     // σύνδεσε σε υπάρχοντα user
    tags: {
      connectOrCreate: [                       // φτιάξε αν δεν υπάρχει, αλλιώς σύνδεσε
        { where: { name: "prisma" }, create: { name: "prisma" } },
      ],
    },
  },
  include: { tags: true },
});
```

- `connect` → σύνδεση σε υπάρχον record (έχεις το id).
- `create` → φτιάξε νέο child μαζί με τον parent.
- `connectOrCreate` → idempotent: σύνδεσε αν υπάρχει, φτιάξε αν όχι.

Όλα τρέχουν σε **μία transaction** — ή όλα ή τίποτα.

## include vs select — το βασικό

```txt
include → ΟΛΑ τα scalar fields + τις σχέσεις που προσθέτεις
select  → ΜΟΝΟ τα fields/σχέσεις που δηλώνεις ρητά (allowlist)
```

```ts
// include: όλα τα post fields + author
prisma.post.findMany({ include: { author: true } });

// select: μόνο τα απαραίτητα (λιγότερα bytes, καμία διαρροή)
prisma.post.findMany({ select: { id: true, title: true } });
```

Προτίμησε `select` σε APIs: επιστρέφεις μόνο ό,τι χρειάζεται ο client (performance + δεν διαρρέεις π.χ. `passwordHash`). Το `include` είναι βολικό αλλά φέρνει τα πάντα.

## Filtering με βάση relations

```ts
// users που έχουν τουλάχιστον ένα published post
prisma.user.findMany({
  where: { posts: { some: { published: true } } },
});
// posts των οποίων ο author είναι admin
prisma.post.findMany({
  where: { author: { role: "admin" } },
});
```

`some` / `every` / `none` για 1-n· nested object για 1-1/n-1. Το φιλτράρισμα γίνεται στη βάση, όχι στη μνήμη.

## Πώς απαντάς στη συνέντευξη

«How do you model and query relations in Prisma?»

> Each relation has two sides in the schema — a relation field for navigation, like `posts Post[]`, which isn't a real column, and a scalar foreign key like `authorId`, which is. For many-to-many I use an implicit relation and let Prisma manage the join table, switching to an explicit join model when I need metadata on the relationship like a timestamp. I create related records with nested writes in a single call — `connect` for existing ids, `create` for new children, and `connectOrCreate` for idempotent tag-style links — all in one transaction. For reads I prefer `select` over `include` in APIs because select is an allowlist returning only the fields the client needs, which is faster and avoids leaking things like a password hash, whereas include returns every scalar field. And I filter on relations in the database with `some`/`every`/`none`, never in memory.
