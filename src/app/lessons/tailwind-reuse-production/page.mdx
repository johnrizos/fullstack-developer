# Components, Reuse & Production

Η νο1 ένσταση στο Tailwind είναι «γεμίζει classes το markup και τις κάνω copy-paste». Αυτό το μάθημα δείχνει τη σωστή λύση (reuse με components), πότε χρησιμοποιείς `@apply`, πώς βγαίνει μικρό production bundle, και πότε *δεν* είναι το Tailwind η σωστή επιλογή.

## Reuse: components, ΟΧΙ copy-paste

Το πρόβλημα δεν είναι το Tailwind — είναι το copy-paste. Όταν ένα κουμπί επαναλαμβάνεται, γράφεις το utility string **μία** φορά σε ένα component:

```jsx
function Button({ children }) {
  return (
    <button className="px-4 py-2 bg-brand text-white rounded-lg hover:bg-blue-700">
      {children}
    </button>
  );
}

// Χρήση παντού — μηδέν επανάληψη
<Button>Αποθήκευση</Button>
```

Σε component-based frameworks (React/Vue/Svelte), αυτή είναι **η** απάντηση: το «φλύαρο» markup ζει σε ένα σημείο, και το χρησιμοποιείς ως καθαρό `<Button>`.

## @apply — και πότε ΟΧΙ

Το `@apply` ομαδοποιεί utilities σε μια κανονική CSS class:

```css
.btn { @apply px-4 py-2 bg-brand text-white rounded-lg hover:bg-blue-700; }
```

Χρήσιμο σε **non-component** περιβάλλοντα (π.χ. plain HTML, server-rendered templates χωρίς components) ή για μερικά global elements. **Αλλά με μέτρο:** αν βάζεις τα πάντα σε `@apply`, ξαναφτιάχνεις «κανονικό CSS με ονόματα» — ακριβώς αυτό που το Tailwind ήρθε να αποφύγει. Σε React, σχεδόν πάντα προτίμησε component αντί για `@apply`.

```txt
Επανάληψη σε React;        → component (το <Button>)
Plain HTML / templates;     → @apply, με μέτρο
Όλα μέσα σε @apply;         → anti-pattern (γύρισες στο "κανονικό CSS")
```

## Production: γιατί το bundle είναι μικρό

Στο build, το JIT engine σκανάρει το `content` (τα αρχεία σου) και κρατά **μόνο** τις classes που εμφανίζονται. Μια app με χιλιάδες διαθέσιμες utilities βγάζει συχνά λίγα KB CSS, γιατί χρησιμοποιείς μικρό υποσύνολο.

**Η μεγάλη παγίδα:** classes που φτιάχνονται **δυναμικά** ως strings δεν εντοπίζονται από τον scanner:

```jsx
// ❌ Ο scanner ΔΕΝ βλέπει το "bg-red-600" — δεν υπάρχει αυτούσιο στον κώδικα
<div className={`bg-${color}-600`} />

// ✅ Πλήρη ονόματα ώστε να εντοπιστούν
<div className={color === "red" ? "bg-red-600" : "bg-blue-600"} />
```

Αν πρέπει να είναι δυναμικό, χρησιμοποίησε **safelist** στο config ή πλήρη class names σε map.

## Πότε ΔΕΝ είναι ιδανικό το Tailwind

- Όταν θες **γρήγορα έτοιμα interactive components** (navbar, modal, dropdown) χωρίς custom design — εκεί ταιριάζει component framework (Bootstrap). Το Tailwind δίνει utilities, όχι έτοιμα components (μπορείς να φέρεις Headless UI/Radix για τη συμπεριφορά).
- Όταν η ομάδα προτιμά **κανονικό CSS** με scope — CSS Modules.
- Για πολύ μικρά/στατικά projects όπου το setup δεν αξίζει.

## Πώς απαντάς στη συνέντευξη

> «Doesn't Tailwind make the markup messy?»

> Only if you copy-paste. In a component framework the fix is a component: the utility string lives once in a `<Button>` and I use it cleanly everywhere — the verbose markup is centralised, not repeated. `@apply` exists for non-component contexts like plain templates, but I use it sparingly, because applying everything recreates the named-CSS problem Tailwind avoids. In production the JIT build scans my files and emits only the classes I actually use, so the bundle stays small — the one gotcha is dynamic class strings like `bg-${x}-500`, which the scanner can't see, so I use full class names or a safelist. And when I just need ready-made interactive components fast, I'd reach for Bootstrap instead.
