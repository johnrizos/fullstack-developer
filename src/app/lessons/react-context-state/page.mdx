# Context & State Management

Καθώς μεγαλώνει η εφαρμογή, το «πέρνα props πέντε επίπεδα κάτω» γίνεται πόνος (prop drilling). Το Context το λύνει — αλλά έχει παγίδες performance. Στα interviews: «πότε Context;», «ποιο είναι το re-render trap;», «πότε χρειάζεσαι external store;».

## Τι πρέπει να μπορείς στο τέλος

- να αναγνωρίζεις prop drilling και πότε αξίζει Context,
- να φτιάχνεις typed Context με provider,
- να εξηγείς γιατί το Context μπορεί να προκαλέσει περιττά re-renders,
- να ξεχωρίζεις «server state» από «UI/client state»,
- να ξέρεις πότε φεύγεις σε Zustand/Redux.

## Prop drilling — το πρόβλημα

```tsx
// theme περνά μέσα από components που δεν το χρειάζονται
<App theme={theme}>
  <Layout theme={theme}>
    <Sidebar theme={theme}>
      <UserMenu theme={theme} /> {/* μόνο εδώ χρειάζεται */}
```

Όταν ένα δεδομένο χρειάζεται «παντού» (theme, locale, current user), το prop drilling το κάνει θόρυβο. Το Context σου επιτρέπει να το διαβάσεις απευθείας όπου χρειάζεται.

## Typed Context + provider

```tsx
type Theme = "light" | "dark";
type ThemeContextValue = { theme: Theme; toggle: () => void };

const ThemeContext = createContext<ThemeContextValue | null>(null);

export function ThemeProvider({ children }: { children: React.ReactNode }) {
  const [theme, setTheme] = useState<Theme>("light");
  const value = useMemo(
    () => ({ theme, toggle: () => setTheme((t) => (t === "light" ? "dark" : "light")) }),
    [theme],
  );
  return <ThemeContext.Provider value={value}>{children}</ThemeContext.Provider>;
}

export function useTheme() {
  const ctx = useContext(ThemeContext);
  if (!ctx) throw new Error("useTheme must be used inside ThemeProvider");
  return ctx;
}
```

Το custom `useTheme` hook με το null check είναι το production pattern: type-safe και ρίχνει καθαρό error αν ξεχάσεις τον provider.

## Server state ≠ client state

Το πιο σημαντικό mental shift. Δεν είναι όλα ίδιο «state»:

```txt
Client/UI state   → modal open, selected tab, form input, theme
                    → useState / useReducer / Context / Zustand

Server state      → users, products, posts (ζουν σε database)
                    → React Query / SWR (caching, refetch, dedup)
```

Συχνό λάθος: να βάζεις server data σε global store και να γράφεις μόνος σου caching, loading, refetch. Αυτό το λύνουν εργαλεία server-state (επόμενο μάθημα). Το Context/Redux είναι για **client** state.

## Πότε external store (Zustand/Redux)

```txt
Context αρκεί:    λίγα global values, σπάνιες αλλαγές (theme, user, locale)
Zustand/Redux:    πολύ shared client state, συχνές αλλαγές, χρειάζεσαι
                  selectors (re-render μόνο όσοι διαβάζουν το συγκεκριμένο slice),
                  middleware, devtools, predictable updates
```

Μην ξεκινάς από Redux. Ξεκίνα τοπικό state → lift up → Context → external store μόνο όταν το Context πονάει (performance, πολυπλοκότητα). Τα modern stores (Zustand) δίνουν selectors που αποφεύγουν το re-render trap του Context.

## Πώς απαντάς στη συνέντευξη

«When do you use Context, and what's the catch?»

> Context solves prop drilling for values that many components need — theme, locale, current user. I type it and expose a custom hook that throws if used outside the provider. The catch is performance: every consumer re-renders when the context value changes, and a classic bug is passing a fresh object as `value` on every render, which makes all consumers re-render even when the data didn't change — I fix that with `useMemo` on the value, or by splitting frequently-changing data into a separate context. The bigger distinction I make is server state versus client state: data that lives in a database belongs in a server-state library like React Query, which gives me caching and refetching, while Context and stores like Zustand or Redux are for client state. I only reach for an external store when shared client state gets large or changes often and I need selectors to avoid Context's re-render problem.
