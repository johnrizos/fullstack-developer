# Queries, Filtering & Pagination

Listing endpoints είναι παντού: αναζήτηση, φίλτρα, ταξινόμηση, σελιδοποίηση. Στη συνέντευξη: «cursor vs offset pagination;», «πώς κάνεις aggregation;», «πότε raw query;». Το Prisma δίνει type-safe API για όλα — με μερικές παγίδες performance.

## Τι πρέπει να μπορείς στο τέλος

- να φιλτράρεις με `where` (operators, AND/OR, search),
- να ταξινομείς και να σελιδοποιείς (offset & cursor),
- να κάνεις aggregations και `groupBy`,
- να ξέρεις πότε κατεβαίνεις σε raw SQL.

## Filtering με where

```ts
prisma.product.findMany({
  where: {
    price: { gte: 10, lte: 100 },          // range
    name: { contains: "phone", mode: "insensitive" },  // case-insensitive search
    category: { in: ["tech", "audio"] },   // IN list
    AND: [{ stock: { gt: 0 } }, { active: true }],
    OR: [{ featured: true }, { discount: { gt: 0 } }],
    NOT: { archived: true },
  },
});
```

Operators: `equals, not, in, notIn, lt, lte, gt, gte, contains, startsWith, endsWith`. Συνδυάζονται με `AND`/`OR`/`NOT`. Όλα μεταφράζονται σε SQL — το φιλτράρισμα γίνεται στη **βάση**.

## Sorting & offset pagination

```ts
prisma.product.findMany({
  orderBy: [{ featured: "desc" }, { createdAt: "desc" }],  // multi-column
  skip: (page - 1) * pageSize,    // offset
  take: pageSize,                  // limit
});
```

Το offset pagination (`skip`/`take`) είναι απλό αλλά **αργό σε βαθιές σελίδες**: `skip: 100000` αναγκάζει τη βάση να διασχίσει 100k rows πριν επιστρέψει. Καλό για μικρά datasets / page numbers.

## Cursor pagination — για scale

```ts
prisma.product.findMany({
  take: pageSize,
  cursor: lastId ? { id: lastId } : undefined,  // ξεκίνα μετά το τελευταίο
  skip: lastId ? 1 : 0,                          // skip το ίδιο το cursor
  orderBy: { id: "asc" },
});
```

Το cursor pagination χρησιμοποιεί το `id` (ή timestamp) του τελευταίου record αντί για offset. **Σταθερή ταχύτητα** ανεξάρτητα από βάθος (χρησιμοποιεί index seek, όχι scan) — ιδανικό για infinite scroll / μεγάλα datasets. Trade-off: δεν πας σε αυθαίρετη «σελίδα 50».

## Aggregations & groupBy

```ts
// συγκεντρωτικά
const stats = await prisma.order.aggregate({
  _count: true,
  _sum: { total: true },
  _avg: { total: true },
  where: { status: "paid" },
});

// ανά κατηγορία
const byCategory = await prisma.product.groupBy({
  by: ["category"],
  _count: { _all: true },
  _avg: { price: true },
  having: { price: { _avg: { gt: 50 } } },   // filter σε aggregate
  orderBy: { _count: { category: "desc" } },
});
```

`aggregate` για συνολικά νούμερα, `groupBy` για ανά-ομάδα. Το `having` φιλτράρει **μετά** το grouping (όπως SQL HAVING).

## Count + data μαζί

```ts
const [items, total] = await prisma.$transaction([
  prisma.product.findMany({ where, skip, take }),
  prisma.product.count({ where }),           // ίδιο where!
]);
// response: { items, total, page, pageSize }
```

Για paginated response χρειάζεσαι και τα δεδομένα και το συνολικό count — σε ένα `$transaction` ώστε να είναι consistent.

## Πότε raw SQL

```ts
const rows = await prisma.$queryRaw`
  SELECT category, COUNT(*) FROM "Product"
  WHERE price > ${minPrice} GROUP BY category
`;  // parameterized — ΟΧΙ string concatenation (SQL injection)
```

Κατεβαίνεις σε raw όταν: πολύπλοκο SQL που δεν εκφράζεται με το API (window functions, CTEs), ή performance tuning. **Πάντα** tagged template (`$queryRaw\`...\``) με interpolation — ποτέ string concatenation, που ανοίγει SQL injection.

## Πώς απαντάς στη συνέντευξη

«How do you build a listing endpoint with Prisma?»

> I filter in the database with `where` — range operators, case-insensitive `contains` for search, and `AND`/`OR`/`NOT` to combine — never in memory. I sort with `orderBy`, supporting multiple columns. For pagination I default to offset `skip`/`take` for small datasets and page numbers, but switch to cursor pagination for large datasets or infinite scroll, because offset forces the database to scan all skipped rows and gets slower the deeper you go, while a cursor does an index seek from the last id at constant time. I return both the page and a total count from a single `$transaction` with the same where clause so they stay consistent. For dashboards I use `aggregate` and `groupBy` with `having`. I only drop to `$queryRaw` for SQL the API can't express, always as a tagged template so values are parameterized and there's no injection.
