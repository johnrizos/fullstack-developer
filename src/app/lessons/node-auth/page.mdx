# Authentication & Sessions

Το auth είναι το πιο συχνό «πες μου πώς θα το έχτιζες» στη συνέντευξη backend. Δύο βασικά μοντέλα: **sessions** (state στον server) και **JWT** (stateless token). Πρέπει να ξέρεις πότε διαλέγεις ποιο, πού μπαίνει το token, και πώς προστατεύεις routes χωρίς να γράφεις το ίδιο `if (!user)` παντού.

## Τι πρέπει να μπορείς στο τέλος

- να κάνεις hash κωδικούς με bcrypt (ποτέ plaintext),
- να εξηγείς **session vs JWT** και πότε ταιριάζει το καθένα,
- να στήνεις login που εκδίδει token/session,
- να προστατεύεις routes με auth middleware,
- να κάνεις access + refresh token flow σωστά.

## Authentication vs authorization

- **Authentication**: *ποιος είσαι;* — login, επαλήθευση credentials.
- **Authorization**: *τι επιτρέπεσαι;* — roles, ownership, permissions.

Δύο διαφορετικά πράγματα: μπορείς να είσαι authenticated (ξέρω ποιος είσαι) αλλά **όχι** authorized (δεν επιτρέπεται να σβήσεις αυτό το resource → `403`).

## Password hashing — bcrypt

```ts
import bcrypt from "bcryptjs";

// Στο register: hash πριν αποθηκεύσεις
const passwordHash = await bcrypt.hash(plainPassword, 12);  // 12 = cost factor

// Στο login: compare χωρίς ποτέ να κάνεις decrypt
const ok = await bcrypt.compare(plainPassword, user.passwordHash);
```

Το bcrypt είναι **one-way**: δεν «ξεκλειδώνεις» τον κωδικό, κάνεις compare. Το cost factor (10–12) κάνει το brute-force ακριβό. **Ποτέ** plaintext, ποτέ `md5`/`sha256` σκέτο — είναι πολύ γρήγορα για passwords.

## Session vs JWT — το βασικό trade-off

```txt
SESSION (stateful)                  JWT (stateless)
─────────────────────               ─────────────────────
server κρατά session store          server δεν κρατά τίποτα
cookie με session id                token με signed payload
revoke = σβήσε από store ✅          revoke = δύσκολο (μέχρι expiry) ⚠️
scaling χρειάζεται shared store      scaling εύκολο (κανένα state)
ιδανικό: classic web app            ιδανικό: APIs, mobile, microservices
```

Ο vuln-killer κανόνας: το **JWT δεν ανακαλείται εύκολα**. Αν κλαπεί, ισχύει μέχρι να λήξει. Γι' αυτό τα access tokens είναι **βραχύβια** (π.χ. 15 λεπτά) και βασίζεσαι σε refresh token για ανανέωση.

## JWT login flow

```ts
import jwt from "jsonwebtoken";

app.post("/login", async (req, res) => {
  const user = await userService.findByEmail(req.body.email);
  if (!user || !(await bcrypt.compare(req.body.password, user.passwordHash))) {
    return res.status(401).json({ error: "Invalid credentials" });  // ίδιο μήνυμα και για τα δύο!
  }

  const accessToken = jwt.sign({ sub: user.id, role: user.role }, process.env.JWT_SECRET!, {
    expiresIn: "15m",
  });
  res.json({ accessToken });
});
```

Πρόσεξε: **ίδιο error message** για «δεν υπάρχει user» και «λάθος password». Αν δώσεις διαφορετικό, κάνεις user enumeration — ο attacker μαθαίνει ποια emails υπάρχουν.

## Auth middleware — προστασία routes

Αντί να γράφεις τον έλεγχο σε κάθε handler, τον βάζεις σε middleware:

```ts
function requireAuth(req, res, next) {
  const header = req.headers.authorization;          // "Bearer <token>"
  const token = header?.startsWith("Bearer ") ? header.slice(7) : null;
  if (!token) return res.status(401).json({ error: "Missing token" });

  try {
    req.user = jwt.verify(token, process.env.JWT_SECRET!);  // throws αν invalid/expired
    next();
  } catch {
    return res.status(401).json({ error: "Invalid or expired token" });
  }
}

app.get("/me", requireAuth, (req, res) => res.json({ user: req.user }));
```

Το middleware βάζει το `req.user` και ο handler το χρησιμοποιεί καθαρά. Για authorization προσθέτεις δεύτερο middleware:

```ts
const requireRole = (role) => (req, res, next) =>
  req.user.role === role ? next() : res.status(403).json({ error: "Forbidden" });

app.delete("/users/:id", requireAuth, requireRole("admin"), deleteUser);
```

## Access + refresh tokens

```txt
access token   → 15 λεπτά, στέλνεται σε κάθε request (Authorization header)
refresh token  → 7 ημέρες, httpOnly cookie, ΜΟΝΟ στο /refresh endpoint

Λήξη access → POST /refresh με το refresh cookie → νέο access token
Logout → σβήσε/blacklist το refresh token
```

Το refresh token πάει σε **httpOnly cookie** (δεν το διαβάζει JavaScript → προστασία από XSS). Το access token μένει βραχύβιο ώστε, αν κλαπεί, να λήγει γρήγορα.

## Πώς απαντάς στη συνέντευξη

«How would you implement auth for a REST API?»

> I hash passwords with bcrypt at a cost factor around 12 — never plaintext, and compare on login without ever decrypting. On a successful login I issue a short-lived JWT access token, 15 minutes, carrying only the user id and role in the payload, since the payload is signed but not encrypted so it's readable. The client sends it as a Bearer token and an auth middleware verifies it and sets `req.user`, so handlers stay clean; a second middleware checks roles for authorization and returns 403 when it's not allowed. For longer sessions I add a refresh token in an httpOnly cookie hitting only a `/refresh` endpoint, which keeps the access token short so a stolen one expires fast. Login failures always return the same 401 to avoid user enumeration. If I need instant revocation I'd lean toward server sessions instead, since a plain JWT can't be revoked before it expires.
