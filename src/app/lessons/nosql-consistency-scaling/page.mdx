# Consistency, Replication & Scaling

Τα distributed NoSQL stores κλιμακώνουν οριζόντια σε πολλά nodes — αλλά αυτό φέρνει το θεμελιώδες πρόβλημα των κατανεμημένων συστημάτων: **consistency**. Στα interviews εξετάζουν αν καταλαβαίνεις eventual consistency, quorum, και τα trade-offs που το CAP κάνει υποχρεωτικά.

## Τι πρέπει να μπορείς στο τέλος

- να εξηγείς eventual consistency και πότε είναι αποδεκτή,
- να υπολογίζεις quorum (R + W > N),
- να εξηγείς replication και sharding/partitioning,
- να χειρίζεσαι conflicts (last-write-wins, vector clocks),
- να ξέρεις το PACELC (επέκταση του CAP).

## Replication — αντίγραφα σε πολλά nodes

Κάθε δεδομένο αποθηκεύεται σε **N replicas** (π.χ. N=3) σε διαφορετικά nodes — για ανθεκτικότητα και διαθεσιμότητα:

```txt
        write → [node A] [node B] [node C]   (3 replicas του ίδιου δεδομένου)
        read  ← οποιοδήποτε replica
```

Αν ένα node πέσει, τα δεδομένα υπάρχουν αλλού. Αλλά: πώς εξασφαλίζεις ότι όλα τα replicas συμφωνούν;

## Eventual consistency

Σε masterless/AP systems, ένα write δεν περιμένει να ενημερωθούν **όλα** τα replicas. Επιστρέφει νωρίς, και τα replicas συγκλίνουν **με τον χρόνο**:

```txt
t=0:   write στο node A → επιστρέφει "ok"
t=0:   read από node C → ΑΚΟΜΑ η παλιά τιμή (δεν προλαβε να συγχρονιστεί)
t=50ms: όλα τα replicas συγχρονισμένα → read δίνει τη νέα τιμή
```

Αποδεκτό για likes, feeds, view counts. **Μη** αποδεκτό για bank balance. Το «read your own writes» (να βλέπει ο χρήστης τις δικές του αλλαγές) χρειάζεται ειδικό χειρισμό (sticky reads ή υψηλότερο consistency level).

## Quorum: R + W > N

Ο μηχανισμός για **ρυθμιζόμενη** consistency χωρίς να θυσιάσεις όλη την ταχύτητα:

```txt
N = πλήθος replicas        (π.χ. 3)
W = replicas που πρέπει να επιβεβαιώσουν ένα WRITE
R = replicas που ρωτάς σε ένα READ

Αν R + W > N → εγγυημένα διαβάζεις την τελευταία τιμή
```

## Sharding / Partitioning — μοίρασμα δεδομένων

Replication = ΑΝΤΙΓΡΑΦΑ ίδιων δεδομένων. **Sharding** = ΜΟΙΡΑΣΜΑ διαφορετικών δεδομένων σε nodes, με βάση ένα shard/partition key:

```txt
users A-H → shard 1
users I-P → shard 2
users Q-Z → shard 3
```

Έτσι κάθε node κρατά μέρος των δεδομένων → οριζόντια κλίμακα. Ο κίνδυνος (όπως και στο SQL sharding): **κακό shard key → hot shards** (ανισοκατανομή). Τα distributed NoSQL το κάνουν αυτόματα, αλλά εσύ διαλέγεις το partition key.

## Conflict resolution

Σε masterless systems, δύο nodes μπορεί να δεχτούν concurrent writes στο ίδιο key. Πώς λύνεται;

```txt
Last-Write-Wins (LWW)  → κρατά το write με το πιο πρόσφατο timestamp
                          (απλό, αλλά χάνει δεδομένα σε true concurrency)
Vector clocks          → ανιχνεύουν causality· εντοπίζουν conflicts για να
                          τα λύσει ο app (π.χ. merge shopping carts)
CRDTs                  → Conflict-free Replicated Data Types: δομές που
                          συγχωνεύονται ντετερμινιστικά (counters, sets)
```

## PACELC — η επέκταση του CAP

Το CAP μιλά μόνο για partitions. Το **PACELC** συμπληρώνει το «τι γίνεται όταν ΔΕΝ υπάρχει partition»:

```txt
PAC  → σε Partition: Availability ή Consistency (το CAP)
ELC  → Else (κανονικά): Latency ή Consistency

π.χ. το DynamoDB είναι PA/EL: προτιμά availability & latency.
     ένα CP store προτιμά consistency και στις δύο περιπτώσεις.
```

Δείχνει ότι το trade-off consistency vs latency υπάρχει **πάντα**, όχι μόνο σε αποτυχίες.

## Πώς απαντάς στη συνέντευξη

«How does consistency work in a distributed NoSQL store?»

> Data is replicated to N nodes for durability and availability, and the question is how the replicas agree. Many NoSQL stores are eventually consistent: a write returns before all replicas are updated, and they converge in milliseconds — fine for likes or feeds, not for a bank balance. I tune it with quorums: with N replicas, if the read set and write set satisfy `R + W > N`, they overlap on at least one replica, so I read the latest write. That lets me pick strong or eventual per query. Replication copies the same data; sharding splits different data across nodes by a partition key, and a bad key causes hot shards just like in SQL. Concurrent writes are resolved with last-write-wins, vector clocks, or CRDTs depending on how much I can tolerate losing. And PACELC reminds me the consistency-versus-latency trade-off exists even without a partition, not just during failures.
