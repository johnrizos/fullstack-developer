# Prompt design & Structured outputs

Σε ένα LLM, το «πώς ρωτάς» αλλάζει δραματικά το αποτέλεσμα. Δύο πράγματα ξεχωρίζουν τον ερασιτέχνη από τον επαγγελματία: (1) ξέρει να δομεί το input με **system prompt** + σαφές user prompt, και (2) όταν χρειάζεται **μηχαναγνώσιμη** έξοδο, δεν κάνει parsing ελεύθερου κειμένου — ζητάει **structured output** με schema. Αυτό το μάθημα καλύπτει και τα δύο, γιατί μαζί κάνουν ένα AI feature αξιόπιστο αντί για «λοταρία».

## System prompt vs user prompt

- **System prompt** — οδηγίες ρόλου/συμπεριφοράς που ισχύουν για **όλη** τη συνομιλία: τόνος, μορφή, όρια, γλώσσα. Μπαίνει στο ξεχωριστό πεδίο `system`, όχι μέσα στα `messages`.
- **User prompt** — το συγκεκριμένο αίτημα. Πες τι θες, σε τι μορφή, με παραδείγματα όπου βοηθά.

```ts
const response = await client.messages.create({
  model: "claude-opus-4-8",
  max_tokens: 1024,
  system: "Είσαι βοηθός που απαντά πάντα στα Ελληνικά, σύντομα και χωρίς preamble.",
  messages: [{ role: "user", content: "Εξήγησε τι είναι το REST σε 2 προτάσεις." }],
});
```

Γιατί ξεχωριστό πεδίο και όχι «βάλ' το στο πρώτο message»; Γιατί το system prompt είναι **σταθερό** σε όλη τη συνομιλία και αντιμετωπίζεται ως οδηγία υψηλότερης προτεραιότητας — επίσης παίζει ρόλο στο prompt caching (το βλέπεις στο production lesson).

## Αρχές καλού prompt design

- **Να είσαι συγκεκριμένος** για τη μορφή: «3 bullet points», «μία πρόταση», «JSON με πεδία x, y».
- **Δώσε παραδείγματα** (few-shot) όταν το task είναι ασαφές — δείξε 1-2 ζευγάρια input→output.
- **Βάλε τα όρια ρητά**: «αν δεν ξέρεις, πες "δεν ξέρω"», «μην εφεύρεις δεδομένα».
- **Πες τον ρόλο**: «είσαι senior code reviewer» καλιμπράρει τον τόνο και το βάθος.
- **Πρόσφατα μοντέλα είναι κυριολεκτικά**: μην γράφεις επιθετικά «CRITICAL!!! ΠΡΕΠΕΙ!!!» — ακολουθούν το prompt πιστά, οπότε ήρεμες σαφείς οδηγίες δουλεύουν καλύτερα.

## Structured outputs: σταμάτα το parsing ελεύθερου κειμένου

Όταν θες να γεμίσεις ένα form, να αποθηκεύσεις σε DB, ή να τρέξεις λογική με βάση την απάντηση, χρειάζεσαι **εγγυημένα έγκυρο JSON** — όχι «συνήθως JSON». Αν κάνεις `JSON.parse()` σε ελεύθερο κείμενο, αργά ή γρήγορα θα σκάσει σε «Φυσικά! Ορίστε το JSON: \`\`\`json ...».

Η λύση: **structured output** με `output_config.format` και JSON schema. Το μοντέλο εγγυάται ότι η απάντηση ταιριάζει στο schema:

```ts
const response = await client.messages.create({
  model: "claude-opus-4-8",
  max_tokens: 1024,
  messages: [{ role: "user", content: "Βγάλε όνομα και email: 'Γιάννης, g@x.com'" }],
  output_config: {
    format: {
      type: "json_schema",
      schema: {
        type: "object",
        properties: {
          name: { type: "string" },
          email: { type: "string" },
        },
        required: ["name", "email"],
        additionalProperties: false,
      },
    },
  },
});
```

Πλεονεκτήματα έναντι του «ζήτα JSON στο prompt και κάνε parse»:

- **Εγγύηση εγκυρότητας** — η έξοδος ταιριάζει στο schema, χωρίς preamble ή markdown fences.
- **Λιγότερος defensive κώδικας** — δεν χρειάζεται try/catch γύρω από κάθε parse και regex καθαρισμού.
- **Σαφές contract** — το schema είναι η τεκμηρίωση του τι περιμένεις.

> Πότε ΟΧΙ structured output; Όταν θες ελεύθερο κείμενο για τον άνθρωπο (chat απάντηση, σύνοψη). Το schema είναι για **μηχαναγνώσιμα** δεδομένα που θα τα φάει κώδικας.

## Classification με enum

Ένα κλασικό use case: ταξινόμηση σε σταθερές κατηγορίες. Το `enum` στο schema εγγυάται ότι η απάντηση είναι μία από τις επιτρεπτές τιμές:

```ts
output_config: {
  format: {
    type: "json_schema",
    schema: {
      type: "object",
      properties: {
        sentiment: { type: "string", enum: ["positive", "negative", "neutral"] },
      },
      required: ["sentiment"],
      additionalProperties: false,
    },
  },
}
```

Έτσι δεν θα πάρεις ποτέ «μάλλον θετικό;» — μόνο `"positive"`, `"negative"` ή `"neutral"`.

## Πώς απαντάς στη συνέντευξη

> «How do you make an LLM's output reliable enough to use in code?»

> Two levers. First, prompt design: I set behavior, tone, format, and limits in the system prompt — which is separate from the messages and applies to the whole conversation — and keep the user prompt specific about what I want and in what shape, with a few-shot example when the task is ambiguous. Second, and more importantly for anything a program consumes: I don't parse free text. I use structured outputs with a JSON schema, so the response is guaranteed to match the shape — no preamble, no markdown fences, no `JSON.parse` blowing up on "Sure, here's the JSON:". For classification I put the allowed labels in an enum so I can never get an off-list value. Free-text prompting is for human-facing answers; schemas are for machine-readable data.
