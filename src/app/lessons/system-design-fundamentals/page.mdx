# System Design Fundamentals

Το system-design interview δεν ελέγχει αν ξέρεις ένα framework — ελέγχει αν μπορείς να σχεδιάσεις ένα σύστημα που **κλιμακώνεται, αντέχει σε βλάβες, και κάνει σωστά tradeoffs**. Αυτό το μάθημα δίνει το **framework** που ακολουθείς σε κάθε τέτοια ερώτηση, τα δομικά «τούβλα» ενός συστήματος, και τις θεμελιώδεις έννοιες (scaling, statelessness, single point of failure) πάνω στις οποίες χτίζουν τα επόμενα μαθήματα (scaling, caching, async, walkthrough).

## Το framework: πώς απαντάς σε κάθε system-design ερώτηση

Μη βουτάς κατευθείαν σε βάσεις και servers. Ακολούθησε βήματα — οι interviewers βαθμολογούν τη **διαδικασία**:

1. **Διευκρίνισε requirements** — functional (τι κάνει) + non-functional (πόσοι χρήστες, reads vs writes, latency, availability). Ρώτα, μη μαντεύεις.
2. **Estimation (back-of-envelope)** — τάξη μεγέθους: QPS (queries/sec), storage, bandwidth. Δείχνει ότι σκέφτεσαι την κλίμακα.
3. **API & data model** — βασικά endpoints + βασικές οντότητες/σχέσεις.
4. **High-level design** — σχεδίασε τα κουτιά: client → load balancer → app servers → cache → database (+ queue/CDN όπου χρειάζεται).
5. **Deep dive & bottlenecks** — πού «σπάει» στην κλίμακα; Σκάλωσε εκεί (caching, sharding, replication, async).
6. **Tradeoffs** — κάθε επιλογή έχει κόστος· δικαιολόγησέ την. Δεν υπάρχει «σωστή» αρχιτεκτονική, μόνο κατάλληλη.

## Functional vs non-functional requirements

- **Functional**: «ο χρήστης ανεβάζει φωτογραφία», «βλέπει feed». Το *τι*.
- **Non-functional**: scalability, availability, latency, consistency, durability. Το *πόσο καλά* — και εδώ κρίνεται το design. Πάντα ρώτα: **πόσοι χρήστες; read-heavy ή write-heavy; πόσο latency ανέχεται; πόσο κρίσιμο είναι το downtime;**

## Vertical vs horizontal scaling

Όταν ένας server δεν φτάνει:

- **Vertical (scale up)** — μεγαλύτερο μηχάνημα (CPU/RAM). Απλό, αλλά έχει **όριο** και είναι single point of failure.
- **Horizontal (scale out)** — **περισσότερα** μηχανήματα παράλληλα. Ουσιαστικά απεριόριστο, ανθεκτικό — αλλά απαιτεί load balancing και **stateless** services.

Το horizontal scaling είναι ο δρόμος για μεγάλη κλίμακα — γι' αυτό σχεδόν κάθε system-design συζήτηση το προϋποθέτει.

## Stateless services — η προϋπόθεση του scaling

Ένα **stateless** service δεν κρατά δεδομένα session στη μνήμη του — κάθε request μπορεί να πάει σε **οποιονδήποτε** server με το ίδιο αποτέλεσμα. Έτσι:

- Προσθέτεις/αφαιρείς servers ελεύθερα (auto-scaling).
- Ένας server που πέφτει δεν χάνει «κατάσταση» χρηστών.

Πού πάει το state τότε; Σε **shared store** έξω από τα app servers — database, ή cache (π.χ. Redis) για sessions. Κανόνας: **κράτα τα app servers stateless, βάλε το state σε shared layer**.

## Single point of failure (SPOF) & availability

Ένα **SPOF** είναι ένα component που, αν πέσει, ρίχνει όλο το σύστημα (ένας database server, ένας load balancer). Η availability («πόσο χρόνο είναι up», π.χ. 99.9%) χτίζεται με **redundancy**: πολλαπλά instances, replication, failover. Κάθε κρίσιμο component θέλει εφεδρεία.

## Latency vs throughput

- **Latency** — πόσο αργεί **ένα** request (ms).
- **Throughput** — πόσα requests/sec αντέχει το σύστημα συνολικά.

Δεν είναι το ίδιο: ένα σύστημα μπορεί να έχει υψηλό throughput αλλά κακή latency (π.χ. batching). Στο design ξεκαθάρισε ποιο βελτιστοποιείς.

## Πώς απαντάς στη συνέντευξη

> «How do you approach a system design question?»

> I follow a structured process because the interview grades the process. First I clarify requirements — functional, but especially non-functional: how many users, read-heavy or write-heavy, latency and availability targets. Then a quick back-of-the-envelope estimate of QPS and storage to anchor the scale. I sketch the API and data model, then a high-level design — client, load balancer, stateless app servers, cache, database, adding a queue or CDN where it helps. Then I look for bottlenecks at scale and go deep where it breaks. Throughout I state tradeoffs, because there's no single correct architecture, only an appropriate one. The two foundations I lean on: scale horizontally with stateless services so any request can hit any server, and remove single points of failure with redundancy.
