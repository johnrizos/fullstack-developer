# API Integrations

Πολλές δουλειές fullstack δεν είναι μόνο “φτιάχνω δικό μου API”. Είναι και “συνδέω τρίτα APIs”.

Παραδείγματα:

- payments,
- maps,
- email providers,
- AI APIs,
- CRMs,
- analytics.

## Auth headers

Πολλά APIs θέλουν token.

```ts
await fetch("https://api.example.com/data", {
  headers: {
    Authorization: `Bearer ${token}`
  }
});
```

Μην βάζεις secrets σε client-side code. Τα secrets πρέπει να μένουν server-side.

Σε Next.js app, ένα συνηθισμένο pattern είναι να καλείς το τρίτο API από server route ή server action και το frontend να καλεί το δικό σου endpoint.

```ts
export async function GET() {
  const response = await fetch("https://api.example.com/data", {
    headers: {
      Authorization: `Bearer ${process.env.EXTERNAL_API_KEY}`,
    },
  });

  if (!response.ok) {
    return Response.json({ error: "External API failed" }, { status: 502 });
  }

  const data = await response.json();
  return Response.json(data);
}
```

Έτσι το secret δεν μπαίνει ποτέ στο browser.

## Rate limits

Rate limit σημαίνει ότι το API επιτρέπει συγκεκριμένο αριθμό requests ανά χρονικό διάστημα.

Χρειάζεσαι:

- caching,
- debounce,
- retries με προσοχή,
- clear error messages.

Αν ένα search input καλεί API σε κάθε keypress, εύκολα χτυπάς rate limit. Γι' αυτό χρησιμοποιείς debounce:

```ts
let timeoutId: ReturnType<typeof setTimeout>;

function onSearchChange(value: string) {
  clearTimeout(timeoutId);
  timeoutId = setTimeout(() => {
    searchApi(value);
  }, 400);
}
```

## Retries

Δεν κάνουμε retry για όλα. Συνήθως έχει νόημα για προσωρινά failures, όχι για validation errors.

Καλή πρακτική:

- retry σε `429`, `500`, `502`, `503`, `504`,
- όχι retry σε `400`, `401`, `403`, `404`,
- μικρό backoff delay,
- όριο στις προσπάθειες.

```ts
async function fetchWithRetry(url: string, attempts = 3) {
  for (let attempt = 1; attempt <= attempts; attempt += 1) {
    const response = await fetch(url);

    if (response.ok) return response;
    if (![429, 500, 502, 503, 504].includes(response.status)) return response;
    if (attempt === attempts) return response;

    await new Promise((resolve) => setTimeout(resolve, attempt * 500));
  }
}
```

## Webhooks: όταν το API καλεί ΕΣΕΝΑ

Στα σοβαρά integrations (Stripe, GitHub, Twilio), δεν ρωτάς εσύ συνέχεια «έγινε η πληρωμή;» — ο provider σου στέλνει **webhook**: ένα POST στο δικό σου endpoint όταν συμβεί το event.

```ts
// app/api/webhooks/stripe/route.ts
export async function POST(request: Request) {
  const payload = await request.text();
  const signature = request.headers.get("stripe-signature");

  // 1. ΕΠΑΛΗΘΕΥΣΗ ΥΠΟΓΡΑΦΗΣ — αλλιώς οποιοσδήποτε POST-άρει "πληρωμές"
  let event;
  try {
    event = stripe.webhooks.constructEvent(payload, signature, process.env.WEBHOOK_SECRET);
  } catch {
    return Response.json({ error: "Invalid signature" }, { status: 400 });
  }

  // 2. ΙΔΕΜΠΟΤΕΝΤΟ ΧΕΙΡΙΣΜΟ — τα webhooks ξανάρχονται (retries)!
  const alreadyProcessed = await db.webhookEvent.findUnique({ where: { id: event.id } });
  if (alreadyProcessed) return Response.json({ received: true });

  // 3. Γρήγορο ACK — βαριά δουλειά σε queue, όχι στο handler
  await db.webhookEvent.create({ data: { id: event.id } });
  await queue.add("process-payment", event);

  return Response.json({ received: true }); // 200 γρήγορα, αλλιώς ο provider κάνει retry
}
```

Τα τρία σημεία που εξετάζονται: **signature verification** (το endpoint είναι δημόσιο — χωρίς επαλήθευση, ο καθένας «πληρώνει»), **idempotency** (οι providers κάνουν retry — το ίδιο event θα έρθει ξανά), και **γρήγορο acknowledgment** (αργό handler → timeout → περισσότερα retries → χιονοστιβάδα).

## Mini project: external API dashboard

Χτίσε ένα μικρό dashboard που καλεί τρίτο API μέσω server endpoint:

1. Το frontend δείχνει loading, success και error state.
2. Το API key μένει server-side.
3. Τα failed responses γυρίζουν καθαρό μήνυμα.
4. Υπάρχει retry για προσωρινά failures.
5. Τα logs δείχνουν status code και provider name, όχι secrets.

## Πώς απαντάς στη συνέντευξη

> For third-party integrations, the secret never reaches the browser — my frontend calls my own server endpoint, which holds the API key and talks to the provider. I add timeouts, retry only transient failures like 429 and 5xx with backoff, and respect rate limits with caching and debouncing. For incoming webhooks I verify signatures, handle events idempotently because providers retry, and acknowledge fast by pushing heavy work to a queue. And I log status codes and provider names — never payloads with secrets.
