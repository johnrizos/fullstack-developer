# Scope, Hoisting & Closures

Αυτό είναι ίσως το πιο συχνό θέμα σε JavaScript συνέντευξη. Δεν αρκεί να ξέρεις τον ορισμό του closure — πρέπει να μπορείς να **προβλέψεις τι θα τυπώσει ένα snippet** χωρίς να το τρέξεις. Σε αυτό το μάθημα θα χτίσουμε ακριβώς αυτή την ικανότητα.

## Τα τρία είδη scope

**Scope** είναι η περιοχή του κώδικα όπου μια μεταβλητή είναι ορατή. Η JavaScript έχει τρία βασικά είδη:

```js
const planet = "Earth"; // 1. Global scope: ορατό παντού

function describe() {
  const continent = "Europe"; // 2. Function scope: ορατό μόνο μέσα στη function

  if (true) {
    const city = "Athens"; // 3. Block scope: ορατό μόνο μέσα στο { } block
    console.log(planet, continent, city); // ✅ όλα ορατά εδώ
  }

  console.log(city); // ❌ ReferenceError: city is not defined
}
```

Ο κανόνας: **ο κώδικας βλέπει προς τα έξω, ποτέ προς τα μέσα**. Ένα inner block βλέπει τις μεταβλητές των outer scopes, αλλά το outer scope δεν βλέπει μέσα στο block.

## var vs let vs const — γιατί έχει σημασία

Αυτό δεν είναι θέμα στυλ. Έχουν **διαφορετικό scope**:

| | Scope | Re-assign | Hoisting συμπεριφορά |
|---|---|---|---|
| `var` | Function | ✅ | Initialized ως `undefined` |
| `let` | Block | ✅ | Hoisted αλλά σε TDZ |
| `const` | Block | ❌ | Hoisted αλλά σε TDZ |

```js
if (true) {
  var a = 1;
  let b = 2;
}

console.log(a); // 1 — το var "δραπέτευσε" από το block!
console.log(b); // ReferenceError — το let έμεινε μέσα
```

Το `var` αγνοεί τα blocks (`if`, `for`, `while`) και κάνει scope μόνο σε functions. Αυτό προκαλεί τα περισσότερα legacy bugs — γι' αυτό σε σύγχρονο κώδικα χρησιμοποιούμε `const` πρώτα, `let` όταν χρειάζεται re-assignment, και `var` ποτέ.

> Σημείωση για το `const`: σημαίνει ότι το **binding** δεν αλλάζει, όχι ότι η τιμή είναι αμετάβλητη. Ένα `const user = { name: "Ada" }` επιτρέπει `user.name = "Grace"`, αλλά όχι `user = otherObject`.

## Hoisting — τι συμβαίνει πραγματικά

Πριν εκτελεστεί ο κώδικας, η JavaScript κάνει ένα «πρώτο πέρασμα» και καταγράφει όλα τα declarations. Αυτό λέγεται **hoisting**, αλλά κάθε τύπος declaration συμπεριφέρεται διαφορετικά:

```js
sayHi(); // ✅ "Hi!" — function declarations hoist-άρονται ολόκληρες

console.log(x); // undefined — το var hoist-άρεται, αλλά ΧΩΡΙΣ την τιμή του
console.log(y); // ❌ ReferenceError — το let είναι σε TDZ

function sayHi() {
  console.log("Hi!");
}

var x = 5;
let y = 10;
```

**TDZ (Temporal Dead Zone)**: η περιοχή από την αρχή του scope μέχρι τη γραμμή του `let`/`const` declaration. Η μεταβλητή «υπάρχει» αλλά κάθε πρόσβαση πετάει `ReferenceError`. Αυτή η λέξη εντυπωσιάζει σε συνέντευξη — χρησιμοποίησέ τη.

Προσοχή και στα **function expressions** — δεν hoist-άρονται όπως τα declarations:

```js
greet(); // ❌ TypeError: greet is not a function

var greet = function () {
  console.log("hello");
};
// Το var greet hoist-άρεται ως undefined,
// οπότε καλείς undefined() → TypeError
```

## Lexical scope — το θεμέλιο των closures

Η JavaScript αποφασίζει τι βλέπει μια function από το **πού γράφτηκε**, όχι από το πού καλείται:

```js
const name = "Ada";

function whoAmI() {
  console.log(name); // κοιτάει στο scope όπου ΓΡΑΦΤΗΚΕ
}

function caller() {
  const name = "Grace";
  whoAmI(); // "Ada" — όχι "Grace"!
}

caller();
```

Η `whoAmI` γράφτηκε στο global scope, άρα βλέπει το global `name`. Το ότι κλήθηκε μέσα από την `caller` δεν αλλάζει τίποτα. Αυτό λέγεται **lexical (static) scoping**.

## Closures

**Closure** είναι όταν μια function κρατάει πρόσβαση στις μεταβλητές του lexical scope της, ακόμα κι αφού το outer function έχει ολοκληρωθεί:

```js
function createCounter() {
  let count = 0; // "private" μεταβλητή

  return function increase() {
    count = count + 1;
    return count;
  };
}

const increase = createCounter(); // η createCounter ΤΕΛΕΙΩΣΕ εδώ
increase(); // 1 — κι όμως το count ζει ακόμα
increase(); // 2
```

Όταν η `createCounter` επιστρέφει, το `count` **δεν** καταστρέφεται — η `increase` το «κουβαλάει» μαζί της. Κάθε κλήση της `createCounter` φτιάχνει **νέο, ανεξάρτητο** closure:

```js
const counterA = createCounter();
const counterB = createCounter();

counterA(); // 1
counterA(); // 2
counterB(); // 1 — δικό του count, δεν επηρεάζεται από το A
```

### Πού χρησιμοποιούνται closures στην πράξη

Δεν είναι ακαδημαϊκό θέμα — τα χρησιμοποιείς καθημερινά:

**1. Data privacy** — μεταβλητές που δεν μπορεί να πειράξει κανείς απ' έξω:

```js
function createWallet(initialBalance) {
  let balance = initialBalance;

  return {
    deposit(amount) {
      if (amount > 0) balance += amount;
      return balance;
    },
    getBalance() {
      return balance;
    },
  };
}

const wallet = createWallet(100);
wallet.balance; // undefined — δεν υπάρχει τρόπος άμεσης πρόσβασης
wallet.deposit(50); // 150 — μόνο μέσα από το API που όρισες
```

**2. Memoization** — cache ακριβών υπολογισμών:

```js
function memoize(fn) {
  const cache = new Map(); // ζει μέσα στο closure

  return function (arg) {
    if (cache.has(arg)) return cache.get(arg);
    const result = fn(arg);
    cache.set(arg, result);
    return result;
  };
}

const slowSquare = (n) => {
  console.log("computing...");
  return n * n;
};

const fastSquare = memoize(slowSquare);
fastSquare(4); // "computing..." → 16
fastSquare(4); // 16 (από το cache, χωρίς log)
```

**3. Event handlers & callbacks** — κάθε handler θυμάται τα δικά του δεδομένα:

```js
function setupButton(label) {
  document.querySelector(`#${label}`).addEventListener("click", () => {
    console.log(`Clicked: ${label}`); // closure πάνω στο label
  });
}
```

Η λύση; Άλλαξε το `var` σε `let`:


Με `let`, κάθε iteration του loop παίρνει **δικό του φρέσκο `i`**. Κάθε closure κλείνει πάνω σε διαφορετική μεταβλητή. Αυτή η εξήγηση — «το var μοιράζεται, το let δημιουργεί νέο binding ανά iteration» — είναι ακριβώς αυτό που θέλει να ακούσει ο interviewer.

> A closure is a function that retains access to variables from its lexical scope, even after the outer function has finished executing. JavaScript uses lexical scoping, so what a function can see is determined by where it was defined. Closures are useful for data privacy, memoization, and callbacks — for example, a counter factory where the count variable is only accessible through the returned methods.
