# Patterns, Testing & Production

Το τελευταίο βήμα: από «δουλεύει στο μηχάνημά μου» σε **production-ready** Vue. Async components, performance, accessibility και testing — τα πράγματα που σε κάνουν να εμπιστεύονται το feature σου σε πραγματικό κώδικα.

## Τι πρέπει να μπορείς στο τέλος

- να φορτώνεις components async με `Suspense`,
- να αναγνωρίζεις και να λύνεις performance προβλήματα,
- να γράφεις βασικά tests με **Vue Test Utils**,
- να ξέρεις τα βασικά accessibility & build βήματα.

## Async components & Suspense

Για βαριά components ή για να δείξεις loading όσο φορτώνει δεδομένα, η Vue έχει async loading και `<Suspense>`:

```vue
<script setup>
import { defineAsyncComponent } from "vue";
const HeavyChart = defineAsyncComponent(() => import("./HeavyChart.vue"));
</script>

<template>
  <Suspense>
    <template #default><HeavyChart /></template>
    <template #fallback><p>Φόρτωση γραφήματος…</p></template>
  </Suspense>
</template>
```

Το `<Suspense>` δείχνει το `#fallback` όσο το async component (ή ένα `async setup`) δεν είναι έτοιμο. Συνδυασμένο με `defineAsyncComponent` σπάει το bundle και βελτιώνει το αρχικό load.

## Performance: τα συνηθισμένα

Η Vue είναι γρήγορη από προεπιλογή, αλλά τα κλασικά λάθη:

- **Σταθερά & μοναδικά `:key`** στο `v-for` — λάθος keys = περιττά re-renders.
- **`v-once`** για περιεχόμενο που δεν αλλάζει ποτέ (renderάρεται μία φορά).
- **`v-memo`** για ακριβές λίστες — παρακάμπτει re-render αν δεν άλλαξαν συγκεκριμένες τιμές.
- **`computed` αντί για methods** για παράγωγες τιμές (caching).
- **Lazy routes/components** για code splitting.
- **`shallowRef`** για μεγάλα objects που δεν χρειάζονται deep reactivity.

> Πρώτα **μέτρα** (Vue DevTools → component render timings) και μετά βελτιστοποίησε. Πρόωρο optimization = χαμένος χρόνος.

## Accessibility (a11y)

Τα βασικά που ελέγχονται σε review:

- Χρησιμοποίησε **semantic HTML** (`<button>`, `<nav>`, `<label>`) — όχι `<div>` με `@click`.
- Κάθε input έχει συνδεδεμένο `<label>`.
- **Focus management** μετά από route change ή άνοιγμα modal.
- `aria-*` attributes όπου το semantic HTML δεν αρκεί.
- Σεβασμός `prefers-reduced-motion` στα transitions.

Η Vue δεν σε εμποδίζει να γράψεις προσβάσιμο HTML — απλώς μην το ξεχνάς πίσω από τα components.

## Testing με Vue Test Utils

Το επίσημο εργαλείο για component tests (συνήθως με **Vitest**):

```js
import { mount } from "@vue/test-utils";
import Counter from "@/components/Counter.vue";

test("increments on click", async () => {
  const wrapper = mount(Counter, { props: { start: 0 } });

  expect(wrapper.text()).toContain("0");
  await wrapper.find("button").trigger("click");
  expect(wrapper.text()).toContain("1");
});
```

Αρχές καλού test:

- Δοκίμασε **συμπεριφορά**, όχι υλοποίηση — τι βλέπει/κάνει ο χρήστης.
- Βρες elements από **ορατό κείμενο/ρόλο**, όχι από CSS classes που αλλάζουν.
- Για events του παιδιού, έλεγξε με `wrapper.emitted()`.
- `await` στα triggers — η ενημέρωση του DOM είναι ασύγχρονη.

## Build & deploy

Η σύγχρονη Vue στήνεται με **Vite**: `npm run build` παράγει στατικά assets (HTML/JS/CSS) στο `dist/`, έτοιμα για οποιοδήποτε static host (Netlify, Vercel, Pages). Βασικά:

- **Environment variables** με πρόθεμα `VITE_` εκτίθενται στον client (μη βάζεις secrets).
- **Tree-shaking & minification** γίνονται αυτόματα στο build.
- Έλεγξε το **bundle size** — lazy load ό,τι δεν χρειάζεται αμέσως.

## Πώς απαντάς στη συνέντευξη

> For production I lazy-load heavy components with `defineAsyncComponent` and `<Suspense>` for a clean loading state, and I keep performance healthy with stable `:key`s, `computed` over methods, and route-level code splitting — but I measure with Vue DevTools before optimizing. I test behavior with Vue Test Utils: find elements by visible text or role, trigger events with `await`, and assert what the user sees, so tests survive refactors. I keep accessibility in mind with semantic HTML, labels, and focus management, and on Vite I'm careful that only `VITE_`-prefixed, non-secret values reach the client.
