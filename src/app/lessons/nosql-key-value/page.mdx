# Key-Value Stores (Redis & DynamoDB)

Τα key-value stores είναι ο απλούστερος και ταχύτερος τύπος database: `get(key)` / `set(key, value)`. Στα interviews εξετάζουν πού ταιριάζουν (cache, sessions, counters, leaderboards) και — στο DynamoDB — πόσο κρίσιμο είναι το **key design**.

## Τι πρέπει να μπορείς στο τέλος

- να εξηγείς το key-value model και γιατί είναι τόσο γρήγορο,
- να αναγνωρίζεις use cases (cache, session, rate limit, counter),
- να χρησιμοποιείς TTL για auto-expiry,
- να σχεδιάζεις keys/partition keys σωστά,
- να ξέρεις τα όρια (όχι ad-hoc queries, όχι joins).

## Το μοντέλο: get / set σε key

```txt
SET  user:42:session   "{...}"   EX 3600    # value + TTL 1 ώρα
GET  user:42:session
DEL  user:42:session
INCR page:home:views               # atomic counter
```

Είναι γρήγορο γιατί είναι **απλό**: ο τύπος πρόσβασης είναι O(1) lookup σε hash map (συχνά in-memory, όπως το Redis). Καμία ανάλυση query, κανένα join — απλώς «δώσε μου την τιμή αυτού του key».

## Use cases — εκεί που λάμπει

```txt
Caching        → cache-άρεις ακριβά query results (key = query hash)
Sessions       → session id → user data, με TTL auto-expiry
Rate limiting   → INCR ανά user/IP, με TTL παράθυρο
Counters        → views, likes (atomic INCR/DECR)
Leaderboards    → sorted sets (Redis ZADD/ZRANGE)
Queues/Pub-Sub  → background jobs, real-time messages (Redis)
```

(Το Redis caching/sessions το εμβαθύνει το μάθημα **Redis, Caching & Sessions** — εδώ εστιάζουμε στο key-value ως data model.)

## TTL (Time To Live) — auto-expiry

Το **TTL** (Time To Live = «χρόνος ζωής») ορίζει πόσο θα ζήσει ένα key πριν σβήσει αυτόματα. Είναι κορυφαίο πλεονέκτημα:

```txt
SET  cart:42  "{...}"  EX 1800     # το cart σβήνει μόνο του σε 30 λεπτά
```

Δεν χρειάζεσαι cron για cleanup — η βάση σβήνει τα expired keys. Ιδανικό για cache, sessions, temporary data, OTP codes.

## DynamoDB: το key design είναι ΤΑ ΠΑΝΤΑ

Στο DynamoDB (managed, distributed key-value/document) κάθε item έχει **partition key** (και optional **sort key**). Το partition key καθορίζει σε ποιον server (partition) ζει το item:

```txt
PK = USER#42        SK = PROFILE        → το προφίλ του user 42
PK = USER#42        SK = ORDER#1001     → παραγγελία του user 42
PK = USER#42        SK = ORDER#1002     → άλλη παραγγελία
```

Query «δώσε όλα τα items με PK=USER#42» → όλα τα δεδομένα ενός χρήστη σε **ένα** γρήγορο query. Αλλά:

## Όρια του key-value

```txt
- Όχι ad-hoc queries: δεν κάνεις "βρες όλους με age > 30" — μόνο by key
- Όχι joins / σχέσεις (τα χειρίζεσαι με key design ή στον app)
- Value συχνά opaque blob (αν θες queries μέσα → document store)
```

Γι' αυτό το key-value θέλει να ξέρεις τα access patterns **εκ των προτέρων** — σχεδιάζεις τα keys γύρω από το «πώς θα διαβάσω».

## Πώς απαντάς στη συνέντευξη

«When do you use a key-value store?»

> When the access pattern is a direct lookup by a known key and I need very high throughput — caching expensive results, sessions, rate-limit counters, leaderboards. It's fast because it's simple: an O(1) lookup, often in memory, with no query planning or joins. TTL is a big win — sessions and caches expire themselves, no cleanup cron. The catch is there are no ad-hoc queries or joins, so I have to know my access patterns up front and design keys around them. In DynamoDB specifically, the partition key is everything: it decides which partition an item lives on, so I pick a high-cardinality, evenly distributed key to avoid hot partitions and throttling — the same idea as avoiding hot shards in sharding.
