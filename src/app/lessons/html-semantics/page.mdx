# HTML Structure & Semantics

Η HTML δεν είναι μόνο `div` και κείμενο. Τα semantic elements περιγράφουν τον ρόλο κάθε περιοχής της σελίδας.

Όταν γράφεις σωστή δομή:

- ο κώδικας διαβάζεται πιο εύκολα,
- οι μηχανές αναζήτησης καταλαβαίνουν καλύτερα τη σελίδα,
- τα screen readers βοηθούν καλύτερα τους χρήστες.

## Όλα τα semantic elements & πώς τα χρησιμοποιείς

### Δομή σελίδας (page layout)

Τα περισσότερα από αυτά δέχονται *flow content* (σχεδόν οποιοδήποτε element), οπότε οι κανόνες φωλιάσματος είναι κυρίως λίγες **σημαντικές απαγορεύσεις**. Για κάθε element: τι κάνει, πού επιτρέπεται να μπει (**parent**), τι λογικά περιέχει (**children**) και πού **δεν** μπαίνει.

> **Λεζάντα μοναδικότητας:** το badge `★ μοναδικό` = ακριβώς **ένα ανά σελίδα**. Το badge `1 στο top level` = ένα για όλη τη σελίδα, αλλά **επαναλαμβάνεται** μέσα σε `article`/`section`. Όσα **δεν** έχουν badge μπαίνουν **όσες φορές** χρειάζεται.

- **`header`** `1 στο top level` — η πάνω περιοχή σελίδας ή ενότητας (logo, τίτλος, κύριο `nav`). Ένα banner στην κορυφή, αλλά μπορείς να έχεις κι άλλα (ένα ανά `article`/`section`).
  - *Parent:* `body`, ή μέσα σε `main`, `article`, `section`, `aside`, `nav`.
  - *Children:* headings `h1`–`h6`, `nav`, `p`, logo `img`, search form — flow content.
  - *ΟΧΙ:* μέσα σε άλλο `header`, σε `footer` ή σε `address`· και **δεν** περιέχει `main`, ούτε άλλο `header`/`footer`.
- **`nav`** — ομάδα από links πλοήγησης (menu, breadcrumbs, table of contents). Μόνο για *σημαντικά* block πλοήγησης, όχι για κάθε link.
  - *Parent:* σχεδόν παντού — `body`, `header`, `main`, `section`, `article`, `aside`, `footer`.
  - *Children:* συνήθως λίστα links (`ul > li > a`), ή heading + links.
  - *ΟΧΙ:* μέσα σε `address`.
- **`main`** `★ μοναδικό` — το μοναδικό, κύριο περιεχόμενο της σελίδας. **Ένα ορατό `main` ανά σελίδα.**
  - *Parent:* ιδανικά απευθείας στο `body` (ή σε `div`/`form` wrapper για layout).
  - *Children:* οτιδήποτε — όλο το κύριο περιεχόμενο.
  - *ΟΧΙ:* **ποτέ** μέσα σε `header`, `nav`, `article`, `aside` ή `footer`.
- **`section`** — θεματική ενότητα περιεχομένου, που (ιδανικά) έχει δικό της heading.
  - *Parent:* `body`, `main`, `article`, `aside`, ή άλλο `section`.
  - *Children:* flow content + heading· μπορεί να περιέχει `article` ή εμφωλευμένα `section`.
  - *ΟΧΙ:* δεν είναι generic wrapper — αν δεν υπάρχει θεματικό heading, βάλε `div`.
- **`article`** — αυτόνομο περιεχόμενο που στέκεται μόνο του (blog post, news item, σχόλιο, product card). Δοκιμή: «βγάζει νόημα αν το βγάλεις από τη σελίδα;».
  - *Parent:* `body`, `main`, `section`, `aside`, ή άλλο `article` (π.χ. σχόλια μέσα σε post).
  - *Children:* δικό του `header`/`footer`, `section`, εμφωλευμένα `article` — flow content.
  - *ΟΧΙ:* μέσα σε `address`.
- **`aside`** — περιεχόμενο δευτερεύον/πλάγιο (sidebar, σχετικά links, διαφήμιση, callout box).
  - *Parent:* `body`, `main`, `article`, `section`.
  - *Children:* flow content — heading, related `nav`, callout.
  - *ΟΧΙ:* μέσα σε `address`· **δεν** περιέχει `main`.
- **`footer`** `1 στο top level` — η κάτω περιοχή σελίδας ή ενότητας (copyright, links, στοιχεία επικοινωνίας, συγγραφέας/ημερομηνία `article`).
  - *Parent:* `body`, `main`, `article`, `section`, `aside`, `nav`.
  - *Children:* `nav`, `address`, copyright, links — flow content.
  - *ΟΧΙ:* μέσα σε άλλο `footer`, σε `header` ή σε `address`· και **δεν** περιέχει `main`, `header` ή άλλο `footer`.

**Χρυσοί κανόνες φωλιάσματος** (αν θυμηθείς μόνο αυτά, καλύπτεις το 90%):

1. `main` → ποτέ μέσα σε `header`/`nav`/`article`/`aside`/`footer`, και ένα ανά σελίδα.
2. `header` & `footer` → δεν φωλιάζουν μέσα στον εαυτό τους ή το ένα μέσα στο άλλο, και δεν περιέχουν `main`.
3. `address` → μόνο contact info· **όχι** sectioning elements (`article`/`section`/`nav`/`aside`), όχι headings, όχι `header`/`footer`.
4. `section`, `article`, `aside`, `nav` → φωλιάζουν ελεύθερα μέσα σε flow content· ο ρόλος τους εξαρτάται από το **πού** μπαίνουν.

### Δέντρο: τι μπαίνει μέσα σε τι

Έτσι φαίνεται μια τυπική σελίδα ως δέντρο. Δίπλα σε κάθε element, το «→ περιέχει:» δείχνει τι μπαίνει συνήθως **μέσα** του:

```text
<body>
│
├─ <header>  [1 στο top level]  → περιέχει: h1, nav, (logo) img
│   ├─ <h1>
│   └─ <nav>          → περιέχει: ul > li > a  (links πλοήγησης)
│
├─ <main>  [★ μοναδικό]  → περιέχει: article, section, aside, headings…
│   │
│   ├─ <article>      → περιέχει: header, section, footer, nested article
│   │   ├─ <header>   → h1–h6, <time>  (τίτλος + ημερομηνία του article)
│   │   ├─ <section>  → περιέχει: heading + p, figure, details…
│   │   │   ├─ <h2>
│   │   │   ├─ <p>
│   │   │   └─ <figure> → <img> + <figcaption>
│   │   ├─ <article>  → nested (π.χ. σχόλια κάτω από το post)
│   │   └─ <footer>   → <address>  (συγγραφέας/ημερομηνία του article)
│   │
│   └─ <aside>        → περιέχει: heading, nav, p  (callout / σχετικά)
│       ├─ <h2>
│       └─ <nav>      → related links
│
└─ <footer>  [1 στο top level]  → περιέχει: nav, address, p  (copyright)
    ├─ <nav>          → footer links
    └─ <address>      → στοιχεία επικοινωνίας
```

Πρόσεξε δύο πράγματα: (1) το `<main>` κάθεται **δίπλα** στα `header`/`footer`, ποτέ μέσα τους· (2) τα `header`, `footer`, `nav` εμφανίζονται σε **δύο επίπεδα** — ένα για όλη τη σελίδα και ένα μέσα στο `article` — και ο ρόλος τους αλλάζει ανάλογα με το πού φωλιάζουν.

### Κείμενο & περιεχόμενο (content semantics)

- `h1`–`h6`: ιεραρχία τίτλων (δες παρακάτω).
- `p`: παράγραφος κειμένου.
- `figure` + `figcaption`: εικόνα/διάγραμμα/κώδικας μαζί με τη λεζάντα του.
- `time`: ημερομηνία ή ώρα σε μηχαναγνώσιμη μορφή με το attribute `datetime`, π.χ. `<time datetime="2026-06-27">27 Ιουν 2026</time>`.
- `mark`: τονισμένο/highlighted κομμάτι κειμένου (π.χ. αποτέλεσμα αναζήτησης).
- `address`: **στοιχεία επικοινωνίας** (όνομα, email, τηλέφωνο, social) του συγγραφέα του κοντινότερου `article` ή όλης της σελίδας. **Όχι** για οποιαδήποτε ταχυδρομική διεύθυνση μέσα στο κείμενο — μόνο contact info.
  - *Parent:* συνήθως μέσα σε `footer` (του `article` ή της σελίδας).
  - *Children:* μόνο απλό inline περιεχόμενο — κείμενο και links (`a`, `mailto:`, `tel:`).
  - *ΟΧΙ:* headings `h1`–`h6`, sectioning elements (`article`/`section`/`nav`/`aside`), `header`/`footer`, ή άλλο `address`.
- `blockquote` / `cite`: παράθεση και αναφορά πηγής.
- `details` + `summary`: αναδιπλούμενο (collapsible) περιεχόμενο — accordion χωρίς JavaScript.
- `abbr`: συντομογραφία με επεξήγηση στο `title`, π.χ. `<abbr title="Cascading Style Sheets">CSS</abbr>`.
- `strong` (σημασία/σοβαρότητα) και `em` (έμφαση) — **όχι** για απλό bold/italic· για εμφάνιση χρησιμοποίησε CSS.

## Βασικός κανόνας

Διάλεξε το tag με βάση το **νόημα** του περιεχομένου, όχι την εμφάνιση. Αν κανένα semantic element δεν ταιριάζει, τότε —και μόνο τότε— χρησιμοποιείς `div` (block) ή `span` (inline) σαν ουδέτερο «κουτί» για styling.

## Landmarks

Elements όπως `header`, `nav`, `main`, `aside` και `footer` δημιουργούν *landmarks*. Τα screen readers τα χρησιμοποιούν σαν “χάρτη” της σελίδας, ώστε ο χρήστης να πηδάει κατευθείαν στο κύριο περιεχόμενο ή στο menu.

Συνήθως μια σελίδα πρέπει να έχει ένα κύριο `main`.

```html
<body>
  <header>...</header>
  <nav>...</nav>
  <main>...</main>
  <aside>...</aside>
  <footer>...</footer>
</body>
```

## Πλήρες παράδειγμα: η δομή μιας σελίδας

Έτσι μπαίνουν όλα μαζί σε μια πραγματική σελίδα blog. Πρόσεξε **πού** κάθεται κάθε semantic element και πώς το ένα φωλιάζει μέσα στο άλλο:

```html
<body>
  <!-- Κορυφή: logo + κύρια πλοήγηση -->
  <header>
    <h1>Tech Blog</h1>
    <nav aria-label="Κύριο menu">
      <a href="/">Αρχική</a>
      <a href="/articles">Άρθρα</a>
      <a href="/about">Σχετικά</a>
    </nav>
  </header>

  <!-- Το κύριο περιεχόμενο της σελίδας -->
  <main>
    <article>
      <header>
        <h2>Τι είναι το semantic HTML;</h2>
        <p>
          Δημοσιεύτηκε στις
          <time datetime="2026-06-27">27 Ιουνίου 2026</time>
        </p>
      </header>

      <section>
        <h3>Εισαγωγή</h3>
        <p>Το semantic HTML περιγράφει τον ρόλο κάθε περιοχής...</p>

        <figure>
          <img src="/diagram.png" alt="Δομή μιας semantic σελίδας" />
          <figcaption>Η ανατομία μιας σελίδας με landmarks.</figcaption>
        </figure>
      </section>

      <section>
        <h3>Συχνές ερωτήσεις</h3>
        <details>
          <summary>Χρειάζομαι πάντα section;</summary>
          <p>Όχι. Χρησιμοποίησέ το όταν έχεις ξεχωριστή θεματική ενότητα.</p>
        </details>
      </section>

      <!-- footer ΜΕΣΑ στο article: αφορά μόνο το άρθρο -->
      <footer>
        <address>Γράφτηκε από τον Γιάννη</address>
      </footer>
    </article>

    <!-- Πλάγιο περιεχόμενο, δευτερεύον σε σχέση με το article -->
    <aside aria-label="Σχετικά άρθρα">
      <h2>Διάβασε επίσης</h2>
      <nav>
        <a href="/css-grid">CSS Grid</a>
        <a href="/accessibility">Accessibility 101</a>
      </nav>
    </aside>
  </main>

  <!-- footer ΟΛΗΣ της σελίδας -->
  <footer>
    <p>© 2026 Tech Blog</p>
    <nav aria-label="Footer">
      <a href="/privacy">Privacy</a>
      <a href="/contact">Επικοινωνία</a>
    </nav>
  </footer>
</body>
```

Δύο πράγματα που ξεχωρίζουν σε αυτό το παράδειγμα:

- Το ίδιο element (π.χ. `header`, `footer`, `nav`) μπορεί να εμφανίζεται **σε διαφορετικά επίπεδα**: ένα για όλη τη σελίδα και ένα μέσα στο `article`. Ο ρόλος του εξαρτάται από το πού φωλιάζει.
- Το `aside` και τα δύο `footer` δεν είναι «λιγότερο σημαντικά» — απλώς έχουν **διαφορετικό ρόλο**. Αυτό ακριβώς βοηθάει browsers και screen readers.

## Section vs article

Χρησιμοποίησε `article` όταν το περιεχόμενο μπορεί να σταθεί μόνο του, όπως blog post, news item ή forum post.

Χρησιμοποίησε `section` όταν χωρίζεις μια σελίδα σε θεματικές ενότητες.

```html
<article>
  <h1>Οδηγός HTML</h1>
  <section>
    <h2>Semantics</h2>
    <p>...</p>
  </section>
</article>
```

## Heading hierarchy

Τα headings πρέπει να δείχνουν δομή:

- `h1`: κύριος τίτλος σελίδας,
- `h2`: μεγάλες ενότητες,
- `h3`: υποενότητες.

Μην διαλέγεις heading μόνο επειδή σου αρέσει το μέγεθος. Για μέγεθος χρησιμοποίησε CSS.

## Common mistake

Το semantic HTML δεν σημαίνει ότι δεν θα χρησιμοποιήσεις ποτέ `div`. Το `div` είναι fine όταν δεν υπάρχει καλύτερο semantic element. Απλώς μην το χρησιμοποιείς για τα πάντα από συνήθεια.
