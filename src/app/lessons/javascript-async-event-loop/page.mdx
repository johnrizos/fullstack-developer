# Async JavaScript & Event Loop

Η JavaScript τρέχει σε **ένα μόνο thread** — μπορεί να κάνει ένα πράγμα τη φορά. Κι όμως, σερβίρει requests, κάνει animations και περιμένει timers ταυτόχρονα. Το «μαγικό» πίσω από αυτό είναι το **event loop**, και είναι από τις πιο σίγουρες ερωτήσεις σε κάθε JavaScript συνέντευξη.

## Το μοντέλο: call stack, queues, event loop

Για να προβλέπεις output order χρειάζεσαι τέσσερα κομμάτια:

1. **Call stack** — εδώ εκτελείται ο κώδικας, μία function τη φορά. Ό,τι είναι synchronous μπαίνει και βγαίνει από εδώ.
2. **Web APIs** — ο browser (ή το Node) αναλαμβάνει timers, fetch, DOM events. Δουλεύουν **έξω** από το thread της JavaScript.
3. **Task queue (macrotasks)** — εδώ περιμένουν callbacks από `setTimeout`, `setInterval`, DOM events.
4. **Microtask queue** — εδώ περιμένουν τα `.then()` callbacks των promises, το `queueMicrotask`, και η συνέχεια μετά από `await`.

Το **event loop** κάνει συνέχεια το εξής:

> Όταν αδειάσει το call stack → τρέξε **ΟΛΑ** τα microtasks → μετά πάρε **ΕΝΑ** macrotask → επανάλαβε.

Αυτή η μία πρόταση απαντά το 90% των async interview ερωτήσεων. Τα microtasks έχουν **πάντα προτεραιότητα** έναντι των macrotasks.

## Βήμα 1: synchronous κώδικας πρώτα

```js
console.log("A");

setTimeout(() => console.log("C"), 0);

console.log("B");
// Output: A, B, C
```

Ακόμα και με `0ms`, το `setTimeout` callback **δεν τρέχει αμέσως**. Πάει στο Web API, μετά στο task queue, και περιμένει να αδειάσει το call stack. Το `0` σημαίνει «το συντομότερο δυνατό», όχι «τώρα».

## Βήμα 2: microtasks πριν από macrotasks

```js
setTimeout(() => console.log("timeout"), 0); // macrotask

Promise.resolve().then(() => console.log("promise")); // microtask

console.log("sync");

// Output: sync → promise → timeout
```

Γιατί; Ο synchronous κώδικας τελειώνει (`sync`). Το event loop κοιτάει πρώτα τη microtask queue (`promise`), και μόνο μετά παίρνει το επόμενο macrotask (`timeout`).

## async/await — promises με ωραίο συντακτικό

Το `async/await` δεν είναι νέος μηχανισμός — είναι **syntax πάνω σε promises**. Δύο κανόνες:

- Μια `async function` επιστρέφει **πάντα** Promise.
- Το `await` παγώνει **μόνο την async function**, όχι το πρόγραμμα. Ό,τι ακολουθεί το `await` συνεχίζει ως microtask.

```js
async function load() {
  console.log("before await"); // synchronous!
  await Promise.resolve();
  console.log("after await"); // microtask
}

console.log("start");
load();
console.log("end");

// Output: start → before await → end → after await
```

Προσοχή στο σημείο-κλειδί: το σώμα της async function **μέχρι το πρώτο await τρέχει synchronously**. Μόνο η συνέχεια μετά το `await` μπαίνει στη microtask queue.

## Error handling σε async κώδικα

Στη συνέντευξη συχνά ακολουθεί ερώτηση για errors. Με `async/await` χρησιμοποιείς `try/catch`:

```js
async function fetchUser(id) {
  try {
    const response = await fetch(`/api/users/${id}`);
    if (!response.ok) {
      throw new Error(`HTTP ${response.status}`);
    }
    return await response.json();
  } catch (error) {
    console.error("Failed to fetch user:", error);
    throw error; // ή επίστρεψε fallback τιμή
  }
}
```

> Κλασικό trap: το `fetch` **δεν** κάνει reject σε HTTP errors όπως 404 ή 500 — μόνο σε network failures. Πάντα έλεγξε το `response.ok`.

## Παράλληλα ή σειριακά; Promise.all

Άλλο αγαπημένο interview θέμα — αυτά τα δύο ΔΕΝ είναι ίδια:

```js
// ΣΕΙΡΙΑΚΑ: ~2 δευτερόλεπτα συνολικά (το ένα περιμένει το άλλο)
const a = await fetchA(); // 1s
const b = await fetchB(); // 1s

// ΠΑΡΑΛΛΗΛΑ: ~1 δευτερόλεπτο συνολικά (ξεκινούν μαζί)
const [a2, b2] = await Promise.all([fetchA(), fetchB()]);
```

Οι βασικές combinator functions:

| Method | Συμπεριφορά |
|---|---|
| `Promise.all` | Περιμένει όλα. Αν ΕΝΑ κάνει reject → όλο reject. |
| `Promise.allSettled` | Περιμένει όλα, ποτέ δεν κάνει reject — δίνει status ανά promise. |
| `Promise.race` | Επιστρέφει το πρώτο που τελειώνει (resolve Ή reject). |
| `Promise.any` | Επιστρέφει το πρώτο που κάνει **resolve** — αγνοεί rejects. |

Πρακτικός κανόνας: ανεξάρτητα requests → `Promise.all`. Αν θες να μη ρίξει όλο το batch ένα αποτυχημένο request → `Promise.allSettled`.

## Πώς απαντάς στη συνέντευξη

Αν σε ρωτήσουν «why does a Promise run before setTimeout?»:

> JavaScript is single-threaded with an event loop. When the call stack empties, the engine drains the entire microtask queue — which includes promise callbacks — before picking up the next macrotask, like a setTimeout callback. That's why microtasks always win, even against setTimeout with 0ms.
