# TypeScript Foundations

Το TypeScript είναι JavaScript με στατικό type checking — τα types εξαφανίζονται στο compile, άρα **δεν** αλλάζουν τη runtime συμπεριφορά. Η αξία του: πιάνει λάθη πριν τρέξει ο κώδικας, τεκμηριώνει contracts μεταξύ modules, και κάνει το refactoring ασφαλές (μετονομάζεις ένα πεδίο και βλέπεις αμέσως κάθε σπασμένη χρήση). Σε fullstack αγγελίες είναι πλέον σχεδόν υποχρεωτικό. Αυτό το πρώτο μάθημα του track χτίζει τη βάση: πώς σκέφτεται ο compiler, βασικοί τύποι & annotations, `type` vs `interface`, και η κρίσιμη ιδέα ότι **τα types εξαφανίζονται στη runtime**. Τα επόμενα μαθήματα μπαίνουν βαθιά σε unions/narrowing, functions, generics, utility/mapped types, safety/config και declarations.

## Βασικοί τύποι & annotations

Δηλώνεις τον τύπο μιας μεταβλητής/παραμέτρου με `: Type`. Συχνά δεν χρειάζεται — ο compiler κάνει **inference**:

```ts
let title: string = "Μάθε TS";  // explicit
let count = 0;                  // inferred ως number — δεν χρειάζεται annotation
const tags: string[] = [];      // array από strings
const point: { x: number; y: number } = { x: 1, y: 2 };

function greet(name: string): string {  // typed param + return
  return `Γεια ${name}`;
}
```

Κανόνας: **άσε το inference να δουλέψει** για τοπικές μεταβλητές· βάλε explicit types στα **όρια** — function signatures, public APIs, exported types. Εκεί το type είναι το «συμβόλαιο».

## Η βάση: structural typing

Το πιο σημαντικό πράγμα για το πώς σκέφτεται ο compiler: συγκρίνει **σχήματα**, όχι ονόματα. Αυτό λέγεται structural typing (ή «duck typing»):

```ts
interface Point {
  x: number;
  y: number;
}

const spot = { x: 1, y: 2, label: "home" };
const p: Point = spot; // ✅ ΟΚ — έχει ό,τι χρειάζεται το Point (και παραπάνω)

function logPoint(point: Point) {}
logPoint({ x: 1, y: 2 });               // ✅
logPoint({ x: 1, y: 2, label: "hi" });  // ❌ object literal: excess property check!
```

Παρατήρησε την ασυμμετρία — όταν περνάς **object literal απευθείας**, το TypeScript κάνει αυστηρότερο έλεγχο για «έξτρα» properties (πιάνει typos). Μέσω μεταβλητής, αρκεί το σχήμα να καλύπτεται. Αντίθετο μοντέλο: το **nominal typing** (Java/C#), όπου μετράει το όνομα της κλάσης.

## type vs interface — η σίγουρη ερώτηση

Και τα δύο περιγράφουν object shapes. Οι πραγματικές διαφορές:

| | `interface` | `type` |
|---|---|---|
| Object shapes | ✅ | ✅ |
| Unions, primitives, tuples | ❌ | ✅ `type ID = string \| number` |
| Επέκταση | `extends` | `&` (intersection) |
| Declaration merging | ✅ — δηλώσεις με ίδιο όνομα ενώνονται | ❌ — διπλό όνομα = error |

Πρακτικός κανόνας που αρκεί σε συνέντευξη: **interface για δημόσια object contracts** (props, models — και για library types όπου το merging βοηθά), **type για unions και συνθέσεις**. Μέσα σε ένα project, η συνέπεια μετράει περισσότερο από την επιλογή.

## Τα types εξαφανίζονται στη runtime

Η πιο παρεξηγημένη ιδέα — και ερώτηση-φίλτρο. Το TypeScript κάνει **type erasure**: μετά το compile, ο κώδικας είναι σκέτη JavaScript χωρίς κανένα type. Συνέπειες:

- **Μηδενικό runtime κόστος** — τα types δεν υπάρχουν όταν τρέχει ο κώδικας.
- **Δεν μπορείς να ελέγξεις types στη runtime** με τον τρόπο των classes — δεν υπάρχει «`if (x is User)`» μαγικά.
- **Τα δεδομένα από έξω (API/JSON) δεν ελέγχονται** από τους τύπους. Ένα `as User` είναι απλώς υπόσχεση προς τον compiler — γι' αυτό χρειάζεσαι **runtime validation** (το βλέπεις στο safety μάθημα).

```ts
interface User { name: string; }
const raw = '{"username": "ada"}';
const user = JSON.parse(raw) as User; // ψέμα στον compiler — δεν ελέγχει τίποτα
user.name.toUpperCase();              // 💥 runtime: name είναι undefined
```

## Πώς απαντάς στη συνέντευξη

«Why TypeScript, and what's the catch?»

> TypeScript catches type errors at compile time, documents the contracts between modules, and makes refactoring safe — renaming a field shows every broken usage instantly. The compiler is structural: compatibility is about an object's shape, not the name of its type. The catch is that types are erased at compile time — there's no runtime cost, but it also means external data still needs runtime validation; an `as` assertion checks nothing. My rule of thumb on `type` vs `interface`: interfaces for object contracts (they support `extends` and declaration merging), type aliases for unions and compositions — and consistency within a codebase above the choice itself.
