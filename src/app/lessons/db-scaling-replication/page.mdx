# Scaling, Replication & Partitioning

Όταν ένα μόνο database server δεν αντέχει το φορτίο, το κλιμακώνεις. Στα interviews εξετάζουν τη σειρά: **πρώτα optimize, μετά scale up, μετά scale out** — και τα trade-offs replication/sharding (consistency, complexity).

## Τι πρέπει να μπορείς στο τέλος

- να ξεχωρίζεις vertical από horizontal scaling,
- να εξηγείς read replicas και replication lag,
- να καταλαβαίνεις partitioning vs sharding,
- να ξέρεις τη θέση του caching στο scaling,
- να ακολουθείς σωστή σειρά (μην κάνεις sharding πρώτο).

## Η σωστή σειρά

```txt
1. Optimize       → indexes, σβήσε N+1, διόρθωσε αργά queries (φθηνό, μεγάλο κέρδος)
2. Cache          → βάλε caching layer για read-heavy δεδομένα (Redis)
3. Scale up       → μεγαλύτερο μηχάνημα (vertical) — απλό, μέχρι ένα όριο
4. Read replicas  → ξεφόρτωσε reads σε αντίγραφα
5. Shard/Partition→ μοίρασε τα δεδομένα (πολύπλοκο — τελευταία λύση)
```

Το πιο συχνό λάθος σε interview: «θα κάνω sharding» ως πρώτη απάντηση. Το sharding είναι η **τελευταία** και πιο ακριβή λύση.

## Vertical vs horizontal

```txt
Vertical (scale up):   μεγαλύτερο μηχάνημα (CPU/RAM/disk). Απλό, αλλά έχει οροφή & cost.
Horizontal (scale out): περισσότερα μηχανήματα (replicas/shards). Χωρίς οροφή, αλλά πολυπλοκότητα.
```

## Read replicas & replication lag

Οι περισσότερες apps είναι **read-heavy**. Στήνεις replicas: ο **primary** δέχεται writes, τα **replicas** εξυπηρετούν reads.

```txt
        writes →  [ PRIMARY ]  → replication →  [ REPLICA ] [ REPLICA ]
                                                     ↑ reads
```

Πρόβλημα: **replication lag** — τα replicas ενημερώνονται με μικρή καθυστέρηση (eventual consistency).

## Partitioning vs sharding

```txt
Partitioning: σπας έναν μεγάλο πίνακα σε κομμάτια στο ΙΔΙΟ database
              (π.χ. orders ανά μήνα). Μικρότερα indexes, ταχύτερα queries/maintenance.

Sharding:     μοιράζεις τα δεδομένα σε ΠΟΛΛΑ database servers με βάση ένα shard key
              (π.χ. user_id % N). Οριζόντια κλίμακα για writes/δεδομένα.
```

Το sharding φέρνει σοβαρή πολυπλοκότητα:

```txt
- Διάλεξε καλό shard key (ομοιόμορφη κατανομή, αλλιώς "hot shards")
- Cross-shard queries/joins είναι δύσκολα/αργά
- Cross-shard transactions σχεδόν αδύνατα
- Rebalancing/re-sharding είναι πόνος
```

## Caching στο scaling

Πριν προσθέσεις replicas/shards, ένα **cache** (Redis) μπροστά από τη βάση απορροφά την πλειονότητα των read-heavy queries. (Λεπτομέρειες στο μάθημα Redis, Caching & Sessions.) Είναι από τα φθηνότερα και πιο αποτελεσματικά βήματα — αρκεί να χειρίζεσαι invalidation.

## Πώς απαντάς στη συνέντευξη

«Your database can't keep up — how do you scale it?»

> I go in order, cheapest and highest-leverage first. I optimize: add the missing indexes, kill N+1 queries, fix slow queries with `EXPLAIN`. Then I put a cache in front for read-heavy data. Then scale up to a bigger machine, which is simple up to a ceiling. Most apps are read-heavy, so next I add read replicas — the primary takes writes, replicas serve reads — but I account for replication lag: right after a user's own write, I read from the primary so they see their changes, otherwise they might get stale data from a lagging replica. Only as a last resort do I shard, splitting data across servers by a shard key, because that brings real complexity: choosing a key that avoids hot shards, cross-shard queries and transactions becoming hard, and painful rebalancing. The mistake I avoid is jumping to sharding first when an index or a cache would have solved it.
