# Composition API in Depth

Το μεγάλο πλεονέκτημα του Composition API είναι η **επαναχρησιμοποίηση λογικής** χωρίς τα προβλήματα του παρελθόντος (mixins). Εδώ μαθαίνεις **composables** — το αντίστοιχο των custom hooks της React — καθώς και `provide`/`inject` και πότε ακόμα συναντάς το Options API.

## Τι πρέπει να μπορείς στο τέλος

- να γράφεις **composables** που μοιράζονται λογική,
- να περνάς δεδομένα βαθιά με `provide`/`inject`,
- να εξηγείς **Options API vs Composition API**,
- να αναγνωρίζεις πότε αξίζει να εξάγεις λογική σε composable.

## Composables: επαναχρησιμοποιήσιμη λογική

Ένα **composable** είναι απλώς μια function που χρησιμοποιεί το reactivity API της Vue (`ref`, `computed`, lifecycle) και **επιστρέφει** reactive state + functions. Σύμβαση: το όνομα ξεκινά με `use`.

```js
// composables/useToggle.js
import { ref } from "vue";

export function useToggle(initial = false) {
  const state = ref(initial);
  const toggle = () => (state.value = !state.value);
  return { state, toggle };
}
```

```vue
<script setup>
import { useToggle } from "@/composables/useToggle";
const { state: isOpen, toggle } = useToggle();
</script>

<template>
  <button @click="toggle">{{ isOpen ? "Κλείσε" : "Άνοιξε" }}</button>
</template>
```

Πιο ρεαλιστικό — ένα `useFetch` με loading/error state και lifecycle:

```js
// composables/useFetch.js
import { ref, onMounted } from "vue";

export function useFetch(url) {
  const data = ref(null);
  const error = ref(null);
  const loading = ref(true);

  onMounted(async () => {
    try {
      const res = await fetch(url);
      data.value = await res.json();
    } catch (e) {
      error.value = e;
    } finally {
      loading.value = false;
    }
  });

  return { data, error, loading };
}
```

Κάθε component που καλεί `useFetch` παίρνει **το δικό του** ανεξάρτητο state — ακριβώς όπως τα custom hooks στη React. Έτσι σπάς μεγάλα components σε μικρά, ονοματισμένα κομμάτια λογικής.

## `provide` / `inject`: δεδομένα βαθιά στο δέντρο

Όταν πρέπει να περάσεις δεδομένα από έναν πρόγονο σε μακρινό απόγονο **χωρίς prop drilling**, χρησιμοποιείς `provide`/`inject` — το αντίστοιχο του React Context:

```js
// πρόγονος
import { provide, ref } from "vue";
const theme = ref("dark");
provide("theme", theme);
```

```js
// οποιοσδήποτε απόγονος, όσο βαθιά κι αν είναι
import { inject } from "vue";
const theme = inject("theme", "light"); // 2ο όρισμα = default
```

Αν δώσεις ένα `ref`, η σύνδεση παραμένει **reactive**: αλλάζει ο πρόγονος → ενημερώνονται οι απόγονοι. Ιδανικό για theme, locale, auth user — πράγματα «καθολικά» για ένα υποδέντρο. Για πραγματικό shared app state, προτιμάς **Pinia** (επόμενο μάθημα).

## Options API vs Composition API

Θα συναντήσεις και τα δύο, ειδικά σε παλαιότερο κώδικα:

```js
// Options API — οργάνωση κατά "τύπο" (data, methods, computed…)
export default {
  data() { return { count: 0 }; },
  methods: { inc() { this.count++; } },
  computed: { double() { return this.count * 2; } },
};
```

```vue
<!-- Composition API — οργάνωση κατά "feature" -->
<script setup>
import { ref, computed } from "vue";
const count = ref(0);
const inc = () => count.value++;
const double = computed(() => count.value * 2);
</script>
```

Διαφορά-κλειδί: το Options API σε αναγκάζει να σκορπίσεις τη λογική ενός feature σε `data`/`methods`/`computed`· το Composition API την κρατά **μαζεμένη** και την κάνει **εξαγώγιμη** σε composable. Για νέο κώδικα → Composition API με `<script setup>`. Το Options API δεν είναι deprecated, αλλά το σύγχρονο στυλ είναι το Composition.

## Πώς απαντάς στη συνέντευξη

> The Composition API lets me extract reusable logic into composables — plain `use*` functions that use Vue's reactivity and return state plus methods, like React custom hooks. Each call gets independent state, and unlike old mixins there are no hidden name collisions because the input and output are explicit. For passing data deep without prop drilling I use `provide`/`inject` — Vue's Context equivalent — keeping it reactive by providing a `ref`. I reach for the Composition API with `<script setup>` on new code because it groups logic by feature instead of scattering it across `data`, `methods`, and `computed`.
