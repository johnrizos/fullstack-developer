# Django Foundations & MVT

Το **Django** είναι το «batteries-included» Python framework: σου δίνει ORM, admin, auth, forms, templates και security έτοιμα. Είναι ουσιαστικά το Python αντίστοιχο του Laravel — convention over configuration.

## Setup

```bash
pip install django
django-admin startproject config .    # το project (ρυθμίσεις)
python manage.py startapp blog         # ένα app (feature)
python manage.py runserver             # dev server στο :8000
```

Το `manage.py` είναι το CLI εργαλείο σου — το χρησιμοποιείς συνέχεια (`runserver`, `migrate`, `createsuperuser`, `startapp`).

## Project vs App — η βασική διάκριση

- **Project**: όλη η εγκατάσταση + ρυθμίσεις (`settings.py`, `urls.py`). Ένα ανά εφαρμογή.
- **App**: ένα αυτοτελές feature (blog, accounts, payments). **Πολλά** apps ανά project, και ιδανικά επαναχρησιμοποιήσιμα.

Αυτή η μοντελοποίηση σε σπρώχνει να σπας το project σε μικρά, εστιασμένα κομμάτια από την αρχή.

## MVT — το νοητικό μοντέλο

Το Django λέγεται **Model–View–Template** (το δικό του spin στο MVC):

- **Model**: τα δεδομένα (μια Python class → πίνακας βάσης).
- **View**: η λογική — δέχεται request, μιλάει με models, επιστρέφει response. (Προσοχή: το «view» του Django είναι ο controller άλλων frameworks!)
- **Template**: το HTML που βλέπει ο χρήστης.

```txt
Request → URLconf → View → Model (DB) → Template → Response
```

> Παγίδα ορολογίας: στο Django το **View = controller**, και το **Template = view** του κλασικού MVC.

## settings.py

Η καρδιά της ρύθμισης:

```python
# config/settings.py
INSTALLED_APPS = [
    "django.contrib.admin",
    "django.contrib.auth",
    # ...
    "blog",                 # το δικό σου app — πρέπει να δηλωθεί εδώ
]

DATABASES = {
    "default": {
        "ENGINE": "django.db.backends.postgresql",
        "NAME": "myapp",
        # ...
    }
}

DEBUG = True   # ΠΟΤΕ True σε production
```

Κάθε νέο app πρέπει να μπει στο `INSTALLED_APPS` για να το «δει» το Django.

## Η πρώτη ροή: URL → View → Template

```python
# blog/views.py
from django.shortcuts import render

def post_list(request):
    posts = ["Πρώτο post", "Δεύτερο post"]
    return render(request, "blog/post_list.html", {"posts": posts})
```

```python
# blog/urls.py
from django.urls import path
from . import views

urlpatterns = [
    path("posts/", views.post_list, name="post_list"),
]
```

```python
# config/urls.py
from django.urls import include, path

urlpatterns = [
    path("", include("blog.urls")),   # σύνδεσε τα urls του app
]
```

```html
<!-- blog/templates/blog/post_list.html -->
<ul>
{% for post in posts %}
    <li>{{ post }}</li>
{% endfor %}
</ul>
```

Πάτα `/posts/` → το URLconf βρίσκει το view → το view δίνει context στο template → ο χρήστης βλέπει HTML.

## Πώς απαντάς στη συνέντευξη

«How is a Django project structured?»

> Django splits a project — the settings and root URLconf — from apps, which are self-contained features you register in `INSTALLED_APPS`. It follows MVT: a request hits the URLconf, which routes to a view (the controller in classic MVC terms), the view talks to models via the ORM and renders a template. Configuration lives in `settings.py`, and I never run with `DEBUG=True` in production. The `manage.py` command-line tool drives everything — `runserver`, `migrate`, `startapp`, `createsuperuser`.
