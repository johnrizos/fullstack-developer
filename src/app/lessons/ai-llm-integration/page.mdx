# AI & LLM Integration

Το 2026, το «βάζω AI feature στην app» είναι mainstream fullstack δουλειά — chat, summarization, semantic search, AI-assisted forms. Δεν χρειάζεται να εκπαιδεύσεις μοντέλο: καλείς ένα **LLM API** (εδώ το Claude API της Anthropic) σαν οποιοδήποτε άλλο τρίτο API, με τη διαφορά ότι η απάντηση είναι κείμενο που παράγεται token-token. Αυτό το μάθημα δείχνει το πλήρες fullstack pattern: backend call, streaming στο UI, prompt design, tool use, RAG basics και cost/latency.

## Ο κανόνας #1: το key ΜΕΝΕΙ στο backend

Ένα LLM API key είναι μυστικό **με χρέωση**. Αν το βάλεις στο frontend, ο καθένας το βλέπει στο Network tab και ξοδεύει τα λεφτά σου. Άρα η αρχιτεκτονική είναι **πάντα**:

```txt
React UI  ──fetch──►  το ΔΙΚΟ σου backend (route handler)  ──API key──►  LLM API
          ◄──stream──                                       ◄──tokens──
```

Το frontend μιλά μόνο στο δικό σου server· ο server κρατά το key (σε env var) και μιλά στο LLM. Ποτέ `ANTHROPIC_API_KEY` σε client component ή σε `NEXT_PUBLIC_` μεταβλητή.

## Η πιο απλή κλήση (backend)

Με το επίσημο SDK, μια κλήση είναι ένα request/response. Το `client` διαβάζει το `ANTHROPIC_API_KEY` από το environment:

```ts
import Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic(); // key από env — ποτέ hardcoded

const response = await client.messages.create({
  model: "claude-opus-4-8",
  max_tokens: 1024,
  messages: [{ role: "user", content: "Σύνοψε σε μία πρόταση: ..." }],
});

// Το content είναι λίστα από blocks — διάλεξε τα text blocks
const text = response.content
  .filter((b) => b.type === "text")
  .map((b) => b.text)
  .join("");
```

Τρία πράγματα που ισχύουν σε **κάθε** LLM call:

- **`messages`** — η συνομιλία ως λίστα `{ role: "user" | "assistant", content }`. Το API είναι **stateless**: στέλνεις όλο το ιστορικό κάθε φορά (όπως το REST που είδες).
- **`max_tokens`** — ανώτατο όριο μήκους απάντησης (κόβει αν το ξεπεράσει).
- **`model`** — ποιο μοντέλο. Διάλεγε το πιο ικανό (`claude-opus-4-8`) εκτός αν θες φθηνότερο/ταχύτερο για απλά tasks.

## Streaming: γιατί και πώς

Μια μεγάλη απάντηση μπορεί να αργήσει δευτερόλεπτα. Αν περιμένεις όλο το κείμενο, ο χρήστης βλέπει «παγωμένη» οθόνη. Το **streaming** στέλνει την απάντηση **token-token**, ώστε το UI να τη γράφει σαν να «πληκτρολογείται» — η εμπειρία που ξέρεις από κάθε AI chat.

**Backend (Next.js route handler)** — κάνει stream από το LLM και προωθεί τα κομμάτια στον client:

```ts
// app/api/chat/route.ts
import Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic();

export async function POST(req: Request) {
  const { messages } = await req.json();

  const stream = client.messages.stream({
    model: "claude-opus-4-8",
    max_tokens: 1024,
    messages,
  });

  // Μετέτρεψε τα text deltas σε ReadableStream για τον browser
  const encoder = new TextEncoder();
  const body = new ReadableStream({
    async start(controller) {
      for await (const event of stream) {
        if (event.type === "content_block_delta" && event.delta.type === "text_delta") {
          controller.enqueue(encoder.encode(event.delta.text));
        }
      }
      controller.close();
    },
  });

  return new Response(body, { headers: { "Content-Type": "text/plain; charset=utf-8" } });
}
```

**Frontend (React)** — διαβάζει το stream και ενημερώνει το state σταδιακά:

```tsx
async function send(messages) {
  const res = await fetch("/api/chat", {
    method: "POST",
    body: JSON.stringify({ messages }),
  });

  const reader = res.body.getReader();
  const decoder = new TextDecoder();
  let text = "";

  while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    text += decoder.decode(value, { stream: true });
    setAnswer(text); // ενημερώνεται σε κάθε chunk → "πληκτρολογείται"
  }
}
```

> Πρόσεξε: το streaming αξιοποιεί ακριβώς τα concepts των προηγούμενων μαθημάτων — route handlers, `ReadableStream`, `fetch` με reader, React state. Το AI feature **είναι** fullstack.

## Prompt design: το input καθορίζει την ποιότητα

Σε ένα LLM, το «πώς ρωτάς» αλλάζει δραματικά το αποτέλεσμα. Δύο εργαλεία:

- **System prompt** — οδηγίες ρόλου/συμπεριφοράς που ισχύουν για όλη τη συνομιλία (τόνος, μορφή, όρια). Μπαίνει στο πεδίο `system`, ξεχωριστά από τα `messages`.
- **Σαφές, δομημένο user prompt** — λες τι θες, σε τι μορφή, με παραδείγματα όπου βοηθά.

```ts
const response = await client.messages.create({
  model: "claude-opus-4-8",
  max_tokens: 1024,
  system: "Είσαι βοηθός που απαντά πάντα στα Ελληνικά, σύντομα και χωρίς preamble.",
  messages: [{ role: "user", content: "Εξήγησε τι είναι το REST σε 2 προτάσεις." }],
});
```

Όταν χρειάζεσαι **μηχαναγνώσιμη** έξοδο (π.χ. για να γεμίσεις ένα form), μην κάνεις parsing ελεύθερου κειμένου — ζήτα **structured output** με schema, ώστε η απάντηση να είναι εγγυημένα έγκυρο JSON:

```ts
const response = await client.messages.create({
  model: "claude-opus-4-8",
  max_tokens: 1024,
  messages: [{ role: "user", content: "Βγάλε όνομα και email: 'Γιάννης, g@x.com'" }],
  output_config: {
    format: {
      type: "json_schema",
      schema: {
        type: "object",
        properties: { name: { type: "string" }, email: { type: "string" } },
        required: ["name", "email"],
        additionalProperties: false,
      },
    },
  },
});
```

## Tool use: όταν το μοντέλο πρέπει να «κάνει» κάτι

Από μόνο του το μοντέλο μόνο παράγει κείμενο — δεν ξέρει τον καιρό σου, ούτε διαβάζει τη βάση σου. Με **tool use** (γνωστό και ως function calling) του δίνεις «εργαλεία»: το μοντέλο **ζητά** να καλέσεις μια συνάρτηση, **εσύ** την εκτελείς, και του επιστρέφεις το αποτέλεσμα.

```txt
1. Ορίζεις tool: getWeather(city)         (όνομα + JSON schema)
2. Μοντέλο: "θέλω getWeather('Αθήνα')"     → stop_reason: "tool_use"
3. ΕΣΥ τρέχεις τη getWeather, παίρνεις 28°C
4. Επιστρέφεις tool_result → το μοντέλο γράφει την τελική απάντηση
```

```ts
const tools = [{
  name: "get_weather",
  description: "Καιρός για μια πόλη. Κάλεσέ το όταν ο χρήστης ρωτά για καιρό.",
  input_schema: {
    type: "object",
    properties: { city: { type: "string" } },
    required: ["city"],
  },
}];

const response = await client.messages.create({
  model: "claude-opus-4-8",
  max_tokens: 1024,
  tools,
  messages: [{ role: "user", content: "Τι καιρό κάνει στην Αθήνα;" }],
});

if (response.stop_reason === "tool_use") {
  // βρες το tool_use block, τρέξε τη συνάρτηση, στείλε tool_result πίσω...
}
```

Έτσι χτίζονται **agents**: το μοντέλο αποφασίζει ποια εργαλεία να χρησιμοποιήσει, εσύ ελέγχεις τι κάνει το καθένα (και βάζεις όρια ασφαλείας σε ό,τι είναι μη αναστρέψιμο).

## RAG basics: δώσε στο μοντέλο τα ΔΙΚΑ σου δεδομένα

Το μοντέλο δεν ξέρει το περιεχόμενο της εταιρείας/εφαρμογής σου. **RAG** (Retrieval-Augmented Generation) λύνει αυτό: αντί να «ξέρει», του **δίνεις** το σχετικό κείμενο μέσα στο prompt.

```txt
1. Ο χρήστης ρωτά κάτι.
2. Ψάχνεις στα δικά σου docs τα πιο σχετικά κομμάτια  (retrieval)
3. Τα βάζεις στο prompt ως context:
   "Με βάση ΜΟΝΟ αυτά τα έγγραφα: [...], απάντησε: <ερώτηση>"
4. Το μοντέλο απαντά γειωμένο στα δεδομένα σου  (augmented generation)
```

Το «πώς βρίσκω τα σχετικά κομμάτια» γίνεται συνήθως με **embeddings** + **vector search**: μετατρέπεις κείμενα σε vectors και βρίσκεις τα πιο «κοντινά» στην ερώτηση. Το κλειδί του RAG ως concept: **grounding** — η απάντηση βασίζεται σε πραγματικά, ελεγμένα δεδομένα αντί για ό,τι «θυμάται» το μοντέλο (μειώνει και τα hallucinations).

## Cost & latency: σκέψου το σαν metered API

Σε αντίθεση με ένα κανονικό API, εδώ πληρώνεις **ανά token** (input + output) και η απάντηση παίρνει **χρόνο** ανάλογο με το μήκος. Πρακτικοί κανόνες:

- **Διάλεξε μοντέλο ανά task** — μεγάλο/ικανό για σύνθετα, μικρότερο/φθηνότερο για classification ή απλά. Μην πληρώνεις premium για κάτι τετριμμένο.
- **Κράτα τα prompts λιτά** — λιγότερα input tokens = φθηνότερα + γρηγορότερα. Στείλε μόνο το αναγκαίο context (γι' αυτό το RAG επιλέγει *σχετικά* κομμάτια, όχι όλα).
- **Όρισε `max_tokens` λογικά** — μην αφήνεις την έξοδο να ξεφεύγει.
- **Stream** για μεγάλες/interactive απαντήσεις — δεν μειώνει το κόστος αλλά κρύβει το latency.
- **Cache** ό,τι επαναλαμβάνεται (σταθερά system prompts/context) όπου το υποστηρίζει το API.

## Πώς απαντάς στη συνέντευξη

> «How would you add an AI chat feature to a web app?»

> The API key stays server-side — the React client talks only to my own backend route, which holds the key and calls the LLM. For chat I stream: the model returns tokens incrementally, my route forwards them as a ReadableStream, and the client reads the stream and updates state so text appears as it's generated. I control quality with a system prompt and structured outputs when I need machine-readable JSON. For actions, I use tool use — the model requests a function call, my code executes it with validation, and returns the result. To ground answers in our own data I use RAG: retrieve the relevant documents, put them in the prompt as context, and the model answers from them. And I treat it as a metered API — pick the model per task, keep prompts lean, cap max_tokens, and cache stable context to manage cost and latency.
