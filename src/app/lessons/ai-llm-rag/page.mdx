# RAG: δώσε στο μοντέλο τα ΔΙΚΑ σου δεδομένα

Το μοντέλο δεν ξέρει το περιεχόμενο της εταιρείας/εφαρμογής σου, ούτε τα πιο πρόσφατα δεδομένα. **RAG** (Retrieval-Augmented Generation) λύνει αυτό: αντί να «ξέρει», του **δίνεις** το σχετικό κείμενο μέσα στο prompt τη στιγμή της ερώτησης. Είναι το πιο συχνό production AI pattern — chat πάνω στα docs σου, customer support, internal knowledge search. Αυτό το μάθημα εξηγεί το pattern, τα embeddings/vector search που το τροφοδοτούν, και τις παγίδες που ρωτάνε στις συνεντεύξεις.

## Το pattern σε 4 βήματα

```txt
1. Ο χρήστης ρωτά κάτι.
2. Ψάχνεις στα δικά σου docs τα πιο σχετικά κομμάτια      (retrieval)
3. Τα βάζεις στο prompt ως context:
   "Με βάση ΜΟΝΟ αυτά τα έγγραφα: [...], απάντησε: <ερώτηση>"
4. Το μοντέλο απαντά γειωμένο στα δεδομένα σου            (augmented generation)
```

Η ουσία του RAG ως concept είναι το **grounding**: η απάντηση βασίζεται σε πραγματικά, ελεγμένα δεδομένα αντί για ό,τι «θυμάται» το μοντέλο. Αυτό (α) μειώνει τα hallucinations, (β) δίνει ενημερωμένη γνώση χωρίς retraining, και (γ) σου επιτρέπει να δείξεις **πηγές** (citations).

## Πώς βρίσκεις τα «σχετικά» κομμάτια: embeddings + vector search

Το «βρες τα σχετικά κομμάτια» δεν γίνεται με keyword matching (χάνεις συνώνυμα και νόημα). Γίνεται με **σημασιολογική** αναζήτηση:

- **Embedding**: μετατρέπεις ένα κείμενο σε vector (λίστα αριθμών) που αποτυπώνει το **νόημά** του. Κείμενα με παρόμοιο νόημα έχουν «κοντινά» vectors.
- **Vector search**: μετατρέπεις και την ερώτηση σε vector και βρίσκεις τα πιο «κοντινά» chunks (cosine similarity). Αυτά είναι τα σχετικά.
- **Vector DB**: αποθηκεύεις τα vectors σε ειδική βάση (pgvector, Pinecone, Qdrant…) που κάνει γρήγορα nearest-neighbor search.

```txt
Offline (indexing, μία φορά):
  docs → chunks → embedding κάθε chunk → αποθήκευση vectors στο vector DB

Online (σε κάθε ερώτηση):
  ερώτηση → embedding → vector search (top-k κοντινά chunks) → context στο prompt
```

## Chunking: το «αόρατο» κρίσιμο βήμα

Δεν βάζεις ολόκληρα έγγραφα — τα σπας σε **chunks** (π.χ. παραγράφους ή ~500 tokens με μικρό overlap). Γιατί:

- **Ακρίβεια**: μικρότερα chunks → πιο στοχευμένο retrieval (το vector αποτυπώνει ένα θέμα, όχι 10).
- **Κόστος/όρια**: βάζεις στο prompt μόνο τα top-k σχετικά chunks, όχι όλη τη βάση — λιγότερα tokens.

Κακό chunking (πολύ μεγάλα ή πολύ μικρά chunks) είναι από τις πιο συχνές αιτίες κακού RAG.

## RAG δεν είναι fine-tuning

Συχνή ερώτηση συνέντευξης. **RAG** = δίνεις γνώση στο prompt τη στιγμή της ερώτησης (δυναμικό, φθηνό, ενημερώνεται αμέσως, δείχνει πηγές). **Fine-tuning** = αλλάζεις τα βάρη του μοντέλου με training (ακριβό, για στυλ/συμπεριφορά, όχι για «φρέσκα» δεδομένα). Για «να ξέρει τα docs μας» → RAG, σχεδόν πάντα.

## Πώς απαντάς στη συνέντευξη

> «How would you build a chatbot that answers questions about our internal docs?»

> RAG. Offline I chunk the documents, embed each chunk into a vector, and store the vectors in a vector database. At query time I embed the user's question, run a vector search to pull the top-k most semantically similar chunks, and put those into the prompt as context — "answer using only these documents." The model's answer is grounded in real, retrieved data instead of what it happens to remember, which cuts hallucinations, keeps knowledge current without retraining, and lets me cite sources. The details that make or break it are chunking — chunks small enough to be focused but with a little overlap — and retrieval quality. And to be clear, this is retrieval, not fine-tuning: fine-tuning changes the model's weights and is for style or behavior, not for feeding it fresh, frequently-changing facts.
