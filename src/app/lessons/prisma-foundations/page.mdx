# Prisma Foundations

Ένα Node API χωρίς αξιόπιστο data layer είναι demo. Στο TypeScript ecosystem, το **Prisma** είναι ο πιο διαδεδομένος type-safe ORM: ορίζεις schema, παίρνεις fully-typed client, και κάνεις migrations με version control. Στη συνέντευξη: «πώς μιλάς στη βάση από Node;», «τι είναι το N+1;», «πώς κάνεις schema changes με ασφάλεια;».

## Τι πρέπει να μπορείς στο τέλος

- να ορίζεις Prisma schema με models και relations,
- να κάνεις type-safe CRUD με τον Prisma Client,
- να αναγνωρίζεις και να λύνεις το **N+1 query problem**,
- να χρησιμοποιείς transactions για atomicity,
- να κάνεις schema migrations με ασφάλεια.

## Schema-first με Prisma

Ορίζεις το data model δηλωτικά· το Prisma παράγει typed client:

```prisma
// schema.prisma
model User {
  id    Int    @id @default(autoincrement())
  email String @unique
  name  String
  posts Post[]                    // relation: ένας user → πολλά posts
}

model Post {
  id       Int    @id @default(autoincrement())
  title    String
  author   User   @relation(fields: [authorId], references: [id])
  authorId Int
}
```

```bash
npx prisma migrate dev --name init   # φτιάχνει migration + ενημερώνει DB + client
```

Ο παραγόμενος client είναι **πλήρως typed** — το autocomplete ξέρει κάθε field, και λάθη πιάνονται στο compile time.

## Type-safe CRUD

```ts
import { PrismaClient } from "@prisma/client";
const prisma = new PrismaClient();

// CREATE
const user = await prisma.user.create({
  data: { email: "ana@x.com", name: "Ana" },
});

// READ — typed filters, typed result
const found = await prisma.user.findUnique({ where: { email: "ana@x.com" } });

// UPDATE
await prisma.user.update({ where: { id: 1 }, data: { name: "Ana B." } });

// DELETE
await prisma.user.delete({ where: { id: 1 } });

// Filtering, pagination, ordering
const posts = await prisma.post.findMany({
  where: { author: { email: "ana@x.com" } },
  orderBy: { id: "desc" },
  take: 10,
  skip: 0,
});
```

## Το N+1 Query Problem — η ερώτηση-κλειδί

Το πιο συχνό ORM performance bug. Όταν φέρνεις σχέσεις σε loop, γίνεται ένα query **ανά** εγγραφή:

```ts
// ❌ N+1: 1 query για users + 1 ΑΝΑ user για posts
const users = await prisma.user.findMany();          // 1 query
for (const user of users) {
  const posts = await prisma.post.findMany({          // +1 query ΑΝΑ user!
    where: { authorId: user.id },
  });
}
// 100 users = 101 queries 💥

// ✓ include — eager loading σε σταθερό αριθμό queries
const users = await prisma.user.findMany({
  include: { posts: true },                           // posts μαζί
});
// 2 queries συνολικά, ανεξάρτητα από πλήθος
```

Το `include` (ή `select`) φέρνει τα σχετιζόμενα δεδομένα μαζί. Το να αναγνωρίζεις το N+1 — 1 query για τη λίστα + N για τις σχέσεις — και να το λύνεις με eager loading είναι από τα πιο πρακτικά senior signals.

## Transactions — atomicity

Πολλαπλές γραφές που πρέπει να πετύχουν ή να αποτύχουν **μαζί**:

```ts
await prisma.$transaction(async (tx) => {
  await tx.account.update({ where: { id: from }, data: { balance: { decrement: 100 } } });
  await tx.account.update({ where: { id: to },   data: { balance: { increment: 100 } } });
  // αν πεταχτεί error εδώ → rollback ΚΑΙ των δύο, καμία μισή μεταφορά
});
```

Το `$transaction` εγγυάται ότι ή και οι δύο updates γράφονται ή καμία — ποτέ ενδιάμεση κατάσταση όπου αφαιρέθηκαν 100 χωρίς να προστεθούν.

## Migrations — schema as code

Όταν αλλάζει το schema, δεν πειράζεις τη βάση με το χέρι:

```bash
npx prisma migrate dev --name add_active_flag   # dev: φτιάχνει & εφαρμόζει migration
npx prisma migrate deploy                        # production: εφαρμόζει pending migrations
```

Τα migrations είναι **versioned αρχεία σε git**, εφαρμόζονται με **ίδια σειρά** σε κάθε environment (dev/staging/prod). Έτσι το schema είναι reproducible και τα changes γίνονται review σαν κώδικας — αντί για ad-hoc `ALTER TABLE` που ξεχνιούνται ή διαφέρουν ανά περιβάλλον.

> Connection pooling: σε serverless/πολλά instances, άνοιγε **έναν** PrismaClient (singleton), όχι νέο ανά request — αλλιώς εξαντλείς τα database connections. Σε serverless χρησιμοποιείς pooler (π.χ. PgBouncer/Prisma Accelerate).

## Πώς απαντάς στη συνέντευξη

«How do you handle persistence in a Node/TypeScript backend?»

> I use Prisma — I define the schema declaratively and get a fully typed client, so filters and results are checked at compile time. The pitfall I watch for is N+1: fetching a list and then loading each row's relation in a loop fires one query per row, so 100 users becomes 101 queries. I fix it with eager loading via `include` or `select`, which fetches the relations in a constant number of queries. Multi-step writes go in `$transaction` for atomicity, schema changes go through versioned migrations applied in the same order across environments rather than manual ALTERs, and I reuse a single PrismaClient instead of one per request so I don't exhaust the connection pool.
