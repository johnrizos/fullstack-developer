# Providers & Dependency Injection

Η DI είναι η καρδιά του Nest — και το πιο συχνό σημείο που μπερδεύει στη συνέντευξη. «Τι είναι provider;», «πώς ξέρει το Nest τι να κάνει inject;», «τι είναι dynamic module;». Εδώ πας πέρα από το `@Injectable()` και καταλαβαίνεις πώς δουλεύει πραγματικά το IoC container.

## Τι πρέπει να μπορείς στο τέλος

- να εξηγείς τι είναι provider και πώς γίνεται resolve,
- να χρησιμοποιείς custom providers (useClass/useValue/useFactory),
- να καταλαβαίνεις injection scopes (singleton vs request),
- να φτιάχνεις dynamic module με `forRoot`/`forRootAsync`,
- να στήνεις typed config με ConfigModule.

## Τι είναι provider

Provider είναι **οτιδήποτε μπορεί να γίνει inject**: services, repositories, factories, values. Το `@Injectable()` το δηλώνει διαθέσιμο στο DI container. Το `providers: []` του module το κάνει register.

```ts
@Injectable()
export class TaskService {
  constructor(private readonly repo: TaskRepository) {}  // inject by type
}

@Module({
  providers: [TaskService, TaskRepository],   // register στο container
  exports: [TaskService],                     // διαθέσιμο σε άλλα modules
})
export class TaskModule {}
```

Το Nest διαβάζει τους τύπους του constructor (μέσω `reflect-metadata`) και κάνει resolve το γράφημα εξαρτήσεων **μία φορά** στο bootstrap. Γι' αυτό χρειάζεσαι `emitDecoratorMetadata: true` στο tsconfig.

## Custom providers — πέρα από το σκέτο class

Όταν το token δεν είναι απλό class, χρησιμοποιείς το αναλυτικό object form:

```ts
@Module({
  providers: [
    TaskService,                                       // shorthand για { provide: TaskService, useClass: TaskService }
    { provide: "CONFIG", useValue: { maxItems: 100 } }, // useValue: σταθερή τιμή
    { provide: PaymentGateway, useClass: StripeGateway }, // useClass: swap implementation
    {
      provide: "DB_CONNECTION",                          // useFactory: dynamic construction
      useFactory: (config: ConfigService) => createPool(config.get("DB_URL")),
      inject: [ConfigService],                           // οι εξαρτήσεις του factory
    },
  ],
})
export class AppModule {}
```

Το `inject` λέει στο Nest τι να περάσει στο factory. Το `useClass` σε αφήνει να αλλάξεις implementation (π.χ. `StripeGateway` σε prod, `FakeGateway` σε test) **χωρίς** να αγγίξεις τους consumers.

## Injecting by token

Όταν inject-άρεις non-class token, χρησιμοποιείς `@Inject()`:

```ts
@Injectable()
export class TaskService {
  constructor(
    @Inject("CONFIG") private readonly config: AppConfig,
    @Inject("DB_CONNECTION") private readonly db: Pool,
  ) {}
}
```

## Injection scopes

```txt
DEFAULT (singleton)  → 1 instance για όλο το app. Default & ταχύτερο.
REQUEST              → νέα instance ανά request. Για per-request state
                       (π.χ. tenant context). Έχει performance κόστος.
TRANSIENT            → νέα instance σε κάθε inject point.
```

```ts
@Injectable({ scope: Scope.REQUEST })
export class RequestContext {
  constructor(@Inject(REQUEST) private readonly req: Request) {}
}
```

Vuln-trap: το REQUEST scope «μολύνει» — αν ένας singleton κάνει inject ένα request-scoped provider, γίνεται κι αυτός request-scoped (bubbling). Χρησιμοποίησέ το **μόνο** όπου πραγματικά χρειάζεσαι per-request state.

## Dynamic modules — forRoot

Όταν ένα module χρειάζεται **configuration**, εκθέτει `forRoot()` που επιστρέφει το module με providers φτιαγμένους από το config:

```ts
@Module({})
export class DatabaseModule {
  static forRoot(options: DbOptions): DynamicModule {
    return {
      module: DatabaseModule,
      providers: [{ provide: "DB_OPTIONS", useValue: options }, DatabaseService],
      exports: [DatabaseService],
    };
  }
}

// app.module.ts
@Module({ imports: [DatabaseModule.forRoot({ url: process.env.DB_URL! })] })
export class AppModule {}
```

Έτσι δουλεύουν τα `ConfigModule.forRoot()`, `JwtModule.forRoot()` κ.λπ. Η `forRootAsync()` παραλλαγή σε αφήνει να φτιάξεις το config **async** με `useFactory` + `inject`.

## Typed config με ConfigModule

```ts
@Module({
  imports: [ConfigModule.forRoot({ isGlobal: true, validationSchema: envSchema })],
})
export class AppModule {}

@Injectable()
export class TaskService {
  constructor(private readonly config: ConfigService) {}
  get max() { return this.config.get<number>("MAX_ITEMS"); }
}
```

Το `validationSchema` (Joi/Zod) απορρίπτει το boot αν λείπει env var — fail fast αντί για mysterious `undefined` σε production.

## Πώς απαντάς στη συνέντευξη

«Explain dependency injection in NestJS.»

> A provider is anything injectable — usually a service marked `@Injectable()` and registered in a module's `providers`. Nest reads the constructor parameter types via reflect-metadata and resolves the whole dependency graph once at bootstrap, so by default every provider is a singleton shared across the app. When the token isn't a plain class I use custom providers — `useValue` for constants, `useClass` to swap implementations between prod and test, and `useFactory` with `inject` for things I build at runtime like a DB pool. Scopes default to singleton; `REQUEST` scope gives a fresh instance per request but it bubbles up to everything that depends on it, so I keep it rare. For configurable modules I expose a `forRoot` that returns a DynamicModule, which is exactly how `ConfigModule` and `JwtModule` work, and I validate env with a schema so the app fails fast on missing config.
