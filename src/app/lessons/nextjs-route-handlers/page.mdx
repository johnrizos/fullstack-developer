# Route Handlers & APIs

Όταν χρειάζεσαι πραγματικά HTTP endpoints (webhooks, public API, third-party callbacks), φτιάχνεις **Route Handlers** με `route.ts`. Στα interviews εξετάζουν τη διαφορά από Server Actions, τα supported HTTP methods, και ότι χρησιμοποιούν τα Web `Request`/`Response` APIs.

## Τι πρέπει να μπορείς στο τέλος

- να φτιάχνεις `route.ts` με GET/POST/PUT/PATCH/DELETE,
- να διαβάζεις request data και να επιστρέφεις `Response.json`,
- να χρησιμοποιείς `NextRequest`/`NextResponse`,
- να τυποποιείς το context με `RouteContext`,
- να ξέρεις πότε route handler vs Server Action.

## Convention

Ένα `route.ts` μέσα στο `app/` ορίζει custom request handler. **Δεν** μπορεί να συνυπάρχει `route.ts` με `page.tsx` στο ίδιο segment.

```ts
// app/api/health/route.ts → GET /api/health
export async function GET() {
  return Response.json({ status: "ok" });
}
```

Route Handlers είναι το ισοδύναμο των παλιών API Routes (`pages/api`) — αλλά μέσα στο `app/`, χτισμένα πάνω στα Web `Request`/`Response`.

## Supported HTTP methods

```ts
// app/api/tasks/route.ts
export async function GET() { /* λίστα */ }
export async function POST(request: Request) {
  const body = await request.json();
  // create
  return Response.json({ id: "1", ...body }, { status: 201 });
}
```

Υποστηρίζονται `GET`, `POST`, `PUT`, `PATCH`, `DELETE`, `HEAD`, `OPTIONS`. Μη υποστηριζόμενο method → αυτόματα `405 Method Not Allowed`. Κάθε `route.ts` αναλαμβάνει όλα τα verbs του route του.

## NextRequest / NextResponse

Το Next.js επεκτείνει τα native με helpers:

```ts
import { type NextRequest, NextResponse } from "next/server";

export async function GET(request: NextRequest) {
  const query = request.nextUrl.searchParams.get("q");
  const token = request.cookies.get("token")?.value;
  return NextResponse.json({ query }, { status: 200 });
}
```

## Typed dynamic params: RouteContext

Για dynamic route handlers, υπάρχει global helper `RouteContext` (παράγεται σε `next dev`/`build`). Το `params` είναι **Promise** (await):

```ts
// app/users/[id]/route.ts
import type { NextRequest } from "next/server";

export async function GET(_req: NextRequest, ctx: RouteContext<"/users/[id]">) {
  const { id } = await ctx.params; // typed & await
  return Response.json({ id });
}
```

## Caching

Τα Route Handlers **δεν είναι cached by default**. Μπορείς να κάνεις opt-in caching μόνο για `GET` με `export const dynamic = "force-static"`. Με Cache Components, τα `GET` ακολουθούν το ίδιο μοντέλο prerendering με τα pages, και cache-άρεις uncached data με `use cache` σε helper (όχι απευθείας στο handler body).

```ts
// app/api/products/route.ts
import { cacheLife } from "next/cache";

export async function GET() {
  return Response.json(await getProducts());
}

async function getProducts() {
  "use cache"; // σε helper, ΟΧΙ μέσα στο GET απευθείας
  cacheLife("hours");
  return db.query("SELECT * FROM products");
}
```

## Πώς απαντάς στη συνέντευξη

«When do you use a Route Handler vs a Server Action?»

> Route Handlers are custom HTTP endpoints defined in a `route.ts` inside `app`, built on the Web `Request`/`Response` APIs — they're the App Router equivalent of API Routes. I export functions named after HTTP methods (`GET`, `POST`, …); an unsupported method returns 405, and a `route.ts` can't sit in the same segment as a `page.tsx`. For dynamic segments I type the context with the global `RouteContext<'/users/[id]'>` helper and `await ctx.params`. I reach for a Route Handler when I need a stable public URL with full control over request and response — webhooks, OAuth callbacks, a REST/JSON API for external clients. Server Actions are for in-app mutations triggered from my own forms and handlers, not as a public API. Route Handlers aren't cached by default; I opt in for `GET` with `force-static` or wrap uncached data in a `use cache` helper.
