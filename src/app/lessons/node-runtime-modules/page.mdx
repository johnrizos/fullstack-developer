# Node Runtime & Modules

Πριν γράψεις Express, πρέπει να καταλαβαίνεις **τι είναι** το Node: όχι «JavaScript στον server» σκέτο, αλλά ένα runtime χτισμένο γύρω από το V8 και ένα event loop με non-blocking I/O. Στις συνεντεύξεις, το «πώς τρέχει το Node χιλιάδες ταυτόχρονα requests με ένα thread;» και «CommonJS ή ESM;» είναι σχεδόν εγγυημένα.

## Τι πρέπει να μπορείς στο τέλος

- να εξηγείς το runtime: V8, libuv, event loop, single-threaded non-blocking I/O,
- να ξεχωρίζεις I/O-bound από CPU-bound και γιατί έχει σημασία,
- να καταλαβαίνεις τη σειρά: macrotasks, microtasks, `process.nextTick`,
- να διαλέγεις ανάμεσα σε CommonJS και ESM modules,
- να χρησιμοποιείς σωστά `process`, env vars και τα core modules.

## Το runtime: V8 + libuv + event loop

Το Node τρέχει JavaScript πάνω στο **V8** (ο engine της Chrome) και προσθέτει το **libuv** — μια C βιβλιοθήκη που δίνει το event loop και ένα thread pool για I/O. Το κρίσιμο σχεδιαστικό χαρακτηριστικό: **single-threaded JavaScript εκτέλεση με non-blocking I/O**.

Παραδοσιακοί servers δεσμεύουν ένα thread ανά request — αν το request περιμένει τη database, το thread κάθεται αδρανές. Το Node δουλεύει αλλιώς: όταν ένα request περιμένει I/O (database, αρχείο, δίκτυο), ο μοναδικός thread **συνεχίζει να εξυπηρετεί άλλα requests** και επιστρέφει όταν έρθει η απάντηση μέσω callback/promise.

Συνέπειες που πρέπει να αρθρώνεις:
- **Δυνατό σε I/O-bound φορτία**: APIs, proxies, real-time — χιλιάδες ταυτόχρονες συνδέσεις με λίγη μνήμη.
- **Αδύναμο σε CPU-bound δουλειά**: ένα βαρύ συγχρονισμένο loop (image processing, crypto) **μπλοκάρει το event loop** και παγώνει όλα τα requests. Λύση: `worker_threads` ή ξεχωριστό service.

## Event loop phases & σειρά εκτέλεσης

Το event loop έχει φάσεις (timers, I/O callbacks, poll, check, close). Δύο ουρές «παρακάμπτουν» τη σειρά:
- **microtasks** (resolved promises, `queueMicrotask`) — τρέχουν μετά από κάθε operation, **πριν** τα timers.
- **`process.nextTick`** — προτεραιότητα ακόμη πιο πάνω, πριν τα microtasks.

```js
console.log("1: sync");
setTimeout(() => console.log("2: timeout (macrotask)"), 0);
Promise.resolve().then(() => console.log("3: promise (microtask)"));
process.nextTick(() => console.log("4: nextTick"));
console.log("5: sync");
// 1, 5, 4, 3, 2
```

Η λογική: όλος ο sync κώδικας πρώτα (1, 5)· μετά `process.nextTick` (4)· μετά microtasks/promises (3)· τέλος macrotasks/timers (2).

## CommonJS vs ESM

Δύο module systems συνυπάρχουν στο Node:

```js
// CommonJS (CJS) — το ιστορικό default, synchronous require
const fs = require("fs");
module.exports = { add };

// ES Modules (ESM) — το standard, static import/export
import fs from "node:fs";
export function add(a, b) { return a + b; }
```

Πώς διαλέγει το Node: `.mjs` = ESM, `.cjs` = CJS, `.js` ακολουθεί το `"type"` στο `package.json` (`"module"` = ESM, `"commonjs"` ή απών = CJS).

Διαφορές που πρέπει να ξέρεις:
- **ESM είναι static**: τα imports αναλύονται πριν τρέξει ο κώδικας → επιτρέπει tree-shaking, top-level `await`. Το CJS `require` είναι dynamic, runtime.
- **ESM είναι το μέλλον**: νέα projects ξεκινούν ESM. Αλλά πολλά packages/legacy κώδικας είναι ακόμη CJS, άρα πρέπει να ξέρεις και τα δύο.
- Στο ESM δεν υπάρχουν `__dirname`/`require` — χρησιμοποιείς `import.meta.url`.

## `process`, env vars & core modules

```js
process.env.NODE_ENV          // environment config — ΠΟΤΕ secrets hardcoded
process.argv                  // command-line arguments
process.exit(1)               // exit με κωδικό (1 = error)
process.on("SIGTERM", ...)    // graceful shutdown signal

// Core modules — χρησιμοποίησε το node: prefix (σαφές ότι είναι built-in)
import { readFile } from "node:fs/promises";
import path from "node:path";
import crypto from "node:crypto";
```

Τα env vars είναι ο σωστός τρόπος για config: τα secrets και οι ρυθμίσεις έρχονται από το περιβάλλον (`.env` τοπικά, secrets manager σε production), ποτέ μέσα στον κώδικα ή στο Git.

## Πώς απαντάς στη συνέντευξη

«How does Node handle thousands of concurrent requests with a single thread?»

> Node runs my JavaScript on a single thread via V8, but I/O is non-blocking thanks to libuv and the event loop. When a request waits on the database, a file, or the network, the thread doesn't sit idle — it serves other requests and comes back via a callback or promise when the data is ready. That's why Node excels at I/O-bound workloads like APIs and real-time. The flip side is CPU-bound work: a heavy synchronous loop blocks the event loop and freezes every request, so I move that to worker threads or a separate service. And I keep the execution order straight — sync code first, then `process.nextTick`, then promise microtasks, then timers and other macrotasks.
