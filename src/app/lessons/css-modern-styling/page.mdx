# Styling at Scale

Ξέρεις box model, flexbox και grid — αλλά σε πραγματικό project δεν γράφεις CSS «όπως-όπως» σε ένα τεράστιο `styles.css`. Οι ομάδες χρειάζονται styling που **κλιμακώνεται**: να μην συγκρούεται, να είναι συνεπές, να αλλάζει θέμα με μια μεταβλητή. Αυτό το μάθημα είναι ο χάρτης: το πρόβλημα του CSS at scale, οι **δύο φιλοσοφίες** frameworks (utility-first vs component-first), και τα cross-cutting εργαλεία — CSS Modules, design tokens, dark mode. Τα δύο frameworks τα δουλεύεις σε βάθος στα επόμενα μαθήματα (Tailwind, Bootstrap).

## Το πρόβλημα: CSS at scale

Το «κανονικό» CSS έχει δύο πόνους σε μεγάλα projects:

- **Global scope:** κάθε class είναι global. Ένα `.card` σε δύο αρχεία συγκρούεται· αλλάζεις κάτι εδώ, σπάει κάτι αλλού. Φόβος να σβήσεις CSS («μήπως το χρησιμοποιεί κάτι;»).
- **Naming fatigue:** ξοδεύεις χρόνο σε ονόματα (`.user-card__header--active`) και πηγαινοέρχεσαι ανάμεσα σε HTML και CSS αρχείο.

Υπάρχουν τρεις οικογένειες λύσεων: **CSS frameworks** (Tailwind, Bootstrap), **scoped CSS** (CSS Modules / CSS-in-JS), και ο συνδυασμός τους με **design tokens**.

## Οι δύο φιλοσοφίες των frameworks

Τα δύο κυρίαρχα CSS frameworks λύνουν το ίδιο πρόβλημα με **αντίθετο** τρόπο:

```txt
Utility-first (Tailwind)            Component-first (Bootstrap)
────────────────────────            ───────────────────────────
μικρά "τουβλάκια" → χτίζεις          έτοιμα components → συναρμολογείς
px-4 py-2 bg-blue-600 rounded-lg     btn btn-primary
custom design, πλήρης έλεγχος        γρήγορο, συνεπές, "έτοιμο"
```

```html
<!-- Tailwind: συνθέτεις το κουμπί από utilities -->
<button class="px-4 py-2 bg-blue-600 text-white rounded-lg hover:bg-blue-700">OK</button>

<!-- Bootstrap: λες "είναι ένα primary button" -->
<button class="btn btn-primary">OK</button>
```

Κανένα δεν είναι «καλύτερο» — είναι **fit**: Tailwind όταν θες μοναδικό, custom design με έλεγχο· Bootstrap όταν θες να βγάλεις γρήγορα ένα λειτουργικό, συνεπές UI χωρίς custom design. Το καθένα έχει το δικό του πλήρες μάθημα παρακάτω.

## Scoped CSS: CSS Modules

Η τρίτη προσέγγιση, ανεξάρτητη από framework: γράφεις **κανονικό CSS**, αλλά οι classes γίνονται **τοπικές** σε ένα component — μηδέν συγκρούσεις.

```css
/* Button.module.css */
.button { padding: 8px 16px; background: #2563eb; color: white; border-radius: 8px; }
```

```jsx
import styles from "./Button.module.css";
// το styles.button γίνεται κάτι σαν "Button_button__x7f9" — εγγυημένα μοναδικό
<button className={styles.button}>Αποθήκευση</button>
```

Πλεονέκτημα: κρατάς γνωστό CSS και πλήρη ελευθερία, με scope ασφάλεια. Μειονέκτημα: ξανά naming + δύο αρχεία. Υπάρχει και **CSS-in-JS** (styled-components/Emotion) με παρόμοια ιδέα, αλλά τα tradeoffs (runtime cost) έχουν στρέψει πολλούς προς Tailwind ή zero-runtime λύσεις.

## Design tokens: μία πηγή αλήθειας για το design

**Design tokens** είναι named μεταβλητές για χρώματα, spacing, typography — η «γλώσσα» του design system σου. Δουλεύουν με **όλες** τις παραπάνω προσεγγίσεις. Αντί για σκόρπια hex codes, ορίζεις σημασιολογικά ονόματα μία φορά:

```css
:root {
  --color-primary: #2563eb;
  --color-bg: #ffffff;
  --color-text: #111827;
  --space-md: 16px;
  --radius: 8px;
}

.button { background: var(--color-primary); border-radius: var(--radius); }
```

Όφελος: αλλάζεις το brand χρώμα **σε ένα σημείο** και ενημερώνεται όλη η εφαρμογή. Στο Tailwind τα tokens ζουν στο `theme` config· στο Bootstrap σε Sass/CSS variables — αλλά η ιδέα είναι ίδια παντού.

## Dark mode — η killer εφαρμογή των tokens

Επειδή τα χρώματα είναι μεταβλητές, το dark mode γίνεται **αλλαγή τιμών**, όχι ξαναγράψιμο:

```css
:root            { --color-bg: #ffffff; --color-text: #111827; }
[data-theme="dark"] { --color-bg: #0b0f19; --color-text: #e5e7eb; }

body { background: var(--color-bg); color: var(--color-text); }
```

```js
// Toggle: άλλαξε ένα attribute, όλα τα tokens αλλάζουν μαζί
document.documentElement.setAttribute("data-theme", "dark");
```

Το μυστικό: **σημασιολογικά tokens** (`--color-bg`, όχι `--white`) ώστε το ίδιο όνομα να σημαίνει «φόντο» σε κάθε theme.

## Πρακτικός κανόνας επιλογής

```txt
Custom, μοναδικό design με πλήρη έλεγχο      → Tailwind (δες το επόμενο μάθημα)
Γρήγορο admin/prototype/internal tool         → Bootstrap (δες το μάθημα του)
Θέλω κανονικό CSS με scope ασφάλεια           → CSS Modules
Component library με dynamic theming           → CSS-in-JS ή Tailwind
Σε ΟΛΑ: design tokens + σημασιολογικά ονόματα για theming/dark mode
```

## Πώς απαντάς στη συνέντευξη

> «How do you handle styling in a large app?»

> I avoid global CSS sprawl. The two dominant frameworks take opposite approaches: utility-first with Tailwind, where styling lives in the markup composed from a fixed design scale, best for a custom design with full control; and component-first with Bootstrap, prebuilt components and a grid that ship a consistent UI fast, best for admin panels and prototypes. When a team prefers writing real CSS, CSS Modules give local scoping so classes never collide. Whatever the approach, I drive everything from design tokens — semantic variables like `--color-bg` or theme config — so rebranding or adding dark mode is changing values in one place, not editing every component.
