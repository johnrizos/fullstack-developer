# Components, Props & Composition

Τα components είναι οι δομικοί λίθοι, και τα **props είναι το δημόσιο API τους**. Στα interviews δεν ρωτάνε «τι είναι component» — ρωτάνε αν ξέρεις να σχεδιάζεις καθαρά contracts, πότε να κάνεις composition αντί για configuration, και πώς ρέουν τα δεδομένα. Όλα typed με TypeScript από την αρχή.

## Τι πρέπει να μπορείς στο τέλος

- να γράφεις typed props ως ρητό contract,
- να χρησιμοποιείς `children` και composition αντί για ατελείωτα boolean flags,
- να εξηγείς one-way data flow (data κάτω, events πάνω),
- να ξεχωρίζεις presentational από container components,
- να αποφεύγεις prop drilling με composition.

## Typed props — το contract

```tsx
type ButtonProps = {
  label: string;
  onClick: () => void;
  variant?: "primary" | "secondary"; // union, όχι σκέτο string
  disabled?: boolean;
};

function Button({ label, onClick, variant = "primary", disabled = false }: ButtonProps) {
  return (
    <button className={variant} onClick={onClick} disabled={disabled}>
      {label}
    </button>
  );
}
```

Το TypeScript κάνει το contract ρητό: ποια props είναι required, ποια optional, ποιες τιμές επιτρέπονται. Το `variant?: "primary" | "secondary"` πιάνει typos στο compile time — αδύνατο με σκέτο `string`.

## children & composition

Αντί να περνάς δεκάδες props για να παραμετροποιήσεις ένα component, **σύνθεσέ** το με `children`:

```tsx
type CardProps = {
  title: string;
  children: React.ReactNode; // οτιδήποτε renderable
};

function Card({ title, children }: CardProps) {
  return (
    <section className="card">
      <h3>{title}</h3>
      <div className="card-body">{children}</div>
    </section>
  );
}

// Χρήση: το περιεχόμενο μπαίνει «μέσα», όχι μέσω props
<Card title="Profile">
  <Avatar user={user} />
  <p>{user.bio}</p>
</Card>;
```

Το `React.ReactNode` είναι ο τύπος για οτιδήποτε μπορεί να renderάρει το React: JSX, strings, numbers, arrays, `null`.

## One-way data flow

Τα δεδομένα ρέουν **προς τα κάτω** μέσω props· τα events **προς τα πάνω** μέσω callbacks. Ένα child δεν αλλάζει ποτέ απευθείας το state του parent — ζητάει την αλλαγή μέσω callback.

```tsx
function TodoList({ todos, onToggle }: { todos: Todo[]; onToggle: (id: string) => void }) {
  return (
    <ul>
      {todos.map((todo) => (
        // data ↓ (todo),  event ↑ (onToggle)
        <TodoItem key={todo.id} todo={todo} onToggle={onToggle} />
      ))}
    </ul>
  );
}
```

Αυτό κάνει τη ροή προβλέψιμη: για να βρεις «ποιος αλλάζει αυτό το δεδομένο», ανεβαίνεις το δέντρο μέχρι το state. Το state ζει στον χαμηλότερο κοινό πρόγονο όσων το χρειάζονται (lifting state up).

## Presentational vs container

```tsx
// Container: ξέρει ΑΠΟ ΠΟΥ έρχονται τα δεδομένα
function UserProfileContainer({ userId }: { userId: string }) {
  const { data, loading } = useUser(userId);
  if (loading) return <Spinner />;
  return <UserProfile user={data} />;
}

// Presentational: ξέρει μόνο ΠΩΣ φαίνονται — εύκολα testable & reusable
function UserProfile({ user }: { user: User }) {
  return <h2>{user.name}</h2>;
}
```

Ο διαχωρισμός κάνει τα presentational components καθαρά (props in → JSX out), άρα εύκολα να τα τεστάρεις και να τα ξαναχρησιμοποιείς.

## Πώς απαντάς στη συνέντευξη

«How do you decide between props and composition?»

> Props are the component's contract — I type them explicitly, use unions instead of bare strings, and mark optionals. But when I notice a growing pile of boolean flags configuring layout or content, that's a smell: the component is doing too much. I switch to composition, exposing slots via `children` or sub-components, so the caller assembles what they need and the component stays open to new requirements without new props. Data always flows down through props and events flow up through callbacks, so a child never mutates a prop — it calls a callback and the state owner applies the change. I keep presentational components pure (props in, JSX out) and push data-fetching into container components, which makes the presentational ones trivial to test and reuse.
