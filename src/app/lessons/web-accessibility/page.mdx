# Accessibility (a11y)

Accessibility σημαίνει: η εφαρμογή σου δουλεύει για **όλους** — όσους χρησιμοποιούν screen reader, πλοηγούνται μόνο με πληκτρολόγιο, έχουν χαμηλή όραση ή κινητικές δυσκολίες. Δεν είναι «extra»: είναι ποιότητα, συχνά **νομική απαίτηση** (WCAG/EU Accessibility Act), και ερώτηση που ξεχωρίζει σοβαρούς frontend developers. Το καλό νέο: το 90% κερδίζεται με σωστό HTML.

## Το θεμέλιο: semantic HTML κάνει τη μισή δουλειά

Πριν αγγίξεις ARIA, θυμήσου ότι τα semantic elements (που είδες στο semantics μάθημα) είναι **εγγενώς προσβάσιμα**:

```html
<!-- ❌ "div soup": ο screen reader δεν ξέρει ότι είναι κουμπί -->
<div onclick="submit()">Αποθήκευση</div>

<!-- ✅ native button: focusable, λειτουργεί με Enter/Space, ανακοινώνεται ως "button" -->
<button onclick="submit()">Αποθήκευση</button>
```

Ένα native `<button>`, `<a>`, `<input>`, `<nav>` σου δίνει **δωρεάν**: keyboard focus, σωστό role, states, και τις σωστές αλληλεπιδράσεις. Ο πρώτος κανόνας του ARIA είναι: «**μη χρησιμοποιείς ARIA** αν υπάρχει native element που το κάνει».

## Keyboard navigation — δοκίμασέ το χωρίς ποντίκι

Πολλοί χρήστες δεν χρησιμοποιούν ποντίκι. Όλα τα interactive πρέπει να δουλεύουν με πληκτρολόγιο:

- **`Tab`** / `Shift+Tab` → μετακίνηση μεταξύ focusable elements.
- **`Enter` / `Space`** → ενεργοποίηση (links/buttons).
- **`Esc`** → κλείσιμο modal/menu.
- **βελάκια** → μέσα σε menus, tabs, radio groups.

Το **tab order** ακολουθεί τη σειρά του DOM. Native elements (`button`, `a[href]`, `input`) είναι focusable αυτόματα. Ένα `<div>` που έκανες «κουμπί» **δεν** είναι — γι' αυτό προτιμάς native.

> Γρήγορο test: κλείσε το ποντίκι. Μπορείς να κάνεις τα πάντα με Tab/Enter/Esc; Βλέπεις πάντα **πού** είσαι (focus ring); Αν όχι, η σελίδα δεν είναι προσβάσιμη.

Ποτέ μην κάνεις `outline: none` χωρίς αντικατάσταση — αφαιρείς τον focus indicator, που είναι ζωτικός:

```css
/* ❌ αφαιρεί το focus ring — ο keyboard χρήστης χάνεται */
button:focus { outline: none; }

/* ✅ custom αλλά ορατό focus */
button:focus-visible { outline: 2px solid #2563eb; outline-offset: 2px; }
```

## ARIA — όταν το HTML δεν φτάνει

**ARIA** (Accessible Rich Internet Applications) είναι attributes που δίνουν σημασία/κατάσταση σε στοιχεία όταν δεν υπάρχει native element. Τρεις κατηγορίες:

- **Roles** — τι είναι κάτι: `role="dialog"`, `role="alert"`, `role="tab"`.
- **Properties** — σχέσεις/ιδιότητες: `aria-label`, `aria-labelledby`, `aria-describedby`.
- **States** — δυναμική κατάσταση: `aria-expanded`, `aria-checked`, `aria-hidden`, `aria-busy`.

```html
<!-- Icon button χωρίς ορατό κείμενο: δώσε accessible name -->
<button aria-label="Κλείσιμο">✕</button>

<!-- Dropdown: ανακοίνωσε αν είναι ανοιχτό -->
<button aria-expanded="false" aria-controls="menu">Μενού</button>
<ul id="menu" hidden>...</ul>
```

**Χρυσός κανόνας:** το ARIA **δεν** προσθέτει συμπεριφορά — μόνο πληροφορία. Το `role="button"` σε `<div>` δεν το κάνει focusable ούτε να δουλεύει με Enter· πρέπει να τα προσθέσεις μόνος σου (`tabindex="0"`, key handlers). Γι' αυτό native > ARIA.

## Focus management — το κλειδί για modals/menus

Όταν ανοίγει ένα modal, ο focus πρέπει να **πάει μέσα** του και να **παγιδεύεται** (focus trap) — αλλιώς ο keyboard χρήστης συνεχίζει να «ταμπάρει» πίσω στη σελίδα από κάτω. Όταν κλείνει, ο focus επιστρέφει στο κουμπί που το άνοιξε.

```txt
Άνοιγμα modal:
  1. αποθήκευσε ποιο element είχε focus (το trigger)
  2. μετακίνησε focus στο modal (ή στο πρώτο focusable μέσα)
  3. Tab μέσα στο modal κυκλικά (trap) — Esc κλείνει
Κλείσιμο:
  4. επίστρεψε focus στο trigger
```

Αυτό το pattern είναι ο νο1 λόγος που τα custom dropdowns/modals «σπάνε» για χρήστες πληκτρολογίου.

## Forms accessibility — όπου χάνονται τα περισσότερα

Οι φόρμες είναι το #1 σημείο όπου σπάει η προσβασιμότητα. Τέσσερα πράγματα που πρέπει να κάνεις σωστά:

**1. Κάθε input με συνδεδεμένο `<label>`** — ο screen reader διαβάζει το label όταν ο χρήστης φτάνει στο πεδίο. Το placeholder **δεν** αντικαθιστά το label (εξαφανίζεται μόλις γράψεις, δεν ανακοινώνεται αξιόπιστα):

```html
<!-- ❌ μόνο placeholder: κανένα accessible name -->
<input type="email" placeholder="Email" />

<!-- ✅ συνδεδεμένο label (for ↔ id) -->
<label for="email">Email</label>
<input id="email" type="email" />
```

**2. Σύνδεσε το error με το input** — αλλιώς ο screen reader χρήστης ακούει «invalid» χωρίς να ξέρει γιατί ή πού. Χρησιμοποίησε `aria-describedby` (δείχνει στο μήνυμα) + `aria-invalid`:

```html
<label for="pwd">Κωδικός</label>
<input id="pwd" type="password" aria-invalid="true" aria-describedby="pwd-err" />
<p id="pwd-err">Ο κωδικός θέλει τουλάχιστον 8 χαρακτήρες.</p>
```

**3. Ομαδοποίησε σχετικά πεδία με `fieldset` + `legend`** — π.χ. radio buttons «Τρόπος πληρωμής»: το `legend` δίνει κοινό context σε όλη την ομάδα.

**4. Ανακοίνωσε errors δυναμικά** — βάλε το error summary σε `aria-live="assertive"` (ή `role="alert"`) ώστε να ακουστεί μόλις αποτύχει το submit, χωρίς να μετακινηθεί ο focus απότομα.

## Άλλα essentials

- **Alt text:** κάθε ουσιαστική `<img>` θέλει `alt` που περιγράφει το νόημα. Διακοσμητική εικόνα → `alt=""` (την αγνοεί ο screen reader).
- **Color contrast:** το κείμενο χρειάζεται αρκετή αντίθεση (WCAG AA: 4.5:1 για normal text). Μην βασίζεσαι **μόνο** στο χρώμα για να μεταφέρεις πληροφορία (π.χ. «κόκκινο = error» + εικονίδιο/κείμενο).
- **Headings hierarchy:** σωστά `h1→h2→h3` — οι screen readers πλοηγούνται από heading σε heading.
- **`aria-live`:** για δυναμικά μηνύματα (toast, validation) ώστε ο screen reader να τα ανακοινώνει χωρίς να αλλάξει focus.
- **Skip link & landmarks:** ένα «Skip to content» link (πρώτο focusable, ορατό σε focus) επιτρέπει στον keyboard χρήστη να προσπεράσει το nav. Τα landmark elements (`header`, `nav`, `main`, `footer`) δίνουν στους screen readers γρήγορη πλοήγηση ανά περιοχή.
- **`prefers-reduced-motion`:** σεβάσου τους χρήστες που έχουν ζητήσει λιγότερη κίνηση (vestibular disorders) — μείωσε/απενεργοποίησε animations σε αυτό το media query (το είδες στο transitions/animations μάθημα).

```css
@media (prefers-reduced-motion: reduce) {
  *, *::before, *::after { animation-duration: 0.01ms !important; transition-duration: 0.01ms !important; }
}
```

## WCAG — το πρότυπο σε 4 λέξεις: POUR

Το **WCAG** οργανώνει την προσβασιμότητα σε 4 αρχές (**POUR**):

- **Perceivable** — μπορείς να το αντιληφθείς (alt text, contrast, captions).
- **Operable** — μπορείς να το χειριστείς (keyboard, αρκετός χρόνος, όχι flashing).
- **Understandable** — είναι κατανοητό (συνεπές, καθαρά errors).
- **Robust** — δουλεύει με assistive tech (valid HTML, σωστά ARIA).

Επίπεδα συμμόρφωσης: A → **AA** (ο συνήθης στόχος, νομική βάση) → AAA.

## Πώς απαντάς στη συνέντευξη

> «How do you make a web app accessible?»

> I start with semantic HTML — native button, a, input, nav give focus, roles and keyboard behaviour for free, so I avoid div soup. I make sure everything works with the keyboard alone: logical tab order, visible focus with focus-visible, Enter/Space/Esc, and focus trapping in modals with focus returned to the trigger on close. I add ARIA only when there's no native equivalent — roles, aria-label for icon buttons, aria-expanded for disclosure, aria-live for dynamic messages — knowing ARIA describes but doesn't add behaviour. Forms get special care: associated labels, errors tied to the input with aria-describedby and aria-invalid, and groups in fieldset/legend. Then the essentials: alt text, AA colour contrast, heading hierarchy, a skip link with landmarks, and respecting prefers-reduced-motion — and I test with keyboard, a screen reader and an axe/Lighthouse audit against WCAG.
