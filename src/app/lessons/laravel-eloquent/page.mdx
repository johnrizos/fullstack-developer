# Eloquent ORM & Migrations

Το **Eloquent** είναι ο ORM του Laravel: κάθε πίνακας της βάσης γίνεται μια PHP class (model), και κάθε γραμμή ένα αντικείμενο. Γράφεις PHP αντί για SQL — αλλά πρέπει να ξέρεις τι συμβαίνει από κάτω.

## Migrations — version control για το schema

```bash
php artisan make:migration create_posts_table
```

```php
<?php
// database/migrations/...create_posts_table.php
public function up(): void {
    Schema::create("posts", function (Blueprint $table) {
        $table->id();
        $table->string("title");
        $table->text("body");
        $table->foreignId("user_id")->constrained()->cascadeOnDelete();
        $table->timestamps();   // created_at, updated_at
    });
}

public function down(): void {
    Schema::dropIfExists("posts");
}
```

```bash
php artisan migrate          # εφάρμοσε
php artisan migrate:rollback  # ανέβα πίσω
```

Οι migrations είναι **version control για τη βάση**: όλη η ομάδα τρέχει τα ίδια migrations και έχει πανομοιότυπο schema.

## Models

```bash
php artisan make:model Post
```

```php
<?php
namespace App\Models;

use Illuminate\Database\Eloquent\Model;

class Post extends Model {
    // mass-assignment protection: ποια πεδία επιτρέπεται να γεμίσουν μαζικά
    protected $fillable = ["title", "body", "user_id"];

    // casts: μετατροπές τύπων
    protected $casts = ["published_at" => "datetime"];
}
```

## CRUD με Eloquent

```php
<?php
// create
$post = Post::create(["title" => "Γεια", "body" => "...", "user_id" => 1]);

// read
$all = Post::all();
$one = Post::find(1);
$one = Post::findOrFail(1);          // 404 αν δεν υπάρχει
$recent = Post::where("user_id", 1)
    ->orderBy("created_at", "desc")
    ->limit(10)
    ->get();

// update
$post->update(["title" => "Νέος τίτλος"]);

// delete
$post->delete();
```

Το `$fillable` είναι **σημαντικό για ασφάλεια**: χωρίς αυτό, ένας χρήστης θα μπορούσε με `create($request->all())` να γεμίσει πεδία που δεν έπρεπε (π.χ. `is_admin`). Αυτό λέγεται mass-assignment vulnerability.

## Relationships

```php
<?php
class User extends Model {
    public function posts() {
        return $this->hasMany(Post::class);     // ένας user → πολλά posts
    }
}

class Post extends Model {
    public function user() {
        return $this->belongsTo(User::class);   // ένα post → ένας user
    }

    public function tags() {
        return $this->belongsToMany(Tag::class); // many-to-many
    }
}
```

```php
<?php
$user = User::find(1);
foreach ($user->posts as $post) {   // χρήση σαν property
    echo $post->title;
}
echo $post->user->name;             // αντίστροφα
```

## Το N+1 problem — η #1 παγίδα performance

```php
<?php
// ❌ N+1: 1 query για posts + 1 query ΑΝΑ post για τον user = 101 queries
$posts = Post::all();
foreach ($posts as $post) {
    echo $post->user->name;   // κρυφό query σε κάθε επανάληψη!
}

// ✅ eager loading: 2 queries συνολικά
$posts = Post::with("user")->get();
foreach ($posts as $post) {
    echo $post->user->name;   // ήδη φορτωμένο
}
```

Το `with()` (**eager loading**) φορτώνει τις σχέσεις με 1 επιπλέον query αντί για ένα ανά γραμμή. Αυτό ρωτιέται σχεδόν σε κάθε Laravel συνέντευξη.

## Query Builder & aggregates

```php
<?php
$count = Post::where("published", true)->count();
$avg = Post::avg("views");
$grouped = Post::selectRaw("user_id, count(*) as total")
    ->groupBy("user_id")
    ->get();
```

## Πώς απαντάς στη συνέντευξη

«What is the N+1 problem and how do you fix it?»

> N+1 happens when you load a list, then access a relationship inside a loop — Eloquent fires one extra query per row, so 100 posts become 101 queries. The fix is eager loading with `with('user')`, which fetches all the related records in a single extra query. I define relationships as `hasMany`/`belongsTo`/`belongsToMany` methods, protect mass assignment with `$fillable`, and use migrations as version control for the schema so the whole team shares an identical database structure.
