# SQL Queries & Joins

Το SQL είναι **declarative**: λες _τι_ θες, όχι _πώς_ να το φέρει. Στα interviews γράφεις queries στον πίνακα και εξηγείς joins, aggregations και subqueries. Είναι από τα πιο εξεταζόμενα πρακτικά skills.

## Τι πρέπει να μπορείς στο τέλος

- να γράφεις SELECT με WHERE, ORDER BY, LIMIT,
- να ξεχωρίζεις INNER/LEFT/RIGHT/FULL JOIN,
- να κάνεις GROUP BY με aggregations και HAVING,
- να γράφεις subqueries και CTEs,
- να καταλαβαίνεις τη σειρά εκτέλεσης ενός query.

## SELECT — τα βασικά

```sql
SELECT id, email, created_at
FROM users
WHERE created_at >= '2026-01-01'
ORDER BY created_at DESC
LIMIT 20;
```

## JOIN types — η καρδιά του relational

```sql
-- INNER: μόνο rows που ταιριάζουν ΚΑΙ στους δύο πίνακες
SELECT o.id, u.email
FROM orders o
INNER JOIN users u ON u.id = o.user_id;

-- LEFT: ΟΛΑ τα orders, + user αν υπάρχει (αλλιώς NULL στις στήλες του user)
SELECT o.id, u.email
FROM orders o
LEFT JOIN users u ON u.id = o.user_id;
```

```txt
INNER JOIN  → τομή: rows που ταιριάζουν και στους δύο
LEFT JOIN   → όλα τα αριστερά + ταιριάσματα δεξιά (NULL αν δεν υπάρχει)
RIGHT JOIN  → όλα τα δεξιά + ταιριάσματα αριστερά
FULL JOIN   → όλα από παντού, NULL όπου δεν ταιριάζει
```

## GROUP BY & aggregations

Aggregation functions (`COUNT`, `SUM`, `AVG`, `MIN`, `MAX`) μαζί με `GROUP BY` συνοψίζουν δεδομένα:

```sql
SELECT u.id, COUNT(o.id) AS order_count, SUM(o.total) AS lifetime_value
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
GROUP BY u.id
HAVING COUNT(o.id) >= 5      -- φίλτρο ΜΕΤΑ το grouping
ORDER BY lifetime_value DESC;
```

**`WHERE` vs `HAVING`**: το `WHERE` φιλτράρει rows _πριν_ το grouping· το `HAVING` φιλτράρει groups _μετά_ (πάνω σε aggregates).

## Subqueries & CTEs

```sql
-- Subquery: users με total πάνω από τον μέσο όρο
SELECT id, email FROM users
WHERE id IN (
  SELECT user_id FROM orders GROUP BY user_id HAVING SUM(total) > 1000
);

-- CTE (Common Table Expression): πιο ευανάγνωστο, βήμα-βήμα
WITH big_spenders AS (
  SELECT user_id, SUM(total) AS spent
  FROM orders GROUP BY user_id
  HAVING SUM(total) > 1000
)
SELECT u.email, b.spent
FROM big_spenders b
JOIN users u ON u.id = b.user_id;
```

Τα CTEs (`WITH`) κάνουν πολύπλοκα queries αναγνώσιμα, σπάζοντάς τα σε λογικά βήματα.

## Σειρά εκτέλεσης (όχι σειρά γραφής!)

```txt
Γράφεις:    SELECT → FROM → WHERE → GROUP BY → HAVING → ORDER BY → LIMIT
Εκτελείται: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY → LIMIT
```

Γι' αυτό δεν μπορείς να χρησιμοποιήσεις alias του `SELECT` μέσα στο `WHERE` (το WHERE τρέχει πριν το SELECT), αλλά μπορείς στο `ORDER BY`.

## Πώς απαντάς στη συνέντευξη

«Walk me through joins and aggregation in SQL.»

> SQL is declarative — I describe the result I want. Joins combine tables: an inner join keeps only matching rows on both sides, while a left join keeps every row from the left and fills the right with NULLs when there's no match, which lets me do anti-joins like "users with no orders" by filtering `WHERE right.id IS NULL`. For summaries I use aggregations with `GROUP BY`, and I'm careful about `WHERE` versus `HAVING`: `WHERE` filters rows before grouping, `HAVING` filters groups afterward on aggregates. When logic gets nested I prefer CTEs over deep subqueries because they read as sequential steps. And I keep the logical execution order in mind — `FROM`, `WHERE`, `GROUP BY`, `HAVING`, `SELECT`, `ORDER BY` — which is why a SELECT alias works in `ORDER BY` but not in `WHERE`.
