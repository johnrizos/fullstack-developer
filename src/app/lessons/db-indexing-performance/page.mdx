# Indexing & Query Performance

Όταν μια βάση «αργεί», 9 στις 10 φορές λείπει ένα index ή ένα query κάνει full table scan. Στα interviews εξετάζουν αν ξέρεις **πώς δουλεύει ένα index**, πώς διαβάζεις ένα query plan, και γιατί τα indexes δεν είναι δωρεάν.

## Τι πρέπει να μπορείς στο τέλος

- να εξηγείς τι είναι ένα B-tree index και γιατί επιταχύνει,
- να διαβάζεις `EXPLAIN` (seq scan vs index scan),
- να φτιάχνεις composite indexes με σωστή σειρά στηλών,
- να ξέρεις πότε ένα index βλάπτει (writes, χώρος),
- να αναγνωρίζεις queries που δεν μπορούν να χρησιμοποιήσουν index.

## Τι είναι index

Ένα index είναι μια **ταξινομημένη δομή** (συνήθως B-tree) που επιτρέπει στη βάση να βρει rows χωρίς να σαρώσει όλο τον πίνακα — σαν το ευρετήριο ενός βιβλίου.

```sql
CREATE INDEX idx_users_email ON users(email);
-- Τώρα: WHERE email = '...' → index lookup (O(log n)) αντί για full scan (O(n))
```

Χωρίς index, το `WHERE email = 'x'` διαβάζει **κάθε** row (sequential/table scan). Με index, πηγαίνει κατευθείαν.

## EXPLAIN — δες τι κάνει η βάση

Το `EXPLAIN` (ή `EXPLAIN ANALYZE`) δείχνει το **query plan**:

```sql
EXPLAIN ANALYZE SELECT * FROM users WHERE email = 'a@b.com';
```

```txt
χωρίς index:  Seq Scan on users  (cost=... rows=1000000)   ← σαρώνει τα πάντα 🐢
με index:     Index Scan using idx_users_email  (cost=... rows=1)  ← στοχευμένο ⚡
```

Όταν βλέπεις **Seq Scan** σε μεγάλο πίνακα με selective `WHERE`, λείπει index.

## Composite indexes & σειρά στηλών

Index σε πολλές στήλες — η **σειρά μετράει** (left-to-right prefix rule):

```sql
CREATE INDEX idx_orders_user_created ON orders(user_id, created_at);
```

```txt
✅ WHERE user_id = 5                          → χρησιμοποιεί το index
✅ WHERE user_id = 5 AND created_at > '...'   → χρησιμοποιεί πλήρως
❌ WHERE created_at > '...'                   → ΔΕΝ χρησιμοποιεί (δεν ξεκινά από user_id)
```

Κανόνας: το composite index καλύπτει queries που ξεκινούν από το **αριστερό prefix** των στηλών του. Βάλε πρώτη τη στήλη ισότητας (`=`), μετά τη range/sort στήλη.

## Πότε ένα index βλάπτει

Τα indexes **δεν είναι δωρεάν**:

```txt
- Κάθε INSERT/UPDATE/DELETE πρέπει να ενημερώσει ΚΑΙ τα indexes → πιο αργά writes
- Καταναλώνουν χώρο (disk + memory)
- Πάρα πολλά/αχρησιμοποίητα indexes = overhead χωρίς όφελος
- Index σε low-cardinality στήλη (π.χ. boolean) συχνά άχρηστο
```

Κανόνας: index στις στήλες που χρησιμοποιείς συχνά σε `WHERE`/`JOIN`/`ORDER BY` με υψηλή selectivity — όχι σε ό,τι να 'ναι.

## Queries που δεν μπορούν να χρησιμοποιήσουν index

```sql
WHERE LOWER(email) = '...'        -- function πάνω στη στήλη
WHERE email LIKE '%@gmail.com'    -- leading wildcard (% στην αρχή)
WHERE status != 'active'          -- negation, χαμηλή selectivity
WHERE created_at + interval '1 day' > now()  -- πράξη πάνω στη στήλη
```

Γενικά: κράτα τη **στήλη «καθαρή»** στη μία πλευρά της σύγκρισης, και απόφυγε leading wildcards.

## Πώς απαντάς στη συνέντευξη

«How do you diagnose and fix a slow query?»

> I start with `EXPLAIN ANALYZE` to see the plan. A sequential scan on a large table with a selective filter usually means a missing index. An index is a sorted structure — typically a B-tree — that lets the database jump to matching rows instead of scanning everything, turning an O(n) scan into an O(log n) lookup. For multi-column filters I use a composite index and order the columns left to right, equality columns first then a range column, because the index only helps queries that match a left prefix. I also watch for things that defeat indexes: a function wrapped around the column, a leading wildcard in `LIKE`, or arithmetic on the column. And I don't index everything — each index slows down writes and costs space, so I index the columns that actually appear in selective `WHERE`, `JOIN`, and `ORDER BY` clauses, guided by the query plan.
