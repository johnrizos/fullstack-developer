# Server & Client Components

Η μεγαλύτερη εννοιολογική αλλαγή του App Router: **τα components είναι Server Components by default**. Στα interviews εξετάζουν αν ξέρεις πότε χρειάζεσαι `'use client'`, τι σημαίνει το boundary, και πώς συνδυάζονται τα δύο είδη χωρίς να φουσκώνεις το client bundle.

## Τι πρέπει να μπορείς στο τέλος

- να εξηγείς τι κάνει το καθένα και πού τρέχει,
- να διαλέγεις Server ή Client Component ανά περίπτωση,
- να καταλαβαίνεις το `'use client'` ως boundary (όχι per-file flag),
- να κάνεις interleave (Server μέσα σε Client μέσω children),
- να αποφεύγεις διαρροή secrets στον client.

## By default: Server Components

Layouts και pages είναι **Server Components**: τρέχουν στον server, μπορούν να κάνουν async data fetching κοντά στην πηγή, κρατούν secrets εκτός client, και στέλνουν λιγότερη JavaScript.

```tsx
// Server Component (default) — async, χωρίς 'use client'
import { getPost } from "@/lib/data";
import LikeButton from "@/app/ui/like-button";

export default async function Page({ params }: { params: Promise<{ id: string }> }) {
  const { id } = await params;
  const post = await getPost(id); // τρέχει στον server
  return (
    <main>
      <h1>{post.title}</h1>
      <LikeButton likes={post.likes} /> {/* client island */}
    </main>
  );
}
```

## Πότε Client Component

Βάζεις `'use client'` όταν χρειάζεσαι:

```txt
- state & event handlers   (useState, onClick, onChange)
- lifecycle / effects      (useEffect)
- browser-only APIs        (window, localStorage, geolocation)
- custom hooks που τα χρησιμοποιούν
```

```tsx
// app/ui/like-button.tsx
"use client";
import { useState } from "react";

export default function LikeButton({ likes }: { likes: number }) {
  const [count, setCount] = useState(likes);
  return <button onClick={() => setCount(count + 1)}>{count}</button>;
}
```

## 'use client' είναι boundary, όχι ετικέτα ανά αρχείο

Μόλις ένα αρχείο έχει `'use client'`, **όλα τα imports του και τα components που renderάρει απευθείας** μπαίνουν στο client bundle. Δεν χρειάζεται να βάλεις τη directive σε κάθε child component — η directive ορίζει το σύνορο Server/Client module graph.

## Interleaving: Server μέσα σε Client

Δεν μπορείς να κάνεις `import` Server Component μέσα σε Client. Μπορείς όμως να το περάσεις ως **children/prop** — τότε renderάρεται στον server και περνά ως έτοιμο output:

```tsx
// app/ui/modal.tsx
"use client";
export default function Modal({ children }: { children: React.ReactNode }) {
  return <div className="modal">{children}</div>;
}

// app/page.tsx (Server Component)
import Modal from "./ui/modal";
import Cart from "./ui/cart"; // Server Component

export default function Page() {
  return (
    <Modal>
      <Cart /> {/* renderάρεται server-side, μπαίνει ως slot στο client Modal */}
    </Modal>
  );
}
```

Τα props που περνάς από Server σε Client πρέπει να είναι **serializable** (όχι functions/Date instances με logic, κλπ — απλά δεδομένα).

## Context providers χρειάζονται Client

Το React Context δεν υποστηρίζεται σε Server Components. Φτιάχνεις provider ως Client Component που δέχεται `children` και τον βάζεις στο (server) layout:

```tsx
// app/theme-provider.tsx
"use client";
import { createContext } from "react";
export const ThemeContext = createContext("light");
export default function ThemeProvider({ children }: { children: React.ReactNode }) {
  return <ThemeContext.Provider value="dark">{children}</ThemeContext.Provider>;
}
```

## Μην διαρρεύσεις secrets

Server Components κρατούν API keys ασφαλή. Μόνο μεταβλητές με πρόθεμα `NEXT_PUBLIC_` μπαίνουν στο client bundle — οι υπόλοιπες αντικαθίστανται με κενό string. Για επιπλέον προστασία, το πακέτο `server-only` ρίχνει build-time error αν server κώδικας εισαχθεί σε client.

## Πώς απαντάς στη συνέντευξη

«Server vs Client Components — when do you use each?»

> In the App Router everything is a Server Component by default, which is great for data fetching close to the source, keeping secrets server-side, and shipping less JavaScript. I add `'use client'` only when I need state, event handlers, effects, or browser APIs. The important nuance is that `'use client'` is a boundary, not a per-file label: once a module is a client module, everything it imports and renders directly goes into the client bundle, so I push the directive as far down the tree as possible — to the interactive leaf, like a search bar — and keep the surrounding layout on the server. I can't import a Server Component into a Client one, but I can pass it as `children` so it renders on the server and slots into the client component. Props crossing that boundary must be serializable, and only `NEXT_PUBLIC_`-prefixed env vars reach the client, so secrets stay safe.
