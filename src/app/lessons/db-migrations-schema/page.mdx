# Schema Migrations & Evolution

Το schema **αλλάζει** συνέχεια: νέα πεδία, renames, νέοι πίνακες. Σε production με ζωντανούς χρήστες, μια λάθος migration ρίχνει το app ή κλειδώνει τη βάση. Στα interviews εξετάζουν πώς κάνεις αλλαγές **χωρίς downtime** και χωρίς data loss.

## Τι πρέπει να μπορείς στο τέλος

- να εξηγείς τι είναι migrations και γιατί version-controlled,
- να γράφεις forward & rollback migrations,
- να κάνεις zero-downtime αλλαγές (expand/contract),
- να χειρίζεσαι backfills σε μεγάλους πίνακες,
- να αποφεύγεις migrations που κλειδώνουν τη βάση.

## Τι είναι migration

Ένα migration είναι ένα **versioned, επαναλήψιμο** βήμα αλλαγής του schema, μέρος του κώδικα (git):

```sql
-- migrations/0007_add_phone_to_users.sql
ALTER TABLE users ADD COLUMN phone TEXT;
```

Τα migration tools (Prisma Migrate, Alembic, Flyway) κρατούν ποια migrations έχουν τρέξει, ώστε κάθε environment (dev/staging/prod) να φτάνει στην ίδια κατάσταση **ντετερμινιστικά**. Ποτέ δεν αλλάζεις το schema «με το χέρι» στην παραγωγή.

## Forward & rollback

```sql
-- up (forward)
ALTER TABLE users ADD COLUMN phone TEXT;

-- down (rollback) — πώς αναιρείς αν κάτι πάει στραβά
ALTER TABLE users DROP COLUMN phone;
```

Κάθε migration ιδανικά έχει και rollback. Προσοχή: destructive rollbacks (DROP COLUMN) χάνουν δεδομένα — σχεδίασε με προσοχή.

## Zero-downtime: expand / contract

Το κρίσιμο pattern. Ο παλιός και ο νέος κώδικας τρέχουν **ταυτόχρονα** κατά το deploy, οπότε το schema πρέπει να είναι συμβατό και με τους δύο. Σπας μια «επικίνδυνη» αλλαγή σε βήματα:

```txt
Παράδειγμα: rename στήλης name → full_name (χωρίς downtime)

1. EXPAND:  ADD COLUMN full_name (nullable). Κώδικας γράφει ΚΑΙ στα δύο.
2. BACKFILL: αντίγραψε name → full_name σε batches.
3. MIGRATE: ο νέος κώδικας διαβάζει full_name. Deploy.
4. CONTRACT: όταν κανείς δεν χρησιμοποιεί name → DROP COLUMN name.
```

## Backfills σε μεγάλους πίνακες

Γέμισμα νέας στήλης σε εκατομμύρια rows με ένα `UPDATE` **κλειδώνει** τον πίνακα και ρίχνει το app. Κάνε το σε **batches**:

```sql
-- ❌ Κλειδώνει τα πάντα
UPDATE users SET full_name = name;

-- ✅ Batched (loop στον app/script), μικρά chunks, με παύσεις
UPDATE users SET full_name = name
WHERE id BETWEEN 1 AND 10000 AND full_name IS NULL;
-- επανάλαβε για τα επόμενα ranges
```

## Migrations που κλειδώνουν — απόφυγέ τα

```txt
- ADD COLUMN με NON-NULL default σε παλιές βάσεις → rewrite όλου του πίνακα (lock)
  → πρόσθεσε nullable, backfill, μετά set NOT NULL
- CREATE INDEX (blocking) σε μεγάλο πίνακα → χρησιμοποίησε CREATE INDEX CONCURRENTLY
- Μεγάλο UPDATE/ALTER σε ένα transaction → batches
```

Γενικός κανόνας: **additive & nullable** αλλαγές πρώτα· destructive αλλαγές τελευταίες, αφού σιγουρευτείς ότι τίποτα δεν τις χρειάζεται.

## Πώς απαντάς στη συνέντευξη

«How do you change a database schema without downtime?»

> Schema changes are versioned migrations checked into git and applied by a tool, so every environment converges deterministically — I never hand-edit production. The key constraint is that during a rolling deploy the old and new code run at the same time, so the schema has to be compatible with both. That's why I never rename or drop directly; I use expand/contract. To rename a column I add the new one nullable, have the code write to both, backfill the old values, switch reads to the new column, and only after the old code is gone do I drop the old column. Backfills on big tables go in small batches, never one giant `UPDATE` that locks the table. I also avoid lock-heavy operations — a non-null default on a huge table or a blocking `CREATE INDEX` — by adding nullable first then setting not-null, and using `CREATE INDEX CONCURRENTLY`. Additive changes first, destructive ones last.
