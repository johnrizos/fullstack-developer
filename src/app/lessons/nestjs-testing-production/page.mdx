# Testing & Production

Το Nest είναι φτιαγμένο για testing — το DI container σού επιτρέπει να swap-άρεις providers με mocks χωρίς πόνο. Στη συνέντευξη: «πώς τεστάρεις ένα service;», «unit vs e2e;», «πώς γίνεται deploy ένα Nest app;».

## Τι πρέπει να μπορείς στο τέλος

- να γράφεις unit test με `Test.createTestingModule` και mocked providers,
- να γράφεις e2e test με Supertest πάνω στο πραγματικό app,
- να ξέρεις τη testing pyramid,
- να στήνεις config validation, health checks και graceful shutdown για production.

## Unit test — mock τα dependencies

Το `Test.createTestingModule` φτιάχνει ένα DI container όπου overrideάρεις providers:

```ts
describe("TaskService", () => {
  let service: TaskService;
  const prismaMock = { task: { findUnique: jest.fn() } };

  beforeEach(async () => {
    const moduleRef = await Test.createTestingModule({
      providers: [
        TaskService,
        { provide: PrismaService, useValue: prismaMock },   // swap το πραγματικό client
      ],
    }).compile();
    service = moduleRef.get(TaskService);
  });

  it("ρίχνει NotFound όταν δεν υπάρχει task", async () => {
    prismaMock.task.findUnique.mockResolvedValue(null);
    await expect(service.findOne("x")).rejects.toThrow(NotFoundException);
  });
});
```

Επειδή το service παίρνει το Prisma μέσω DI, το αντικαθιστάς με ένα plain object mock — **κανένα** πραγματικό DB call. Γρήγορο, deterministic, τεστάρει τη logic.

## E2E test — το πραγματικό app

Το e2e σηκώνει ολόκληρο το Nest app και χτυπά HTTP με Supertest:

```ts
describe("Tasks (e2e)", () => {
  let app: INestApplication;

  beforeAll(async () => {
    const moduleRef = await Test.createTestingModule({ imports: [AppModule] }).compile();
    app = moduleRef.createNestApplication();
    app.useGlobalPipes(new ValidationPipe({ whitelist: true }));   // ίδιο setup με production
    await app.init();
  });
  afterAll(async () => await app.close());

  it("POST /tasks → 400 σε invalid body", () => {
    return request(app.getHttpServer()).post("/tasks").send({ title: "" }).expect(400);
  });
  it("POST /tasks → 201 με valid body", () => {
    return request(app.getHttpServer()).post("/tasks").send({ title: "Buy milk" }).expect(201);
  });
});
```

Το e2e περνά μέσα από όλο το pipeline (pipes, guards, interceptors) — επαληθεύει ότι τα κομμάτια δουλεύουν **μαζί**, όχι μόνο απομονωμένα.

## Testing pyramid

```txt
        ╱╲   λίγα E2E       (αργά, ολόκληρο app + HTTP)
       ╱──╲
      ╱ Int ╲ μερικά        (service + πραγματική/test βάση)
     ╱──────╲
    ╱  Unit  ╲ πολλά        (γρήγορα, mocked deps, business logic)
   ╱──────────╲
```

Βάση από γρήγορα unit tests (logic, mocked deps), λίγα integration, ελάχιστα e2e για τα critical paths. Αντίστροφη πυραμίδα (όλα e2e) = αργό, flaky CI.

## Production essentials

```ts
// main.ts
async function bootstrap() {
  const app = await NestFactory.create(AppModule);
  app.useGlobalPipes(new ValidationPipe({ whitelist: true, transform: true }));
  app.enableShutdownHooks();        // τρέχει onModuleDestroy → $disconnect, κλείσιμο pools
  app.enableCors({ origin: process.env.CLIENT_URL });
  await app.listen(process.env.PORT ?? 3000);
}
```

- **Config validation**: `ConfigModule.forRoot({ validationSchema })` — fail fast αν λείπει env var.
- **Graceful shutdown**: `enableShutdownHooks()` ώστε σε SIGTERM να κλείνουν connections καθαρά (αλλιώς leaked connections/in-flight requests).
- **Health checks**: `@nestjs/terminus` → `/health` που ελέγχει DB/dependencies (για k8s/load balancer).
- **Logging**: structured logger (Pino) αντί για `console.log`· interceptor για request logging.

## Πώς απαντάς στη συνέντευξη

«How do you test and ship a NestJS app?»

> The DI container makes testing straightforward. For unit tests I build a `Test.createTestingModule` and override real dependencies with mocks via `useValue`, so a service test exercises pure logic with zero I/O — fast and deterministic. For e2e I import the real `AppModule`, apply the same global pipes as production, and hit it with Supertest so the whole pipeline — pipes, guards, interceptors — is exercised together. I follow the testing pyramid: many unit tests, a few integration, a handful of e2e on critical paths, because an all-e2e suite is slow and flaky. For production I validate env with a config schema so it fails fast, enable shutdown hooks so SIGTERM closes connections cleanly via `$disconnect`, add a `/health` endpoint with terminus, configure CORS, and use a structured logger instead of console.
