# Blueprints & App Factory

Μια Flask app σε ένα `app.py` δεν κλιμακώνει. Δύο patterns την κάνουν συντηρήσιμη: τα **Blueprints** (σπάσιμο σε modules) και το **application factory** (καθαρή δημιουργία της app). Είναι το «πώς δομώ μεγάλο Flask project».

## Το πρόβλημα με ένα αρχείο

Όλα τα routes, models, config σε ένα `app.py` → χάος, circular imports, αδύνατο testing με διαφορετικές ρυθμίσεις. Χρειάζεσαι δομή.

## Blueprints — modular routes

Ένα blueprint είναι ένα «υπο-app»: ομαδοποιεί σχετιζόμενα routes, templates και static files.

```python
# app/blog/routes.py
from flask import Blueprint, render_template

blog = Blueprint("blog", __name__, url_prefix="/blog")

@blog.route("/")
def index():
    return render_template("blog/index.html")

@blog.route("/<int:post_id>")
def detail(post_id):
    return f"Post {post_id}"
```

Πρόσεξε: `@blog.route` (όχι `@app.route`). Το `url_prefix="/blog"` βάζει αυτόματα πρόθεμα σε όλα τα routes.

## Application factory

Αντί να φτιάχνεις την `app` σε global scope, τη φτιάχνεις σε **function** — έτσι μπορείς να δημιουργείς πολλαπλά instances (π.χ. με test config):

```python
# app/__init__.py
from flask import Flask
from flask_sqlalchemy import SQLAlchemy

db = SQLAlchemy()   # ορίζεται έξω, αρχικοποιείται μέσα

def create_app(config_object="config.Config"):
    app = Flask(__name__)
    app.config.from_object(config_object)

    db.init_app(app)                        # σύνδεσε extensions στην app

    from app.blog.routes import blog        # import ΜΕΣΑ (αποφυγή circular)
    app.register_blueprint(blog)            # σύνδεσε το blueprint

    return app
```

```python
# run.py
from app import create_app
app = create_app()
```

Το factory pattern λύνει δύο πράγματα: **circular imports** (imports μέσα στη function) και **testing** (φτιάχνεις app με test config χωρίς να μολύνεις την production).

## Extensions στο factory pattern

Τα extensions (`db`, `migrate`, `login_manager`) ορίζονται **έξω** από το factory και αρχικοποιούνται **μέσα** με `init_app(app)`:

```python
db = SQLAlchemy()
login_manager = LoginManager()

def create_app():
    app = Flask(__name__)
    db.init_app(app)
    login_manager.init_app(app)
    return app
```

Έτσι το ίδιο `db` object είναι προσβάσιμο από παντού (models, blueprints) χωρίς circular import με την `app`.

## Προτεινόμενη δομή

```txt
myapp/
  app/
    __init__.py        ← create_app() factory
    extensions.py      ← db, login_manager (ορισμοί)
    blog/
      __init__.py
      routes.py        ← Blueprint
      models.py
    auth/
      routes.py        ← άλλο Blueprint
  config.py            ← config classes
  run.py
```

Κάθε feature είναι ένα blueprint package — παρόμοια φιλοσοφία με τα Django apps, αλλά χτισμένη χειροκίνητα (micro-framework).

## Πώς απαντάς στη συνέντευξη

«How do you structure a larger Flask app?»

> Two patterns. Blueprints split the app into feature modules — each registers its own routes with `@blueprint.route` and can have a `url_prefix`, like mini-apps. The application factory creates the app inside a `create_app()` function instead of at import time, which solves circular imports and lets me build an app with test config without polluting production. Extensions like SQLAlchemy are defined at module level and wired in with `init_app(app)` inside the factory, so the same `db` is importable everywhere. The result is a package per feature — conceptually similar to Django apps, but assembled by hand.
