# GraphQL Basics

GraphQL είναι API style όπου ο client ζητά ακριβώς τα fields που χρειάζεται.

## Query

Query σημαίνει read operation. Ο client δηλώνει το shape της απάντησης.

```graphql
query {
  tasks {
    id
    title
    completed
  }
}
```

Αν ο client δεν ζητήσει `completed`, ο server δεν χρειάζεται να το επιστρέψει. Αυτό μειώνει overfetching, ειδικά όταν mobile και web clients χρειάζονται διαφορετικά fields.

## Mutation

Mutation σημαίνει write operation: create, update ή delete.

```graphql
mutation {
  createTask(title: "Learn GraphQL") {
    id
    title
  }
}
```

Σε production API, η mutation συνήθως δέχεται input object:

```graphql
mutation CreateTask($input: CreateTaskInput!) {
  createTask(input: $input) {
    id
    title
    completed
  }
}
```

## Schema

Το schema είναι contract.

```graphql
type Task {
  id: ID!
  title: String!
  completed: Boolean!
}
```

Στο schema δηλώνεις types, queries, mutations και ποια fields είναι required με `!`.

```graphql
type Query {
  tasks: [Task!]!
  task(id: ID!): Task
}

input CreateTaskInput {
  title: String!
}
```

## Resolver

Resolver είναι η function που φέρνει τα δεδομένα για ένα field.

```ts
const resolvers = {
  Query: {
    tasks: async () => db.task.findMany(),
  },
  Mutation: {
    createTask: async (_parent, args) => {
      return db.task.create({ data: { title: args.input.title } });
    },
  },
};
```

Το συχνό performance πρόβλημα είναι το **N+1 query** — και στο GraphQL είναι ακόμα πιο ύπουλο, γιατί το προκαλεί ο client με ένα nested query:

```graphql
query {
  tasks {        # 1 query για τα tasks
    title
    author {     # +1 query ΓΙΑ ΚΑΘΕ task — ο resolver του author
      name       #  τρέχει ξεχωριστά για κάθε γονικό task!
    }
  }
}
```

Η καθιερωμένη λύση είναι το **DataLoader**: μαζεύει όλα τα ζητούμενα ids μέσα σε ένα tick και κάνει ΕΝΑ batched query:

```ts
const authorLoader = new DataLoader(async (userIds) => {
  const users = await db.user.findMany({ where: { id: { in: userIds } } });
  return userIds.map((id) => users.find((u) => u.id === id));
});

const resolvers = {
  Task: {
    author: (task) => authorLoader.load(task.userId), // batching + per-request cache
  },
};
```

Από 1+N queries → 2 queries. Το να αναφέρεις το DataLoader με τη λέξη «batching» είναι από τα πιο δυνατά σήματα σε GraphQL συζήτηση.

## REST vs GraphQL

REST είναι απλό και predictable για πολλά APIs. GraphQL βοηθά όταν έχεις πολλά clients ή σύνθετα data requirements και θέλεις να αποφύγεις overfetching/underfetching.

Καλή χρήση GraphQL:

- dashboard που χρειάζεται data από πολλά resources,
- mobile client που θέλει μικρότερα responses,
- προϊόν με διαφορετικά clients και shared schema.

Πιθανή κακή χρήση:

- πολύ απλό CRUD API όπου REST είναι αρκετό,
- ομάδα χωρίς εμπειρία σε schema design/caching,
- endpoints που κυρίως κάνουν file uploads ή streaming.

## Authorization

Μην θεωρείς ότι επειδή ένα field δεν φαίνεται στο UI είναι ασφαλές. Ο client μπορεί να γράψει δικό του query.

Authorization πρέπει να γίνεται στους resolvers ή στο service layer:

```ts
const resolvers = {
  Query: {
    task: async (_parent, args, context) => {
      const task = await db.task.findUnique({ where: { id: args.id } });

      if (!task || task.userId !== context.currentUser.id) {
        throw new Error("Forbidden");
      }

      return task;
    },
  },
};
```

## Caching

REST caching συχνά βασίζεται εύκολα σε URLs. GraphQL είναι πιο σύνθετο επειδή πολλά διαφορετικά queries πάνε στο ίδιο endpoint, συνήθως `/graphql`. Γι' αυτό client caches όπως Apollo/Relay χρησιμοποιούν normalized cache με IDs και types.

## Πώς απαντάς στη συνέντευξη

«REST or GraphQL?»

> GraphQL gives clients a typed schema and lets them request exactly the fields they need — that solves overfetching and underfetching, which matters with multiple clients like web and mobile. The cost is real: caching is harder than REST's URL-based caching, nested queries create N+1 problems that need DataLoader batching, and authorization must live in resolvers because clients can query anything in the schema. For a simple CRUD API with one client, REST stays my default; GraphQL earns its complexity when the data graph and client diversity justify it.
