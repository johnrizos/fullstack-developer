# Schema Design & Interview Practice

Το τελευταίο βήμα του database track: να **συνθέτεις** όλα τα προηγούμενα σε ένα πραγματικό design. Στις συνεντεύξεις το «σχεδίασε το schema για X» είναι ανοιχτό — δεν υπάρχει μία σωστή απάντηση, αλλά υπάρχει σωστή **διαδικασία**.

## Τι πρέπει να μπορείς στο τέλος

- να ακολουθείς δομημένη διαδικασία σχεδιασμού,
- να ξεκινάς από access patterns & requirements,
- να επιλέγεις τύπους, keys, σχέσεις και indexes συνειδητά,
- να εξηγείς trade-offs (normalization, consistency, scale),
- να απαντάς σε κλασικές db ερωτήσεις με δομή.

## Η διαδικασία σχεδιασμού

```txt
1. Requirements   → τι entities; τι σχέσεις; read-heavy ή write-heavy;
2. Access patterns→ ποια queries θα τρέχουν συχνά; (αυτά οδηγούν το design)
3. Entities/tables→ ένα table ανά entity, σωστοί τύποι, primary keys
4. Relationships  → FKs, 1-N / N-N (join tables)
5. Constraints    → NOT NULL, UNIQUE, CHECK, referential integrity
6. Indexes        → στις στήλες των συχνών WHERE/JOIN/ORDER BY
7. Scale/trade-offs→ normalization vs denormalization, SQL vs NoSQL, caching
```

Μην ορμήσεις στα tables — ξεκίνα από **requirements & access patterns**. Αυτό δείχνει ωριμότητα.

## Παράδειγμα: σχεδίασε ένα «URL shortener»

```txt
Requirements: short code → long URL, click analytics, πολύ read-heavy.
Access patterns:
  - GET /:code → redirect (ΠΟΛΥ συχνό, πρέπει να είναι γρήγορο)
  - δημιουργία νέου short link
  - analytics ανά link (λιγότερο συχνό)
```

```sql
CREATE TABLE links (
  id          BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  code        TEXT NOT NULL UNIQUE,      -- index για το hot lookup
  long_url    TEXT NOT NULL,
  user_id     BIGINT REFERENCES users(id),
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE clicks (             -- ξεχωριστά: write-heavy, μη μπλοκάρει το read
  id         BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
  link_id    BIGINT NOT NULL REFERENCES links(id),
  clicked_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

Σχεδιαστικές αποφάσεις: το `code` έχει `UNIQUE` index (το hot `WHERE code = ?`)· τα clicks σε ξεχωριστό πίνακα (διαφορετικό write pattern)· caching layer μπροστά για το redirect· analytics ίσως aggregated table αν τα raw clicks γίνουν τεράστια.

## Trade-offs που πρέπει να αρθρώνεις

```txt
- Normalize vs denormalize → ακεραιότητα vs ταχύτητα reads (με μέτρηση)
- SQL vs NoSQL             → σχέσεις/ACID vs scale/ευελιξία/access patterns
- Strong vs eventual       → consistency vs availability/latency (CAP)
- Index ή όχι              → γρήγορα reads vs αργά writes & χώρος
- Sync vs async            → π.χ. analytics γράψε async για να μη μπλοκάρει το hot path
```

Καλή απάντηση = «εξαρτάται, και να γιατί». Δείξε ότι ξέρεις και τις δύο πλευρές.

## Κλασικές db ερωτήσεις — δομημένα

```txt
"SQL ή NoSQL;"        → access patterns, σχέσεις, ACID, scale (μάθημα NoSQL)
"Πώς θα έκανες αυτό το query πιο γρήγορο;" → EXPLAIN → index → query rewrite
"N+1;"               → όρισέ το, δείξε eager loading
"Πώς αλλάζεις schema χωρίς downtime;" → expand/contract + batched backfill
"Race condition σε αγορά;" → atomic update / FOR UPDATE / optimistic lock
"Πώς κλιμακώνεις;"   → optimize → cache → scale up → replicas → shard
```

## Πώς απαντάς στη συνέντευξη

«Design the database schema for [a feature].»

> I don't jump to tables — I start by clarifying requirements and the access patterns: what entities exist, how they relate, which queries are hot, and whether it's read- or write-heavy, plus any consistency and scale needs. Those patterns drive the design. Then I lay out a table per entity with proper types and primary keys, model relationships with foreign keys and join tables for many-to-many, and push integrity into constraints. I index the columns that show up in frequent `WHERE`, `JOIN`, and `ORDER BY` clauses, and I separate tables with very different access patterns — like keeping append-heavy analytics out of a hot read path. Throughout, I think out loud about trade-offs: normalize for integrity versus denormalize for read speed when measured, SQL versus NoSQL based on relationships and access patterns, strong versus eventual consistency. There's rarely one right schema, so I justify each decision and name what I'd revisit at scale.
