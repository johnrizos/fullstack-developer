# Docker Compose

Μια πραγματική εφαρμογή σπάνια είναι **ένας** container. Έχεις το backend, μια **βάση**, ίσως ένα **Redis**, ίσως έναν worker. Με σκέτο `docker run` θα έγραφες 4 μεγάλες εντολές με σωστές πόρτες, volumes, env vars και δικτύωση — κάθε φορά, με το σωστό σκορ. Το **Docker Compose** το κάνει αυτό σε **ένα αρχείο** και **μία εντολή**.

## Το πρόβλημα που λύνει

Χωρίς Compose, για να σηκώσεις app + database θα έτρεχες κάτι σαν:

```bash
docker run -d --name db -e POSTGRES_PASSWORD=secret -v pgdata:/var/lib/postgresql/data postgres:16
docker run -d --name api -p 8080:3000 -e DATABASE_URL=... --link db my-app
```

Επίπονο, εύκολο να κάνεις λάθος, δύσκολο να το μοιραστείς. Με Compose περιγράφεις **δηλωτικά** ό,τι θες σε ένα `compose.yaml`, και το σηκώνεις όλο με `docker compose up`.

## Το compose.yaml

```yaml
services:
  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: secret
      POSTGRES_DB: appdb
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 5s
      retries: 5

  api:
    build: .
    ports:
      - "8080:3000"
    environment:
      DATABASE_URL: postgres://postgres:secret@db:5432/appdb
    depends_on:
      db:
        condition: service_healthy

volumes:
  pgdata:
```

Ας το διαβάσουμε:

- **`services:`** — κάθε «κουτί» (container) της εφαρμογής σου. Εδώ δύο: `db` και `api`.
- **`image: postgres:16`** — το `db` χρησιμοποιεί έτοιμο image από το registry.
- **`build: .`** — το `api` **χτίζεται** από το `Dockerfile` του τρέχοντος φακέλου (αντί για έτοιμο image).
- **`environment:`** — env vars, ακριβώς όπως το `-e`.
- **`volumes:`** — μόνιμη αποθήκευση για τη βάση (το ίδιο concept με το προηγούμενο μάθημα). Το `pgdata` δηλώνεται και στο κάτω `volumes:` block.
- **`ports: "8080:3000"`** — port mapping, όπως το `-p host:container`.
- **`depends_on`** — σειρά εκκίνησης (θα το δούμε παρακάτω).

## Το «μαγικό» κομμάτι: δικτύωση με ονόματα

Πρόσεξε το `DATABASE_URL`:

```text
postgres://postgres:secret@db:5432/appdb
                          ^^
```

Το host δεν είναι `localhost` ούτε κάποια IP — είναι **`db`**, το **όνομα του service**. Το Compose φτιάχνει αυτόματα ένα **ιδιωτικό δίκτυο** και βάζει όλα τα services μέσα του, όπου **βρίσκουν το ένα το άλλο με το όνομά τους** (service discovery μέσω DNS). Έτσι το `api` μιλά στη βάση απλώς λέγοντας `db:5432` — χωρίς IPs, χωρίς χειροκίνητο linking.

Σημαντικό: το `db:5432` δουλεύει **μεταξύ των containers**. Από τον **δικό σου browser** μιλάς μέσω της host πόρτας (`localhost:8080`) που εκτέθηκε με το `ports`. Δεν χρειάζεται (και συνήθως δεν θέλεις) να εκθέσεις την 5432 της βάσης έξω.

## depends_on: σειρά εκκίνησης

Το `depends_on` λέει «σήκωσε πρώτα το `db`, μετά το `api`». **Προσοχή** σε μια κλασική παγίδα: το σκέτο `depends_on: [db]` περιμένει μόνο να **ξεκινήσει** ο container της βάσης — **όχι** να είναι έτοιμη να δεχτεί συνδέσεις. Η PostgreSQL κάνει λίγα δευτερόλεπτα να σηκωθεί, οπότε το `api` μπορεί να σκάσει προσπαθώντας να συνδεθεί σε μια βάση που «ξεκίνησε αλλά δεν είναι έτοιμη».

Η σωστή λύση είναι ο συνδυασμός **`healthcheck` + `condition: service_healthy`** (όπως στο παράδειγμα): το `api` ξεκινά μόνο όταν το healthcheck της βάσης περάσει.

## Οι εντολές που θα χρησιμοποιείς καθημερινά

```bash
docker compose up            # σήκωσε τα πάντα (foreground, βλέπεις logs)
docker compose up -d         # ...στο παρασκήνιο
docker compose up --build    # ξαναχτίσε τα images πρώτα
docker compose ps            # τι τρέχει
docker compose logs -f api   # logs ενός service
docker compose exec api sh   # μπες μέσα σε ένα service
docker compose down          # σταμάτα & σβήσε τα πάντα
docker compose down -v       # ...ΚΑΙ σβήσε τα volumes (χάνεις τα δεδομένα!)
```

Με **μία** εντολή (`up`) όλη η ομάδα έχει ολόιδιο περιβάλλον. Ένας νέος developer κάνει `git clone` + `docker compose up` και δουλεύει σε λεπτά — αυτό είναι το μεγάλο win.

## Compose ≠ production orchestration

Ξεκαθάρισέ το για να μη μπερδευτείς (και για τη συνέντευξη): το Compose είναι ιδανικό για **local development** και απλά setups — τρέχει σε **ένα μηχάνημα**. Δεν κάνει auto-scaling σε πολλούς servers, self-healing σε cluster, ή rolling updates χωρίς downtime. Γι' αυτά υπάρχει το **Kubernetes** (επόμενο βήμα στη ροή). Compose = «σήκωσε το stack μου τοπικά, εύκολα».

## Πού ζει η βάση σε production;

Συχνή απορία: «θέλω PostgreSQL, αλλά ο server που ανεβάζω το app **δεν έχει** Postgres — τι γίνεται;». Η απάντηση διαλύει μια βασική παρανόηση: **ο server δεν χρειάζεται να έχει Postgres εγκατεστημένη — ποτέ.**

Το app σου δεν «περιέχει» βάση· απλώς **της μιλάει μέσω δικτύου** με ένα connection string. Το μόνο που έχει μέσα το app image είναι ένας **client** (`pg`, Prisma) που στέλνει queries σε ένα `HOST:5432`:

```
DATABASE_URL=postgres://user:password@HOST:5432/appdb
```

**Πού ζει** η βάση είναι ξεχωριστή απόφαση. Δύο δρόμοι:

**Δρόμος Α — Postgres ως δικό του container** (αυτό που κάνεις στο Compose). Δεν εγκαθιστάς τίποτα στον server· τρέχεις ένα `postgres:16` container και το image κατεβαίνει από το registry φέρνοντας τα πάντα. Ο server χρειάζεται **μόνο Docker**. Το τίμημα: **εσύ** αναλαμβάνεις backups/upgrades, και **υποχρεωτικά** volume για persistence — αλλιώς ένα `docker rm` (ή `down -v`) σβήνει τα δεδομένα. Και το volume ζει σε **αυτόν τον έναν server**.

**Δρόμος Β — Managed database** (το σύνηθες σε σοβαρή παραγωγή). Η βάση **δεν τρέχει καν στον server σου** — την παρέχει υπηρεσία cloud (**AWS RDS, Google Cloud SQL, Neon, Supabase, Railway**). Βάζεις στο `DATABASE_URL` τη διεύθυνση που σου δίνουν· το app container δεν έχει **κανένα** Postgres μέσα, μόνο τον client. Backups, failover, patching, replication τα κάνει ο provider.

| Περίπτωση | Επιλογή |
|---|---|
| Local development / demo / μικρό project | **Postgres ως container** (Compose) — γρήγορο, μηδέν κόστος |
| Σοβαρή παραγωγή με πραγματικά δεδομένα | **Managed database** — ο provider αναλαμβάνει backups/failover/replication |

⚠️ **Anti-pattern:** μη βάζεις το Postgres **μέσα** στο ίδιο image με το app. Ισχύει ο κανόνας **μία διεργασία ανά container** — app σε ένα container, βάση σε άλλο (ή managed). Με μια φράση: ο server θέλει μόνο **Docker + δικτυακή πρόσβαση** στη βάση· το app τη βρίσκει με **μία** env var.

## Πώς απαντάς στη συνέντευξη

> «What is Docker Compose and when do you use it?»

> Docker Compose describes a multi-container app declaratively in one compose.yaml and brings it up with a single docker compose up. Each service is a container — say an api that builds from my Dockerfile and a db from the postgres image — with its ports, environment, volumes, and dependencies in one place. Compose creates a private network where services find each other by service name, so the api connects to the database at db:5432, no IPs or manual linking. I use depends_on with a healthcheck and condition service_healthy so the api waits until the database actually accepts connections, not just until its container starts. The big win is onboarding: clone the repo, run compose up, and the whole stack is identical for everyone. But Compose runs on a single machine — for multi-node scaling, self-healing, and zero-downtime rollouts I'd reach for Kubernetes instead.
