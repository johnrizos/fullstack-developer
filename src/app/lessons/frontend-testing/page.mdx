# React Testing

Frontend testing πρέπει να ελέγχει συμπεριφορά χρήστη, όχι implementation details.

Καλή ερώτηση για κάθε test:

> Αν ήμουν χρήστης, τι βλέπω και τι κάνω;

## React Testing Library

Η React Testing Library σε σπρώχνει να βρίσκεις elements όπως θα τα έβρισκε ο χρήστης ή screen reader:

```ts
screen.getByLabelText("Email");
screen.getByRole("button", { name: "Login" });
```

Αυτό ενισχύει και accessibility.

## userEvent

Το `userEvent` προσομοιώνει αλληλεπιδράσεις πιο κοντά στον χρήστη από απλά fire events.

```ts
await user.type(input, "ada@example.com");
await user.click(button);
```

## Τι να τεστάρεις

Τέσταρε:

- submit flows,
- validation messages,
- loading/error states,
- conditional rendering,
- critical user journeys.

Μην τεστάρεις:

- internal state variables απευθείας,
- private implementation details,
- styling που αλλάζει συχνά χωρίς business σημασία.

## Async UI: findBy, waitFor και mock του network

Τα περισσότερα πραγματικά components φορτώνουν δεδομένα — και εδώ ξεχωρίζουν τα καλά tests:

```tsx
test("shows the loaded tasks", async () => {
  render(<TaskList />); // κάνει fetch στο mount

  // getBy*: σύγχρονο — σκάει αμέσως αν δεν υπάρχει
  expect(screen.getByText("Loading…")).toBeInTheDocument();

  // findBy*: ΠΕΡΙΜΕΝΕΙ να εμφανιστεί (returns Promise) — για async UI
  expect(await screen.findByText("Learn testing")).toBeInTheDocument();

  // queryBy*: επιστρέφει null αντί για error — για να ελέγξεις ΑΠΟΥΣΙΑ
  expect(screen.queryByText("Loading…")).not.toBeInTheDocument();
});
```

Η τριάδα **getBy / findBy / queryBy** είναι σίγουρη ερώτηση: getBy για ό,τι πρέπει να υπάρχει τώρα, findBy για ό,τι θα εμφανιστεί, queryBy για ό,τι ΔΕΝ πρέπει να υπάρχει.

Το network δεν χτυπιέται ποτέ αληθινά στα tests — το καθιερωμένο εργαλείο είναι το **MSW (Mock Service Worker)**: κάνει intercept τα requests σε επίπεδο δικτύου, οπότε το component τρέχει τον ΠΡΑΓΜΑΤΙΚΟ fetch κώδικά του χωρίς να ξέρει ότι το API είναι ψεύτικο.

## E2E με Playwright: τα κρίσιμα flows

```ts
import { test, expect } from "@playwright/test";

test("user can add a todo", async ({ page }) => {
  await page.goto("/");
  await page.getByLabel("New todo").fill("Buy milk");
  await page.getByRole("button", { name: "Add" }).click();
  await expect(page.getByText("Buy milk")).toBeVisible();
});
```

Ίδια φιλοσοφία queries (role/label), αλλά σε πραγματικό browser με πραγματικό app. Τα κρατάς λίγα — signup, login, core flow — γιατί είναι αργά και πιο εύθραυστα από τα component tests.

## Πώς απαντάς στη συνέντευξη

«What do you test in React?»

> I test user-visible behavior: rendering, interactions, validation, loading and error states — never internal state or implementation details, so refactoring doesn't break the suite. I query by role and label, which doubles as an accessibility check, use findBy queries for async UI, and mock the network with MSW so components run their real fetch code. A few Playwright tests cover the critical journeys end to end.
