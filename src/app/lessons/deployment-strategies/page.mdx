# Deployment Strategies & Zero-Downtime Releases

«Πώς βγάζεις νέο release χωρίς να ρίξεις το site;» — η ερώτηση που ξεχωρίζει όποιον έχει κάνει deploy σε production. Ένα απλό «σταμάτα το παλιό, ανέβασε το νέο» σημαίνει downtime και ρίσκο. Αυτό το μάθημα καλύπτει τις στρατηγικές zero-downtime (rolling, blue-green, canary), τα **feature flags**, τα **rollbacks**, και γιατί τα **health checks** + **backward-compatible migrations** είναι προϋπόθεση για όλα.

## Το πρόβλημα: «recreate» = downtime

Η αφελής προσέγγιση («σβήσε το παλιό, σήκωσε το νέο») έχει ένα παράθυρο όπου **τίποτα δεν τρέχει**, και αν το νέο release είναι χαλασμένο, έχεις downtime μέχρι να το πιάσεις. Οι στρατηγικές παρακάτω εξαλείφουν αυτό το παράθυρο και μειώνουν το blast radius.

## Rolling deployment

Αντικαθιστάς τα instances **σταδιακά**, λίγα-λίγα: ανεβάζεις το νέο σε κάποια, τα βγάζεις από τον load balancer, μετά συνεχίζεις στα υπόλοιπα.

- **+** Χωρίς downtime, χωρίς διπλάσιους πόρους.
- **−** Για λίγο τρέχουν **παλιά + νέα** έκδοση μαζί → πρέπει να είναι συμβατές (δες migrations).
- **−** Πιο αργό rollback (πρέπει να γυρίσεις σταδιακά).

## Blue-green deployment

Κρατάς **δύο πανομοιότυπα περιβάλλοντα**: **blue** (τρέχον) και **green** (νέο). Ανεβάζεις στο green, το δοκιμάζεις, και μετά **γυρνάς όλο το traffic** στο green με μια αλλαγή στον router/LB.

- **+** Ακαριαίο switch· **ακαριαίο rollback** (γύρνα πίσω στο blue).
- **−** Διπλάσιοι πόροι (δύο πλήρη περιβάλλοντα).

## Canary deployment

Στέλνεις το νέο release σε **μικρό ποσοστό** χρηστών (π.χ. 5%), παρακολουθείς metrics/errors, και αν όλα καλά **αυξάνεις σταδιακά** μέχρι 100%.

- **+** Πιάνεις προβλήματα με ελάχιστο blast radius (μόνο 5% επηρεάζεται).
- **−** Απαιτεί καλό monitoring + traffic splitting.

## Feature flags — αποσύνδεσε deploy από release

Ένα **feature flag** είναι ένα if που ενεργοποιεί/απενεργοποιεί λειτουργικότητα **χωρίς νέο deploy**. Έτσι «ξεκλειδώνεις» ένα feature όταν θες (ή μόνο σε beta users), και αν χαλάσει το **σβήνεις άμεσα** χωρίς rollback.

```txt
deploy (ο κώδικας μπαίνει, flag OFF) ≠ release (flag ON για χρήστες)
```

## Rollbacks & health checks

- **Rollback** — η ικανότητα να γυρίσεις γρήγορα στην προηγούμενη γνωστή-καλή έκδοση. Blue-green το κάνει ακαριαία· πάντα να μπορείς να κάνεις rollback.
- **Health checks** — endpoints (`/health`, readiness/liveness) που λένε στον LB/orchestrator αν ένα instance είναι έτοιμο. **Χωρίς αυτά**, καμία στρατηγική δεν δουλεύει — ο LB θα στείλει traffic σε instance που δεν σηκώθηκε ακόμα.

## Backward-compatible migrations — ο κρυφός κανόνας

Σε rolling/canary, **παλιός + νέος** κώδικας τρέχουν ταυτόχρονα πάνω στην **ίδια** βάση. Άρα οι DB migrations πρέπει να είναι **backward-compatible**: π.χ. για να αφαιρέσεις στήλη → πρώτα σταμάτα να τη γράφεις (deploy), μετά σβήσε τη στήλη (επόμενο deploy). Ποτέ «breaking» migration μαζί με το deploy που τη χρειάζεται — αλλιώς ο παλιός κώδικας σπάει.

## Πώς απαντάς στη συνέντευξη

> «How do you release without downtime?»

> I never recreate — that means a window of downtime and risk. The three strategies: rolling replaces instances gradually, no extra resources but old and new run together for a while; blue-green keeps two identical environments and flips the router to the new one, which gives an instant switch and instant rollback at the cost of double resources; canary sends the new version to a small percentage, watches metrics, and ramps up, catching issues with minimal blast radius. I decouple deploy from release with feature flags so I can turn a feature on or off without shipping, and kill it instantly if it misbehaves. Two things make all of this work: health checks so the load balancer only sends traffic to ready instances, and backward-compatible migrations — expand-then-contract — because old and new code share the same database during the rollout, and I always keep a fast rollback path.
