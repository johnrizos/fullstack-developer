# State, Events & Forms

Το state είναι η «μνήμη» ενός component και η πιο συχνή πηγή bugs. Στα interviews εξετάζουν αν καταλαβαίνεις γιατί το `setState` είναι async, πότε χρειάζεσαι functional update, και γιατί τα controlled inputs είναι ο default τρόπος για forms.

## Τι πρέπει να μπορείς στο τέλος

- να διαλέγεις σωστά τι ανήκει σε state και τι όχι,
- να κάνεις immutable updates σε arrays/objects,
- να χρησιμοποιείς functional updates όταν το νέο state εξαρτάται από το παλιό,
- να φτιάχνεις controlled inputs με typed events,
- να αποφεύγεις duplicate/derived state.

## Τι ανήκει σε state

State είναι μόνο ό,τι **αλλάζει με τον χρόνο ΚΑΙ δεν μπορεί να υπολογιστεί** από κάτι άλλο. Αν προκύπτει από props ή άλλο state, δεν είναι state — είναι derived.

```tsx
const [todos, setTodos] = useState<Todo[]>([]);
const [filter, setFilter] = useState<Filter>("all");

// ❌ ΟΧΙ state — υπολογίζεται
const [activeCount, setActiveCount] = useState(0);

// ✅ derived στο render
const activeCount = todos.filter((t) => !t.completed).length;
```

Διπλό state που «καθρεφτίζει» άλλο state ξεσυγχρονίζεται αναπόφευκτα. Ένα source of truth, τα υπόλοιπα υπολογισμένα.

## setState είναι async & batched

```tsx
function handleClick() {
  setCount(count + 1);
  console.log(count); // ΑΚΟΜΑ η παλιά τιμή — το render δεν έγινε ακόμα
}
```

Το `setState` προγραμματίζει render· δεν αλλάζει την τοπική μεταβλητή αμέσως. Το React μαζεύει (batches) πολλά setState σε ένα render για performance.

## Immutable updates σε arrays & objects

```tsx
// Προσθήκη
setTodos([...todos, newTodo]);

// Ενημέρωση ενός
setTodos(todos.map((t) => (t.id === id ? { ...t, completed: !t.completed } : t)));

// Διαγραφή
setTodos(todos.filter((t) => t.id !== id));

// Nested object
setUser({ ...user, address: { ...user.address, city: "Athens" } });
```

Ποτέ `todos.push()` ή `user.address.city = ...` — αυτό κάνει mutate την ίδια reference και το React δεν βλέπει αλλαγή. Πάντα νέα reference σε κάθε επίπεδο που αλλάζεις.

## Controlled inputs

Σε controlled input, το `value` έρχεται από state και ο χρήστης δεν μπορεί να αλλάξει τίποτα χωρίς να περάσει από `onChange`. Το React είναι το source of truth.

```tsx
function NameForm() {
  const [name, setName] = useState("");

  function handleSubmit(e: React.FormEvent) {
    e.preventDefault(); // αλλιώς το browser κάνει full reload
    console.log(name.trim());
  }

  return (
    <form onSubmit={handleSubmit}>
      <input
        value={name}
        onChange={(e: React.ChangeEvent<HTMLInputElement>) => setName(e.target.value)}
      />
      <button type="submit" disabled={name.trim().length < 2}>Save</button>
    </form>
  );
}
```

Πλεονεκτήματα: validation σε πραγματικό χρόνο, conditional disable, εύκολος έλεγχος/μετασχηματισμός της τιμής. Το `e.preventDefault()` στο submit είναι υποχρεωτικό — αλλιώς η σελίδα κάνει reload.

## Πώς απαντάς στη συνέντευξη

«Why is setState asynchronous, and when do you need a functional update?»

> `setState` schedules a re-render instead of mutating a variable, and React batches multiple updates into one render for performance — so right after calling it, the local variable still holds the old value. That's fine when I'm setting an independent value, but when the new state depends on the previous one — incrementing a counter, toggling, appending — I pass a function, `setCount(c => c + 1)`. The updater receives the latest queued value, so three of them in a row correctly produce +3, whereas reading `count` directly would read the same stale snapshot three times. For forms I default to controlled inputs: the value comes from state and changes go through `onChange`, which gives me validation and conditional submit, and I always `preventDefault` on submit and update state immutably.
