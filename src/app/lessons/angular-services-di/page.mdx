# Services & Dependency Injection

Το **Dependency Injection (DI)** είναι ίσως το πιο χαρακτηριστικό κομμάτι της Angular. Αντί ένα component να φτιάχνει μόνο του τα εργαλεία που χρειάζεται, η Angular του τα **εγχέει** (injects). Αυτό κάνει τον κώδικα testable, επαναχρησιμοποιήσιμο και loosely coupled — και ρωτιέται **πάντα** σε συνέντευξη Angular.

## Τι πρέπει να μπορείς στο τέλος

- να γράφεις ένα **service** με `@Injectable`,
- να το εγχέεις με τη function `inject()`,
- να εξηγείς το DI system και τα providers,
- να καταλαβαίνεις **singleton vs scoped** instances.

## Τι είναι ένα service

Ένα **service** είναι μια απλή class που κρατά λογική ή δεδομένα που **δεν ανήκουν** σε ένα συγκεκριμένο component — π.χ. API calls, auth state, ένα cart. Το `@Injectable` τη δηλώνει «εγχύσιμη»:

```ts
import { Injectable } from "@angular/core";

@Injectable({ providedIn: "root" })
export class UserService {
  private users = ["Ada", "Grace"];

  getUsers() {
    return this.users;
  }
  addUser(name: string) {
    this.users.push(name);
  }
}
```

Το **`providedIn: "root"`** λέει στην Angular: «φτιάξε **ένα** instance για όλη την εφαρμογή» (singleton). Αυτό είναι το συνηθισμένο.

## Inject το service σε component

Με το σύγχρονο API χρησιμοποιείς τη function **`inject()`**:

```ts
import { Component, inject } from "@angular/core";
import { UserService } from "./user.service";

@Component({
  selector: "app-user-list",
  standalone: true,
  template: `
    @for (u of users; track u) { <li>{{ u }}</li> }
  `,
})
export class UserListComponent {
  private userService = inject(UserService);
  users = this.userService.getUsers();
}
```

Δεν κάνεις ποτέ `new UserService()` — η Angular σου δίνει το instance. Έτσι **πολλά** components μοιράζονται το **ίδιο** service (και άρα το ίδιο state), και στα tests μπορείς να βάλεις ένα **mock** στη θέση του.

> Το παλιό στυλ ήταν injection μέσω constructor: `constructor(private userService: UserService) {}`. Δουλεύει ακόμα, αλλά η function `inject()` είναι πιο ευέλικτη (δουλεύει και εκτός constructor, σε composables-style functions).

## Πώς δουλεύει το DI system

Όταν η Angular φτιάχνει ένα component, κοιτά τι ζητάει (μέσω `inject()` ή constructor types) και ψάχνει σε έναν **injector** να βρει ή να δημιουργήσει το instance:

1. Ζητάς `UserService`.
2. Ο injector ψάχνει αν υπάρχει ήδη instance· αν όχι, το **δημιουργεί** (resolving και τα δικά του dependencies).
3. Σου το δίνει — και το **ξαναχρησιμοποιεί** στις επόμενες αιτήσεις (στο ίδιο scope).

Το **provider** είναι η «συνταγή»: τι class/τιμή να δώσει ο injector για ένα συγκεκριμένο token. Το `providedIn: "root"` είναι ο πιο απλός provider.

## Singleton vs scoped

- **`providedIn: "root"`** → **ένα** instance για όλη την εφαρμογή (singleton). Όλα τα components βλέπουν το ίδιο state. Αυτό θες για auth, cart, caches.
- **Provided σε επίπεδο component** (`providers: [SomeService]` στο `@Component`) → **νέο** instance ανά component (και τα παιδιά του). Χρήσιμο όταν κάθε instance χρειάζεται **δικό** του απομονωμένο state.

```ts
@Component({
  selector: "app-widget",
  standalone: true,
  providers: [WidgetState], // νέο instance ανά <app-widget>
  template: `…`,
})
export class WidgetComponent {
  state = inject(WidgetState);
}
```

## Πώς απαντάς στη συνέντευξη

> Angular's dependency injection lets a component declare what it needs instead of constructing it. I put reusable logic and state in a service marked `@Injectable({ providedIn: 'root' })`, which makes an app-wide singleton, and I pull it in with the `inject()` function — never `new`. The injector resolves and caches instances per scope, so multiple components share the same state and tests can swap in a mock. When a component needs its own isolated instance I provide the service at the component level instead of root, which gives a new instance per component subtree.
