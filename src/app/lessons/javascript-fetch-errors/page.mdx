# Fetch, Errors & API Handling

Το μισό frontend δουλειά είναι «φέρε δεδομένα, δείξε τα, και μην σπάσεις όταν κάτι πάει στραβά». Στις συνεντεύξεις, η ποιότητα ενός υποψηφίου φαίνεται ακριβώς εδώ: όλοι γράφουν το happy path — οι καλοί χειρίζονται **κάθε** πιθανή κατάσταση του request.

## Η ανατομία ενός fetch

```js
const response = await fetch("/api/users/1");

response.status;  // 200, 404, 500...
response.ok;      // true ΜΟΝΟ για status 200-299
response.headers; // Headers object

const data = await response.json(); // ΚΙ ΑΥΤΟ είναι async — μπορεί να αποτύχει!
```

Δύο σημεία που οι περισσότεροι δεν προσέχουν:

1. Το `response.json()` είναι **ξεχωριστό await** — διαβάζει και κάνει parse το body. Αν ο server γύρισε HTML error page αντί για JSON, **εδώ** σκάει το exception.
2. Το body διαβάζεται **μία φορά** — δεύτερο `response.json()` πετάει error.

## Το trap που πέφτει σε κάθε συνέντευξη

Το `fetch` **ΔΕΝ κάνει reject για HTTP errors**. Ένα 404 ή 500 είναι «επιτυχία» από τη σκοπιά του fetch — το request ολοκληρώθηκε και ήρθε response:

```js
// ❌ Αυτό ΔΕΝ πιάνει το 404 — το catch δεν θα τρέξει!
try {
  const response = await fetch("/api/users/999"); // 404
  const user = await response.json(); // ίσως σκάσει εδώ, ίσως όχι
} catch (error) {
  // φτάνει εδώ ΜΟΝΟ για network errors
}

// ✅ Έλεγχος response.ok — μετατρέπεις HTTP errors σε exceptions
const response = await fetch("/api/users/999");
if (!response.ok) {
  throw new Error(`HTTP ${response.status}`);
}
```

Το fetch κάνει reject **μόνο** όταν το request δεν ολοκληρώθηκε καθόλου: χαμένο δίκτυο, DNS failure, CORS block, abort. Αυτή η διάκριση — **network error vs HTTP error** — είναι η απάντηση-κλειδί.

## Το πλήρες, σωστό pattern

```js
async function fetchJson(url, options = {}) {
  let response;
  try {
    response = await fetch(url, options);
  } catch (error) {
    // Εδώ: ΜΟΝΟ network-level αποτυχίες
    throw new Error("Network error — check your connection");
  }

  if (!response.ok) {
    // Προσπάθησε να διαβάσεις το error body του API (αν υπάρχει)
    const body = await response.json().catch(() => null);
    throw new Error(body?.error?.message ?? `HTTP ${response.status}`);
  }

  return response.json();
}
```

Αυτό το helper κωδικοποιεί όλη τη γνώση: ξεχωριστός χειρισμός network vs HTTP errors, ασφαλές parsing του error body, καθαρό error μήνυμα προς τα πάνω.

## POST, headers και body

```js
const response = await fetch("/api/tasks", {
  method: "POST",
  headers: {
    "Content-Type": "application/json",
    Authorization: `Bearer ${token}`,
  },
  body: JSON.stringify({ title: "Learn fetch" }),
});
```

Τα τρία κλασικά λάθη: ξεχνάς το `JSON.stringify` (στέλνεις `[object Object]`), ξεχνάς το `Content-Type` (ο server δεν κάνει parse το body), και ξεχνάς ότι το `body` δεν επιτρέπεται σε GET.

## Timeouts & ακύρωση: AbortController

Το fetch **δεν έχει default timeout** — ένα κολλημένο request μπορεί να περιμένει για πάντα:

```js
const controller = new AbortController();
const timeout = setTimeout(() => controller.abort(), 5000);

try {
  const response = await fetch("/api/slow", { signal: controller.signal });
  // ...
} catch (error) {
  if (error.name === "AbortError") {
    console.log("Request timed out or was cancelled");
  }
} finally {
  clearTimeout(timeout);
}
```

Το ίδιο `AbortController` λύνει και το **race condition του search**: ο χρήστης πληκτρολογεί «re», μετά «react» — αν το πρώτο (αργό) response έρθει μετά το δεύτερο, θα πατήσει τα σωστά αποτελέσματα. Ακυρώνεις το προηγούμενο request πριν στείλεις νέο.

## Retries — με μέτρο και backoff

```js
async function fetchWithRetry(url, retries = 3) {
  for (let attempt = 1; attempt <= retries; attempt++) {
    try {
      const response = await fetch(url);
      // Retry ΜΟΝΟ σε server errors (5xx) — όχι σε 4xx!
      if (response.status >= 500 && attempt < retries) {
        await new Promise((r) => setTimeout(r, 2 ** attempt * 300)); // 600ms, 1.2s...
        continue;
      }
      return response;
    } catch (error) {
      if (attempt === retries) throw error; // network error: retry μέχρι το όριο
      await new Promise((r) => setTimeout(r, 2 ** attempt * 300));
    }
  }
}
```

Οι κανόνες που δείχνουν ωριμότητα: retry μόνο σε **5xx και network errors** (το 404 δεν θα γίνει 200 όσο και να ξαναρωτάς), **exponential backoff** (όχι σφυροκόπημα του ταλαιπωρημένου server), και retry μόνο σε **idempotent requests** — ένα POST πληρωμής θέλει idempotency key πριν τολμήσεις retry.

## Τα UI states: η πλήρης λίστα

Κάθε async οθόνη έχει **τουλάχιστον 4 καταστάσεις** — γράψε τις πριν γράψεις κώδικα:

| State | Τι δείχνεις |
|---|---|
| `loading` | Spinner/skeleton — και disable το submit |
| `success` με δεδομένα | Τα αποτελέσματα |
| `success` αλλά **κενά** | «Δεν βρέθηκαν αποτελέσματα» — ΟΧΙ κενή οθόνη |
| `error` | Καθαρό μήνυμα + κουμπί retry |

Το **empty state** είναι αυτό που ξεχνούν όλοι — και αυτό που οι interviewers προσέχουν στα live coding.

## Πώς απαντάς στη συνέντευξη

«How do you handle API errors in JavaScript?»

> First, fetch only rejects on network-level failures — HTTP errors like 404 or 500 resolve normally, so I always check response.ok and throw with the status. I parse JSON defensively since an error page might not be valid JSON, I use AbortController for timeouts and to cancel stale requests, and I retry only server errors with exponential backoff — never 4xx. On the UI side, every async view handles four states: loading, success, empty, and error with a retry action.
