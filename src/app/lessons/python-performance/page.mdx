# Performance & Profiling

«Make it work, make it right, make it fast» — με αυτή τη σειρά. Ένας expert δεν βελτιστοποιεί στα τυφλά· **μετράει πρώτα**, βρίσκει το πραγματικό bottleneck, και μετά διορθώνει. Στη συνέντευξη, το «ο κώδικάς μου είναι αργός, τι κάνεις;» έχει μία σωστή πρώτη απάντηση: «profile it». Το να ξέρεις τα εργαλεία και τις βασικές αρχές απόδοσης της Python σε ξεχωρίζει.

## Τι πρέπει να μπορείς στο τέλος

- να μετράς χρόνο σωστά (`timeit`) και να κάνεις profile (`cProfile`),
- να ξέρεις τα Big-O των βασικών δομών της Python,
- να αναγνωρίζεις κοινά performance traps (string concat, list membership, περιττά copies),
- να εφαρμόζεις caching (`lru_cache`) και να ξέρεις πότε βοηθά,
- να ξέρεις πότε φεύγεις από pure Python (numpy, C-extensions, άλλο process).

## Πρώτος κανόνας: μέτρα, μη μαντεύεις

Η διαίσθηση για το «τι είναι αργό» είναι σχεδόν πάντα λάθος. Μέτρα:

```py
import timeit

# timeit — για μικρά snippets, τρέχει πολλές φορές για ακρίβεια
timeit.timeit("sum(range(100))", number=100000)

# cProfile — για ολόκληρο πρόγραμμα: ποια function τρώει τον χρόνο
import cProfile
cProfile.run("main()")          # δείχνει calls, total time, per-function time
```

Το `cProfile` δείχνει **πού** πάει ο χρόνος — και συχνά η απάντηση είναι έκπληξη (π.χ. ένα αθώο `json.loads` σε loop, όχι ο «βαρύς» υπολογισμός που υποψιαζόσουν). Βελτιστοποιείς **μόνο** το αποδεδειγμένο bottleneck — το 90% του υπόλοιπου κώδικα δεν αξίζει τον κόπο (premature optimization).

## Big-O των δομών της Python

Η σωστή δομή κάνει τη μεγαλύτερη διαφορά — πολύ περισσότερο από micro-optimizations:

```txt
list:    index O(1) | append O(1) | x in list O(n)   | insert(0,x)/pop(0) O(n)
dict:    get/set O(1) | key in dict O(1)
set:     add O(1) | x in set O(1)
deque:   appendleft/popleft O(1)  (list pop(0) O(n))
```

Το πιο συχνό λάθος: επαναλαμβανόμενο `x in some_list` σε μεγάλα δεδομένα. Αν κάνεις πολλά membership checks, μετέτρεψε σε `set` — από O(n) σε O(1) ανά έλεγχο. Σε 50.000 lookups, αυτό είναι η διαφορά δευτερολέπτων από millisecond.

## Κοινά traps

```py
# ❌ String concatenation σε loop — O(n²): κάθε += φτιάχνει νέο string
result = ""
for s in parts:
    result += s
# ✓ join — O(n)
result = "".join(parts)

# ❌ Membership σε list μέσα σε loop — O(n*m)
for item in items:
    if item in big_list:    # O(n) κάθε φορά
        ...
# ✓ set lookup — O(1)
lookup = set(big_list)
for item in items:
    if item in lookup:      # O(1)
        ...

# ❌ Περιττό copy μεγάλης δομής σε loop
# ✓ generator/iterator για single pass, in-place όπου ασφαλές
```

Το `"".join(...)` αντί για `+=` σε loop είναι κλασικό: τα strings είναι immutable, οπότε κάθε `+=` δημιουργεί νέο string αντιγράφοντας — quadratic. Το `join` το κάνει σε ένα πέρασμα.

## Caching — υπολόγισε μία φορά

```py
from functools import lru_cache

@lru_cache(maxsize=1024)
def expensive(n: int) -> int:
    # ακριβός pure υπολογισμός — cache-άρεται ανά argument
    ...
```

Το `lru_cache` βοηθά όταν: η function είναι **pure** (ίδιο input → ίδιο output), καλείται **επανειλημμένα με ίδια arguments**, και ο υπολογισμός είναι **ακριβός**. ΔΕΝ βοηθά (και βλάπτει) όταν τα arguments είναι σχεδόν πάντα διαφορετικά (το cache απλώς τρώει μνήμη) ή η function έχει side effects.

## Πότε φεύγεις από pure Python

Η Python είναι αργή σε tight numeric loops (interpreter overhead). Αν το profiling δείξει ότι το bottleneck είναι αριθμητικό:

1. **Vectorize με numpy** — η πρώτη και συχνά μόνη λύση που χρειάζεσαι (είδες στο data μάθημα).
2. **C-extensions / Cython / native libs** — όταν numpy δεν αρκεί.
3. **Ξεχωριστό process / άλλη γλώσσα** — για CPU-bound που πρέπει να γίνει παράλληλα (θυμήσου το GIL).
4. **Async/concurrency** — αν το bottleneck είναι I/O, όχι CPU (διαφορετικό πρόβλημα — δες async μάθημα).

Το κλειδί: **διάγνωσε πρώτα** τι είδους bottleneck έχεις (CPU-bound vs I/O-bound), γιατί η λύση είναι τελείως διαφορετική.

## Πώς απαντάς στη συνέντευξη

«Your Python code is slow — how do you approach it?»

> First I measure instead of guessing — `cProfile` for the whole program, `timeit` for snippets — because intuition about hotspots is usually wrong, and I only optimize the bottleneck the profiler actually points to. Most wins come from algorithms and data structures, not micro-tweaks: swapping an O(n) `list` membership check for an O(1) `set`, using `"".join` instead of `+=` in a loop, or `deque` for queues. I add `lru_cache` for expensive pure functions called with repeating arguments. If the profiler shows a tight numeric loop, I vectorize with numpy, and only if that's not enough do I reach for C extensions or a separate process — and I first confirm whether the bottleneck is CPU-bound or I/O-bound, because the fix is completely different.
