# Scaling, Load Balancing & Data Distribution

Όταν ένα σύστημα μεγαλώνει, δύο πράγματα γίνονται bottleneck: τα **app servers** και η **βάση**. Αυτό το μάθημα δείχνει πώς κλιμακώνεις και τα δύο — load balancing & horizontal scaling για το compute, replication & sharding για τα δεδομένα — και την έννοια που τα διέπει όλα στην κλίμακα: **CAP theorem** και consistency models.

## Load balancing — μοίρασε το φορτίο

Ένας **load balancer** στέκεται μπροστά από πολλά (stateless) app servers και μοιράζει τα requests:

- **Algorithms**: round-robin (με τη σειρά), least-connections (στον λιγότερο φορτωμένο), IP hash (ίδιος χρήστης → ίδιος server).
- **Health checks**: σταματά να στέλνει σε server που δεν απαντά → αυτόματο failover.
- Είναι και ο ίδιος υποψήφιο **SPOF** — σε production τρέχει σε εφεδρεία (active-passive ή πολλαπλοί).

Επειδή τα app servers είναι stateless (προηγούμενο μάθημα), ο LB μπορεί να στείλει κάθε request οπουδήποτε — έτσι κλιμακώνεις το compute οριζόντια απλώς προσθέτοντας servers (auto-scaling με βάση CPU/traffic).

## Database scaling — το πραγματικό bottleneck

Τα app servers κλιμακώνονται εύκολα· η **βάση** είναι το δύσκολο. Τρεις τεχνικές:

### 1. Replication (αντίγραφα)

Ένα **primary** (writes) + πολλά **replicas** (reads). Επειδή τα περισσότερα συστήματα είναι **read-heavy**, στέλνεις τα reads στα replicas → κλιμακώνεις τα reads μαζικά.

- **Tradeoff: replication lag** — τα replicas ενημερώνονται με μικρή καθυστέρηση, οπότε ένα read αμέσως μετά από write μπορεί να δει «παλιά» δεδομένα (eventual consistency).
- Δίνει και **availability**: αν πέσει το primary, ένα replica προάγεται (failover).

### 2. Partitioning / Sharding (διαμοιρασμός)

Όταν τα **δεδομένα** ή τα **writes** δεν χωράνε σε ένα μηχάνημα, σπας τη βάση σε **shards** — κάθε shard κρατά ένα **υποσύνολο** των δεδομένων (π.χ. users A–M στο shard 1, N–Z στο shard 2), με βάση ένα **shard key**.

- Κλιμακώνει **writes** και **storage** (όχι μόνο reads όπως η replication).
- **Tradeoffs**: queries που χρειάζονται πολλά shards γίνονται ακριβά· κακό shard key → **hot shards** (ανισοκατανομή). Διάλεξε shard key με ομοιόμορφη κατανομή.

### 3. Caching (επόμενο μάθημα)

Το πιο φθηνό «scaling»: μη χτυπάς καθόλου τη βάση για συχνά δεδομένα.

## SQL vs NoSQL στην κλίμακα

- **SQL (relational)** — strong consistency, joins, transactions· κάθετη κλιμάκωση πιο εύκολη από οριζόντια (sharding είναι πιο δύσκολο).
- **NoSQL** — σχεδιασμένα για horizontal scaling & τεράστιο throughput, ευέλικτο schema· συχνά eventual consistency, χωρίς joins. Διάλεξε βάσει access patterns, όχι hype.

## CAP theorem — το αναπόφευκτο tradeoff

Σε ένα **κατανεμημένο** σύστημα, όταν συμβεί **network partition** (P) — δηλαδή κόβεται η επικοινωνία μεταξύ nodes — πρέπει να διαλέξεις **ένα** από τα δύο:

- **Consistency (C)** — κάθε read βλέπει την πιο πρόσφατη τιμή (ή σφάλμα). Προτίμηση σε τραπεζικά/πληρωμές.
- **Availability (A)** — το σύστημα πάντα απαντά, αλλά ίσως με «παλιά» δεδομένα. Προτίμηση σε feeds/likes.

Το P είναι αναπόφευκτο σε κατανεμημένα συστήματα, άρα στην πράξη διαλέγεις **CP** ή **AP**. Παράδειγμα: ένα banking ledger θέλει C (καλύτερα error παρά λάθος υπόλοιπο)· ένα social feed θέλει A (καλύτερα λίγο παλιό feed παρά κανένα).

## Consistency models

- **Strong consistency** — μετά από write, κάθε read βλέπει τη νέα τιμή. Ακριβό στην κλίμακα.
- **Eventual consistency** — τα αντίγραφα «συγκλίνουν» με τον χρόνο· προσωρινά μπορεί να δεις παλιά δεδομένα. Φθηνό, scalable — αποδεκτό για likes/views/feeds, όχι για υπόλοιπα λογαριασμού.

## Πώς απαντάς στη συνέντευξη

> «How do you scale the database when one server isn't enough?»

> First I separate the two bottlenecks. App servers are stateless, so I scale them horizontally behind a load balancer that health-checks and fails over. The database is the hard part. Since most systems are read-heavy, I start with replication — a primary for writes and read replicas for reads — accepting replication lag, which means a read right after a write might see stale data unless I read-your-writes from the primary. When writes or storage outgrow one machine, I shard by a key with even distribution, which scales writes too but makes cross-shard queries expensive and risks hot shards with a bad key. Underlying all of this is CAP: in a partition I choose consistency or availability — strong consistency for money, eventual consistency for feeds and likes — and I pick SQL or NoSQL by access patterns, not hype.
