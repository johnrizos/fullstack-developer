# CI/CD

CI/CD εμφανίζεται σχεδόν σε κάθε αγγελία backend/fullstack γιατί δείχνει ότι μπορείς να παραδώσεις λογισμικό **με προβλέψιμο τρόπο** — αυτόματα ελεγμένο, όχι «το ανέβασα και σταυρώνω τα δάχτυλα». Αυτό το μάθημα ξεκαθαρίζει τι είναι το CI, τι το CD, φτιάχνεις ένα πραγματικό GitHub Actions pipeline, και μαθαίνεις να διαβάζεις ένα αποτυχημένο build.

## CI: Continuous Integration

**Continuous Integration** σημαίνει ότι κάθε φορά που κάποιος ανοίγει PR ή κάνει push, τρέχουν **αυτόματα** μια σειρά από ελέγχους — **πριν** το merge. Είναι το **quality gate** που προστατεύει το main branch: αν κάτι σπάσει, φαίνεται αμέσως, όχι μια βδομάδα μετά σε production.

Ένα τυπικό CI pipeline για fullstack app:

```text
install → lint → typecheck → unit tests → build → integration tests
```

Αν **οποιοδήποτε** βήμα αποτύχει, το PR **δεν πρέπει** να γίνεται merge μέχρι να διορθωθεί. Το «Integration» στο όνομα αναφέρεται στην ιδέα ότι ενσωματώνεις τη δουλειά σου **συχνά** και σε **μικρά κομμάτια** — έτσι τα conflicts και τα regressions πιάνονται νωρίς και είναι μικρά.

## CD: Continuous Delivery vs Deployment

Το **CD** έχει δύο εκδοχές που μπερδεύονται συνεχώς — ξεκαθάρισέ τες (κλασική ερώτηση):

- **Continuous Delivery**: το pipeline ετοιμάζει ένα **έτοιμο, δοκιμασμένο release** — αλλά **άνθρωπος** πατά το κουμπί «deploy». Έλεγχος στο πότε βγαίνει.
- **Continuous Deployment**: όταν περάσουν **όλα** τα checks, το deploy γίνεται **αυτόματα**, χωρίς ανθρώπινο βήμα. Πλήρης αυτοματοποίηση.

Και οι δύο **προϋποθέτουν αξιόπιστο CI** — δεν κάνεις auto-deploy αν δεν εμπιστεύεσαι τα tests σου.

## Ένα πραγματικό GitHub Actions workflow

Το GitHub Actions τρέχει workflows που ορίζεις σε YAML μέσα στο `.github/workflows/`. Ας φτιάξουμε ένα CI που τρέχει σε κάθε PR:

```yaml
name: CI

on:
  pull_request:
  push:
    branches: [main]

jobs:
  checks:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm
      - run: npm ci
      - run: npm run lint
      - run: npm run typecheck
      - run: npm test
      - run: npm run build
```

Ας το διαβάσουμε:

- **`on:`** — **πότε** τρέχει: σε κάθε `pull_request` και σε κάθε `push` στο `main`.
- **`jobs:`** — μία ή περισσότερες δουλειές. Εδώ μία (`checks`).
- **`runs-on: ubuntu-latest`** — σε **καθαρό** μηχάνημα που στήνει το GitHub — γι' αυτό το CI πιάνει το «works on my machine»: ξεκινά από το μηδέν.
- **`steps:`** — τα βήματα με τη σειρά. Τα `uses:` είναι έτοιμα actions (checkout του κώδικα, setup του Node με cache)· τα `run:` είναι δικές σου εντολές.

Αν κάποιο `run` επιστρέψει non-zero exit code, το job **αποτυγχάνει** και το PR σημειώνεται ως failing.

## npm ci vs npm install — γιατί στο CI;

Πρόσεξες `npm ci` και όχι `npm install`. Στο CI θέλεις **αναπαραγωγιμότητα**:

- `npm ci` κάνει **καθαρό** install **ακριβώς** από το `package-lock.json`: σβήνει το `node_modules`, **αποτυγχάνει** αν το lock και το `package.json` διαφωνούν, και **ποτέ** δεν τροποποιεί το lock.
- `npm install` μπορεί να αλλάξει το lock και να τραβήξει ελαφρώς διαφορετικές εκδόσεις.

Σε clean environment (όπως το CI runner), το `npm ci` είναι και ταχύτερο και ντετερμινιστικό — ακριβώς ό,τι θες.

## Ταχύτητα: caching & matrix

Ένα αργό pipeline είναι ένα pipeline που η ομάδα αρχίζει να **αγνοεί** ή να κάνει bypass. Δύο εργαλεία που το κρατούν γρήγορο:

- **Dependency caching** — μη ξανακατεβάζεις τα ίδια packages σε κάθε run. Το `setup-node` με `cache: npm` (που ήδη έχεις) το κάνει: αν το `package-lock.json` δεν άλλαξε, επαναχρησιμοποιεί το cached `~/.npm`.
- **Matrix** — τρέξε το **ίδιο** job σε πολλούς συνδυασμούς **παράλληλα** (π.χ. πολλές εκδόσεις Node), χωρίς copy-paste:

```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node: [20, 22]
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node }}
          cache: npm
      - run: npm ci
      - run: npm test
```

Αυτό σηκώνει **δύο παράλληλα** jobs (Node 20 & 22). Ανεξάρτητα jobs τρέχουν ταυτόχρονα — άρα σπάσε τα βαριά βήματα σε παράλληλα jobs αντί για ένα μακρύ σειριακό.

## CD στην πράξη: ένα deploy job με approval

Το CI workflow που είδες σταματά στο build. Ας δούμε πώς μοιάζει το **CD** — ένα `deploy` job που τρέχει **μόνο** μετά το πράσινο CI και **μόνο** στο `main`:

```yaml
jobs:
  checks:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: npm }
      - run: npm ci
      - run: npm test
      - run: npm run build

  deploy:
    needs: checks                     # τρέχει ΜΟΝΟ αν πέρασε το checks
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    environment: production           # ← το κλειδί για approval
    steps:
      - uses: actions/checkout@v4
      - run: ./scripts/deploy.sh
```

Δύο κρίσιμα σημεία:

- **`needs: checks`** — το `deploy` δεν ξεκινά αν το `checks` απέτυχε. Ποτέ deploy από σπασμένο build.
- **`environment: production`** — δηλώνεις ένα GitHub **Environment**. Πάνω του βάζεις **protection rules**: **required reviewers** (κάποιος πρέπει να πατήσει «Approve» πριν τρέξει το deploy), wait timers, ή περιορισμό branches.

Εδώ γίνεται **χειροπιαστή** η διάκριση Delivery vs Deployment:

- Βάζεις **required reviewer** στο environment → **Continuous Delivery** (άνθρωπος πατά το κουμπί).
- Το αφήνεις **χωρίς** reviewer → **Continuous Deployment** (μόλις περάσει το CI, βγαίνει αυτόματα).

Ίδιο pipeline — η μόνη διαφορά είναι το protection rule.

## Secrets & OIDC: πρόσβαση χωρίς μόνιμα κλειδιά

Το `deploy` job χρειάζεται credentials (π.χ. για AWS/Vercel). **Ποτέ hardcoded** — δύο σωστοί τρόποι, από τον καλό στον καλύτερο:

**1. GitHub Actions Secrets** — αποθηκεύεις το secret στο repo/environment settings και το διαβάζεις ως `${{ secrets.NAME }}`. Το GitHub τα **κρύβει (masks)** στα logs.

```yaml
- run: ./deploy.sh
  env:
    DEPLOY_TOKEN: ${{ secrets.DEPLOY_TOKEN }}
```

**2. OIDC (το σύγχρονο best practice)** — αντί να αποθηκεύεις ένα **μακρόβιο** cloud key (που μπορεί να διαρρεύσει και ζει για πάντα), το GitHub Actions ανταλλάσσει ένα **βραχύβιο, per-run** token με τον cloud provider. Δεν υπάρχει μόνιμο μυστικό να κλαπεί:

```yaml
    permissions:
      id-token: write        # επιτρέπει στο job να ζητήσει OIDC token
      contents: read
    steps:
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789:role/deploy
          aws-region: eu-central-1
```

Ο cloud provider εμπιστεύεται το GitHub ως identity provider και δίνει προσωρινά credentials μόνο σε **αυτό** το workflow run. Μηδέν long-lived keys στο repo — αυτό ψάχνει ένας security-minded reviewer.

## Πώς διαβάζεις ένα αποτυχημένο pipeline

Όταν «κοκκινίσει» το CI, μην πανικοβάλλεσαι — διάβασέ το με σειρά:

1. **Ποιο job** απέτυχε;
2. **Ποιο step/command** μέσα στο job;
3. Ποιο **error message** εμφανίστηκε **πρώτο** (όχι το τελευταίο — το πρώτο είναι συνήθως η ρίζα).
4. **Αναπαράγεται τοπικά;** Τρέξε την ίδια εντολή στο μηχάνημά σου.

Και ο χρυσός κανόνας: **μην κάνεις bypass τα checks χωρίς λόγο.** Τα checks υπάρχουν για να προστατεύουν το main. Αν κάτι είναι flaky, διόρθωσε την αιτία — μην το απενεργοποιείς.

## Καλές πρακτικές

- **Κράτα τα CI jobs γρήγορα** — αν αργούν, η ομάδα σταματά να τα εμπιστεύεται/χρησιμοποιεί (cache dependencies, τρέξε βήματα παράλληλα).
- **Μην τυπώνεις secrets στα logs** — χρησιμοποίησε GitHub Actions secrets, όχι hardcoded τιμές.
- **Το build πρέπει να περνά πριν το deploy** — ποτέ deploy από σπασμένο branch.
- **Branch protection**: απαίτησε τα checks να περνούν πριν επιτραπεί το merge.
- **Migrations με rollback plan** — μην κάνεις deploy schema change χωρίς σχέδιο επαναφοράς.

## Πώς απαντάς στη συνέντευξη

> «Explain CI/CD.»

> CI, continuous integration, runs automated checks on every PR and push — install, lint, typecheck, tests, and build — on a clean machine, before anything merges. It's the quality gate that protects main and catches regressions early because everyone integrates small changes often. I run these in GitHub Actions with npm ci for reproducible, lockfile-exact installs, and keep them fast with dependency caching and matrix jobs running in parallel. CD comes in two flavors: continuous delivery produces an always-ready, tested release but a human clicks deploy, while continuous deployment ships automatically once all checks pass — both depend on trustworthy CI. In practice the deploy job uses needs so it only runs after green checks, and an Environment with a required reviewer is what turns deployment into delivery — same pipeline, just a protection rule. For cloud credentials I prefer OIDC over long-lived keys so each run gets short-lived, per-run access with no permanent secret to leak. When a pipeline fails I read which job and command failed and the first error, reproduce it locally, and fix the root cause rather than bypassing the check.
