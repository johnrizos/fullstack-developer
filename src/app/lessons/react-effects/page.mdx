# Effects & Lifecycle

Το `useEffect` είναι το πιο παρεξηγημένο hook. Οι περισσότεροι το χρησιμοποιούν υπερβολικά — για πράγματα που ανήκουν στο render. Στα interviews εξετάζουν αν ξέρεις **πότε ΔΕΝ χρειάζεσαι effect**, πώς δουλεύει το cleanup, και πώς αποφεύγεις race conditions.

## Τι πρέπει να μπορείς στο τέλος

- να ξεχωρίζεις «συγχρονισμό με εξωτερικό σύστημα» από «αντίδραση σε state»,
- να διαβάζεις σωστά το dependency array,
- να γράφεις cleanup functions για subscriptions/timers/requests,
- να λύνεις race conditions με AbortController/flag,
- να αναγνωρίζεις effects που είναι περιττά.

## Τι είναι effect

Το `useEffect` συγχρονίζει το component με **εξωτερικά συστήματα**: network, subscriptions, timers, DOM APIs, browser storage. Δεν είναι «τρέξε κώδικα μετά το render για να αλλάξεις state» — αυτό συνήθως είναι bug.

```tsx
useEffect(() => {
  const id = setInterval(() => setTick((t) => t + 1), 1000);
  return () => clearInterval(id); // cleanup: σταμάτα το προηγούμενο
}, []); // [] = μία φορά στο mount
```

## Το dependency array

```txt
useEffect(fn)            → κάθε render            (σχεδόν πάντα bug)
useEffect(fn, [])        → μία φορά στο mount
useEffect(fn, [filter])  → στο mount + όποτε αλλάξει το filter
```

Το array λέει στο React **πότε** να ξανατρέξει το effect. Κάθε τιμή που χρησιμοποιείς μέσα στο effect και μπορεί να αλλάξει πρέπει να μπει στο array — αλλιώς δουλεύεις με stale τιμές. Το ESLint `exhaustive-deps` το ελέγχει.

## Cleanup — πριν το επόμενο effect & στο unmount

Η returned function τρέχει **πριν** ξανατρέξει το effect και στο unmount. Εκεί ακυρώνεις ό,τι έστησες.

```tsx
useEffect(() => {
  const sub = chatRoom.subscribe(roomId);
  return () => sub.unsubscribe(); // αλλιώς διπλά subscriptions / memory leak
}, [roomId]);
```

Όταν αλλάζει το `roomId`: cleanup του παλιού room → setup του νέου. Χωρίς cleanup μένεις subscribed σε πολλά rooms ταυτόχρονα.

```tsx
useEffect(() => {
  let ignore = false;
  fetch(`/api/user/${userId}`)
    .then((r) => r.json())
    .then((data) => {
      if (!ignore) setUser(data); // αγνόησε αν άλλαξε το effect
    });
  return () => {
    ignore = true; // ή controller.abort()
  };
}, [userId]);
```

## Πότε ΔΕΝ χρειάζεσαι effect

Το πιο σημαντικό κομμάτι. Δεν χρειάζεσαι effect για:

```tsx
// ❌ Effect για derived τιμή
useEffect(() => setFullName(`${first} ${last}`), [first, last]);
// ✅ Υπολόγισέ το στο render
const fullName = `${first} ${last}`;

// ❌ Effect για να «αντιδράσεις» σε event
useEffect(() => { if (submitted) showToast(); }, [submitted]);
// ✅ Κάν' το στον event handler
function handleSubmit() { ...; showToast(); }
```

Κανόνας: αν κάτι μπορεί να υπολογιστεί στο render ή να γίνει σε event handler, **δεν** πάει σε effect. Τα effects είναι μόνο για εξωτερικό συγχρονισμό.

## Πώς απαντάς στη συνέντευξη

«When should you use useEffect, and when not?»

> useEffect is for synchronizing with external systems — data fetching, subscriptions, timers, DOM APIs. It is not for reacting to state or computing values: anything derivable goes in the render body, and anything triggered by a user action goes in the event handler. The dependency array controls when the effect re-runs, and every changing value I read inside has to be listed, or I get stale data. Every effect that sets something up returns a cleanup that runs before the next run and on unmount — that's where I unsubscribe, clear timers, or abort requests. For fetches that's critical: if the user switches fast, a slow earlier response can overwrite a newer one, so I guard with an AbortController or an `ignore` flag in cleanup so a stale response is dropped.
