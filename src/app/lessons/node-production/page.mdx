# Performance, Scaling & Production

Το να τρέχει το API στο laptop σου είναι η αρχή. Το να αντέχει φορτίο, να αξιοποιεί όλους τους πυρήνες, να κάνει graceful shutdown και να σου λέει τι συμβαίνει όταν χαλάσει — αυτό είναι production. Στη συνέντευξη: «πώς κλιμακώνεις ένα Node app;», «πώς αξιοποιείς πολλούς πυρήνες αφού είναι single-threaded;».

## Τι πρέπει να μπορείς στο τέλος

- να κλιμακώνεις με cluster/πολλαπλά instances (το Node είναι single-threaded),
- να βγάζεις CPU-bound δουλειά σε `worker_threads`,
- να προσθέτεις caching σωστά (Redis, με TTL),
- να κάνεις graceful shutdown και health checks,
- να στήνεις structured logging και observability για production.

## Scaling — το Node είναι single-threaded

Ένα Node process χρησιμοποιεί **έναν** πυρήνα. Για να αξιοποιήσεις 8 πυρήνες, τρέχεις **8 instances** πίσω από load balancer:

```ts
// Επιλογή 1: cluster module — fork workers ανά πυρήνα
import cluster from "node:cluster";
import os from "node:os";

if (cluster.isPrimary) {
  for (let i = 0; i < os.availableParallelism(); i++) cluster.fork();
} else {
  startServer();   // κάθε worker τρέχει το app
}
```

Στην πράξη, σε production συνήθως αφήνεις την **ορchestration έξω από το app**: πολλαπλά container replicas (Kubernetes/ECS) ή έναν process manager (PM2). Το concept που πρέπει να ξέρεις: το Node scale-άρει **οριζόντια** (περισσότερα instances), όχι κάθετα μέσα σε ένα process — γι' αυτό τα instances πρέπει να είναι **stateless** (sessions/cache σε Redis, όχι στη μνήμη του process).

## CPU-bound → worker_threads

Όπως είδες στο runtime μάθημα, βαριά συγχρονισμένη CPU δουλειά μπλοκάρει το event loop. Βγάλ' την σε worker:

```ts
import { Worker } from "node:worker_threads";

function runHeavyTask(data) {
  return new Promise((resolve, reject) => {
    const worker = new Worker("./heavy-worker.js", { workerData: data });
    worker.on("message", resolve);
    worker.on("error", reject);
  });
}
// Το main thread μένει ελεύθερο να εξυπηρετεί requests
```

Για πολύ βαριά/μακροχρόνια δουλειά, προτίμησε **job queue** (BullMQ + Redis) με ξεχωριστό worker process — έτσι το API επιστρέφει αμέσως και η δουλειά γίνεται ασύγχρονα (ίδιο pattern με το Python jobs μάθημα).

## Caching με Redis

Cache για ακριβά reads που δεν αλλάζουν συχνά, με **TTL** ώστε να μη μένουν stale:

```ts
async function getUser(id: string) {
  const cached = await redis.get(`user:${id}`);
  if (cached) return JSON.parse(cached);          // cache hit

  const user = await db.user.findUnique({ where: { id } });
  await redis.set(`user:${id}`, JSON.stringify(user), "EX", 300);   // TTL 300s
  return user;
}
```

Κανόνες: πάντα **TTL** (να μη μένει stale για πάντα)· **invalidate** το cache όταν αλλάζει το δεδομένο· cache **stateless** (σε Redis, όχι στη μνήμη του process — αλλιώς κάθε instance έχει διαφορετικό cache).

## Graceful shutdown & health checks

Όταν το orchestration στέλνει `SIGTERM` (deploy/scale-down), το app πρέπει να **τελειώσει τα τρέχοντα requests** πριν κλείσει — όχι να τα κόψει στη μέση:

```ts
const server = app.listen(config.PORT);

process.on("SIGTERM", async () => {
  server.close(async () => {          // σταμάτα να δέχεσαι νέα requests
    await db.$disconnect();           // κλείσε connections καθαρά
    await redis.quit();
    process.exit(0);
  });
});

// Health check — το χρησιμοποιεί ο orchestrator/load balancer
app.get("/health", (_req, res) => res.json({ status: "ok" }));
```

Το `/health` (και ένα `/ready` που ελέγχει db/redis) λέει στον load balancer πότε το instance είναι έτοιμο να δεχτεί traffic.

## Structured logging & observability

`console.log` με strings δεν φιλτράρεται. Σε production: **structured JSON logs** (pino/winston) που πάνε σε aggregator:

```ts
import pino from "pino";
const log = pino();

log.info({ reqId, userId, route: "/tasks", ms: 42 }, "request handled");
log.error({ reqId, err }, "task creation failed");
```

Τα τρία pillars της observability: **logs** (τι έγινε), **metrics** (πόσο/πόσο γρήγορα — latency, error rate, throughput), **traces** (η διαδρομή ενός request μέσα από services). Πρόσθεσε error tracking (Sentry) και θα ξέρεις τι σπάει στην παραγωγή πριν σου το πει ο χρήστης.

## Πώς απαντάς στη συνέντευξη

«How do you scale and run a Node app in production?»

> Node executes JavaScript on a single thread, so I scale horizontally — multiple instances behind a load balancer, via the cluster module or, more commonly, container replicas. That requires the instances to be stateless, so sessions and cache live in Redis, not process memory. CPU-bound work goes to worker threads or, if it's heavy, a job queue with a separate worker so the API stays responsive. I cache expensive reads in Redis with a TTL and invalidate on writes. For reliability I implement graceful shutdown on SIGTERM — stop accepting new requests, drain in-flight ones, close connections — and expose health and readiness endpoints. And I run structured JSON logging plus metrics, traces, and error tracking so I can see what's happening in production instead of guessing.
