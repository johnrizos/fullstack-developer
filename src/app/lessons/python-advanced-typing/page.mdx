# Advanced Typing

Στο OOP/Typing μάθημα είδες τα βασικά type hints. Εδώ είναι το επίπεδο που χρησιμοποιούν βιβλιοθήκες και μεγάλα codebases: **generics, protocols, type narrowing, strict mypy**. Όταν μια ομάδα τρέχει `mypy --strict` στο CI, αυτά είναι η καθημερινότητα — και η ικανότητα να γράφεις σωστά τυποποιημένο, επαναχρησιμοποιήσιμο κώδικα είναι ξεκάθαρο σημάδι senior Python developer.

## Τι πρέπει να μπορείς στο τέλος

- να γράφεις generic functions/classes με `TypeVar` και `Generic`,
- να χρησιμοποιείς `Protocol` για structural (duck) typing,
- να καταλαβαίνεις type narrowing και `TypeGuard`,
- να χρησιμοποιείς `Optional`, `Union`, `Literal`, `Final`, `TypedDict`,
- να ρυθμίζεις και να ικανοποιείς `mypy --strict`.

## Generics — τύποι που «θυμούνται»

Χωρίς generics, μια function που επιστρέφει «ό,τι της έδωσες» χάνει τον τύπο. Το `TypeVar` τον κρατά:

```py
from typing import TypeVar

T = TypeVar("T")

def first(items: list[T]) -> T:        # ο τύπος εξόδου = τύπος στοιχείου
    return items[0]

first([1, 2, 3])        # ο checker ξέρει: int
first(["a", "b"])       # ο checker ξέρει: str  (όχι "object")
```

Χωρίς το `T`, θα έπρεπε να γράψεις `-> object` ή `-> Any` και να χάσεις κάθε ασφάλεια. Με το `T`, ο checker «δένει» είσοδο και έξοδο.

Στη σύγχρονη Python (3.12+) το syntax έγινε καθαρότερο:

```py
def first[T](items: list[T]) -> T:     # 3.12+ — δηλώνεις το T inline
    return items[0]

class Stack[T]:                        # generic class
    def __init__(self) -> None:
        self._items: list[T] = []
    def push(self, item: T) -> None:
        self._items.append(item)
    def pop(self) -> T:
        return self._items.pop()

s: Stack[int] = Stack()
s.push(1)
reveal_type(s.pop())   # mypy: int
```

## Protocols — structural typing (duck typing με δόντια)

Αντί να απαιτείς συγκεκριμένο base class (nominal typing), ένα `Protocol` λέει «δέχομαι ΟΤΙΔΗΠΟΤΕ έχει αυτές τις μεθόδους» — στατικά ελεγμένο duck typing:

```py
from typing import Protocol

class Closeable(Protocol):
    def close(self) -> None: ...

def shutdown(resource: Closeable) -> None:
    resource.close()

# Οποιοδήποτε object με close() δουλεύει — ΧΩΡΙΣ να κληρονομεί το Closeable
class File:
    def close(self) -> None: ...
class Socket:
    def close(self) -> None: ...

shutdown(File())     # ✓ ο mypy το δέχεται
shutdown(Socket())   # ✓ — δομικά ταιριάζει
```

Αυτό είναι ισχυρό: γράφεις functions που εξαρτώνται από **συμπεριφορά**, όχι από ιεραρχία κληρονομικότητας — το «αληθινό» Pythonic duck typing, αλλά τώρα ο checker το επιβάλλει.

## Type narrowing — ο checker «μαθαίνει» από τους ελέγχους

Ο mypy παρακολουθεί τους ελέγχους σου και στενεύει τον τύπο:

```py
def process(value: int | str | None) -> str:
    if value is None:
        return "empty"          # εδώ: None
    if isinstance(value, int):
        return str(value * 2)   # εδώ ο checker ξέρει: int
    return value.upper()        # εδώ απομένει: str — το .upper() είναι ασφαλές
```

Μετά από `if value is None: return`, ο checker αφαιρεί το `None` από τους πιθανούς τύπους. Μετά το `isinstance(value, int)`, ξέρει ότι στο `else` απομένει μόνο `str`. Γι' αυτό δεν χρειάζεσαι casts — αρκούν οι σωστοί έλεγχοι.

## Χρήσιμα typing constructs

```py
from typing import Literal, Final, TypedDict

# Literal — μόνο συγκεκριμένες τιμές επιτρέπονται
def set_mode(mode: Literal["r", "w", "a"]) -> None: ...
set_mode("x")    # mypy error! μόνο "r"/"w"/"a"

# Final — σταθερά που δεν αλλάζει
MAX_RETRIES: Final = 3

# TypedDict — dict με γνωστά keys/τύπους (π.χ. JSON shape)
class UserDict(TypedDict):
    id: int
    name: str
    active: bool

u: UserDict = {"id": 1, "name": "Ana", "active": True}
u["age"]   # mypy error! δεν υπάρχει 'age' στο schema
```

Το `Literal` είναι ιδανικό για enums-σαν-strings (status, mode), το `TypedDict` για JSON payloads όπου ξέρεις το σχήμα αλλά θες να μείνει dict.

## mypy `--strict` — το πραγματικό φίλτρο

Σε production codebase τρέχεις:

```bash
mypy --strict src/
```

Το `--strict` ενεργοποιεί όλους τους αυστηρούς ελέγχους: απαγορεύει σιωπηλό `Any`, απαιτεί type hints σε κάθε function, πιάνει `None` που δεν χειρίζεσαι. Είναι ο τρόπος που ένα μεγάλο Python codebase μένει ασφαλές παρά το dynamic typing — και η συνήθης ερώτηση «πώς κρατάτε type safety σε Python project;» έχει απάντηση: strict mypy στο CI.

## Πώς απαντάς στη συνέντευξη

«How do you keep a large Python codebase type-safe when types aren't enforced at runtime?»

> I lean on a static checker — mypy in strict mode — running in CI, so type errors are caught before merge even though the interpreter ignores annotations. For reusable code I use generics with `TypeVar` so types flow from input to output instead of collapsing to `Any`, and I prefer `Protocol` over base classes when I just need an object with the right shape — structural typing that matches Python's duck-typing spirit but is checked statically. I narrow unions with `isinstance`/`is None` checks so the checker tracks the type without casts, and I reach for `Literal`, `TypedDict`, and `Final` to make string modes, JSON shapes, and constants precise. The payoff is dynamic-language flexibility with static-language safety.
