# Express Fundamentals

Το Express είναι ο πιο διαδεδομένος τρόπος να φτιάξεις HTTP API σε Node — minimal, unopinionated, και η βάση που πρέπει να κατέχεις πριν πας σε NestJS. Στη συνέντευξη: «τι κάνει το `express.json()`;», «πώς δουλεύει το routing;», «route handler vs middleware;».

## Τι πρέπει να μπορείς στο τέλος

- να στήνεις Express app με routes και σωστά HTTP methods/status codes,
- να καταλαβαίνεις το `req`/`res` lifecycle,
- να χρησιμοποιείς route parameters, query strings και request body,
- να κρατάς το route handler «λεπτό» (μόνο HTTP concerns),
- να στέλνεις σωστά JSON responses και status codes.

## Το πρώτο API

```ts
import express from "express";

const app = express();
app.use(express.json());   // ΧΩΡΙΣ αυτό, το req.body είναι undefined!

app.get("/tasks", async (req, res) => {
  const tasks = await taskService.list({ completed: req.query.completed === "true" });
  res.json({ data: tasks });
});

app.post("/tasks", async (req, res, next) => {
  try {
    const title = req.body.title;
    if (typeof title !== "string" || title.trim().length < 2) {
      return res.status(422).json({ error: "Title must be at least 2 characters" });
    }
    const task = await taskService.create({ title: title.trim() });
    res.status(201).json({ data: task });   // 201 Created για νέο resource
  } catch (error) {
    next(error);   // προώθηση σε centralized error handler
  }
});

app.listen(3000);
```

Το `express.json()` είναι middleware που κάνει parse το JSON body — χωρίς αυτό, το `req.body` είναι `undefined`. Κλασικό «γιατί δεν δουλεύει το POST μου» bug.

## req & res — τα βασικά

```ts
app.get("/users/:id", (req, res) => {
  req.params.id        // route parameter: /users/42 → "42"
  req.query.sort       // query string: ?sort=name → "name"
  req.body             // parsed body (χρειάζεται express.json())
  req.headers          // request headers
  req.method, req.path // GET, /users/42

  res.status(200)      // status code (chainable)
     .set("X-Custom", "value")   // response header
     .json({ data });            // JSON + τερματισμός
});
```

Το `res` τερματίζει το request: `res.json()`, `res.send()`, `res.end()`, `res.redirect()`. **Μόνο ένα** ανά request — διπλό `res.json` ρίχνει «headers already sent».

## HTTP methods & status codes — σωστά

```txt
GET     /tasks       → 200  λίστα (idempotent, χωρίς side effects)
GET     /tasks/:id   → 200 ή 404
POST    /tasks       → 201  δημιουργία (επιστρέφει το νέο resource)
PUT     /tasks/:id   → 200  πλήρης αντικατάσταση
PATCH   /tasks/:id   → 200  μερική ενημέρωση
DELETE  /tasks/:id   → 204  διαγραφή (No Content)

422 Unprocessable   → invalid input (validation failed)
401 Unauthorized    → δεν είσαι authenticated
403 Forbidden       → είσαι authenticated αλλά δεν επιτρέπεται
500 Internal Error  → unexpected server error
```

Τα σωστά status codes είναι μέρος του API contract — ο client βασίζεται σε αυτά. Επιστροφή `200` με `{error}` μέσα είναι anti-pattern: ο client δεν μπορεί να ξεχωρίσει success από failure χωρίς να διαβάσει το body.

## Router — οργάνωση routes

Καθώς μεγαλώνει το API, σπας τα routes σε modules με `express.Router`:

```ts
// routes/tasks.routes.ts
import { Router } from "express";
const router = Router();

router.get("/", listTasks);
router.post("/", createTask);
router.patch("/:id", updateTask);

export default router;

// app.ts
app.use("/tasks", tasksRouter);   // όλα τα routes με prefix /tasks
```

Έτσι το `app.ts` μένει καθαρό και κάθε resource έχει το δικό του router file.

## Το route handler μένει λεπτό

Το route handler κρατά **μόνο** τα HTTP concerns: parsing, validation, status codes. Η δουλειά γίνεται στο service:

```ts
app.post("/tasks", async (req, res, next) => {
  try {
    const task = await taskService.create(req.body);  // λογική στο service
    res.status(201).json({ data: task });             // handler: μόνο HTTP
  } catch (error) {
    next(error);
  }
});
```

Γιατί; Το service τεστάρεται **χωρίς HTTP** και επαναχρησιμοποιείται (π.χ. από cron ή queue worker). Ο handler διαβάζεται σε 5 γραμμές. (Περισσότερα στο μάθημα Layered Architecture.)

## Πώς απαντάς στη συνέντευξη

«Walk me through handling a POST request in Express.»

> First I register `express.json()` so the JSON body gets parsed — without it `req.body` is undefined, which is the classic beginner bug. In the handler I read params, query, and body off `req`, validate the input and return `422` with a clear message if it's bad, then delegate the actual work to a service so the handler only deals with HTTP concerns. On success I respond with the right status — `201` for a creation, returning the new resource — and `res.json` ends the request, so I only send one response. Any unexpected error I forward with `next(error)` to a centralized error handler instead of catching it inline everywhere. As the API grows I split routes into `express.Router` modules per resource.
