# Migrations & Seeding

Το schema αλλάζει συνεχώς: νέα πεδία, νέοι πίνακες, renames. Οι migrations είναι το versioned ιστορικό αυτών των αλλαγών — και η πηγή πολλών production ατυχημάτων αν τις χειριστείς λάθος. Στη συνέντευξη: «migrate dev vs deploy;», «τι είναι schema drift;», «πώς κάνεις safe migration σε production;».

## Τι πρέπει να μπορείς στο τέλος

- να διακρίνεις `migrate dev` από `migrate deploy`,
- να καταλαβαίνεις το shadow database και το drift,
- να κάνεις safe schema changes σε production (expand/contract),
- να γράφεις seed script.

## migrate dev vs deploy

```bash
# DEVELOPMENT — φτιάχνει & εφαρμόζει migration, regenerate client
npx prisma migrate dev --name add_user_role

# PRODUCTION — εφαρμόζει ΜΟΝΟ υπάρχουσες migrations, καμία δημιουργία
npx prisma migrate deploy
```

```txt
migrate dev     → συγκρίνει schema, δημιουργεί νέο migration file,
                  το εφαρμόζει, regenerate client. Χρησιμοποιεί shadow DB.
                  ΜΟΝΟ για development.
migrate deploy  → εφαρμόζει pending migrations με τη σειρά. Idempotent,
                  μη-interactive. Αυτό τρέχεις σε CI/CD & production.
```

Κρίσιμο: **ποτέ** `migrate dev` σε production — μπορεί να ζητήσει reset/να χάσει δεδομένα. Σε production μόνο `migrate deploy`.

## Shadow database & drift

Το `migrate dev` φτιάχνει προσωρινά μια **shadow database**: εφαρμόζει εκεί όλες τις migrations από την αρχή για να ανιχνεύσει **drift** — αν η πραγματική βάση δεν ταιριάζει με το migration ιστορικό (π.χ. κάποιος άλλαξε schema με το χέρι).

```txt
DRIFT = η βάση διαφέρει από αυτό που περιγράφουν οι migrations.
Αιτίες: manual ALTER TABLE, edited migration file, partial apply.
Σύμπτωμα: "Drift detected" στο migrate dev.
```

Ο κανόνας: **ποτέ μην πειράζεις τη βάση με το χέρι** ούτε edit-άρεις applied migration. Κάθε αλλαγή → νέο migration.

## Safe production changes — expand/contract

Destructive αλλαγές (drop column, rename, NOT NULL χωρίς default) μπορούν να σπάσουν το running app ή να χάσουν δεδομένα. Το pattern είναι **expand → migrate → contract** σε ξεχωριστά deploys:

```txt
RENAME column oldName → newName, με zero downtime:
1. EXPAND:   πρόσθεσε newName (nullable). Deploy.
2. BACKFILL: αντίγραψε oldName → newName (data migration).
3. Ο κώδικας γράφει & στα δύο / διαβάζει newName. Deploy.
4. CONTRACT: drop oldName, μόλις τίποτα δεν το χρησιμοποιεί. Deploy.
```

Adding NOT NULL: πρώτα πρόσθεσε με default ή nullable + backfill, μετά κάνε το NOT NULL. Ποτέ NOT NULL χωρίς default σε γεμάτο πίνακα — σκάει.

## Seeding

```ts
// prisma/seed.ts
const prisma = new PrismaClient();
async function main() {
  await prisma.role.createMany({
    data: [{ name: "admin" }, { name: "user" }],
    skipDuplicates: true,        // idempotent: ξανατρέχει χωρίς error
  });
}
main().finally(() => prisma.$disconnect());
```

```json
// package.json
"prisma": { "seed": "tsx prisma/seed.ts" }
```

`npx prisma db seed` γεμίζει reference data (roles, categories) ή dev fixtures. Κάνε το **idempotent** (`skipDuplicates` / `upsert`) ώστε να ξανατρέχει χωρίς να σπάει.

## db push vs migrate

```txt
prisma db push   → συγχρονίζει schema στη βάση ΧΩΡΙΣ migration file.
                   Γρήγορο για prototyping/early dev. ΟΧΙ ιστορικό.
prisma migrate   → versioned migration files. Για ομάδες & production.
```

`db push` για γρήγορο πειραματισμό σόλο· `migrate` μόλις χρειάζεσαι versioned, shareable ιστορικό (δηλαδή σχεδόν πάντα σε ομάδα).

## Πώς απαντάς στη συνέντευξη

«How do you handle database migrations safely?»

> In development I use `migrate dev`, which diffs the schema, creates a versioned migration file, applies it through a shadow database that detects drift, and regenerates the client. In production I only ever run `migrate deploy` — it's idempotent and non-interactive, applies pending migrations in order, and never resets, whereas `migrate dev` can propose a destructive reset on drift. The golden rule is never to touch the database by hand or edit an applied migration, since that's what causes drift. For risky changes like renames or new NOT NULL columns I do expand/contract across separate deploys — add the new column nullable, backfill, switch the code, then drop the old one — so the running app stays compatible the whole time and there's zero downtime. I keep seed scripts idempotent with `skipDuplicates` or `upsert` for reference data.
