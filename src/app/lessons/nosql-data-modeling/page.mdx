# NoSQL Data Modeling

Εδώ είναι το πιο δύσκολο mental shift για όποιον έρχεται από SQL: στο NoSQL **δεν** μοντελοποιείς τα δεδομένα και μετά κάνεις queries — μοντελοποιείς γύρω από τα **queries**. Στα interviews, το «σχεδίασε ένα NoSQL schema» αποκαλύπτει αμέσως αν το έχεις καταλάβει.

## Τι πρέπει να μπορείς στο τέλος

- να σχεδιάζεις από access patterns, όχι από entities,
- να χρησιμοποιείς denormalization συνειδητά,
- να εξηγείς single-table design (DynamoDB),
- να μοντελοποιείς σχέσεις χωρίς joins,
- να χειρίζεσαι το trade-off duplication vs ενημέρωση.

## SQL vs NoSQL modeling — αντεστραμμένη σειρά

```txt
SQL:    Entities → normalized tables → όποιο query θες (joins, ad-hoc)
        "σχεδιάζω τα δεδομένα σωστά, τα queries προκύπτουν"

NoSQL:  Access patterns → model ανά pattern → denormalized
        "ξέρω ΠΩΣ θα διαβάσω, φτιάχνω το model να το εξυπηρετεί σε 1 read"
```

Γι' αυτό το πρώτο βήμα σε NoSQL design **δεν** είναι «τι entities έχω;» αλλά **«ποια queries θα τρέχω;»**.

## Denormalization — σκόπιμη επανάληψη

Στο NoSQL, η επανάληψη δεδομένων είναι **εργαλείο**, όχι λάθος. Αντί για join, κρατάς τα δεδομένα μαζί:

```js
// Αντί για orders + users (join), κρατάς ό,τι χρειάζεται το read μαζί:
{
  orderId: "1001",
  userId: "42",
  userName: "Ana",        // ← αντιγραμμένο από τον user (denormalized)
  items: [...],
  total: 49.90
}
```

Το read «δείξε την παραγγελία με όνομα πελάτη» γίνεται **ένα** lookup, χωρίς join. Το κόστος: αν αλλάξει το `userName`, πρέπει να ενημερώσεις πολλά orders (ή να το αφήσεις ως snapshot).

## Σχέσεις χωρίς joins

```txt
1-1, 1-N (bounded)    → embed μέσα στο parent document
1-N (unbounded)       → reference (id) + ξεχωριστή collection/πίνακας
N-N                   → references και στις δύο πλευρές, ή denormalized copies
                        ανά access pattern
```

Όταν χρειάζεσαι «join», έχεις 3 επιλογές: (1) denormalize ώστε να μη χρειάζεται, (2) δεύτερο query στον app («application-side join»), (3) αναθεώρησε αν το NoSQL είναι σωστό για αυτό.

## Single-table design (DynamoDB)

Το πιο προχωρημένο NoSQL pattern: **όλες οι οντότητες σε έναν πίνακα**, διαφοροποιημένες από το key schema. Στόχος: να φέρνεις σχετιζόμενα δεδομένα διαφορετικών τύπων σε **ένα** query.

```txt
PK            SK              attributes
USER#42       PROFILE         name, email
USER#42       ORDER#1001      total, date
USER#42       ORDER#1002      total, date
ORDER#1001    ITEM#1          product, qty
```

Query `PK = USER#42` → προφίλ + όλες οι παραγγελίες του χρήστη, **μαζί**, σε ένα round-trip. Είναι ισχυρό αλλά δύσκολο: απαιτεί να ξέρεις **όλα** τα access patterns εκ των προτέρων.

## Πότε σταματάς το denormalization

Denormalize για reads, αλλά **όχι ατελείωτα**:

```txt
- Αν τα δεδομένα αλλάζουν συχνά & αντιγράφονται σε πολλά μέρη → πόνος ενημέρωσης
- Αν αρχίζεις να συντηρείς δεκάδες copies → ίσως ήθελες relational
- Snapshot data (τιμή/όνομα τη στιγμή του γεγονότος) → denormalize ελεύθερα
```

## Πώς απαντάς στη συνέντευξη

«How is data modeling different in NoSQL?»

> The order is reversed. In SQL I model entities into normalized tables and then run whatever queries I want, leaning on joins. In NoSQL I start from the access patterns — how will I read this? — and design the model so each query is a single fast lookup, which usually means denormalizing. Duplication is a tool, not a mistake: I copy a user's name into an order so reading "order with customer name" is one lookup with no join, and often that copy is a deliberate snapshot of the value at that moment. Relationships are handled by embedding bounded data, referencing unbounded data, or doing an application-side join. The advanced version is single-table design in DynamoDB, where all entities share one table and the key schema lets me fetch related items of different types in one query — powerful, but it demands knowing every access pattern up front. I stop denormalizing when frequently-changing data starts living in too many places.
