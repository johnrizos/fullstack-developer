# Document Databases (MongoDB)

Τα document stores (MongoDB, Firestore) αποθηκεύουν δεδομένα ως **JSON-like documents** με ευέλικτο schema. Είναι ο πιο διαδεδομένος τύπος NoSQL σε fullstack apps. Στα interviews εξετάζουν την κρίσιμη απόφαση **embed vs reference** και ότι το modeling οδηγείται από τα access patterns.

## Τι πρέπει να μπορείς στο τέλος

- να σχεδιάζεις documents γύρω από το πώς διαβάζεις,
- να διαλέγεις embed vs reference συνειδητά,
- να γράφεις βασικά queries και updates,
- να ξέρεις πότε χρειάζεσαι index,
- να εξηγείς το aggregation pipeline.

## Document = ένα aggregate

Ένα document κρατά μαζί δεδομένα που **διαβάζονται μαζί** — όχι σπασμένα σε tables:

```js
// collection: users — ένα document ανά χρήστη
{
  _id: ObjectId("..."),
  email: "ana@x.com",
  name: "Ana",
  address: { city: "Athens", zip: "11525" },   // nested object
  roles: ["user", "editor"]                     // array
}
```

Ευέλικτο schema: δύο documents στην ίδια collection μπορούν να έχουν διαφορετικά πεδία. Καλό για εξελισσόμενα δεδομένα — αλλά η πειθαρχία (validation στον app ή schema rules) μένει δική σου ευθύνη.

## Embed vs Reference — η κρίσιμη απόφαση

```js
// EMBED: τα related μέσα στο document — ένα read, χωρίς "join"
{
  _id: "post_1",
  title: "Hello",
  author: { name: "Ana", avatar: "..." },        // embedded
  comments: [{ text: "Nice!", by: "Bob" }]       // embedded array
}

// REFERENCE: κρατάς id και κάνεις δεύτερο query (σαν FK)
{ _id: "post_1", title: "Hello", authorId: "user_42" }
```

```txt
EMBED όταν:                          REFERENCE όταν:
- διαβάζονται πάντα μαζί             - μοιράζονται (ο author σε πολλά posts)
- ανήκουν στο ίδιο aggregate         - μεγαλώνουν απεριόριστα (unbounded)
- bounded (λίγα comments)            - ενημερώνονται ανεξάρτητα/συχνά
```

## Queries & updates

```js
// Find με φίλτρο και projection (μόνο τα πεδία που θες)
db.users.find({ "address.city": "Athens" }, { name: 1, email: 1 });

// Update — atomic operators
db.users.updateOne(
  { _id: id },
  { $set: { name: "Ana B." }, $push: { roles: "admin" } }
);

// Πάντα φίλτραρε με access pattern + ownership (όπως στο SQL)
db.posts.find({ authorId: currentUserId });
```

## Indexes — ίδια λογική με SQL

Χωρίς index, ένα query κάνει **collection scan** (σαρώνει όλα τα documents). Index στα πεδία που φιλτράρεις/ταξινομείς συχνά:

```js
db.posts.createIndex({ authorId: 1, createdAt: -1 }); // composite, όπως SQL
```

Ίδιοι κανόνες με το relational indexing: composite index ακολουθεί left-prefix, indexes επιβαρύνουν writes, indexάρεις selective πεδία.

## Aggregation pipeline

Για analytics/μετασχηματισμούς, η MongoDB έχει **pipeline** από στάδια (αντίστοιχο του GROUP BY/JOIN του SQL):

```js
db.orders.aggregate([
  { $match: { status: "paid" } },                    // ~ WHERE
  { $group: { _id: "$userId", total: { $sum: "$amount" } } }, // ~ GROUP BY
  { $sort: { total: -1 } },                          // ~ ORDER BY
  { $limit: 10 }
]);
```

Το `$lookup` κάνει join-like ένωση collections — αλλά αν το χρειάζεσαι συχνά, ίσως το data model έπρεπε να είναι διαφορετικό (ή relational).

## Πώς απαντάς στη συνέντευξη

«How do you model data in a document database?»

> I model around how the data is read, not how it's normalized. A document is an aggregate that holds together things read together. The key decision is embed versus reference: I embed when related data is always read with the parent, belongs to the same aggregate, and is bounded — like an address on a user. I reference when data is shared across documents, grows unbounded, or is updated independently — comments on a popular post go in their own collection with a `postId` and index, because embedding tens of thousands would blow past the document size limit and break pagination. Indexing follows the same rules as SQL — composite indexes with a left prefix, and `explain` to verify — and for analytics I use the aggregation pipeline rather than pulling everything into the app.
