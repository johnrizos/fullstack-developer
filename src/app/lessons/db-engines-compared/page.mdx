# Choosing a Database Engine

Αφού μάθεις τις έννοιες (SQL, NoSQL, indexing, transactions), έρχεται η πρακτική ερώτηση: **ποιο συγκεκριμένο engine;** PostgreSQL, MySQL, SQLite ή MongoDB; Στα interviews εξετάζουν αν ξέρεις τις πραγματικές διαφορές — όχι preferences, αλλά πότε ποιο και γιατί.

## Τι πρέπει να μπορείς στο τέλος

- να ξεχωρίζεις embedded από server databases,
- να εξηγείς πότε SQLite, πότε PostgreSQL/MySQL,
- να ξέρεις τις βασικές διαφορές PostgreSQL vs MySQL,
- να τοποθετείς το MongoDB σωστά,
- να διαλέγεις engine με βάση το project, όχι το hype.

## Embedded vs Server

Η πρώτη μεγάλη διάκριση:

```txt
EMBEDDED (SQLite)      → η βάση είναι ΕΝΑ ΑΡΧΕΙΟ, τρέχει μέσα στην app σου.
                         Καμία εγκατάσταση/server. Zero-config.

SERVER (Postgres/MySQL)→ ξεχωριστή διεργασία/μηχάνημα. Πολλοί clients,
                         concurrency, network access, users/permissions.
                         MongoDB είναι επίσης server (document).
```

## SQLite — πότε ένα αρχείο αρκεί

```txt
✅ Τέλειο για:
   - Mobile apps (το πιο διαδεδομένο db στον κόσμο — σε κάθε τηλέφωνο)
   - Desktop apps, CLI tools, embedded devices
   - Tests / local dev (γρήγορο, χωρίς setup)
   - Μικρά sites με κυρίως reads / λίγους concurrent writers

❌ Όχι για:
   - Πολλούς ταυτόχρονους writers (single-writer lock στο write)
   - Πολλά app servers που μοιράζονται μία βάση μέσω δικτύου
   - Μεγάλη κλίμακα / high write concurrency
```

## PostgreSQL vs MySQL

Και τα δύο είναι ώριμα, open-source, relational servers. Οι κύριες διαφορές:

```txt
PostgreSQL                          MySQL
- Πιο "feature-rich" & standards     - Ιστορικά πιο απλό/γρήγορο σε reads
- Άριστο JSONB (document μέσα σε SQL)- JSON support υπάρχει αλλά λιγότερο ισχυρό
- Advanced types, CTEs, window fns,  - Πολύ διαδεδομένο, τεράστιο ecosystem
  full-text, extensions (PostGIS)      (WordPress, LAMP stack)
- Αυστηρότερο σε data integrity      - Replication ώριμο & εύκολο
- Default επιλογή για νέα projects    - Καλό όταν η ομάδα/stack το ξέρει ήδη
```

Πρακτικά: για **νέο** project, το PostgreSQL είναι η συνηθισμένη default (ευελιξία + JSONB + standards). Το MySQL όταν το ecosystem/η ομάδα το επιβάλλει ή υπάρχει legacy.

## MongoDB — πότε το προϊόν, όχι μόνο το μοντέλο

Το MongoDB είναι document server. Διάλεξέ το όταν:

```txt
✅ Το document model ταιριάζει φυσικά (nested, ευέλικτο schema, evolving)
✅ Access patterns δεμένα με aggregates που διαβάζονται μαζί
✅ Θες οριζόντια κλίμακα με sharding out of the box

⚠️ Αλλά: για σχεσιακά δεδομένα με transactions/joins, ένα relational
   engine (ή Postgres+JSONB) είναι συχνά καλύτερη επιλογή.
```

Συχνό λάθος: επιλογή MongoDB «επειδή είναι μοντέρνο» για δεδομένα που είναι ξεκάθαρα σχεσιακά → καταλήγεις να κάνεις application-side joins που το relational θα έκανε καλύτερα.

## Ο πρακτικός κανόνας

```txt
Default νέο web app        → PostgreSQL
Mobile / desktop / tests   → SQLite
Legacy / LAMP / ομάδα ξέρει → MySQL
Document-first / ευέλικτο  → MongoDB (ή Postgres JSONB πρώτα)
+ Redis δίπλα για cache/sessions σχεδόν πάντα
```

## Πώς απαντάς στη συνέντευξη

«PostgreSQL, MySQL, SQLite, or MongoDB — how do you choose?»

> First, embedded versus server: SQLite is a single file running inside the app — perfect for mobile, desktop, CLI tools, and tests, but it has a single-writer lock, so it's not for high-concurrency web backends or many app servers sharing it. For a concurrent web app I want a server database. Between PostgreSQL and MySQL, both are mature relational servers; I default to PostgreSQL for new projects because of its richer features, standards compliance, and excellent JSONB, which covers flexible fields without leaving SQL — MySQL when the ecosystem or team already lives there. MongoDB I pick when the document model genuinely fits — nested, evolving data read as aggregates — but for clearly relational data with joins and transactions, a relational engine or Postgres with JSONB is usually better than doing application-side joins. And almost always there's Redis alongside for caching and sessions. The choice follows the project, not the hype.
