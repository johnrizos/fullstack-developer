# Processes, Logs & Services

Το app σου τρέχει. Μετά πέφτει στις 3 το πρωί. Κανείς δεν το ξανασηκώνει χειροκίνητα, και πρέπει να βρεις τι έγινε χωρίς να ήσουν εκεί. Αυτό το μάθημα είναι για το «πώς κρατάς ένα app ζωντανό σε production και πώς διαβάζεις τι έκανε».

## Τι θα μάθεις

- να βλέπεις και να σκοτώνεις **processes** σωστά (signals, graceful shutdown),
- να βρίσκεις **ποιος ακούει σε ποιο port**,
- να τρέχεις το app ως **systemd service** που auto-restartάρει,
- να διαβάζεις **logs** με `journalctl` και αρχεία στο `/var/log`,
- να παρακολουθείς **πόρους** (CPU, μνήμη, δίσκο) και να προγραμματίζεις jobs με **cron**.

## Processes — τι τρέχει αυτή τη στιγμή

```bash
ps aux                  # όλα τα processes με CPU/μνήμη/user
ps aux | grep node      # μόνο τα node
top                     # live πίνακας (q για έξοδο)
htop                    # ωραιότερο top (αν είναι εγκατεστημένο)
pgrep -fl node          # PIDs που ταιριάζουν στο 'node'
```

Κάθε process έχει ένα μοναδικό **PID** (process ID). Με αυτό το ελέγχεις.

## Foreground, background & nohup

```bash
node app.js             # foreground — δεσμεύει το terminal
node app.js &           # background — σου επιστρέφει τον έλεγχο
jobs                    # τι τρέχει σε background αυτής της session
fg                      # φέρε το πίσω σε foreground
nohup node app.js &     # τρέξε ανεξάρτητα από το terminal (επιβιώνει στο logout)
```

> Το `&`/`nohup` είναι «πρόχειρη» λύση. Σε production **δεν** κρατάς απλώς ένα background process — χρησιμοποιείς service manager (παρακάτω) που το ξανανοίγει αν πέσει.

## Signals & kill — graceful vs forced

Όταν «σκοτώνεις» ένα process, στέλνεις ένα **signal**:

```bash
kill <PID>          # στέλνει SIGTERM (15) — "κλείσε ευγενικά"
kill -9 <PID>       # στέλνει SIGKILL (9)  — "πέθανε τώρα", χωρίς cleanup
kill -HUP <PID>     # SIGHUP — συχνά "reload config"
```

Η διάκριση είναι **κρίσιμη** σε interview:

- **SIGTERM** δίνει στο app χρόνο να κλείσει σωστά: να τελειώσει τα ενεργά requests, να κλείσει DB connections, να γράψει logs. Αυτό είναι **graceful shutdown**.
- **SIGKILL** (`-9`) δεν μπορεί να το πιάσει το app — κόβεται απότομα στη μέση, με κίνδυνο μισές εγγραφές/corrupt state. Είναι η έσχατη λύση όταν δεν ανταποκρίνεται.

Σε Node:

```js
process.on("SIGTERM", async () => {
  await server.close();   // σταμάτα να δέχεσαι νέα requests
  await db.disconnect();  // καθάρισε connections
  process.exit(0);
});
```

## Ποιος ακούει στο port;

Το κλασικό «EADDRINUSE: port 3000 already in use»:

```bash
ss -tlnp                  # όλα τα listening TCP ports + process
ss -tlnp | grep :3000     # ποιος κρατά το 3000;
lsof -i :3000             # εναλλακτικά, ποιος έχει το 3000
kill <PID>                # ελευθέρωσέ το
```

## systemd — να κρατάς το app ζωντανό

Το **systemd** είναι ο init/service manager των περισσότερων Linux. Φτιάχνεις ένα **unit file** και το app γίνεται service που ξεκινά στο boot και auto-restartάρει αν πέσει:

```ini
# /etc/systemd/system/myapp.service
[Unit]
Description=My Node App
After=network.target

[Service]
WorkingDirectory=/var/www/app
ExecStart=/usr/bin/node server.js
Restart=always
User=www-data
Environment=NODE_ENV=production

[Install]
WantedBy=multi-user.target
```

Και το ελέγχεις:

```bash
sudo systemctl daemon-reload        # μετά από αλλαγή unit file
sudo systemctl enable myapp         # ξεκίνα αυτόματα στο boot
sudo systemctl start myapp          # ξεκίνα τώρα
sudo systemctl status myapp         # τρέχει; πότε ξεκίνησε; τελευταία logs
sudo systemctl restart myapp        # μετά από deploy
```

Το `Restart=always` είναι ο λόγος που το app «ξανασηκώνεται μόνο του» — δεν χρειάζεσαι κανέναν στις 3 το πρωί.

## Logs — journalctl & /var/log

Αν το app τρέχει ως systemd service, τα logs του πάνε στο **journal**:

```bash
journalctl -u myapp              # όλα τα logs του service
journalctl -u myapp -f           # live follow (το #1 για debugging)
journalctl -u myapp --since "1 hour ago"
journalctl -u myapp -p err       # μόνο error-level και πάνω
```

Πολλά services γράφουν και σε αρχεία στο `/var/log`:

```bash
tail -f /var/log/nginx/error.log     # live errors του nginx
grep "500" /var/log/nginx/access.log # ψάξε HTTP 500
```

Τα logs μεγαλώνουν — γι' αυτό υπάρχει **log rotation** (`logrotate`): σπάει, συμπιέζει και σβήνει παλιά logs ώστε να μη γεμίσει ο δίσκος.

## Πόροι — δίσκος, μνήμη, CPU

```bash
df -h                # ελεύθερος χώρος ανά disk/partition
du -sh /var/log/*    # ποιοι φάκελοι τρώνε χώρο
free -h              # μνήμη: used / free / available
uptime               # load average + πόσο τρέχει το μηχάνημα
```

Το «το app έπεσε ξαφνικά» πολλές φορές είναι **disk full** (logs που δεν έγιναν rotate) ή **out of memory** (OOM killer). Το `df -h` και το `free -h` είναι τα πρώτα που τρέχεις.

## cron — προγραμματισμένα jobs

Για επαναλαμβανόμενες δουλειές (backups, cleanups, reports):

```bash
crontab -e            # επεξεργασία των cron jobs σου
crontab -l            # λίστα
```

Η σύνταξη (5 πεδία: λεπτό ώρα μέρα μήνας μέρα-εβδομάδας):

```bash
# ┌ min  ┌ hour  ┌ day  ┌ month  ┌ weekday
  0      3       *      *        *      /usr/local/bin/backup.sh
# κάθε μέρα στις 03:00

*/15 * * * * /usr/local/bin/healthcheck.sh
# κάθε 15 λεπτά
```
