# Transactions & Performance

Εδώ χωρίζονται οι junior από τους mid backend engineers. Στη συνέντευξη: «τι είναι το N+1 problem;», «πώς δουλεύει το connection pooling;», «πότε χρειάζεσαι transaction;». Το Prisma δίνει τα εργαλεία — εσύ πρέπει να ξέρεις πότε και γιατί.

## Τι πρέπει να μπορείς στο τέλος

- να χρησιμοποιείς transactions (sequential & interactive),
- να αναγνωρίζεις και να λύνεις το N+1 problem,
- να καταλαβαίνεις connection pooling και τα όριά του,
- να ξέρεις πού βοηθούν indexes.

## Transactions — atomicity

Όταν πολλές γραφές πρέπει να πετύχουν **όλες ή καμία**:

```ts
// Sequential ($transaction array): ανεξάρτητες λειτουργίες, atomic
const [order, payment] = await prisma.$transaction([
  prisma.order.create({ data: orderData }),
  prisma.payment.create({ data: paymentData }),
]);

// Interactive: όταν το επόμενο εξαρτάται από το προηγούμενο
await prisma.$transaction(async (tx) => {
  const account = await tx.account.findUnique({ where: { id } });
  if (account.balance < amount) throw new Error("Insufficient funds");  // rollback
  await tx.account.update({ where: { id }, data: { balance: { decrement: amount } } });
  await tx.transfer.create({ data: { amount, accountId: id } });
});
```

Στο interactive transaction, ένα `throw` κάνει **rollback** όλων — η βάση μένει consistent. Πρόσεξε: κράτα τα transactions **σύντομα** (κρατούν locks)· μη βάζεις HTTP calls μέσα.

## Το N+1 problem — ο κλασικός killer

```ts
// ΛΑΘΟΣ: 1 query για posts + N queries (μία ανά post) για author
const posts = await prisma.post.findMany();          // 1 query
for (const post of posts) {
  post.author = await prisma.user.findUnique({ where: { id: post.authorId } });  // N queries!
}
// 100 posts → 101 queries
```

```ts
// ΣΩΣΤΟ: ένα query με join
const posts = await prisma.post.findMany({ include: { author: true } });  // 1 query
```

Το `include`/`select` με relations κάνει το join στη βάση — **1-2 queries** αντί για N+1. Είναι το πιο συχνό performance bug σε ORMs.

## Connection pooling

```txt
Κάθε query χρειάζεται connection. Το άνοιγμα connection είναι ακριβό,
οπότε το Prisma κρατά POOL από ανοιχτά connections και τα ανακυκλώνει.

connection_limit (default ~ num_cpus * 2 + 1)
  πολύ μικρό → requests περιμένουν σε ουρά (timeout)
  πολύ μεγάλο → εξαντλείς τα connections της βάσης
```

```bash
DATABASE_URL="postgresql://...?connection_limit=10&pool_timeout=20"
```

Σε **serverless** (Lambda/Vercel), κάθε instance ανοίγει pool → εκατοντάδες connections γρήγορα. Λύση: external pooler (PgBouncer / Prisma Accelerate / Data Proxy) που μοιράζεται connections ανάμεσα σε instances.

## Indexes — πού βοηθούν

```prisma
model Post {
  id        String   @id
  authorId  String
  slug      String   @unique          // unique index
  createdAt DateTime
  @@index([authorId])                  // index για το συχνό where
  @@index([authorId, createdAt])       // composite για where + orderBy
}
```

Index βάζεις σε columns που μπαίνουν συχνά σε `where`, `orderBy`, ή σε joins (foreign keys). Trade-off: indexes επιταχύνουν reads αλλά **επιβραδύνουν writes** και πιάνουν χώρο — μη βάζεις παντού. Composite index για queries που φιλτράρουν+ταξινομούν μαζί.

## Μέτρα, μη μαντεύεις

```ts
const prisma = new PrismaClient({ log: ["query"] });   // δες τα πραγματικά SQL
```

Ενεργοποίησε query logging για να δεις πόσα/ποια queries τρέχουν στ' αλήθεια — έτσι πιάνεις N+1. Σε production, χρησιμοποίησε `EXPLAIN ANALYZE` για slow queries πριν προσθέσεις index.

## Πώς απαντάς στη συνέντευξη

«How do you keep a Prisma data layer fast and correct?»

> For correctness, when several writes must all succeed or none I wrap them in a transaction — a `$transaction` array for independent ops, or an interactive transaction when one step depends on the previous, where a throw rolls everything back; I keep them short because they hold locks and never put HTTP calls inside. The biggest performance trap is N+1 — fetching a list then a query per row for its relation, so 100 posts become 101 queries; I fix it with `include`/`select` so the join happens in one database query, and `Promise.all` doesn't help because it's still N queries. I tune the connection pool with `connection_limit`, and in serverless I put an external pooler like PgBouncer or Accelerate in front since each instance otherwise opens its own pool and exhausts connections. I add indexes on columns used in where, orderBy and foreign keys — composite when I filter and sort together — knowing they speed reads but slow writes. And I turn on query logging to measure rather than guess.
