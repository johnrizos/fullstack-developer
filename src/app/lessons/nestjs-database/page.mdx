# Database with Prisma

Στο Nest, η σύνδεση με βάση δεν είναι «κάνε import το client και τρέξε queries» — την κάνεις **injectable** ώστε να τεστάρεται και να μοιράζεται καθαρά. Στη συνέντευξη: «πώς integrate-άρεις Prisma στο Nest;», «γιατί repository pattern;», «πού κάνεις connect/disconnect;».

## Τι πρέπει να μπορείς στο τέλος

- να φτιάχνεις injectable PrismaService με σωστό lifecycle,
- να εκθέτεις το PrismaModule σε άλλα modules,
- να γράφεις repository/service layer πάνω από το Prisma,
- να καταλαβαίνεις γιατί δεν κάνεις queries μέσα στους controllers.

> Αυτό το μάθημα δένει με το αυτόνομο **Prisma track** (schema, relations, migrations, performance). Εδώ εστιάζουμε στο *πώς ζει* το Prisma μέσα στο Nest.

## PrismaService — injectable client με lifecycle

```ts
@Injectable()
export class PrismaService extends PrismaClient implements OnModuleInit, OnModuleDestroy {
  async onModuleInit() {
    await this.$connect();      // connect στο bootstrap
  }
  async onModuleDestroy() {
    await this.$disconnect();   // καθαρό κλείσιμο στο shutdown
  }
}
```

Το `PrismaService` **επεκτείνει** το `PrismaClient`, οπότε έχει όλα τα `.task`, `.user` κ.λπ. Τα lifecycle hooks (`OnModuleInit`/`OnModuleDestroy`) εξασφαλίζουν ότι το connection ανοίγει/κλείνει με το app — όχι ένα νέο client ανά request (που θα εξαντλούσε το connection pool).

## PrismaModule — μοιράζεται παντού

```ts
@Global()                          // διαθέσιμο σε όλα τα modules χωρίς re-import
@Module({
  providers: [PrismaService],
  exports: [PrismaService],
})
export class PrismaModule {}
```

Το `@Global()` το κάνει διαθέσιμο σε όλο το app χωρίς να το κάνεις import σε κάθε feature module. (Χρησιμοποίησε `@Global` φειδωλά — εδώ ταιριάζει γιατί η βάση είναι cross-cutting.)

## Service layer — όχι queries στον controller

```ts
@Injectable()
export class TaskService {
  constructor(private readonly prisma: PrismaService) {}

  findAll(userId: string) {
    return this.prisma.task.findMany({ where: { userId } });
  }

  async findOne(id: string) {
    const task = await this.prisma.task.findUnique({ where: { id } });
    if (!task) throw new NotFoundException(`Task ${id} not found`);  // → 404
    return task;
  }

  create(userId: string, dto: CreateTaskDto) {
    return this.prisma.task.create({ data: { ...dto, userId } });
  }
}
```

```ts
@Controller("tasks")
export class TaskController {
  constructor(private readonly tasks: TaskService) {}

  @Get(":id")
  findOne(@Param("id") id: string) {
    return this.tasks.findOne(id);   // controller = μόνο HTTP, καμία query
  }
}
```

Ο controller κρατά μόνο HTTP concerns· το service κρατά business logic + data access. Έτσι το service τεστάρεται χωρίς HTTP και ξαναχρησιμοποιείται από jobs/CLI.

## Repository pattern — πότε αξίζει

Για μεγαλύτερες εφαρμογές βάζεις ένα repository ανάμεσα σε service και Prisma, ώστε η business logic να μην ξέρει το ORM:

```ts
@Injectable()
export class TaskRepository {
  constructor(private readonly prisma: PrismaService) {}
  findByUser(userId: string) { return this.prisma.task.findMany({ where: { userId } }); }
  save(data: Prisma.TaskCreateInput) { return this.prisma.task.create({ data }); }
}
```

Trade-off: extra abstraction. Σε μικρό app, το service-καλεί-Prisma είναι αρκετό. Το repository αξίζει όταν θες να αλλάξεις data source ή να κρύψεις πολύπλοκα queries πίσω από καθαρό interface.

## Πώς απαντάς στη συνέντευξη

«How do you integrate Prisma with NestJS?»

> I wrap the client in an injectable `PrismaService` that extends `PrismaClient` and implements `OnModuleInit`/`OnModuleDestroy` to `$connect` and `$disconnect` with the app lifecycle — that keeps a single connection pool instead of spawning a client per request, which would exhaust the database. I expose it from a `@Global` PrismaModule so feature modules can inject it without re-importing. Queries live in services, never in controllers: the service holds business logic and data access, throws `NotFoundException` when a record is missing so Nest maps it to 404, and stays testable without HTTP. For larger apps I add a repository layer between the service and Prisma so business logic doesn't depend on the ORM directly, but in a small app a service calling Prisma is enough.
