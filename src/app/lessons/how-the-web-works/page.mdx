# How the Web Really Works

Όταν γράφεις `https://example.com` και πατάς Enter, συμβαίνουν δεκάδες πράγματα πριν δεις τη σελίδα. Οι developers που ξέρουν *τι* ακριβώς, ξεμπλοκάρουν μόνοι τους CORS errors, καταλαβαίνουν γιατί ένα deploy «δεν φρεσκάρει» (caching), και απαντούν με σιγουριά στο κλασικό interview opener: «what happens when you type a URL and hit Enter?».

## Το ταξίδι ενός request — η μεγάλη εικόνα

```txt
1. DNS      example.com ──────────► 93.184.216.34   (όνομα → IP)
2. TCP      «άνοιξε σύνδεση στο 93.184.216.34:443»
3. TLS      handshake → κρυπτογραφημένο κανάλι (το "s" στο https)
4. HTTP     GET / HTTP/2  +  headers ──────────► server
5. Response 200 OK + headers + HTML body ◄────── server
6. Render   browser: parse HTML → CSS → JS → ζητά εικόνες/fonts (πάλι 1-5)
```

Κάθε βήμα είναι ένα concept που θα δούμε. Κράτα ότι **ένα URL = πολλά requests**: το HTML φέρνει references σε CSS, JS, images — το καθένα ξεκινά νέο ταξίδι.

## DNS — το τηλεφωνικό κατάλογος του internet

Οι υπολογιστές μιλάνε με **IP addresses** (`93.184.216.34`), όχι ονόματα. Το **DNS** (Domain Name System) μεταφράζει `example.com` → IP. Γίνεται με αλυσίδα από caches:

```txt
browser cache → OS cache → router → ISP resolver → root → TLD (.com) → authoritative
```

Το πρώτο που έχει την απάντηση τη δίνει — γι' αυτό η δεύτερη επίσκεψη είναι ταχύτερη. Το DNS έχει **TTL**: πόση ώρα επιτρέπεται να μείνει cached. Γι' αυτό μια αλλαγή DNS «αργεί να φανεί» — κάποιοι resolvers κρατούν την παλιά τιμή μέχρι να λήξει το TTL.

## TCP & TLS — σύνδεση και ασφάλεια

Μετά το IP, ο browser ανοίγει **TCP** σύνδεση (αξιόπιστο κανάλι). Σε `https`, ακολουθεί **TLS handshake**: ανταλλαγή certificate και κλειδιών ώστε τα δεδομένα να είναι **κρυπτογραφημένα**. Το certificate αποδεικνύει ότι ο server είναι όντως ο `example.com` (υπογεγραμμένο από Certificate Authority).

> Γι' αυτό το `https` δεν είναι «απλώς ένα λουκέτο»: εμποδίζει κάποιον στο ίδιο WiFi να διαβάσει ή να αλλοιώσει το traffic σου. Ποτέ login form σε `http`.

## HTTP — το πρωτόκολλο request/response

Το **HTTP** είναι το «λεξιλόγιο». Ένα request έχει method, path, headers (και ίσως body):

```http
GET /tasks HTTP/2
Host: example.com
Authorization: Bearer eyJ...
Accept: application/json
```

Η απάντηση έχει status code, headers, body:

```http
HTTP/2 200 OK
Content-Type: application/json
Cache-Control: max-age=3600
Set-Cookie: session=abc; HttpOnly; Secure

{ "data": [...] }
```

Τα **methods** (GET/POST/PUT/PATCH/DELETE) και τα **status codes** (200/404/500…) τα είδες στο REST APIs μάθημα. Εδώ εστιάζουμε στα **headers** — εκεί κρύβεται το 80% της μαγείας: caching, auth, cookies, content negotiation.

## Caching headers — γιατί κάτι «δεν φρεσκάρει»

Ο browser (και τα CDNs) αποθηκεύουν responses για να μην τα ξανακατεβάζουν. Το ελέγχουν headers:

- **`Cache-Control: max-age=3600`** → «κράτησέ το φρέσκο για 1 ώρα, μη με ξαναρωτήσεις».
- **`Cache-Control: no-store`** → ποτέ cache (π.χ. τραπεζικά δεδομένα).
- **`ETag` / `If-None-Match`** → ο browser ρωτά «άλλαξε;» και ο server απαντά **304 Not Modified** χωρίς να ξαναστείλει το body (εξοικονόμηση bandwidth).

```txt
Πρώτο request:   GET /logo.png  → 200 OK + ETag: "v3"  (κατεβαίνει)
Επόμενο request: GET /logo.png  + If-None-Match: "v3"
                                → 304 Not Modified      (δεν ξανακατεβαίνει)
```

Το κλασικό μυστήριο «έκανα deploy αλλά βλέπω την παλιά έκδοση» = aggressive caching. Λύση: **cache busting** — το build δίνει νέο όνομα αρχείου (`app.4f9a.js`) ώστε να θεωρηθεί νέο resource.

## CDN — φέρε το περιεχόμενο κοντά στον χρήστη

Ένα **CDN** (Content Delivery Network) είναι δίκτυο από servers σε όλο τον κόσμο που κρατούν cached αντίγραφα των στατικών σου αρχείων. Ο χρήστης στην Αθήνα τα παίρνει από edge server στην Ευρώπη αντί από το origin στην Αμερική → πολύ μικρότερο latency. Γι' αυτό deploy σε Vercel/Cloudflare = αυτόματο CDN για assets.

## Cookies & sessions — πώς ο server «σε θυμάται»

Το HTTP είναι **stateless** — κάθε request ξεκινά από το μηδέν. Πώς τότε μένεις logged in; Με **cookies**: ο server στέλνει `Set-Cookie`, ο browser το στέλνει πίσω σε κάθε επόμενο request.

```http
Set-Cookie: session=abc123; HttpOnly; Secure; SameSite=Lax
```

Τα flags είναι **security-critical**:
- **`HttpOnly`** → το cookie δεν διαβάζεται από JavaScript (προστασία από XSS token theft).
- **`Secure`** → στέλνεται μόνο σε https.
- **`SameSite`** → περιορίζει αποστολή σε cross-site requests (προστασία από CSRF).

## CORS — το πιο παρεξηγημένο error

Από ασφάλεια, ο browser εφαρμόζει **Same-Origin Policy**: JavaScript στο `app.com` δεν μπορεί να διαβάσει απαντήσεις από `api.other.com` (διαφορετικό origin), εκτός αν ο server το **επιτρέψει ρητά** με **CORS** headers.

```http
# Response από api.other.com που επιτρέπει το app.com:
Access-Control-Allow-Origin: https://app.com
Access-Control-Allow-Methods: GET, POST
Access-Control-Allow-Headers: Authorization, Content-Type
```

Κρίσιμη παρανόηση: το CORS **δεν** είναι περιορισμός του server σου — είναι κανόνας που επιβάλλει ο **browser**. Γι' αυτό:

- Το `curl` ή Postman δεν δίνουν ποτέ CORS error (δεν είναι browsers).
- Το CORS error δεν διορθώνεται στο frontend — διορθώνεται προσθέτοντας τα σωστά **headers στον server**.

```txt
❌ "CORS error" → ΔΕΝ φταίει ο fetch σου
✅ Ο server (api.other.com) πρέπει να στείλει Access-Control-Allow-Origin
```

## Πώς απαντάς στη συνέντευξη

> «What happens when you type a URL and press Enter?»

> The browser resolves the domain to an IP via DNS, opens a TCP connection, and for HTTPS performs a TLS handshake to get an encrypted channel and verify the server's certificate. It sends an HTTP request with method, path and headers; the server responds with a status code, headers and a body. The browser parses the HTML and fires more requests for CSS, JS, images and fonts — often served from a CDN edge close to the user and cached via Cache-Control and ETags. Cookies with HttpOnly, Secure and SameSite flags keep the otherwise stateless session, and cross-origin calls only succeed if the server returns the right CORS headers.
