# Wide-Column Stores (Cassandra)

Τα wide-column stores (Cassandra, Bigtable, ScyllaDB) είναι φτιαγμένα για **τεράστια κλίμακα write-heavy** δεδομένων κατανεμημένα σε πολλά nodes. Στα interviews εξετάζουν την αντεστραμμένη λογική τους: σχεδιάζεις **query-first**, και το partition key καθορίζει τα πάντα.

## Τι πρέπει να μπορείς στο τέλος

- να εξηγείς το wide-column μοντέλο (partition + clustering keys),
- να σχεδιάζεις πίνακα **από το query**, όχι από τα δεδομένα,
- να καταλαβαίνεις γιατί είναι write-optimized,
- να εξηγείς tunable consistency,
- να ξέρεις πότε ταιριάζει (και πότε όχι).

## Το μοντέλο: partition key + clustering key

Δεδομένα οργανώνονται σε **partitions** (κατανεμημένα σε nodes). Μέσα σε ένα partition, τα rows ταξινομούνται από το **clustering key**:

```cql
CREATE TABLE messages_by_room (
  room_id    uuid,        -- partition key: ΠΟΥ ζουν τα δεδομένα
  sent_at    timestamp,   -- clustering key: ΣΕΙΡΑ μέσα στο partition
  message_id uuid,
  body       text,
  PRIMARY KEY (room_id, sent_at)
) WITH CLUSTERING ORDER BY (sent_at DESC);
```

Query «τα τελευταία 50 μηνύματα ενός room» → ένα partition, ήδη ταξινομημένα → πανγρήγορο:

```cql
SELECT * FROM messages_by_room WHERE room_id = ? LIMIT 50;
```

## Query-first design — η αντεστραμμένη λογική

Στο relational: ένα normalized schema, μετά κάνεις όποιο query θες. Στο Cassandra: **ξεκινάς από τα queries** και φτιάχνεις έναν πίνακα **ανά query pattern** — ακόμα κι αν διπλασιάζεις δεδομένα.

```txt
Relational:  1 table → πολλά queries (joins, ad-hoc)
Cassandra:   1 query → 1 table (denormalized, data duplication ΑΠΟΔΕΚΤΗ)
```

## Γιατί write-optimized

Το Cassandra γράφει με **append** σε log (commit log + memtable → SSTable), χωρίς in-place updates ή read-before-write. Αυτό το κάνει εξαιρετικό για:

```txt
- Time-series data (sensors, metrics, IoT)
- Event logs / activity feeds
- Messaging (huge write volume)
- Tracking/analytics ingestion
```

Δεν υπάρχει single master — **όλα τα nodes δέχονται writes** (masterless, peer-to-peer), άρα κλιμακώνει οριζόντια γραμμικά προσθέτοντας nodes.

## Tunable consistency

Δεν διαλέγεις «strong ή eventual» μια για πάντα — το ρυθμίζεις **ανά query** με consistency level:

```txt
Write/Read consistency: ONE, QUORUM, ALL
- ONE     → γρήγορο, λιγότερη συνέπεια
- QUORUM  → η πλειοψηφία των replicas (R + W > N → strong-ish)
- ALL     → όλα τα replicas, αργό αλλά πιο συνεπές
```

Αν `W + R > N` (replicas που γράφουν + διαβάζουν > σύνολο replicas), έχεις εγγυημένα τελευταία τιμή. Έτσι ισορροπείς latency vs consistency ανά περίπτωση.

## Πότε ΟΧΙ Cassandra

```txt
- Όταν χρειάζεσαι joins, ad-hoc queries, ή ad-hoc filtering
- Όταν τα access patterns αλλάζουν συχνά (το schema είναι query-locked)
- Όταν χρειάζεσαι strong ACID transactions across rows
- Για μικρά/μεσαία δεδομένα — είναι overkill· relational αρκεί
```

## Πώς απαντάς στη συνέντευξη

«When and how do you use a wide-column store like Cassandra?»

> Cassandra is for massive, write-heavy, distributed data — time-series, event logs, messaging. The model is partition key plus clustering key: the partition key decides which node the data lives on, the clustering key orders rows within a partition. The mental shift is query-first design: instead of one normalized table and arbitrary queries, I design a table per access pattern and duplicate data across them, because there are no joins and you can't efficiently query non-key columns — storage is cheap, fast reads aren't. Writes are append-based with no read-before-write and it's masterless, so every node accepts writes and it scales horizontally and linearly. Consistency is tunable per query with levels like ONE or QUORUM, and `W + R > N` gives me strong reads when I need them. I avoid it when I need joins, ad-hoc queries, or strong cross-row transactions.
