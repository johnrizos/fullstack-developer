# Authentication & Authorization

Από τα πιο κρίσιμα θέματα σε fullstack συνέντευξη — και από αυτά όπου ένα λάθος στην απάντηση «καίει». Ξεκινάμε από τη θεμελιώδη διάκριση:

- **Authentication (AuthN)**: ποιος είσαι; — login, sessions, tokens.
- **Authorization (AuthZ)**: τι επιτρέπεται να κάνεις; — roles, permissions, ownership.

Ένας 401 status code αφορά authentication, ένας 403 αφορά authorization. Αν τα μπερδέψεις σε συνέντευξη, μετράει αρνητικά.

## Passwords: το πρώτο layer

Ποτέ plain text, ποτέ απλό hash. Τα passwords αποθηκεύονται με **αργό, salted hashing αλγόριθμο** — bcrypt, scrypt ή argon2:

```ts
import bcrypt from "bcrypt";

// Εγγραφή: hash με cost factor (12 = ~250ms ανά hash)
const passwordHash = await bcrypt.hash(plainPassword, 12);
await db.user.create({ data: { email, passwordHash } });

// Login: σύγκριση σταθερού χρόνου
const user = await db.user.findUnique({ where: { email } });
const valid = user && (await bcrypt.compare(plainPassword, user.passwordHash));
if (!valid) {
  // Ίδιο μήνυμα είτε δεν υπάρχει ο user είτε το password είναι λάθος —
  // δεν αποκαλύπτεις ποια emails έχουν λογαριασμό
  return res.status(401).json({ error: "Invalid credentials" });
}
```

Γιατί όχι SHA-256; Γιατί είναι **σχεδιασμένο να είναι γρήγορο** — ένας attacker δοκιμάζει δισεκατομμύρια hashes/δευτερόλεπτο. Το bcrypt είναι σκόπιμα αργό και έχει ενσωματωμένο salt που εξουδετερώνει τα rainbow tables.

## Sessions — η κλασική, στιβαρή προσέγγιση

Μετά το login, ο server δημιουργεί **session record** (σε DB ή Redis) και δίνει στον browser ένα cookie με το session id:

```text
1. POST /login (email, password)
2. Server: επαλήθευση → δημιουργία session στο store → Set-Cookie: sid=abc123
3. Κάθε επόμενο request: ο browser στέλνει αυτόματα το cookie
4. Server: lookup του sid στο store → ξέρει ποιος είσαι
5. Logout: διαγραφή του session από το store → το cookie αχρηστεύεται
```

Το cookie πρέπει να έχει τα σωστά flags — αυτά ρωτιούνται ευθέως:

```http
Set-Cookie: sid=abc123; HttpOnly; Secure; SameSite=Lax; Max-Age=604800; Path=/
```

- `HttpOnly` — η JavaScript **δεν** μπορεί να το διαβάσει → προστασία από XSS theft.
- `Secure` — αποστολή μόνο μέσω HTTPS.
- `SameSite=Lax/Strict` — δεν στέλνεται σε cross-site requests → προστασία από CSRF.
- `Max-Age` — λογικό expiration.

Δυνατά σημεία των sessions: **άμεσο revoke** (σβήνεις το record και ο χρήστης είναι έξω), όλα τα secrets server-side. Αδυναμία: κάθε request θέλει lookup στο session store — γι' αυτό συχνά μπαίνει Redis.

## JWT — stateless tokens

Το **JWT (JSON Web Token)** είναι ένα signed token με τρία μέρη: `header.payload.signature`.

```json
{
  "sub": "user_123",
  "role": "admin",
  "iat": 1735603200,
  "exp": 1735689600
}
```

Ο server το υπογράφει με secret (HMAC) ή private key (RSA). Σε κάθε request, ο client το στέλνει στο `Authorization: Bearer <token>` και ο server **επαληθεύει την υπογραφή χωρίς database lookup** — αυτό είναι το stateless πλεονέκτημα.

Τα τρία πράγματα που ΠΡΕΠΕΙ να πεις σωστά σε συνέντευξη:

1. **Το payload ΔΕΝ είναι encrypted** — είναι base64-encoded, ο καθένας το διαβάζει. Η υπογραφή εγγυάται μόνο ότι δεν **τροποποιήθηκε**. Ποτέ ευαίσθητα δεδομένα μέσα.
2. **Revocation είναι το αδύνατο σημείο** — δεν μπορείς να «σβήσεις» ένα εκδομένο JWT. Γι' αυτό: short-lived access tokens (5-15 λεπτά) + refresh tokens.
3. **Πάντα επαληθεύεις signature ΚΑΙ expiration** server-side — και ορίζεις ρητά τον αλγόριθμο (η επίθεση `alg: none` είναι κλασική).

### Access + Refresh token pattern

```text
Login → access token (15 λεπτά) + refresh token (7 ημέρες, αποθηκευμένο σε DB)
API requests → access token στο Authorization header
Access token έληξε → POST /refresh με το refresh token → νέο access token
Logout / κλοπή → revoke το refresh token στη DB → μετά από ≤15 λεπτά ο χρήστης είναι έξω
```

Έτσι συνδυάζεις stateless ταχύτητα (κάθε request) με δυνατότητα revoke (στο refresh).

### Πού αποθηκεύεις το token στον browser;

| Αποθήκευση | Κίνδυνος |
|---|---|
| `localStorage` | Ευάλωτο σε **XSS** — κάθε injected script το διαβάζει |
| Cookie χωρίς flags | Ευάλωτο σε **CSRF** |
| `HttpOnly` + `SameSite` cookie | Η ασφαλέστερη συνήθης επιλογή |
| Μνήμη (in-memory) | Ασφαλές αλλά χάνεται στο refresh — συνδυάζεται με refresh cookie |

## Sessions vs JWT — η ερώτηση-κλειδί

| | Sessions | JWT |
|---|---|---|
| State | Server-side (DB/Redis) | Stateless — όλα στο token |
| Revoke | Άμεσο | Δύσκολο — θέλει short expiry + refresh |
| Κόστος ανά request | Store lookup | Μόνο verify signature |
| Microservices | Θέλουν κοινό store | Κάθε service επαληθεύει μόνο του |
| Ιδανικό για | Web apps, server-rendered | APIs, mobile, distributed systems |

Καλή συνεντευξιακή απάντηση: «Για ένα κλασικό web app ξεκινάω με sessions + HttpOnly cookies γιατί το revoke είναι άμεσο και απλό. JWT όταν έχω πολλά services ή mobile clients — με short-lived access tokens και refresh rotation.»

## OAuth 2.0 — «Login with Google»

Το OAuth είναι **delegation protocol**, όχι password login — η εφαρμογή σου δεν βλέπει ΠΟΤΕ το Google password του χρήστη:

```text
1. Ο χρήστης πατά "Login with GitHub"
2. Redirect στο GitHub: client_id, redirect_uri, scope, state
3. Ο χρήστης εγκρίνει στο GitHub (εκεί βάζει το password του — όχι σε σένα)
4. GitHub redirect πίσω: ?code=AUTH_CODE&state=...
5. Ο server σου ανταλλάσσει SERVER-SIDE το code + client_secret για access token
6. Με το token ζητάς profile data → βρίσκεις/δημιουργείς τον δικό σου user
7. Από εκεί και πέρα: ΔΙΚΟ σου session/JWT όπως πάντα
```

Σημεία που ξεχωρίζουν τον καλό υποψήφιο: το **authorization code** ανταλλάσσεται server-side (το `client_secret` δεν αγγίζει ποτέ τον browser), το **state** parameter προστατεύει από CSRF, και το **PKCE** είναι η επέκταση για mobile/SPA clients που δεν μπορούν να κρατήσουν secret.

## Authorization: το δεύτερο μισό που όλοι ξεχνούν

Το login λέει ποιος είσαι. Η εξουσιοδότηση πρέπει να ελέγχεται **σε κάθε ενέργεια, server-side**:

```ts
// ❌ ΛΑΘΟΣ: ο έλεγχος βασίζεται σε κρυμμένο UI
// (ο attacker καλεί το API απευθείας με curl)

// ✅ ΣΩΣΤΟ: ownership check στο data layer
async function getTask(taskId: string, currentUserId: string) {
  const task = await db.task.findUnique({ where: { id: taskId } });

  if (!task) throw new NotFoundError();
  if (task.userId !== currentUserId) throw new ForbiddenError(); // 403

  return task;
}
```

Η πιο διάσημη ευπάθεια αυτής της κατηγορίας: **IDOR (Insecure Direct Object Reference)** — ο χρήστης αλλάζει το `/tasks/42` σε `/tasks/43` στο URL και βλέπει ξένα δεδομένα, επειδή ο server έλεγξε μόνο «είναι logged in;» και όχι «του ανήκει;».

Επίπεδα προστασίας σε fullstack app — και τα τρία:

1. **UI routes** — redirect αν δεν υπάρχει session (UX, όχι security).
2. **API routes** — middleware που απαιτεί έγκυρο session/token (security).
3. **Data queries** — ownership/role checks σε κάθε query (η τελική γραμμή άμυνας).

## Mini project: protected dashboard

Σχεδίασε (και ιδανικά υλοποίησε) πλήρες auth flow:

1. `POST /register` με bcrypt hash και validation.
2. `POST /login` → session cookie (HttpOnly, Secure, SameSite) ή access+refresh tokens.
3. Middleware που προστατεύει το `/api/me` και το dashboard.
4. API route που επιστρέφει **μόνο** τα δεδομένα του current user (όχι IDOR!).
5. `POST /logout` που καταστρέφει το session ή κάνει revoke το refresh token.
6. Rate limiting στο login endpoint (προστασία από brute force).

## Πώς απαντάς στη συνέντευξη

> Authentication verifies identity; authorization checks permissions — and I enforce both server-side. Passwords are hashed with bcrypt, sessions live in HttpOnly SameSite cookies, and if I use JWTs I keep them short-lived with refresh token rotation since JWTs can't be revoked. Every data query includes an ownership check, because hiding a button in the UI is not security — that's how you get IDOR vulnerabilities.
