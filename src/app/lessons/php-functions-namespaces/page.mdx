# Functions, Includes & Namespaces

Όσο μεγαλώνει ένα project, η οργάνωση του κώδικα μετράει περισσότερο από τη σύνταξη. Εδώ μαθαίνεις functions, πώς να σπας τον κώδικα σε αρχεία, και πώς τα namespaces αποτρέπουν συγκρούσεις ονομάτων.

## Functions με τύπους

```php
<?php
declare(strict_types=1);

function add(int $a, int $b): int {
    return $a + $b;
}

// nullable τύπος + default τιμή
function greet(?string $name = null): string {
    return "Γεια " . ($name ?? "ξένε");
}
```

Πάντα βάζεις **type hints** στα parameters και **return type** μετά το `:`. Το `?string` σημαίνει «string ή null».

## Default & named arguments

```php
<?php
function makeUser(string $name, int $age = 18, bool $admin = false): array {
    return compact("name", "age", "admin");
}

// named arguments (PHP 8): δίνεις μόνο ό,τι θες, με όνομα
$u = makeUser(name: "Άννα", admin: true);
```

Τα **named arguments** σε γλιτώνουν από το να θυμάσαι σειρά ή να περνάς `null, null, true`.

## Arrow functions & closures

```php
<?php
$tax = 0.24;

// arrow function: κρατά αυτόματα το $tax από το περιβάλλον
$withTax = fn(float $price): float => $price * (1 + $tax);

// closure: θέλει ρητό use για να «δει» εξωτερικές μεταβλητές
$withTax2 = function (float $price) use ($tax): float {
    return $price * (1 + $tax);
};
```

Η διαφορά: το `fn() =>` κληρονομεί αυτόματα τις μεταβλητές του scope, ενώ το `function () use (...)` τις θέλει ρητά.

## Σπάσε τον κώδικα σε αρχεία

```php
<?php
// helpers.php
function slugify(string $text): string {
    return strtolower(str_replace(" ", "-", trim($text)));
}
```

```php
<?php
// index.php
require __DIR__ . "/helpers.php";
echo slugify("Modern PHP"); // modern-php
```

- `require`: σταματά με fatal error αν λείπει το αρχείο.
- `include`: συνεχίζει με warning.
- `require_once` / `include_once`: αποτρέπουν διπλό include.

Κανόνας: για απαραίτητο κώδικα → `require_once`. (Στην πράξη σχεδόν ποτέ δεν γράφεις χειροκίνητα require — το αναλαμβάνει ο Composer autoloader, δες παρακάτω μάθημα.)

## Namespaces — γιατί υπάρχουν

Δύο βιβλιοθήκες μπορεί να έχουν class `Logger`. Χωρίς namespaces, σύγκρουση. Το namespace είναι «επώνυμο» για τον κώδικά σου.

```php
<?php
// src/Support/Str.php
namespace App\Support;

class Str {
    public static function slug(string $text): string {
        return strtolower(str_replace(" ", "-", trim($text)));
    }
}
```

```php
<?php
// index.php
require ...;

use App\Support\Str;          // import με όνομα

echo Str::slug("Hello World"); // hello-world
// ή πλήρως: \App\Support\Str::slug(...)
```

- `namespace App\Support;` δηλώνει πού «ζει» η class.
- `use App\Support\Str;` την εισάγει ώστε να γράφεις σκέτο `Str`.
- Το `\` μπροστά σημαίνει «από τη ρίζα» (global namespace).

Η σύμβαση: το namespace **αντικατοπτρίζει τη δομή φακέλων** (`App\Support\Str` → `src/Support/Str.php`). Αυτό είναι το PSR-4, που κάνει δυνατό το autoloading.

## Πώς απαντάς στη συνέντευξη

«How do you organize PHP code across files?»

> I split reusable logic into classes, one per file, under a namespace that mirrors the folder structure — that's the PSR-4 convention. I `use` the classes I need at the top of a file instead of fully-qualifying them everywhere. In real projects I don't write manual `require` calls; Composer's autoloader resolves `App\Support\Str` to `src/Support/Str.php` automatically. For functions I always add type hints and return types, and I prefer arrow functions for short callbacks since they capture scope automatically.
