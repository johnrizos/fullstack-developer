# NoSQL Foundations

Το «NoSQL» δεν είναι ένα πράγμα — είναι μια οικογένεια από databases με διαφορετικά data models, που γεννήθηκαν όταν το relational δεν κάλυπτε συγκεκριμένες ανάγκες (κλίμακα, ευελιξία, throughput). Στα interviews εξετάζουν αν ξέρεις **ποιον τύπο** και **πότε** — όχι «NoSQL» ως γενική απάντηση.

## Τι πρέπει να μπορείς στο τέλος

- να ονομάζεις τους 4 τύπους NoSQL και τι λύνει ο καθένας,
- να εξηγείς SQL vs NoSQL με βάση ανάγκες, όχι hype,
- να ξεχωρίζεις ACID από BASE,
- να εξηγείς το CAP theorem,
- να ξέρεις τι θυσιάζεις φεύγοντας από relational.

## Οι 4 τύποι NoSQL

```txt
Document    (MongoDB, Firestore)   → JSON-like documents, ευέλικτο schema
Key-Value   (Redis, DynamoDB)      → απλό get/set σε key, πανγρήγορο
Wide-Column (Cassandra, Bigtable)  → τεράστια κλίμακα writes, column families
Graph       (Neo4j)                → nodes & edges, σχέσεις first-class
```

Καθένας λύνει διαφορετικό πρόβλημα. Στο υπόλοιπο track εμβαθύνουμε σε document, key-value και wide-column (οι πιο συχνοί στο fullstack). Το graph το αναφέρεις όταν το domain είναι κυρίως σχέσεις (social, recommendations, fraud).

## SQL vs NoSQL — πότε τι

```txt
Διάλεξε RELATIONAL (SQL) όταν:
- Τα δεδομένα έχουν σχέσεις & χρειάζεσαι joins
- Χρειάζεσαι ACID transactions (πληρωμές, αποθέματα)
- Τα access patterns αλλάζουν / ad-hoc queries
- Ακεραιότητα/constraints είναι κρίσιμα

Διάλεξε NoSQL όταν:
- Γνωστά, σταθερά access patterns με ανάγκη για κλίμακα
- Ευέλικτο/εξελισσόμενο schema (document)
- Πολύ υψηλό write/read throughput, horizontal scaling
- key-value/caching/session/event data
```

Σημαντικό: σύγχρονες relational βάσεις (Postgres) έχουν `JSONB`, που καλύπτει πολλές «document» ανάγκες χωρίς να φύγεις από SQL. Default: **ξεκίνα relational· πήγαινε NoSQL όταν ένα συγκεκριμένο pattern το απαιτεί**.

## ACID vs BASE

Το relational εγγυάται **ACID**. Πολλά NoSQL ανταλλάσσουν αυστηρότητα για κλίμακα/διαθεσιμότητα — το μοντέλο τους λέγεται **BASE**:

```txt
ACID  → Atomicity, Consistency, Isolation, Durability
        "πάντα σωστό, ακόμα κι αν αργήσει ή μπλοκάρει"

BASE  → Basically Available, Soft state, Eventually consistent
        "πάντα διαθέσιμο, συγκλίνει στο σωστό με τον χρόνο"
```

Δεν είναι «καλό vs κακό» — είναι trade-off. Ένα banking ledger θέλει ACID. Ένα «likes counter» ή feed αντέχει eventual consistency για να είναι πάντα γρήγορο/διαθέσιμο.

## CAP theorem

Σε ένα **distributed** σύστημα, κατά τη διάρκεια ενός **network partition** (P, αναπόφευκτο), διαλέγεις:

```txt
C (Consistency)  → όλοι βλέπουν τα ίδια δεδομένα· άρνηση αν δεν γίνεται
A (Availability) → πάντα απαντάς, ακόμα κι αν τα δεδομένα είναι stale
```

Δεν μπορείς να έχεις πλήρως και τα δύο σε partition. Παραδείγματα: παραδοσιακές SQL τείνουν σε **CP**· πολλά NoSQL προσφέρουν **AP** με eventual consistency.

## Τι θυσιάζεις φεύγοντας από relational

```txt
- Joins (συχνά τα κάνεις στον app ή με denormalization)
- ACID transactions across documents (περιορισμένα/πιο αδύναμα σε πολλά)
- Ad-hoc queries (το model είναι δεμένο με τα γνωστά access patterns)
- Strong consistency (συχνά eventual)
```

Σε αντάλλαγμα κερδίζεις: horizontal scale, ευελιξία schema, throughput για **γνωστά** patterns. Αν δεν κερδίζεις κάτι από αυτά, μάλλον θες relational.

## Πώς απαντάς στη συνέντευξη

«SQL or NoSQL — how do you choose?»

> NoSQL isn't one thing — it's document, key-value, wide-column, and graph stores, each for a different shape of problem, so I name the specific one and why. I reach for relational when data has relationships I need to join, when I need ACID transactions and integrity, or when queries are ad-hoc. I reach for NoSQL when access patterns are known and stable but need to scale, or for flexible schemas and very high throughput. The deeper trade-off is ACID versus BASE: relational gives strong guarantees, many NoSQL stores choose availability with eventual consistency, which CAP makes explicit — in a partition you pick consistency or availability. Often Postgres with JSONB covers document needs without leaving SQL, so I default to relational and move when a pattern genuinely demands it.
