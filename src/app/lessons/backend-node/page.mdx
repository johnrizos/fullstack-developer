# Node.js, Express & NestJS

Backend developer σημαίνει να ξέρεις πώς στήνεται server, πώς περνά ένα request μέσα από middleware μέχρι το response, και πώς οργανώνεις τον κώδικα ώστε να μη γίνει το route handler ένα τεράστιο function. Στις συνεντεύξεις, οι σταθερές ερωτήσεις: «γιατί Node;», «τι είναι middleware;», «Express ή NestJS;».

## Γιατί Node.js — η σωστή απάντηση

Το Node τρέχει JavaScript στον server πάνω στο V8, με ένα κρίσιμο σχεδιαστικό χαρακτηριστικό: **non-blocking I/O σε ένα thread**.

Παραδοσιακά servers δεσμεύουν ένα thread ανά request — αν το request περιμένει τη database, το thread κάθεται. Το Node δουλεύει αλλιώς: όταν ένα request περιμένει I/O (database, αρχείο, εξωτερικό API), ο μοναδικός thread **συνεχίζει να εξυπηρετεί άλλα requests** και επιστρέφει όταν έρθει η απάντηση — ο μηχανισμός είναι ο ίδιος event loop που ξέρεις από τον browser.

Συνέπειες που πρέπει να αρθρώνεις:

- **Δυνατό σε I/O-bound φορτία**: APIs, proxies, real-time — χιλιάδες ταυτόχρονες συνδέσεις με λίγη μνήμη.
- **Αδύναμο σε CPU-bound δουλειά**: ένα βαρύ συγχρονισμένο loop (image processing, crypto) **μπλοκάρει όλα τα requests** — γι' αυτό βαριά δουλειά πάει σε worker threads ή ξεχωριστό service.
- Ίδια γλώσσα με το frontend: κοινά types, κοινά validation schemas, ένα οικοσύστημα.

Η ροή ενός request σε σωστά οργανωμένο backend:

```txt
Client ──> Middleware (logging, auth, parsing)
       ──> Route handler (HTTP concerns: params, status codes)
       ──> Service (business logic)
       ──> Data layer (database queries)
       <── JSON response
```

## Express: minimal αλλά με κανόνες

```ts
import express from "express";

const app = express();
app.use(express.json()); // χωρίς αυτό, το req.body είναι undefined!

app.get("/tasks", async (req, res) => {
  const tasks = await taskService.list({ completed: req.query.completed === "true" });
  res.json({ data: tasks });
});

app.post("/tasks", async (req, res, next) => {
  try {
    const title = req.body.title;
    if (typeof title !== "string" || title.trim().length < 2) {
      return res.status(422).json({ error: "Title must be at least 2 characters" });
    }
    const task = await taskService.create({ title: title.trim() });
    res.status(201).json({ data: task });
  } catch (error) {
    next(error); // προώθηση στο centralized error handler
  }
});

app.listen(3000);
```

Το route handler κρατά **μόνο** τα HTTP concerns: parsing, validation, status codes. Η δουλειά γίνεται στο service — έτσι η λογική τεστάρεται χωρίς HTTP και επαναχρησιμοποιείται.

## Middleware: το pipeline

Middleware = functions που τρέχουν **με τη σειρά δήλωσής τους** πριν φτάσει το request στο handler. Κάθε ένα είτε καλεί `next()` είτε τερματίζει το request:

```ts
// Logger — τρέχει για όλα τα requests
app.use((req, _res, next) => {
  console.info({ method: req.method, path: req.path });
  next();
});

// Auth guard — μόνο για /api routes
app.use("/api", (req, res, next) => {
  const user = verifyToken(req.headers.authorization);
  if (!user) return res.status(401).json({ error: "Unauthorized" });
  req.user = user; // διαθέσιμο σε όλα τα επόμενα στάδια
  next();
});

// Error handler — ΤΕΛΕΥΤΑΙΟ, με 4 παραμέτρους (έτσι το αναγνωρίζει το Express)
app.use((error, _req, res, _next) => {
  console.error(error);
  res.status(500).json({ error: "Internal server error" }); // όχι stack traces έξω!
});
```

> Trap που αξίζει να ξέρεις: στο Express 4, ένα exception μέσα σε **async** handler χωρίς try/catch **δεν** φτάνει στον error handler — το request κρεμάει. ## Η αρχιτεκτονική σε στρώματα

```txt
src/
  routes/tasks.routes.ts     ← HTTP: paths, status codes, parsing
  services/tasks.service.ts  ← business logic (testable, χωρίς HTTP)
  repositories/tasks.repo.ts ← data access (queries)
  middleware/auth.ts
  config.ts                  ← env vars ΕΝΑ σημείο (process.env μόνο εδώ)
```

```ts
// services/tasks.service.ts — δεν ξέρει τίποτα από req/res
export async function create(input: { title: string }) {
  // business rules εδώ: όρια, ιδιοκτησία, side effects
  return taskRepo.insert({ ...input, completed: false });
}
```

Γιατί; Το service τεστάρεται με απλές κλήσεις. Το repository αλλάζει (SQLite → Postgres) χωρίς να αγγίξεις λογική. Και το route διαβάζεται σε 10 γραμμές.

## NestJS: η ίδια ιδέα, επιβεβλημένη από το framework

Το NestJS παίρνει αυτά τα στρώματα και τα κάνει conventions με decorators και **dependency injection**:

```ts
@Controller("tasks")
export class TasksController {
  // DI: το Nest ΔΙΝΕΙ το service — δεν κάνεις new TasksService()
  constructor(private readonly tasksService: TasksService) {}

  @Get()
  findAll(@Query("completed") completed?: string) {
    return this.tasksService.findAll({ completed: completed === "true" });
  }

  @Post()
  @UsePipes(new ValidationPipe())
  create(@Body() dto: CreateTaskDto) {
    return this.tasksService.create(dto);
  }
}
```

Οι έννοιες-κλειδιά: **Controller** (HTTP), **Service/Provider** (logic), **Module** (οργάνωση + DI scope), **Guard** (auth), **Pipe** (validation/transform), **Interceptor** (logging, response shaping).

Το «γιατί DI;» είναι η ερώτηση-φίλτρο: επειδή ο controller **δηλώνει τι χρειάζεται** αντί να το κατασκευάζει, στο test δίνεις mock service με μία γραμμή — και οι εξαρτήσεις του συστήματος είναι ορατές στα constructors αντί κρυμμένες σε imports.

**Express ή NestJS;** Express: μικρά APIs, microservices, πλήρης ελευθερία. NestJS: μεγαλύτερα apps και ομάδες — η δομή είναι έτοιμη και κοινή για όλους, με κόστος boilerplate και καμπύλη εκμάθησης.

## Mini project: small tasks API

Χτίσε το βήμα-βήμα:

1. `GET /tasks` με optional `?completed=` filter.
2. `POST /tasks` με validation (422 σε άκυρο input, 201 με το νέο task).
3. `PATCH /tasks/:id` για το `completed` (404 αν δεν υπάρχει).
4. Logging middleware + auth middleware στο `/api`.
5. Centralized error handler — δοκίμασε να πετάξεις error από service και δες ότι βγαίνει καθαρό JSON, όχι stack trace.
6. Χώρισε σε routes/service/repository — και μετά πρόσθεσε `DELETE /tasks/:id` μετρώντας πόσα αρχεία άγγιξες.

## Πώς απαντάς στη συνέντευξη

«Express ή NestJS;»

> Express is minimal and unopinionated — great for small services where I set up my own layering: routes for HTTP concerns, services for business logic, repositories for data access, and a centralized error handler. NestJS bakes that same architecture into the framework with modules, controllers, providers, and dependency injection, which pays off in larger codebases and teams because the structure is consistent and testing is easy — you inject mocks instead of patching imports.
