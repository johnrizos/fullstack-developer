# Containers & Orchestration (Kubernetes)

Το Docker σου δίνει ένα container· αλλά σε production τρέχεις **δεκάδες/εκατοντάδες** containers σε πολλά μηχανήματα — και κάποιος πρέπει να τα ξεκινά, να τα κλιμακώνει, να τα ξανασηκώνει όταν πέφτουν, και να τα ενημερώνει χωρίς downtime. Αυτό κάνει ένα **orchestrator** (κυρίως **Kubernetes**). Αυτό το μάθημα είναι concept-first: τι λύνει το orchestration, οι core έννοιες του Kubernetes, και — εξίσου σημαντικό — **πότε ΔΕΝ το χρειάζεσαι**.

## Από container σε orchestration

Ένα container είναι φορητό και απομονωμένο, αλλά μόνο του δεν ξέρει:
- να τρέχει **πολλά αντίγραφα** σε πολλούς servers,
- να **ξανασηκώνεται** αν πέσει (self-healing),
- να **κλιμακώνεται** αυτόματα με το φορτίο,
- να γίνεται **rolling update** χωρίς downtime,
- να βρίσκει άλλα services (service discovery) και να μοιράζει traffic.

Ένας **orchestrator** τα κάνει όλα αυτά δηλωτικά: του λες **τι** θες (desired state), και αυτός το διατηρεί.

## Declarative & self-healing — η βασική ιδέα

Με Kubernetes δηλώνεις την επιθυμητή κατάσταση (π.χ. «θέλω 3 αντίγραφα αυτού του image»). Ο orchestrator συνεχώς **συγκρίνει actual vs desired** και διορθώνει: αν ένα pod πέσει, σηκώνει νέο· αν ζητήσεις 5 αντί για 3, προσθέτει 2. Δεν λες **πώς** — λες **τι**.

## Core Kubernetes concepts

| Concept | Τι είναι |
|---|---|
| **Pod** | Η μικρότερη μονάδα — ένα (ή λίγα) containers που τρέχουν μαζί. Εφήμερο. |
| **Deployment** | Δηλώνει «θέλω N αντίγραφα αυτού του pod» + κάνει rolling updates & rollbacks. |
| **Service** | Σταθερό δικτυακό endpoint + load balancing μπροστά από τα (εφήμερα) pods. |
| **Ingress** | Δρομολόγηση εξωτερικού HTTP traffic σε services (host/path routing, TLS). |
| **ConfigMap / Secret** | Config & μυστικά, ξεχωριστά από το image (έξω από τον κώδικα). |
| **Namespace** | Λογικός διαχωρισμός (π.χ. dev/staging/prod). |

Το pod είναι εφήμερο (μπορεί να πεθάνει/μετακινηθεί), γι' αυτό μιλάς πάντα μέσω **Service** — αυτό δίνει σταθερή διεύθυνση και μοιράζει traffic στα ζωντανά pods.

## Self-healing με health probes

Το Kubernetes χρησιμοποιεί δύο probes:
- **Liveness probe** — «είναι ζωντανό;» Αν αποτύχει → **restart** του pod.
- **Readiness probe** — «είναι έτοιμο να δεχτεί traffic;» Αν αποτύχει → το βγάζει από το Service (δεν του στέλνει requests) μέχρι να ετοιμαστεί.

Έτσι υλοποιούνται τα zero-downtime deploys που είδες: νέα pods μπαίνουν στο Service μόνο όταν είναι ready.

## Autoscaling

- **Horizontal Pod Autoscaler (HPA)** — αυξάνει/μειώνει τον αριθμό των pods με βάση CPU/μετρικές.
- **Cluster autoscaler** — προσθέτει/αφαιρεί μηχανήματα (nodes) όταν χρειάζονται.

## Πότε ΔΕΝ χρειάζεσαι Kubernetes

Κρίσιμη (και ώριμη) απάντηση συνέντευξης: το Kubernetes έχει **μεγάλη πολυπλοκότητα**. Για μικρές/μεσαίες εφαρμογές, ένα **PaaS** (Vercel, Render, Railway, Fly.io) ή απλά managed containers κάνουν την ίδια δουλειά με κλάσμα του operational κόστους. Διάλεξε Kubernetes όταν έχεις **πολλά services**, ανάγκη για fine-grained control/portability, ή ομάδα που μπορεί να το συντηρήσει — όχι by default.

## Πώς απαντάς στη συνέντευξη

> «What does Kubernetes give you, and when would you avoid it?»

> A container alone doesn't run multiple replicas across machines, restart on failure, scale with load, roll out without downtime, or do service discovery — an orchestrator does. Kubernetes is declarative and self-healing: I declare desired state, like three replicas of an image, and it continuously reconciles actual against desired, recreating a pod if one dies. The core objects: pods are the ephemeral unit, a Deployment maintains N replicas and handles rolling updates and rollbacks, a Service is a stable endpoint with load balancing in front of the ephemeral pods, and Ingress routes external HTTP. Liveness probes restart unhealthy pods and readiness probes gate traffic, which is what enables zero-downtime deploys, and HPA autoscales pods on metrics. But I wouldn't reach for it by default — it's heavy operationally. For small to mid apps a PaaS like Vercel or Render does the same job for a fraction of the cost; Kubernetes earns its complexity when you have many services or need fine control and have a team to run it.
