# Docker Foundations

Πριν γράψεις ούτε ένα `Dockerfile`, πρέπει να καταλάβεις **τι πρόβλημα λύνει το Docker** — αλλιώς οι εντολές μοιάζουν μαγικά ξόρκια. Αυτό το μάθημα ξεκινά από το μηδέν: γιατί υπάρχει το Docker, ποιες είναι οι 3 λέξεις-κλειδιά (image, container, registry), σε τι διαφέρει από ένα virtual machine, και τρέχεις τον πρώτο σου container.

## Το πρόβλημα: «στο μηχάνημά μου δουλεύει»

Φαντάσου ότι έφτιαξες μια εφαρμογή στον υπολογιστή σου. Δουλεύει τέλεια. Τη στέλνεις σε συνάδελφο — και **σκάει**. Γιατί;

- Εσύ έχεις Node 20, αυτός Node 18.
- Εσύ έχεις PostgreSQL εγκατεστημένη, αυτός όχι.
- Εσύ έχεις μια environment variable που ξέχασες ότι υπάρχει.
- Ο server της παραγωγής τρέχει άλλη έκδοση Linux με άλλες βιβλιοθήκες.

Αυτό είναι το κλασικό **«works on my machine»** πρόβλημα: η εφαρμογή σου εξαρτάται από **αόρατες προϋποθέσεις** του δικού σου μηχανήματος. Όσο περισσότερα μηχανήματα εμπλέκονται (ο δικός σου, του συναδέλφου, το CI, η παραγωγή), τόσο πιο εύθραυστο γίνεται.

## Η λύση: πακετάρω τα ΠΑΝΤΑ μαζί

Το Docker λέει: αντί να ελπίζω ότι κάθε μηχάνημα έχει τις σωστές προϋποθέσεις, **βάζω τα πάντα σε ένα πακέτο** — τον κώδικά μου, το runtime (π.χ. Node), τις βιβλιοθήκες, τα system dependencies, ακόμα και ένα κομμάτι λειτουργικού. Αυτό το πακέτο τρέχει **ακριβώς το ίδιο** παντού: στον υπολογιστή σου, στο CI, στην παραγωγή.

Αυτό λέγεται **containerization**. Το αποτέλεσμα: «αν δουλεύει εδώ, δουλεύει παντού» — γιατί το «εδώ» ταξιδεύει μαζί με την εφαρμογή.

## Οι 3 λέξεις που πρέπει να ξεκαθαρίσεις

Το 90% της αρχικής σύγχυσης με το Docker είναι ότι μπερδεύεις αυτούς τους 3 όρους. Ας τους καρφώσουμε:

| Όρος | Τι είναι | Αναλογία |
|---|---|---|
| **Image** | Το αμετάβλητο «πακέτο»: app + dependencies + runtime. Ένα αρχείο-πρότυπο, δεν «τρέχει». | Σαν **class** στον προγραμματισμό — ή σαν μια συνταγή. |
| **Container** | Ένα **τρέχον instance** ενός image. Ζωντανή διεργασία με δικό της filesystem. | Σαν **object** — ή σαν το φαγητό που μαγείρεψες από τη συνταγή. |
| **Registry** | Αποθήκη με images, από όπου κάνεις push/pull (π.χ. Docker Hub, GHCR). | Σαν το **npm/GitHub** αλλά για images. |

Το κλειδί: από **ένα** image μπορείς να σηκώσεις **πολλά** containers, ακριβώς όπως από μία class φτιάχνεις πολλά objects. Το image είναι νεκρό/αμετάβλητο· ο container ζει.

```text
Registry (Docker Hub)
    │  docker pull node:20
    ▼
  Image  (node:20)  ── docker run ──▶  Container 1 (ζωντανός)
                     ── docker run ──▶  Container 2 (ζωντανός)
```

## Container vs Virtual Machine — η ερώτηση που θα σου κάνουν

Και τα δύο απομονώνουν εφαρμογές, αλλά με πολύ διαφορετικό κόστος. Αυτή είναι **σχεδόν σίγουρη ερώτηση συνέντευξης**:

- Ένα **VM** σηκώνει **ολόκληρο guest λειτουργικό** πάνω σε έναν hypervisor. Βαρύ: GBs στο δίσκο, δεκάδες δευτερόλεπτα boot, πολλή RAM. Αλλά **ισχυρή απομόνωση** (ξεχωριστός kernel).
- Ένα **container μοιράζεται τον kernel του host** και απομονώνεται σε επίπεδο διεργασίας (μέσω Linux namespaces & cgroups). Ελαφρύ: MBs, ξεκινά σε **milliseconds**, ελάχιστη μνήμη. Πιο αδύναμη απομόνωση από VM, αλλά αρκετή για τις περισσότερες περιπτώσεις.

```text
VM:                          Container:
┌─────────┐ ┌─────────┐      ┌─────────┐ ┌─────────┐
│  App A  │ │  App B  │      │  App A  │ │  App B  │
│ Guest OS│ │ Guest OS│      └─────────┘ └─────────┘
├─────────┴─┴─────────┤      ├─────────────────────┤
│      Hypervisor     │      │  Docker Engine       │
├─────────────────────┤      ├─────────────────────┤
│      Host OS         │      │  Host OS (κοινός kernel)│
└─────────────────────┘      └─────────────────────┘
```

Με απλά λόγια: **το VM κουβαλά ολόκληρο OS, ο container μοιράζεται το OS του host** — γι' αυτό ο container είναι δραματικά ελαφρύτερος και γρηγορότερος.

## «Μοιράζεται το OS» — τι ΑΚΡΙΒΩΣ σημαίνει;

Εδώ μπερδεύονται σχεδόν όλοι, οπότε ας το ξεκαθαρίσουμε. Ένα λειτουργικό σύστημα έχει **δύο μέρη**:

- **Kernel** — ο «κινητήρας». Μιλάει στο υλικό (CPU, μνήμη, δίσκος, δίκτυο) και διαχειρίζεται τις διεργασίες. Η καρδιά του OS.
- **Userspace** — τα προγράμματα, οι βιβλιοθήκες, τα αρχεία, το shell, ο package manager. «Κάθονται πάνω» στον kernel και του ζητάνε δουλειές.

Όταν λέμε ότι ο container «μοιράζεται το OS του host», εννοούμε συγκεκριμένα ότι **μοιράζεται τον kernel** — δεν κουβαλάει δικό του.

### Ένα image περιέχει userspace, ΟΧΙ kernel

Αυτό είναι το κλειδί: ένα image (`ubuntu`, `node:20-alpine`) περιέχει **μόνο το userspace** μιας διανομής Linux — τα αρχεία και τις βιβλιοθήκες της. **Δεν περιέχει kernel.** Γι' αυτό ένα Alpine image είναι ~5MB και ένα Ubuntu ~30MB, ενώ ένα ολόκληρο Ubuntu ISO (με kernel) είναι ~4GB. Το image δεν είναι «όλο το Ubuntu» — είναι τα προγράμματά του, που δανείζονται τον kernel του host για να τρέξουν.

### Ο κανόνας συμβατότητας

Επειδή ο container δανείζεται τον kernel του host, πρέπει να είναι **συμβατός** μαζί του:

> Ένας **Linux container** χρειάζεται **Linux kernel**· ένας **Windows container** χρειάζεται **Windows kernel**.

Άρα, τι γίνεται με ένα **Linux container σε Windows Server** (ή στον Windows/Mac υπολογιστή σου); Με τη «στεγνή» λογική δεν θα έπρεπε να δουλεύει — ο Windows kernel δεν τρέχει Linux userspace, είναι άλλος κινητήρας. Και όμως δουλεύει, χάρη σε ένα «κόλπο»:

> Σε Windows/Mac, το Docker στήνει από κάτω ένα **μικρό, κρυφό Linux VM** (στα Windows μέσω **WSL2** ή Hyper-V). Ο Linux container σου **δεν** χρησιμοποιεί τον Windows kernel — δανείζεται τον **Linux kernel αυτού του VM**.

```text
Windows Server / ο Windows υπολογιστής σου
 └── κρυφό Linux VM (WSL2)   ← έχει ΑΛΗΘΙΝΟ Linux kernel
       └── ο Linux container δανείζεται ΑΥΤΟΝ τον kernel
```

Τα Windows δεν «προσποιούνται» ότι είναι Linux — υπάρχει ένας πραγματικός (αλλά μικρός και κρυμμένος) Linux kernel από κάτω. Γι' αυτό ακριβώς, μόλις τρέξεις `docker run -it ubuntu bash` σε Windows, βρίσκεσαι «μέσα σε Linux»: το Docker Desktop έχει ήδη στήσει αυτό το Linux VM για σένα.

### Το take-away για την παραγωγή

Αυτό το κρυφό VM υπάρχει **μόνο** όταν το OS family του host δεν ταιριάζει με του container. Στην πραγματική παραγωγή σχεδόν πάντα τρέχεις **Linux containers σε Linux servers** → **κανένα VM**, άμεσο kernel sharing, όλο το όφελος (ms εκκίνηση, MBs μνήμης). Το VM στον υπολογιστή σου είναι απλώς για development· ο container που φτιάχνεις είναι Linux και θα τρέξει σε Linux server χωρίς κανένα VM.

## Ο πρώτος σου container

Αν έχεις εγκατεστημένο το Docker Desktop, μία εντολή αρκεί:

```bash
docker run hello-world
```

Τι συνέβη βήμα-βήμα:

1. Το Docker έψαξε το image `hello-world` **τοπικά**. Δεν το βρήκε.
2. Το κατέβασε (**pull**) από το Docker Hub (registry).
3. Δημιούργησε έναν **container** από αυτό το image και τον έτρεξε.
4. Ο container τύπωσε ένα μήνυμα και **τερμάτισε**.

Δοκίμασε κάτι πιο χειροπιαστό — τρέξε μια πλήρη Ubuntu και μπες μέσα της:

```bash
docker run -it ubuntu bash
# τώρα είσαι μέσα σε ένα Linux container:
#   cat /etc/os-release   → Ubuntu, ακόμα κι αν είσαι σε Windows/Mac!
#   exit                  → βγαίνεις, ο container σταματά
```

Μέσα σε δευτερόλεπτα έχεις ένα καθαρό Linux περιβάλλον, απομονωμένο από το μηχάνημά σου. Αυτή είναι η δύναμη: **μιας χρήσης, φορητά, καθαρά περιβάλλοντα** χωρίς να «λερώνεις» τον υπολογιστή σου.

## Πού κολλάει αυτό στη δουλειά σου

- **Onboarding:** νέος developer κάνει `docker compose up` και έχει όλο το stack (app + database) σε λεπτά, χωρίς οδηγίες 3 σελίδων.
- **Ίδιο περιβάλλον παντού:** το image που πέρασε τα tests στο CI είναι το **ίδιο** που πάει παραγωγή — όχι «παρόμοιο».
- **Καθαριότητα:** δοκιμάζεις PostgreSQL 16 χωρίς να την εγκαταστήσεις μόνιμα· σβήνεις τον container και δεν έμεινε τίποτα.

## Πώς απαντάς στη συνέντευξη

> «What is Docker and what problem does it solve?»

> Docker solves the "works on my machine" problem by packaging an app together with its runtime, dependencies, and system libraries into an immutable image, so it runs identically on my laptop, in CI, and in production. The three core terms: an image is the immutable template — like a class; a container is a running instance of that image — like an object, and you can start many containers from one image; a registry like Docker Hub is where images are pushed and pulled. Unlike a virtual machine, which boots a full guest OS on a hypervisor and costs gigabytes and seconds, a container shares the host kernel and isolates at the process level with namespaces and cgroups, so it starts in milliseconds and uses megabytes. That lightweight, reproducible environment is why the image that passed tests in CI is the exact same one that ships to production.
