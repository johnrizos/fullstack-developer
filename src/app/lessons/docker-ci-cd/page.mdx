# Docker & CI/CD

Docker και CI/CD εμφανίζονται συχνά σε αγγελίες γιατί δείχνουν ότι μπορείς να παραδώσεις λογισμικό με προβλέψιμο τρόπο — όχι «στο μηχάνημά μου δουλεύει».

## Docker: οι βασικές έννοιες

Πρώτα η ορολογία που πρέπει να είναι καθαρή:

- **Image**: το αμετάβλητο «πακέτο» — app + dependencies + runtime, χτισμένο σε layers. Σαν class.
- **Container**: ένα τρέχον instance ενός image. Σαν object — μπορείς να τρέξεις πολλά από το ίδιο image.
- **Registry** (Docker Hub, GHCR): εκεί κάνεις push/pull τα images.

Και η διάκριση από VMs που ρωτιέται συχνά: τα containers **μοιράζονται τον kernel του host** και απομονώνονται σε επίπεδο διεργασίας — ξεκινούν σε milliseconds με MBs μνήμης, ενώ ένα VM σηκώνει ολόκληρο OS.

```dockerfile
FROM node:20-alpine
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build
CMD ["npm", "start"]
```

### Γιατί COPY package.json ΠΡΙΝ τον υπόλοιπο κώδικα;

Αυτή η σειρά δεν είναι τυχαία — είναι **layer caching**, και είναι κλασική interview ερώτηση. Κάθε εντολή του Dockerfile φτιάχνει ένα layer που γίνεται cache. Αν αλλάξει μόνο ο κώδικάς σου (το συχνότερο), το `COPY package*.json` + `RUN npm ci` layers είναι ίδια → **το npm install δεν ξανατρέχει**. Αν έκανες `COPY . .` πρώτο, κάθε αλλαγή κώδικα θα ακύρωνε το cache και θα ξανακατέβαζες όλα τα dependencies.

### Multi-stage builds: μικρά production images

```dockerfile
# Stage 1: build — με όλα τα devDependencies
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci
COPY . .
RUN npm run build

# Stage 2: runtime — ΜΟΝΟ ό,τι χρειάζεται για να τρέξει
FROM node:20-alpine
WORKDIR /app
ENV NODE_ENV=production
COPY package*.json ./
RUN npm ci --omit=dev
COPY --from=builder /app/dist ./dist
USER node
CMD ["node", "dist/server.js"]
```

Το τελικό image δεν κουβαλά compiler, devDependencies και source — μικρότερο, ταχύτερο στο pull, μικρότερη επιφάνεια επίθεσης. Συν δύο λεπτομέρειες που μετράνε: `USER node` (όχι root μέσα στο container) και ποτέ secrets/`.env` μέσα στο image — τα secrets έρχονται ως env vars στο runtime.

## CI

Continuous Integration σημαίνει ότι checks τρέχουν αυτόματα σε κάθε PR.

```yaml
- run: npm ci
- run: npm run lint
- run: npm run build
```

Ένα τυπικό CI pipeline για fullstack app μπορεί να έχει:

- install dependencies,
- lint,
- typecheck,
- unit tests,
- build,
- integration tests,
- artifact ή preview deploy.

Αν κάποιο βήμα αποτύχει, το PR δεν πρέπει να γίνεται merge μέχρι να διορθωθεί.

## CD

Continuous Deployment/Delivery σημαίνει ότι verified αλλαγές μπορούν να πάνε σε environment με προβλέψιμο workflow.

Υπάρχουν δύο συχνά μοντέλα:

- **Continuous Delivery**: το pipeline ετοιμάζει release, αλλά άνθρωπος πατάει deploy.
- **Continuous Deployment**: όταν περάσουν τα checks, το deploy γίνεται αυτόματα.

## Docker Compose

Σε local development, συχνά χρησιμοποιείς Docker Compose για app + database.

```yaml
services:
  db:
    image: postgres:16
    environment:
      POSTGRES_PASSWORD: postgres
    ports:
      - "5432:5432"
```

Δεν χρειάζεται να τρέχουν όλα σε Docker από την πρώτη μέρα. Αλλά πρέπει να καταλαβαίνεις πότε βοηθά: ίδια database για όλους, εύκολο reset, predictable setup.

## Pipeline failures

Όταν αποτύχει CI, διάβασε:

1. Ποιο job απέτυχε.
2. Ποιο command απέτυχε.
3. Ποιο error message εμφανίστηκε πρώτο.
4. Αν το failure αναπαράγεται local.

Μην κάνεις bypass checks χωρίς λόγο. Τα checks υπάρχουν για να προστατεύουν το main branch.

## Mini project: GitHub Actions workflow

Φτιάξε ένα `.github/workflows/ci.yml` με βήματα:

```yaml
name: CI

on:
  pull_request:
  push:
    branches: [main]

jobs:
  checks:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm
      - run: npm ci
      - run: npm run lint
      - run: npm run build
```

## Τι να προσέχεις

- Μην κάνεις deploy αν δεν περνάει build.
- Μην τυπώνεις secrets στα logs.
- Μην κάνεις deploy migrations χωρίς rollback plan.
- Κράτα τα CI jobs γρήγορα ώστε η ομάδα να τα χρησιμοποιεί πραγματικά.

## Πώς απαντάς στη συνέντευξη

> Docker packages the app with its dependencies into an immutable image, so every environment runs the same thing. I order Dockerfile instructions for layer caching — dependencies before source — and use multi-stage builds so the production image carries only the runtime, running as a non-root user with secrets injected at runtime, never baked in. CI runs install, lint, typecheck, tests, and build on every PR as a quality gate; CD then promotes the verified artifact — delivery with a manual approve, or deployment fully automated.
