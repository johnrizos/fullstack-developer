# Performance & Optimization

«Το React είναι αργό» σχεδόν ποτέ δεν ισχύει — αργά είναι τα **περιττά re-renders** και η δουλειά μέσα τους. Στα interviews εξετάζουν αν ξέρεις να μετράς πριν βελτιστοποιήσεις, τι κάνει το `React.memo`, και πότε `useMemo`/`useCallback` βοηθούν πραγματικά.

## Τι πρέπει να μπορείς στο τέλος

- να εντοπίζεις την αιτία ενός περιττού re-render,
- να χρησιμοποιείς `React.memo` σωστά (και να ξέρεις πότε δεν βοηθά),
- να εξηγείς πότε `useMemo`/`useCallback` έχουν νόημα,
- να βελτιστοποιείς μεγάλες λίστες (virtualization),
- να μετράς με το Profiler αντί να μαντεύεις.

## Πρώτα μέτρα, μετά βελτιστοποίησε

Ο νούμερο ένα κανόνας: μην βελτιστοποιείς στα τυφλά. Το React DevTools **Profiler** δείχνει ποια components renderάρουν, πόσο συχνά και γιατί. Premature `memo` παντού προσθέτει πολυπλοκότητα και κόστος χωρίς όφελος.

```txt
1. Μέτρησε με Profiler / "Highlight updates"
2. Βρες ποιο component renderάρει περιττά ή κάνει ακριβή δουλειά
3. Βελτιστοποίησε ΜΟΝΟ αυτό
4. Ξαναμέτρα ότι όντως βελτιώθηκε
```

## Γιατί renderάρει ένα component

```txt
- άλλαξε το state του
- ξανα-renderάρισε ο parent  ← η πιο συχνή αιτία περιττών renders
- άλλαξε context που καταναλώνει
```

Όταν renderάρει ο parent, renderάρουν **όλα** τα παιδιά by default — ακόμα κι αν τα props τους δεν άλλαξαν. Αυτό συνήθως είναι φθηνό· γίνεται πρόβλημα μόνο όταν το child είναι ακριβό ή η λίστα μεγάλη.

## React.memo — skip render αν τα props είναι ίδια

```tsx
const TodoItem = React.memo(function TodoItem({ todo, onToggle }: Props) {
  return <li>{todo.title}</li>;
});
```

Το `memo` κάνει shallow compare τα props· αν είναι ίδια (`Object.is`), παρακάμπτει το render. **Αλλά** σπάει αν περνάς νέα reference κάθε φορά:

## useMemo / useCallback — πότε όντως βοηθούν

```tsx
// 1) Ακριβός υπολογισμός που δεν θες να ξανατρέχει σε κάθε render
const sorted = useMemo(() => bigList.slice().sort(compare), [bigList]);

// 2) Σταθερή reference για memoized child ή dependency array
const handleSelect = useCallback((id: string) => setSelected(id), []);
```

Δεν τα βάζεις παντού: έχουν κόστος (memory + σύγκριση deps). Αξίζουν όταν (α) ο υπολογισμός είναι μετρήσιμα ακριβός, ή (β) η σταθερή reference αποτρέπει re-render σε `memo` child ή ξαναεκτέλεση effect.

## Μεγάλες λίστες — virtualization

Render 10.000 rows = 10.000 DOM nodes = αργό. **Windowing/virtualization** renderάρει μόνο τα ορατά rows (π.χ. με `react-window`/`@tanstack/react-virtual`):

```tsx
// Αντί για 10.000 <Row>, renderάρεις ~20 (όσα χωράνε στην οθόνη)
// και τα ανακυκλώνεις καθώς ο χρήστης κάνει scroll.
```

Πριν φτάσεις εκεί: σωστά `key`s, `memo` στο Row, και αποφυγή ακριβών υπολογισμών ανά row.

## Πώς απαντάς στη συνέντευξη

«How do you optimize React performance?»

> First I measure — the React Profiler shows which components render, how often, and why, so I don't optimize blindly. Most performance problems are unnecessary re-renders: when a parent renders, all children render by default, which is cheap until a child is expensive or a list is large. For those I wrap the child in `React.memo` so it skips rendering when props are shallow-equal, but that only works if I keep prop references stable — an inline function or object is a new reference every render and defeats memo, so I pair it with `useCallback`/`useMemo`. I reserve `useMemo` for genuinely expensive computations, not everything, because memoization has its own cost. For very large lists I use windowing so only the visible rows are in the DOM. Then I re-measure to confirm it actually helped.
