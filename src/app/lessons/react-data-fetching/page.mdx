# Data Fetching & Server State

Το data fetching με σκέτο `useEffect` δουλεύει σε demos, αλλά σε production θες caching, refetch, dedup, retries και error handling. Στα interviews εξετάζουν αν καταλαβαίνεις γιατί το **server state είναι διαφορετικό** και πώς το διαχειρίζεσαι σωστά.

## Τι πρέπει να μπορείς στο τέλος

- να εξηγείς γιατί το server state χρειάζεται διαφορετική μεταχείριση,
- να γράφεις σωστά loading/error/empty/success states,
- να χρησιμοποιείς React Query/SWR concepts (cache, staleness, invalidation),
- να αποφεύγεις race conditions και waterfalls,
- να βάζεις error boundaries γύρω από data UI.

## Γιατί το server state είναι ιδιαίτερο

Το server state δεν «ανήκει» στο component — είναι ένα **cache** μιας αλήθειας που ζει αλλού (database) και μπορεί να αλλάξει χωρίς να το ξέρεις.

```txt
Client state:  το ελέγχεις πλήρως, sync, δικό σου
Server state:  async, shared, μπορεί να γίνει stale, χρειάζεται
               caching / refetch / invalidation / dedup
```

Γι' αυτό το «βάλ' το σε useState μέσω useEffect» δεν φτάνει: δεν cache-άρεται μεταξύ components, ξανα-φέρνεται περιττά, και κάθε component ξαναγράφει loading/error.

## Η αφελής προσέγγιση και τα όριά της

```tsx
function useUser(id: string) {
  const [data, setData] = useState<User | null>(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState<Error | null>(null);

  useEffect(() => {
    let ignore = false;
    setLoading(true);
    fetch(`/api/users/${id}`)
      .then((r) => { if (!r.ok) throw new Error(`HTTP ${r.status}`); return r.json(); })
      .then((d) => { if (!ignore) { setData(d); setError(null); } })
      .catch((e) => { if (!ignore) setError(e); })
      .finally(() => { if (!ignore) setLoading(false); });
    return () => { ignore = true; };
  }, [id]);

  return { data, loading, error };
}
```

Δουλεύει και χειρίζεται race condition (`ignore` flag) — αλλά **δεν** έχει caching, dedup, ή background refetch. Δύο components με ίδιο `id` κάνουν δύο requests.

## React Query / SWR — server state σωστά

```tsx
function UserProfile({ id }: { id: string }) {
  const { data, isLoading, error } = useQuery({
    queryKey: ["user", id],
    queryFn: () => fetch(`/api/users/${id}`).then((r) => r.json()),
  });

  if (isLoading) return <Spinner />;
  if (error) return <ErrorBox error={error} />;
  return <h2>{data.name}</h2>;
}
```

Τι σου δίνει έτοιμο:

- **Caching ανά `queryKey`** — δύο components με ίδιο key μοιράζονται ένα request (dedup).
- **Stale-while-revalidate** — δείχνει cached δεδομένα αμέσως, ξανα-φέρνει στο background.
- **Invalidation** — μετά από mutation, `invalidateQueries(["user", id])` ανανεώνει.
- **Retries, refetch on focus/reconnect**, loading/error states.

## Πάντα τέσσερα states

Κάθε async UI έχει τέσσερις καταστάσεις — μη ξεχνάς καμία:

```tsx
if (isLoading) return <Spinner />;        // loading
if (error) return <ErrorBox error={error} />; // error
if (data.length === 0) return <Empty />;  // empty (success χωρίς δεδομένα)
return <List items={data} />;             // success με δεδομένα
```

Το «μόνο το happy path» είναι ο νούμερο ένα λόγος για buggy UI σε review.

## Error boundaries

Τα errors στο render δεν τα πιάνει το try/catch. Ένα **error boundary** πιάνει crashes στο subtree και δείχνει fallback UI αντί για λευκή οθόνη:

```tsx
<ErrorBoundary fallback={<p>Κάτι πήγε στραβά</p>}>
  <UserProfile id={id} />
</ErrorBoundary>
```

Συνδυάζεται με `Suspense` για declarative loading: `<Suspense fallback={<Spinner/>}>`.

## Πώς απαντάς στη συνέντευξη

«How do you handle data fetching in React?»

> I treat server state as different from client state: it's an async cache of data that lives in a database and can go stale, so it needs caching, deduplication, refetching and invalidation — things plain `useEffect` doesn't give you. A bare effect works but two components fetching the same thing make two requests and each re-implements loading and error. So I use a server-state library like React Query: queries are keyed, identical keys are deduped and share a cache, it serves cached data immediately while revalidating in the background, and after a mutation I invalidate the relevant key to refresh. I always render all four states — loading, error, empty, and success — and wrap data UI in an error boundary, optionally with Suspense, so a render-time failure shows a fallback instead of a blank screen.
