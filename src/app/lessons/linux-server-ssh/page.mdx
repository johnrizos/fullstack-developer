# Linux Server & SSH

Το app σου δεν τρέχει στο laptop των χρηστών — τρέχει σε ένα Linux μηχάνημα σε κάποιο data center, χωρίς οθόνη. Όλη η επικοινωνία μαζί του γίνεται μέσω **SSH**. Αυτό το μάθημα είναι η γέφυρα ανάμεσα στο «γράφω κώδικα» και στο «το app μου ζει σε production».

## Τι θα μάθεις

- τι είναι ένας Linux server / VPS και πώς συνδέεσαι με **SSH**,
- γιατί τα **SSH keys** είναι ασφαλέστερα από passwords και πώς στήνονται,
- να μεταφέρεις αρχεία με **scp/rsync**,
- **users, sudo και permissions** (`chmod`/`chown`) — το πιο συχνό μπέρδεμα σε deploys,
- βασικό hardening: firewall, ports, και τι σημαίνει «reverse proxy».

## Τι είναι ένας server

Ένας server είναι απλώς ένας υπολογιστής που τρέχει συνέχεια, συνήθως χωρίς GUI (**headless**), με ένα Linux distro — τυπικά **Ubuntu Server**, Debian ή Amazon Linux. Όταν νοικιάζεις «VPS» (Virtual Private Server) από DigitalOcean/Hetzner/AWS EC2, παίρνεις ένα τέτοιο μηχάνημα και ένα IP.

Το μόνο που έχεις στην αρχή: ένα IP, ένα username, και έναν τρόπο authentication.

## SSH — η ασφαλής πόρτα εισόδου

**SSH** (Secure Shell) σου δίνει κρυπτογραφημένο terminal πάνω στο απομακρυσμένο μηχάνημα:

```bash
ssh john@203.0.113.10
# john     = ο user στον server
# 203...   = το IP (ή domain) του server
```

Με το που συνδεθείς, το terminal σου ελέγχει τον **remote** server — κάθε εντολή τρέχει εκεί, όχι τοπικά.

## Password vs SSH keys

Δύο τρόποι να αποδείξεις ποιος είσαι:

- **Password**: ευάλωτο σε brute-force, φαίνεται σε κάθε bot που σαρώνει το internet.
- **SSH key pair**: ασύμμετρη κρυπτογραφία — ένα **private key** που μένει ΜΟΝΟ στο μηχάνημά σου και ένα **public key** που βάζεις στον server. Ασφαλέστερο και χωρίς να πληκτρολογείς password κάθε φορά.

Φτιάχνεις ζευγάρι κλειδιών:

```bash
ssh-keygen -t ed25519 -C "john@laptop"
# φτιάχνει:
#   ~/.ssh/id_ed25519       → PRIVATE key (ΠΟΤΕ δεν το μοιράζεσαι/ανεβάζεις)
#   ~/.ssh/id_ed25519.pub   → PUBLIC key (αυτό πάει στον server)
```

Στέλνεις το public key στον server:

```bash
ssh-copy-id john@203.0.113.10
# προσθέτει το .pub σου στο ~/.ssh/authorized_keys του server
```

Από εκεί και πέρα το `ssh john@203.0.113.10` σε μπάζει χωρίς password.

> Ο κανόνας που δεν παραβιάζεις ποτέ: **το private key δεν φεύγει από το μηχάνημά σου**. Αν διαρρεύσει, όποιος το έχει είναι «εσύ» για κάθε server που το εμπιστεύεται.

## ~/.ssh/config — σταμάτα να πληκτρολογείς IPs

Αντί να θυμάσαι IPs και users, φτιάχνεις aliases:

```bash
# ~/.ssh/config
Host prod
    HostName 203.0.113.10
    User john
    IdentityFile ~/.ssh/id_ed25519
```

Τώρα: `ssh prod`. Καθαρό, και αλλάζεις IP σε ένα σημείο.

## Μεταφορά αρχείων

```bash
scp build.zip john@prod:/var/www/app/        # local → server
scp john@prod:/var/log/app.log ./            # server → local
rsync -avz ./dist/ john@prod:/var/www/app/   # συγχρονισμός φακέλου (μόνο διαφορές)
```

Το **rsync** προτιμάται για deploys: στέλνει μόνο ό,τι άλλαξε, αντέχει διακοπές, και είναι πολύ ταχύτερο για μεγάλους φακέλους.

## Users, root & sudo

Σε Linux ο **root** είναι ο παντοδύναμος χρήστης. Δεν δουλεύεις ως root στην καθημερινότητα — κάνεις login ως κανονικός user και ανεβάζεις δικαιώματα μόνο όταν χρειάζεται:

```bash
whoami                        # ποιος είμαι
sudo apt update               # τρέξε ΑΥΤΗ την εντολή ως root
sudo systemctl restart nginx  # χρειάζεται root για services
```

**Κανόνας ασφαλείας:** φτιάχνεις non-root user με sudo δικαιώματα και απενεργοποιείς το direct root login. Αν κάτι κάνει compromise έναν app process που τρέχει ως root, ελέγχει όλο το μηχάνημα.

## Permissions — rwx, το #1 μπέρδεμα

Κάθε αρχείο έχει δικαιώματα για τρεις ομάδες: **owner**, **group**, **others**. Κάθε ομάδα έχει **r**ead, **w**rite, e**x**ecute:

```bash
ls -l app.sh
# -rwxr-xr--  1 john  www-data  ...
#  │└┬┘└┬┘└┬┘
#  │ │  │  └── others: r--  (μόνο διάβασμα)
#  │ │  └───── group:   r-x  (διάβασμα + εκτέλεση)
#  │ └──────── owner:   rwx  (όλα)
#  └────────── τύπος: - αρχείο, d φάκελος
```

Τα δικαιώματα ως αριθμοί (octal): **r=4, w=2, x=1**, τα προσθέτεις ανά ομάδα:

```bash
chmod 755 deploy.sh    # owner=rwx(7) group=r-x(5) others=r-x(5)
chmod 644 config.json  # owner=rw-(6) group=r--(4) others=r--(4)
chmod 600 ~/.ssh/id_ed25519   # owner=rw- μόνο — υποχρεωτικό για private keys
chmod +x script.sh     # symbolic: πρόσθεσε execute
```

Και ο **owner**:

```bash
chown john:www-data app.log          # owner=john, group=www-data
chown -R www-data:www-data /var/www  # -R αναδρομικά για όλο τον φάκελο
```

> Το πιο συχνό production μυστήριο: «το app δεν μπορεί να γράψει στο /var/www». Σχεδόν πάντα είναι λάθος **owner** ή permissions στον φάκελο, όχι bug στον κώδικα.

## Package managers

Σε Ubuntu/Debian εγκαθιστάς λογισμικό με `apt`:

```bash
sudo apt update                 # ανανέωσε τη λίστα πακέτων
sudo apt upgrade                # αναβάθμισε ό,τι είναι εγκατεστημένο
sudo apt install nginx          # εγκατάστησε πακέτο
```

Το `apt update` φέρνει τον κατάλογο, το `apt install` εγκαθιστά. Αρχάριοι ξεχνούν το πρώτο και απορούν γιατί «δεν βρίσκει» ένα πακέτο.

## Firewall, ports & reverse proxy

Ένας server ακούει σε **ports**: 22 (SSH), 80 (HTTP), 443 (HTTPS). Με firewall αφήνεις ανοιχτά μόνο όσα χρειάζονται:

```bash
sudo ufw allow 22       # SSH
sudo ufw allow 80       # HTTP
sudo ufw allow 443      # HTTPS
sudo ufw enable
```

Το Node app σου τρέχει συνήθως σε port 3000 — **δεν** το εκθέτεις απευθείας. Μπροστά βάζεις έναν **reverse proxy** (συνήθως nginx) που:

- ακούει στο 80/443 και προωθεί στο 3000,
- κάνει terminate το HTTPS (TLS certificates),
- σερβίρει static αρχεία και κάνει load balancing.

Έτσι ο κόσμος μιλά με το nginx, και το nginx με το app σου εσωτερικά.
