# Databases

Fullstack σημαίνει ότι δεν αρκεί να εμφανίζεις δεδομένα — πρέπει να ξέρεις πώς αποθηκεύονται, πώς σχεδιάζονται και πώς ανακτώνται γρήγορα. Στις συνεντεύξεις τα database θέματα είναι σταθερά: JOINs, indexes, N+1 problem, transactions, SQL vs NoSQL. Όλα εδώ.

## SQL: tables, constraints, σχέσεις

SQL databases όπως η PostgreSQL οργανώνουν δεδομένα σε tables με αυστηρό schema:

```sql
CREATE TABLE users (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  email TEXT NOT NULL UNIQUE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE tasks (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  title TEXT NOT NULL,
  completed BOOLEAN NOT NULL DEFAULT false,
  user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

Η μεγάλη δύναμη του SQL: τα **constraints** προστατεύουν τα δεδομένα ακόμα κι αν το backend έχει bug:

- `PRIMARY KEY` — μοναδική ταυτότητα κάθε row.
- `NOT NULL` — required πεδία.
- `UNIQUE` — δεν θα υπάρξουν ποτέ δύο users με ίδιο email, ό,τι κι αν κάνει το app.
- `FOREIGN KEY ... REFERENCES` — δεν μπορεί να υπάρξει task για ανύπαρκτο user.
- `CHECK (price >= 0)` — business rules στο επίπεδο της βάσης.
- `ON DELETE CASCADE` — σβήνεις τον user, σβήνονται και τα tasks του (εναλλακτικά: `RESTRICT`, `SET NULL`).

## Τα queries που πρέπει να γράφεις με κλειστά μάτια

```sql
-- Φιλτράρισμα, ταξινόμηση, σελιδοποίηση
SELECT id, title, completed
FROM tasks
WHERE user_id = $1 AND completed = false
ORDER BY created_at DESC
LIMIT 20 OFFSET 0;

-- Aggregation: πόσα tasks ανά user
SELECT user_id, COUNT(*) AS task_count
FROM tasks
GROUP BY user_id
HAVING COUNT(*) > 5;
```

> Προσοχή: το `WHERE` φιλτράρει **rows πριν** το grouping, το `HAVING` φιλτράρει **groups μετά**. Κλασική interview ερώτηση.

## JOINs — η πιο σίγουρη SQL ερώτηση συνέντευξης

```sql
-- Tasks μαζί με το email του ιδιοκτήτη τους
SELECT tasks.title, users.email
FROM tasks
INNER JOIN users ON users.id = tasks.user_id;

-- ΟΛΟΙ οι users, με τα tasks τους ΑΝ έχουν
SELECT users.email, tasks.title
FROM users
LEFT JOIN tasks ON tasks.user_id = users.id;
```

| JOIN | Τι επιστρέφει |
|---|---|
| `INNER JOIN` | Μόνο rows που ταιριάζουν **και στους δύο** πίνακες |
| `LEFT JOIN` | Όλα τα rows του αριστερού + matches (ή NULL) από τον δεξιό |
| `RIGHT JOIN` | Το αντίστροφο του LEFT |
| `FULL OUTER JOIN` | Όλα τα rows και των δύο, με NULL όπου δεν υπάρχει match |

Πρακτικό τεστ κατανόησης: «βρες τους users **χωρίς κανένα task**» — η απάντηση συνδυάζει LEFT JOIN με NULL check:

```sql
SELECT users.email
FROM users
LEFT JOIN tasks ON tasks.user_id = users.id
WHERE tasks.id IS NULL;
```

## Indexes — γιατί το query σου είναι αργό

Χωρίς index, η βάση κάνει **full table scan** — διαβάζει κάθε row. Με index (B-tree), βρίσκει τα rows σχεδόν αμέσως, όπως το ευρετήριο ενός βιβλίου:

```sql
CREATE INDEX tasks_user_id_idx ON tasks(user_id);

-- Composite index: για queries που φιλτράρουν και στα δύο
CREATE INDEX tasks_user_status_idx ON tasks(user_id, completed);
```

Τα indexes **δεν είναι δωρεάν**: κάθε INSERT/UPDATE πρέπει να ενημερώσει και τα indexes, και καταναλώνουν χώρο. Πότε προσθέτεις index:

- Columns σε συχνά `WHERE` / `JOIN ON` / `ORDER BY`.
- Foreign keys (σχεδόν πάντα).
- **Όχι** σε columns με λίγες διακριτές τιμές (π.χ. boolean μόνο του) ή σε tables που δέχονται κυρίως writes.

Για το composite index ισχύει ο **leftmost prefix κανόνας**: το `(user_id, completed)` εξυπηρετεί queries με `WHERE user_id = ?` και `WHERE user_id = ? AND completed = ?`, αλλά **όχι** σκέτο `WHERE completed = ?`.

Το εργαλείο διάγνωσης: `EXPLAIN ANALYZE SELECT ...` — αν δεις «Seq Scan» σε μεγάλο table, λείπει index.

## Το N+1 problem — το πιο συχνό performance bug

Εμφανίζεται παντού όπου υπάρχει ORM, και οι interviewers το λατρεύουν:

```js
// ❌ N+1: 1 query για τα posts + 1 query ΓΙΑ ΚΑΘΕ post
const posts = await db.post.findMany();          // 1 query
for (const post of posts) {
  post.author = await db.user.findUnique({       // N queries!
    where: { id: post.authorId },
  });
}

// ✅ Λύση: eager loading — 1-2 queries συνολικά
const postsWithAuthors = await db.post.findMany({
  include: { author: true }, // ο ORM κάνει JOIN ή ένα IN query
});
```

Με 100 posts, το πρώτο κάνει **101 queries**, το δεύτερο 1-2. Σε raw SQL η λύση είναι ένα JOIN ή `WHERE id IN (...)`.

## Transactions & ACID

**Transaction** = πολλά βήματα εκτελούνται όλα μαζί ή κανένα:

```sql
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
UPDATE accounts SET balance = balance + 100 WHERE id = 2;
COMMIT; -- ή ROLLBACK αν κάτι πήγε στραβά
```

Αν ο server πέσει ανάμεσα στα δύο UPDATEs, χωρίς transaction θα είχες χαμένα χρήματα. Οι εγγυήσεις συνοψίζονται στο **ACID**:

- **A**tomicity: όλα ή τίποτα.
- **C**onsistency: η βάση πάει από έγκυρη κατάσταση σε έγκυρη (constraints τηρούνται).
- **I**solation: παράλληλα transactions δεν βλέπουν τα μισοτελειωμένα δεδομένα του άλλου.
- **D**urability: μετά το COMMIT, τα δεδομένα επιβιώνουν από crash.

## SQL vs NoSQL — πώς απαντάς στο «ποιο να διαλέξω;»

NoSQL (π.χ. MongoDB) αποθηκεύει ευέλικτα documents:

```json
{
  "_id": "task_1",
  "title": "Learn databases",
  "completed": false,
  "tags": ["backend", "sql"],
  "subtasks": [{ "title": "Read about indexes", "done": true }]
}
```

| | SQL (PostgreSQL) | NoSQL (MongoDB) |
|---|---|---|
| Schema | Αυστηρό, στη βάση | Ευέλικτο, ζει στο app layer |
| Σχέσεις | JOINs, foreign keys | Embedding ή manual references |
| Transactions | Πλήρη ACID παντού | Υπάρχουν, αλλά λιγότερο κεντρικά |
| Scaling | Κάθετο + read replicas | Οριζόντιο (sharding) πιο φυσικό |
| Ιδανικό για | Σχεσιακά δεδομένα, συναλλαγές, reporting | Ευέλικτα documents, μεγάλο write throughput |

Η σωστή default απάντηση σε συνέντευξη: **«Ξεκινάω με PostgreSQL εκτός αν υπάρχει συγκεκριμένος λόγος όχι»** — σχέσεις, constraints και transactions καλύπτουν το 90% των εφαρμογών. NoSQL όταν το σχήμα είναι πραγματικά ευμετάβλητο ή το scale το απαιτεί. Και θυμήσου: NoSQL **δεν** σημαίνει «χωρίς schema thinking» — το schema απλώς μετακομίζει στο app layer και στα access patterns.

## Πώς απαντάς στη συνέντευξη

> I default to PostgreSQL because relationships, constraints, and ACID transactions cover most application needs. I design schemas around access patterns, index foreign keys and frequent WHERE columns, and watch out for N+1 queries by using eager loading. When a query is slow, EXPLAIN ANALYZE tells me whether it's a missing index or a bad query plan.
