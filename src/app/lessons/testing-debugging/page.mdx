# Testing & Debugging

Testing δεν σημαίνει «γράφω tests για το coverage report». Σημαίνει **μπορώ να αλλάζω κώδικα χωρίς φόβο** — τα tests είναι το δίχτυ ασφαλείας που κάνει το refactoring δυνατό. Debugging δεν σημαίνει «console.log παντού» — σημαίνει υπόθεση, έλεγχος, απομόνωση. Και τα δύο εξετάζονται σε συνεντεύξεις: το πρώτο με «πώς τεστάρεις;», το δεύτερο συχνά live, με ένα πραγματικό bug μπροστά σου.

## Η πυραμίδα των tests

Η κλασική απάντηση στο «τι tests γράφεις;» — από τα πολλά/γρήγορα στα λίγα/αργά:

```txt
        ┌─────────┐
        │   E2E   │  ← λίγα: πλήρη user flows σε πραγματικό browser
        │(Playwright)│   (αργά, εύθραυστα, ακριβά — αλλά ρεαλιστικά)
       ┌┴─────────┴┐
       │Integration│  ← μέτρια: components + συνεργασίες
       │   (RTL)   │   (render component, click, assert τι βλέπει ο χρήστης)
      ┌┴───────────┴┐
      │    Unit     │  ← πολλά: pure functions, business logic
      │(Vitest/Jest)│   (χιλιάδες σε δευτερόλεπτα)
      └─────────────┘
```

Ο κανόνας: **όσο πιο κάτω μπορεί να ζήσει ένα test, τόσο το καλύτερο** — τρέχει γρηγορότερα και σπάει σπανιότερα. Το E2E το κρατάς για τα 3-5 flows που «αν σπάσουν, χάνουμε λεφτά»: signup, login, checkout.

## Unit tests: η ανατομία

Κάθε καλό test έχει τρία μέρη — **Arrange, Act, Assert**:

```ts
import { describe, it, expect } from "vitest";

describe("applyDiscount", () => {
  it("applies a 50% discount", () => {
    const total = 200;                          // Arrange
    const result = applyDiscount(total, 50);    // Act
    expect(result).toBe(100);                   // Assert
  });

  // Τα boundary values είναι εκεί που ζουν τα bugs:
  it("handles 0% and 100%", () => {
    expect(applyDiscount(200, 0)).toBe(200);
    expect(applyDiscount(200, 100)).toBe(0);
  });

  it("throws for out-of-range discount", () => {
    expect(() => applyDiscount(200, 120)).toThrow();
    //      ↑ wrapper function — αλλιώς το throw σκάει ΠΡΙΝ το expect
  });
});
```

Δύο λεπτομέρειες που προσέχουν οι reviewers:

- **Ένα test, μία συμπεριφορά** — το όνομα του test διαβάζεται σαν spec: «applies a 50% discount».
- Τα **edge cases** είναι το ζητούμενο: κενό array, 0, αρνητικά, όρια. Το happy path το γράφουν όλοι.

## Mocking: πότε και πόσο

Όταν η μονάδα που τεστάρεις μιλά με τον έξω κόσμο, αντικαθιστάς τον έξω κόσμο:

```ts
import { vi, it, expect } from "vitest";

it("sends a welcome email after signup", async () => {
  const emailService = { send: vi.fn() }; // mock: καταγράφει τις κλήσεις
  const userService = createUserService({ emailService });

  await userService.signup({ email: "ada@example.com" });

  expect(emailService.send).toHaveBeenCalledWith(
    expect.objectContaining({ to: "ada@example.com" })
  );
});
```

Ο χρυσός κανόνας: **mock τα boundaries (network, DB, time, random), όχι τη λογική σου**. Αν χρειάζεσαι 10 mocks για ένα test, το design φταίει — η λογική είναι μπλεγμένη με τα side effects (θυμήσου το μάθημα των modules). Και το anti-pattern που κόβει σε code review: tests που ελέγχουν **πώς** δουλεύει ο κώδικας (internals, ποια private function κλήθηκε) αντί για **τι** κάνει — σπάνε σε κάθε refactoring χωρίς να πιάνουν bugs.

## Component tests: τεστάρεις ό,τι βλέπει ο χρήστης

Με React Testing Library, η φιλοσοφία είναι «queries όπως ο χρήστης»:

```tsx
import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";

it("adds a todo", async () => {
  render(<TodoApp />);

  await userEvent.type(screen.getByLabelText("New todo"), "Learn testing");
  await userEvent.click(screen.getByRole("button", { name: "Add" }));

  expect(screen.getByText("Learn testing")).toBeInTheDocument();
});
```

Βρίσκεις στοιχεία με **role/label/text** (όπως ο χρήστης), όχι με CSS classes ή test-ids όπου γίνεται — έτσι το test επιβιώνει από restyling και πιάνει και accessibility προβλήματα δωρεάν.

## Debugging: μέθοδος, όχι μαντεψιά

Το επιστημονικό loop που περιγράφεις και σε συνέντευξη:

1. **Αναπαραγωγή** — αν δεν μπορείς να το προκαλέσεις κατά βούληση, δεν μπορείς να το φτιάξεις.
2. **Σμίκρυνση** — αφαίρεσε κομμάτια μέχρι το μικρότερο σενάριο που ακόμα αποτυγχάνει.
3. **Διάβασε το error πραγματικά** — το stack trace δείχνει αρχείο και γραμμή· το πρώτο δικό σου frame στο trace είναι ο πρώτος ύποπτος.
4. **Υπόθεση → έλεγχος** — «νομίζω ότι το Χ είναι null εδώ» → βάλε breakpoint/log ΕΚΕΙ → επιβεβαίωσε ή απόρριψε. Ένα ερώτημα τη φορά.
5. **Fix + regression test** — το bug γίνεται failing test ΠΡΙΝ τη διόρθωση· έτσι δεν ξαναγυρνά ποτέ.

Εργαλεία πέρα από το console.log: **breakpoints** στο DevTools/VS Code (βλέπεις όλο το state, όχι ό,τι θυμήθηκες να τυπώσεις), **Network tab** (τι ΠΡΑΓΜΑΤΙΚΑ στάλθηκε/ήρθε), **git bisect** (ποιο commit το έσπασε), και `console.table` για arrays από objects.

## Πώς απαντάς στη συνέντευξη

«How do you approach testing?»

> I follow the testing pyramid: lots of fast unit tests for business logic, component tests with React Testing Library that interact the way a user would, and a handful of end-to-end tests for the flows that matter most — signup, login, checkout. I mock at the boundaries — network, database, time — never my own logic, and I test behavior, not implementation details, so refactoring doesn't break the suite. Every real bug becomes a failing test first, then a fix, so it can never silently return. CI runs everything before merge.
