# TypeScript: types, interfaces, unions και generics

Το TypeScript είναι JavaScript με στατικό type checking — τα types εξαφανίζονται στο compile, άρα **δεν** αλλάζουν τη runtime συμπεριφορά. Η αξία του: πιάνει λάθη πριν τρέξει ο κώδικας, τεκμηριώνει contracts, και κάνει το refactoring ασφαλές. Σε fullstack αγγελίες είναι πλέον σχεδόν υποχρεωτικό, και στις συνεντεύξεις οι ερωτήσεις πάνε αρκετά πιο βαθιά από «τι είναι interface».

## Η βάση: structural typing

Πρώτο πράγμα που πρέπει να ξέρεις: το TypeScript συγκρίνει **σχήματα**, όχι ονόματα. Αυτό λέγεται structural typing (ή «duck typing»):

```ts
interface Point {
  x: number;
  y: number;
}

const spot = { x: 1, y: 2, label: "home" };
const p: Point = spot; // ✅ ΟΚ — έχει ό,τι χρειάζεται το Point (και παραπάνω)

function logPoint(point: Point) {}
logPoint({ x: 1, y: 2 });               // ✅
logPoint({ x: 1, y: 2, label: "hi" });  // ❌ object literal: excess property check!
```

Παρατήρησε την ασυμμετρία — όταν περνάς **object literal απευθείας**, το TypeScript κάνει αυστηρότερο έλεγχο για «έξτρα» properties. Μέσω μεταβλητής, αρκεί το σχήμα να καλύπτεται.

## Union types & narrowing — το καθημερινό εργαλείο

```ts
type TaskStatus = "todo" | "doing" | "done";
```

Τα **literal unions** αντικαθιστούν τα enums στα περισσότερα modern codebases: απλά strings στη runtime, πλήρης έλεγχος στο compile. Με unions έρχεται το **narrowing** — το TypeScript «στενεύει» τον τύπο καθώς ελέγχεις:

```ts
function describe(value: string | number | null) {
  if (value === null) return "empty";        // εδώ: null
  if (typeof value === "string") {
    return value.toUpperCase();              // εδώ: string
  }
  return value.toFixed(2);                   // εδώ: number — δεν χρειάστηκε cast!
}
```

## Discriminated unions — το πιο δυνατό pattern

Αυτό είναι το pattern που ξεχωρίζει όσους «γράφουν TypeScript» από όσους τον αξιοποιούν. Ένα κοινό literal field (ο **discriminant**) κάνει τα invalid states αδύνατα να γραφτούν:

```ts
type ApiResult<T> =
  | { status: "loading" }
  | { status: "success"; data: T }
  | { status: "error"; message: string };

function render(result: ApiResult<Task[]>) {
  switch (result.status) {
    case "loading":
      return "Loading…";
    case "success":
      return result.data.length;     // ✅ το data υπάρχει ΜΟΝΟ εδώ
    case "error":
      return result.message;         // ✅ το message υπάρχει ΜΟΝΟ εδώ
  }
}
```

Σύγκρινε με το «αφελές» μοντέλο `{ loading: boolean; data?: T; error?: string }` — εκεί τίποτα δεν εμποδίζει `loading: true` μαζί με `error`. Με το discriminated union, ο compiler **αποδεικνύει** ότι χειρίστηκες κάθε περίπτωση. Κι αν αργότερα προσθέσεις `status: "empty"`, κάθε switch που το ξέχασε γίνεται compile error — αυτό λέγεται **exhaustiveness checking**:

```ts
default: {
  const unreachable: never = result; // error αν έμεινε case εκτός switch
  return unreachable;
}
```

## type vs interface — η σίγουρη ερώτηση

Και τα δύο περιγράφουν object shapes. Οι πραγματικές διαφορές:

| | `interface` | `type` |
|---|---|---|
| Object shapes | ✅ | ✅ |
| Unions, primitives, tuples | ❌ | ✅ `type ID = string \| number` |
| Επέκταση | `extends` | `&` (intersection) |
| Declaration merging | ✅ — δηλώσεις με ίδιο όνομα ενώνονται | ❌ — διπλό όνομα = error |

Πρακτικός κανόνας που αρκεί σε συνέντευξη: **interface για δημόσια object contracts** (props, models — και για library types όπου το merging βοηθά), **type για unions και συνθέσεις**. Μέσα σε ένα project, η συνέπεια μετράει περισσότερο από την επιλογή.

## Generics — επαναχρησιμοποιήσιμα types

Τα generics είναι «παράμετροι τύπων». Χωρίς αυτά, θα έγραφες το ίδιο type ξανά και ξανά:

```ts
interface ApiResponse<T> {
  data: T;
  error?: string;
}

type TasksResponse = ApiResponse<Task[]>;
type UserResponse = ApiResponse<User>;

// Σε functions — με inference, δεν χρειάζεται να δηλώσεις το T:
function first<T>(items: T[]): T | undefined {
  return items[0];
}
const task = first(tasks);  // T = Task, αυτόματα
```

Με **constraints** περιορίζεις τι δέχεται το generic:

```ts
function getById<T extends { id: string }>(items: T[], id: string): T | undefined {
  return items.find((item) => item.id === id);
}

getById(tasks, "1");      // ✅ τα Task έχουν id
getById([1, 2, 3], "1");  // ❌ τα numbers δεν έχουν id
```

## Utility types — το λεξιλόγιο που εξετάζεται

Το TypeScript έχει ενσωματωμένα «εργαλεία μετασχηματισμού» — αυτά τα τέσσερα-πέντε πέφτουν συνέχεια:

```ts
interface Task {
  id: string;
  title: string;
  status: TaskStatus;
  createdAt: Date;
}

type TaskDraft = Omit<Task, "id" | "createdAt">; // χωρίς server-generated fields
type TaskPatch = Partial<TaskDraft>;             // όλα optional — για PATCH body
type TaskPreview = Pick<Task, "id" | "title">;   // μόνο όσα δείχνει η λίστα
type TasksById = Record<string, Task>;           // map από id σε Task

function createTask(draft: TaskDraft): Task { /* ... */ }
type Created = ReturnType<typeof createTask>;    // Task — παράγεται, δεν αντιγράφεται
```

Το πραγματικό μάθημα εδώ: **ένα source of truth**. Ορίζεις το `Task` μία φορά και **παράγεις** τα συγγενικά types, αντί να τα γράφεις (και να τα ξεσυγχρονίζεις) με το χέρι.

## any vs unknown — η ερώτηση-φίλτρο

```ts
function parse(json: string): unknown {
  return JSON.parse(json);
}

const data = parse('{"title":"hi"}');
data.title;                       // ❌ error — το unknown ΔΕΝ επιτρέπει πρόσβαση
if (
  typeof data === "object" && data !== null && "title" in data
) {
  // εδώ μπορείς να το χρησιμοποιήσεις — απέδειξες το σχήμα
}
```

- `any`: **απενεργοποιεί** τον έλεγχο — ό,τι κάνεις περνάει, και «μολύνει» ό,τι αγγίζει.
- `unknown`: «δεν ξέρω τι είναι» — δεν μπορείς να το αγγίξεις **μέχρι να το αποδείξεις** με narrowing ή validation.

Για δεδομένα από έξω (API responses, JSON, query params), το σωστό pattern σε production είναι `unknown` + runtime validation (π.χ. με Zod) — γιατί τα types εξαφανίζονται στη runtime και ένα `as User` δεν ελέγχει τίποτα στην πραγματικότητα. Το **type assertion (`as`) λέει στον compiler «εμπιστέψου με»** — δεν μετατρέπει δεδομένα.

## Πώς απαντάς στη συνέντευξη

«Why TypeScript?»

> TypeScript catches type errors at compile time, documents contracts between modules, and makes refactoring safe — renaming a field shows every broken usage instantly. Types are erased at runtime, so there's no performance cost, but that also means external data still needs runtime validation.

«type vs interface?»

> Both describe object shapes. Interfaces support extends and declaration merging, which matters for public APIs and libraries. Type aliases handle unions, tuples, and compositions that interfaces can't. My default: interfaces for object contracts, type aliases for unions — and consistency within the codebase above all.
