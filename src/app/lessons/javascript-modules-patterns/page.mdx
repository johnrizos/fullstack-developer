# Modules & Code Organization

Όταν ο κώδικας μεγαλώνει, το πρόβλημα δεν είναι «να δουλεύει» — είναι να παραμένει κατανοητός, testable και εύκολος να αλλάξει. Στις συνεντεύξεις αυτό εξετάζεται με δύο τρόπους: ερωτήσεις για ES modules (imports/exports) και η ποιότητα οργάνωσης του κώδικά σου στο live coding. Εδώ θα δεις και τα δύο.

## ES Modules: το σύστημα οργάνωσης της γλώσσας

Κάθε αρχείο είναι module με δικό του scope — τίποτα δεν είναι global εκτός αν το κάνεις export:

```js
// cart.js
export function calculateTotal(items) {
  return items.reduce((total, item) => total + item.price * item.quantity, 0);
}

export const TAX_RATE = 0.24;

export default class Cart { /* ... */ }
```

```js
// app.js
import Cart, { calculateTotal, TAX_RATE } from "./cart.js";
import * as cart from "./cart.js"; // namespace import: cart.calculateTotal(...)
```

**Named vs default exports** — η κλασική ερώτηση:

| | Named | Default |
|---|---|---|
| Ανά αρχείο | Πολλά | Ένα |
| Import όνομα | Πρέπει να ταιριάζει (`{ calculateTotal }`) | Ό,τι θες (`import Whatever`) |
| Refactoring/autocomplete | Καλύτερο — τα ονόματα ελέγχονται | Ευάλωτο σε ασυνέπειες |

Πολλά teams (και style guides) προτιμούν **named exports παντού**: το όνομα ελέγχεται από τον bundler, το rename γίνεται με ασφάλεια, και το autocomplete δουλεύει καλύτερα.

Δύο ακόμα ιδιότητες των ES modules που ρωτιούνται:

- **Τα modules είναι singletons**: το `import` του ίδιου αρχείου από 10 σημεία εκτελεί το module **μία φορά** — όλοι μοιράζονται το ίδιο instance. (Γι' αυτό ένα `export const config = {...}` λειτουργεί ως κοινή κατάσταση.)
- **ESM vs CommonJS**: το `import/export` είναι το standard (στατικό, αναλύσιμο στο build — επιτρέπει tree-shaking). Το `require()/module.exports` είναι το παλιό σύστημα του Node (δυναμικό, runtime). Σήμερα γράφεις ESM σχεδόν παντού.

## Pure functions: ο πυρήνας του testable κώδικα

Μια **pure function**: (1) ίδια inputs → πάντα ίδιο output, (2) **κανένα side effect** — δεν αλλάζει τίποτα έξω από αυτήν:

```js
// ✅ Pure: μόνο inputs → output
function calculateLineTotal(item) {
  return item.price * item.quantity;
}

// ❌ Impure: διαβάζει/γράφει εξωτερική κατάσταση
let total = 0;
function addToTotal(item) {
  total += item.price;        // side effect: αλλάζει εξωτερική μεταβλητή
  document.title = total;     // side effect: αγγίζει το DOM
}

// ❌ Impure με ύπουλο τρόπο: ΜΕΤΑΛΛΑΣΣΕΙ το input!
function applyDiscount(item) {
  item.price = item.price * 0.9; // ο caller δεν το περιμένει αυτό
  return item;
}

// ✅ Pure εκδοχή: επιστρέφει νέο object
function applyDiscount(item) {
  return { ...item, price: item.price * 0.9 };
}
```

Γιατί τόση φασαρία; Οι pure functions **τεστάρονται με μία γραμμή** (input → expected output, χωρίς mocks), συνδυάζονται ελεύθερα, και δεν κρύβουν εκπλήξεις. Όσο περισσότερη λογική ζει σε pure functions, τόσο μικρότερο και πιο απλό είναι το «βρώμικο» κομμάτι (DOM, network, storage).

## Separation of concerns: τα τρία στρώματα

Ακόμα και σε μικρό app, χώρισε τον κώδικα σε ρόλους:

```txt
┌─────────────────────────────────────────┐
│ UI layer (events, DOM)                  │ ← «βρώμικο», λεπτό στρώμα
│   click → καλεί logic → ενημερώνει DOM  │
├─────────────────────────────────────────┤
│ Business logic (pure functions)         │ ← ο «εγκέφαλος», 100% testable
│   calculateTotal, applyDiscount, ...    │
├─────────────────────────────────────────┤
│ Data/IO layer (fetch, localStorage)     │ ← side effects στα όρια
│   saveCart, loadCart, fetchPrices       │
└─────────────────────────────────────────┘
```

Το anti-pattern που βλέπεις σε κάθε αρχάριο codebase: **όλα μέσα στο click handler** — υπολογισμοί, validation, DOM updates, fetch, όλα μαζί. Δεν τεστάρεται, δεν επαναχρησιμοποιείται, δεν διαβάζεται.

```js
// ✅ Το handler είναι "συντονιστής" — λεπτός και βαρετός
document.querySelector("#calculate").addEventListener("click", () => {
  const total = calculateCartTotal(cartItems);        // logic (pure)
  renderTotal(formatCurrency(total));                  // UI
});
```

Ο πρακτικός κανόνας: **side effects στα boundaries**. Η καρδιά του προγράμματος είναι pure· το διάβασμα/γράψιμο του κόσμου (DOM, δίκτυο, δίσκος) συμβαίνει στις άκρες.

## Πώς χωρίζεις σε modules — με ευθύνη, όχι με μέγεθος

```txt
src/
  cart/
    calculations.js   ← pure: totals, discounts, tax
    storage.js        ← IO: save/load από localStorage
    ui.js             ← DOM: render, event listeners
  utils/
    currency.js       ← pure: formatCurrency
```

Κριτήριο διαχωρισμού: **«τι αλλάζει μαζί;»** — όχι «πόσες γραμμές έχει το αρχείο». Αν αλλάξεις τον τρόπο αποθήκευσης (localStorage → API), πρέπει να αγγίξεις ΜΟΝΟ το `storage.js`. Αν αλλάξει ο ΦΠΑ, μόνο το `calculations.js`. Αυτό είναι το πρακτικό νόημα του «loose coupling, high cohesion» που ακούγεται στις συνεντεύξεις:

- **High cohesion**: ό,τι είναι μαζί σε ένα module, ανήκει πραγματικά μαζί.
- **Loose coupling**: τα modules μιλούν μεταξύ τους μέσω μικρών, καθαρών APIs — όχι ο καθένας με τα εσωτερικά του άλλου.

## Πώς απαντάς στη συνέντευξη

«How do you organize JavaScript code?»

> I separate business logic from UI and IO. The core logic lives in small pure functions — same input, same output, no side effects — which makes it trivially testable without mocks. Side effects like DOM updates, fetch, and storage stay at the boundaries, in thin modules with a clear responsibility each. I use ES modules with named exports, and I split by responsibility — what changes together stays together — not by file size.
